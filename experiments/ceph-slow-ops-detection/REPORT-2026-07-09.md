# 只靠 Prometheus 偵測 BlueStore 暫態卡頓與 slow ops：研究報告

> 2026-07-09。研究方法：`skills/researching-system-behavior`（Frame → Enumerate → Falsify → Automate → Synthesize）。
> 完整 hypothesis backlog（25 條、含證據錨點）見 `HYPOTHESES.md`；實驗 bundle 索引見 `EVIDENCE-SUMMARY-2026-07-09.md`。
> 版本錨點：ceph v19.2.3（Squid）、Prometheus v2.51.0、cephadm 部署、scrape 10s。

## TL;DR

1. **你的 firmware 事件（5~8 顆 OSD 同秒卡 5~8 秒）用現有 SLOW_OPS 類 alert 永遠抓不到**——SLOW_OPS 只計「在飛且齡 >30s」的 op（`osd_op_complaint_time`），5~8s 的暫態連門檻的零頭都不到（真機實證 E-01：注入 8s 卡頓，SLOW_OPS 兩路全程 0）。
2. **解法是 Squid 才有的 4 個 BlueStore slow counter**（`ceph_bluestore_slow_{aio_wait,committed_kv,read_onode_meta,read_wait_aio}_count`）：op 超過 `bluestore_log_op_age`（預設 5s）就 +1、累積不歸零、ceph-exporter 預設就匯出。以它做的 R1 規則在真機上**卡頓開始後 14 秒可判真**（+ rule eval ≤15s ≈ 30 秒內 fire）。
3. **「同 node ≥3 顆 OSD 同窗出現 slow op」= RAID 卡/firmware 層級事件的指紋**（R2）：真機注入同 node 3 顆同秒卡 8s，R2 在 +20 秒判真、且只有事發 node 達標——單碟壞不會長這樣，共用元件（RAID 卡、同型號 firmware、背板）才會。
4. **「會卡住 client 的一般 slow ops」偵測延遲從 ~2 分鐘壓到 45~60 秒**（R3/R4，for:0 + keep_firing_for），且真機發現三個現有規則的漏接模式（見「三層盲區」）。
5. 研究過程推翻了我自己 4 條預測（H-007/H-009/H-015 + E-04 兩輪設計）——被推翻的每一條都變成了規則設計的修正（詳見「被推翻的預測」）。

## 緣起：你的事件在監控鏈路上為什麼是隱形的

事件形態：OSD node（10x SAS SSD 掛同一張 RAID 卡）上 5~8 顆 OSD 在同一秒回報 BlueStore op 卡 5~8 秒，疑似 SSD firmware。你看到的「slow ops in bluestore」log 來自 `BlueStore::log_latency()`（`BlueStore.cc:18476`）：任何 BlueStore 內部階段超過 `bluestore_log_op_age`（預設 **5s**）就打一行 `slow operation observed`。

這行 log 同時做三件事（v19.2.3，`BlueStore.cc:18476-18484`）：

1. 打 log（你已經在看的）。
2. `_add_slow_op_event()` → 事件進 in-memory queue → 超過 `bluestore_slow_ops_warn_threshold`（預設 1）就掛出 `BLUESTORE_SLOW_OP_ALERT` health 警告，保留 `bluestore_slow_ops_warn_lifetime`（預設 **24 小時**，latch 不自動消）。
3. **部分呼叫點**（讀路徑 ×2、txc commit、aio wait）額外 inc 一個專用 perf counter——這就是 Prometheus 可用的快訊號。

而現有 alert 依賴的 SLOW_OPS 是完全不同的機制：OSD op tracker 每秒檢查「在飛 op 的年齡 > `osd_op_complaint_time`（30s）」（`OSD.cc:7825-7832`）→ 經 mgr → mon health。**5~8s 的暫態 op 完成得太快，永遠不會被它看到。**

## 訊號地圖（哪些 metric 什麼時候動）

| 訊號 | 來源鏈 | 門檻 | 對 5~8s 暫態 | 對持續慢盤 | 對永久卡死 | 定位能力 |

**版本依賴（H-019 實證，GitHub 原始碼逐版比對）**：4 個 slow counter = **Squid v19.2.0+ 才有**（Reef 全系列沒有）；`BLUESTORE_SLOW_OP_ALERT` = v19.2.1+ / Reef v18.2.6+。生產若還在 Reef，R1/R2 不可用，fallback 只剩 R3/R4 + node_disk + commit_latency_ms。

## 三層盲區（真機實證，都是現有 alert 的漏接模式）

1. **暫態盲區（H-001）**：op 齡 <30s 的單發卡頓，op-tracker 路徑物理上不可見。E-01 注入 8s，SLOW_OPS 兩路全程 0。
2. **持續性 sub-30s 劣化盲區（H-024）**：E-04 v2 注入 150s 持續節流，op_w 均值峰值 15.2s——client 痛了兩分半，SLOW_OPS 仍全程 0，因為**每個 op 都在 30s 內完成**。這正是 SSD 劣化/firmware 問題最常見的表現形態。
3. **採樣漏失盲區（H-023）**：daemon 端 SLOW_OPS 是「快照 gauge」——op 完成即歸零，可見窗=「op 齡 −30s」那幾秒，要穿過 mgr 15s cache + 10s scrape。實測一個 max latency 44.7s 的真事件：mon 端 `ceph_health_detail` 抓到 2 個樣本（撐不過舊規則的 for:1m），**per-daemon metric 整段漏掉**。44 秒的真 slow op、現有兩條 alert 全部沉默。

加一個定位陷阱：**H-025——SLOW_OPS 會怪錯人**。E-04 卡住 osd.0（osd-01 上），daemon SLOW_OPS 亮的是 osd.4/6/7/8（別台 node 的 primary）：client op 記在 primary 的 op tracker 上，等被卡 replica 的 ack 等到超齡。oncall 看到 SLOW_OPS 亮 osd.4 就去查 osd.4 的碟＝方向錯誤。R1 是 device-local 訊號、直接指向元兇。

## 建議規則（rules/ceph-slow-ops-fast.yml，promtool 測試全綠）

與既有 27 條規則並存，不取代（R1 補暫態/持續劣化盲區；R3/R4 加速既有 SLOW_OPS 路徑；R2 做硬體歸因升級）：

| 規則 | 表達式核心 | severity | 語意 |
|---|---|---|---|
| R1 `CephOSDTransientStall` | 4 counter 的 `increase[1m]` 相加 >0，for:0，keep_firing_for:10m | warning | 「這顆 OSD 剛有 op 超過 5s」——症狀偵測，不是硬體定罪 |
| R2 `CephNodeMultiOSDStall` | `count by (instance)`(per-OSD increase[2m]>0) ≥3，keep_firing_for:15m | critical | **firmware/RAID 卡指紋**：單碟壞不會同時打中多顆 OSD |
| R3 `CephDaemonSlowOpsFast` | `max_over_time(daemon_health_metrics{SLOW_OPS}[1m])>0`，for:0 | critical | client 已被卡 >30s，立即 page（不等舊規則的 for:1m） |
| R4 `CephClusterSlowOpsFast` | `max_over_time(health_detail{SLOW_OPS}[1m])>0`，for:0 | critical | R3 的 mon 端備援（H-023：daemon 路徑會漏採） |

設計上的三個非顯然決策（都來自真機翻車）：

- **4 個 counter 用顯式相加，不能用 `__name__` regex + `increase()`**：regex 選出的多 counter 經 increase 丟掉 `__name__` 後同 OSD 變重複 labelset，真 Prometheus 直接回錯（promtool 測試恰好避開了這個碰撞——測試輸入每 OSD 只給一個 counter 就會漏抓，已補「4 counter 齊備」的真實形態 case）。
- **for: 0 + keep_firing_for**：事件是單發脈衝（increase 只在一個窗內 >0），for >0 會直接殺掉偵測能力；keep_firing_for（Prometheus 2.42+）讓 oncall 來得及看到。
- **counter reset 安全**：OSD 重啟 counter 歸零不會讓 `increase()` 產生假陽性（promtool case + finale 真機驗證）。

## 真機偵測延遲數據（本研究的核心產出）

注入方法：`dmsetup suspend`（device 層 IO 全排隊、OSD/heartbeat/admin socket 正常——最貼近 firmware 卡頓語意）；持續事件用 suspend/resume duty-cycle。時間以 lab 內 NTP 同步時鐘（skew 實測 ≤1s）。「可判真」= 表達式第一次為真的 scrape 時刻；實際 fire 另加 rule eval interval（≤15s）。

| 事件 | 訊號 | 可判真時刻（相對卡頓開始） | 舊規則對照 |
|---|---|---|---|
| 單 OSD 卡 8s（E-01，寫負載） | R1 | **+14s**（suspend 結束 +6s） | 永遠不 fire |
| 單 OSD 卡 8s（E-01） | SLOW_OPS 兩路 | 永遠 0 | 永遠不 fire |
| 同 node 3 OSD 同秒卡 8s（E-02） | R2（count=3，僅事發 node 達標） | **+20s** | 永遠不 fire |
| 持續慢盤 ~143s（E-04，op 齡破 30s） | R1 | 全程連續增量（首增 ≤+20s） | — |
| 持續慢盤 ~143s（E-04） | 三路 SLOW_OPS 首度非零 | **+45s** → R3/R4 可 fire ≈ +45~60s | 舊規則（for:1m）≈ +120s；<60s 的事件整段漏 |
| idle OSD 卡 15s（E-03） | 全部訊號 | **全靜默**（結構性盲區：無 IO 即無偵測） | 同樣靜默 |

## 給你的環境的落地建議

1. **先確認版本**：R1/R2 需要 Squid（v19.2.0+）。`curl -s exporter:9926/metrics | grep bluestore_slow` 有東西就能用。
2. **部署 R1~R4**（`rules/ceph-slow-ops-fast.yml`），既有規則不動。R2 的 `>=3` 門檻是 per-node OSD 數的函數——你一台 10 顆，建議維持 3（10 顆同卡上 5~8 顆同時卡，3 綽綽有餘；設太高反而漏掉部分受影響的事件）。
3. **R1 的 FP 心理預期**：過載（recovery、backfill、深佇列）也會觸發（H-015）。它的語意是「有 op >5s」——這在 SAS SSD 上本來就不該發生，過載到這個程度本身值得看。真正的分級靠 R2（單 node 聚集 = 硬體）與持續性（keep_firing_for 窗內反覆 = 劣化中）。
4. **想抓 <5s 的卡頓**：`ceph config set osd bluestore_log_op_age 2`（runtime 生效，E-06 驗證）——SAS SSD 正常 op 遠低於 2s，調低門檻可把偵測面推進到 2~5s 的早期徵兆。代價：log 量與 FP 上升，建議先在一台 node 試。
5. **RAID 卡歸因**：R2 fire 時，用 `ceph_disk_occupation` join `node_disk_io_time_weighted_seconds_total` 看是「多 VD 同窗異常」（→控制器/firmware/背板）還是單 VD（→單碟）。注意 RAID 卡 write-back cache 可能吸收寫延遲、讓 node_disk 低估——讀路徑訊號（`slow_read_wait_aio_count`）不受 cache 掩護，是更誠實的 device 指標。
6. **結構性限制要知道**：(a) idle OSD 上的卡頓無訊號（E-03）——低流量時段的 firmware 事件會漏，若在意可加低頻 synthetic IO（如每分鐘一次小寫入的 cron）；(b) op 永久卡死只有 SLOW_OPS 看得到（R3/R4 負責）；(c) OSD 若在事件後 ~15s 內 crash，counter 增量來不及被 scrape（此時 OSD down alert 接手）。
7. **BLUESTORE_SLOW_OP_ALERT 當盤後追查器**：它 latch 24h，適合「昨晚有沒有發生過」的事後盤點，不適合 pager。要手動清：restart 該 OSD（finale 驗證）。
