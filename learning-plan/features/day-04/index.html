Day 04 — Controller Manager 與 self-heal

刪掉一個 ReplicaSet 管理的 Pod，親眼看 controller-manager 在 1-2 秒內重建，理解 informer / work queue 的角色。

今日目標

  • 建立 Deployment，觀察 ReplicaSet self-heal 的速度
  • 用 kubectl get pods -w 即時看 Pod 重建過程
  • 在 kube-controller-manager log 找到 reconcile 相關訊息
  • 理解 informer cache 與 work queue 的設計哲學
  • 知道 kube-controller-manager 裡大約有多少個 controller 在同時跑

今日在 30 天計畫中的位置

Day 3 我們看到 scheduler 如何把 Pod 綁到 node。今天的 self-heal 示範讓這條線更完整：一個 Pod 消失 → ReplicaSet controller 感知到 → 建立新 Pod（進入 etcd）→ scheduler 排程 → kubelet 啟動。每個元件各司其職，透過 API server 上的 watch event 串在一起。Day 5 會繼續追 kubelet 那一段。

5. 清理

觀念深化

kube-controller-manager 是單一 process，但裡面同時跑約 30 個 controller（ReplicaSet、Deployment、Job、StatefulSet、Namespace、Node 等），每個 controller 是獨立的 goroutine。

每個 controller 有三個核心元件：

  1. Informer：第一次啟動時對 API server 做 List（拿全量資料），之後轉為 Watch（只收 delta event）。Informer 把資料快取在本地 in-memory store，讓 controller 查詢時不用每次都打 API server。
  2. Work Queue：Informer 收到 event 後把 object key（namespace/name）放進 work queue。Work queue 內建 dedup（同一個 key 不會重複排隊）和 rate-limit（避免錯誤時 tight loop）。
  3. Reconcile function：從 work queue 拿出 key → 從 informer cache 讀當前狀態 → 計算與期望狀態的 diff → 呼叫 API server 補齊差距。

這個 informer → work queue → reconcile 三角形，是整個 Kubernetes controller 生態系的共同 pattern，kubebuilder 寫的 custom operator 也完全遵循同一套。
