動手做

1. 寫 VirtualMachine YAML

2. apply + 等 VMI 起來

NAME        AGE   STATUS    READY
fedora-vm   5s    Starting  False

等到 Running：

fedora-vm   45s   Running    10.244.x.x   worker-01

VirtualMachine（vm）= 持久定義；VirtualMachineInstance（vmi）= 實際在跑的 VM 實例。Stop VM 時 vmi 會消失但 vm 還在。

3. 看 launcher Pod

NAME                            READY   STATUS    ...
virt-launcher-fedora-vm-xxxxx   2/2     Running

每個 VMI 對應一個 virt-launcher Pod，Pod 內跑 libvirt + qemu。

4. 用 virtctl console 進 VM

Successfully connected to fedora-vm console. The escape sequence is ^]

Fedora Linux 39 (Cloud Edition)
Kernel 6.x.x on an x86_64 (ttyS0)

fedora-vm login:

帳號 fedora 密碼 fedora（cloud-init 注入）。登入後試 ip a 看 NIC、uname -a 看 kernel。離開按 Ctrl+]。

5. Stop / Start 看 VM 與 VMI 的關係

NAME                                  AGE   STATUS    READY
virtualmachine.kubevirt.io/fedora-vm  5m    Stopped   False
# vmi 不見了

幾秒後 VMI 重新出現。原本的 cloud-init 會再執行一次（每次 VMI 都是新 instance）。

觀念深化

VirtualMachine 對 VirtualMachineInstance 的關係，類似 Deployment 對 Pod：

持久定義短暫實例
Deployment / ReplicaSetPod
VirtualMachineVirtualMachineInstance

但 VM 跟 Deployment 有兩個重要差別：

  1. 單實例：VM 永遠最多 1 個 VMI（不是 N 個 replica）。VM 的 runStrategy 控制何時啟動 VMI（Always / RerunOnFailure / Manual / Halted）。
  2. 狀態保留位置：磁碟內容存在 PVC（Day 24 會講）。VMI 死掉重起，磁碟保留；但 RAM 不保留 — 重起就是 cold boot。Live migration（Day 27）才能保留 RAM。

containerDisk 是 Day 23 的捷徑：image 包在 OCI image 裡，每次 VMI 啟動都從 image 拉一份新的（read-only），改動不會持久化。生產應該用 persistentVolumeClaim 或 dataVolume。
