##### identity-and-policy.mdx 骨架

```mdx
---
layout: doc
title: Cilium — Identity 與 Policy
description: SecurityIdentity 為什麼能取代 IP；PolicyMap 是怎樣的 BPF map；L7 / FQDN policy 怎麼透過 proxy redirect 達成。
---

## 場景
你寫 NetworkPolicy 用 podSelector 指定來源 Pod。Pod 換了 IP，policy 還是對的。Cilium 不靠 IP — 那靠什麼？答：SecurityIdentity。再追一層：identity 怎麼變成 kernel 內的 forwarding decision？

## SecurityIdentity = label set 的雜湊
- 兩個 Pod 同 label → 同 identity → 同 numeric ID（16-bit, 全 cluster 唯一）
- 換 IP 不影響 identity，policy 不需要重算
- reserved identities（pkg/identity/reserved.go）：world, host, init, unmanaged

```go
// File: pkg/identity/identity.go (line 27)
type Identity struct {
    ID         NumericIdentity
    Labels     labels.Labels
    LabelArray labels.LabelArray
    ...
}
```

## Identity allocator
- pkg/identity/cache/cache.go
- 用 kvstore（etcd 或 k8s CRD 模式）做 cluster-wide unique ID 配發

## Policy compute
- 來源：standard NetworkPolicy + cilium 自己的 CiliumNetworkPolicy / CiliumClusterwideNetworkPolicy
- agent watch 這些 CRD，對每個 endpoint 算「這個 endpoint 對其他 identity 各自允許哪些 (port, proto, L7 redirect)」
- 結果由 endpoint regeneration 寫進 PolicyMap

## PolicyMap 結構
- per-endpoint BPF hash map（不是 cluster-wide table）
- key = (identity, dest_port, proto, direction[, port_prefix_len])
- value = (flags: deny? proxy_port?)
- bpf_lxc.c 在 forwarding 時對它做 lookup

```go

type PolicyKey struct {
    ...
}

// File: pkg/maps/policymap/policymap.go (line 155)

    ...
}
```

## L7 policy 與 proxy redirect
- DNS / HTTP 規則 → policy 中 proxy_port != 0
- bpf_lxc 看到 proxy_port → redirect 到 host 的 cilium-agent envoy / built-in proxy
- proxy 解析後決定 allow/deny
- DNS proxy 還會把回應 IP 寫進 ipcache，並關聯 identity

## FQDN policy
- toFQDNs: ["github.com"] 寫在 CNP
- DNS proxy 攔截 Pod 的 DNS 解析，把回應 IP 加進 ipcache 並關聯 identity（pkg/fqdn/）
- 之後 Pod 連那個 IP → ipcache lookup → identity → PolicyMap → allow/deny

## 收穫檢查
