## 場景

scheduler 把 virt-launcher Pod 放到 worker-X。從這時起到 QEMU 真的跑起來、guest OS boot，發生了什麼？virt-handler 跟 virt-launcher 怎麼分工？為什麼不直接讓 virt-handler 自己 spawn QEMU process？

## virt-handler：per-node 的 VM 管家

virt-handler 是 DaemonSet，每個 node 一個。它跟 kubelet 平起平坐：kubelet 管 container，virt-handler 管 VM（precisely: 它管 VMI 的 lifecycle、跟 virt-launcher Pod 內的 libvirt 對話）。

```go

    // 起 metrics + heartbeat
}
```

## virt-launcher：per-VMI Pod 內的執行者

每個 VMI 對應一個 virt-launcher Pod。Pod 內 process tree:

```
PID 1: virt-launcher-monitor       ← watchdog
       └─ exec → virt-launcher      ← cmd-server, lifecycle wrapper
                  ├─ libvirtd / virtqemud (子 process)
                  │   └─ qemu-kvm (孫 process, running guest)
                  └─ notify-client (連回 virt-handler 報 event)
```

`virt-launcher-monitor` 是 PID 1，唯一工作是「watch virt-launcher，掛了就讓 Pod 結束」。這樣可以避免 zombie 進程，也讓 Pod 有明確的 termination semantics。

`virt-launcher` 本身做幾件事：

```go
// File: cmd/virt-launcher/virt-launcher.go (line 343)
func main() {
    // 1. 起 cmd-server 監聽 unix socket (給 virt-handler 連)
    // 2. 連 libvirtd
    // 3. 啟 startDomainEventMonitoring (line 136) 從 libvirt 收 event
    // 4. wait for SyncVMI gRPC, 收到後翻 spec、call libvirt
    // 5. domain 啟動後 markReady (line 77)
}
```

## cmd-server gRPC：跨 Pod 的合理介面

為什麼 virt-handler 不直接連 libvirt？因為 libvirt 預設只監聽 unix socket，且在 virt-launcher Pod 的 mount/network namespace 內。要跨 Pod 用 libvirt 需要 TCP socket + auth + cert，相當麻煩。

更乾淨的做法：在 virt-launcher 內起一個 gRPC server (cmd-server)，定義 KubeVirt 自己的 RPC 集合 (SyncVMI / KillVMI / MigrateVMI / FreezeVMI / ...)，由 virt-launcher 把 RPC 翻成 libvirt API call。

```
virt-handler  ──unix socket gRPC──>  virt-launcher cmd-server
                                          │
                                          │ libvirt-go-xml
                                          ▼
                                       libvirtd
                                          │
                                          ▼
                                       QEMU
```

socket path 在 host 的 `/var/run/kubevirt/sockets/<vmi-uid>_sock`，由 virt-launcher Pod 用 hostPath volume 暴露給 virt-handler。

事件方向相反：virt-launcher 透過 notify-client (`pkg/virt-launcher/notify-client/`) 連回 virt-handler 的事件接收 socket，把 libvirt domain event 推回去。virt-handler 看到 event → 更新 in-memory cache → 觸發 reconcile → 把 status PATCH 回 apiserver。

protobuf 在 `pkg/handler-launcher-com/`。

## VMI spec → libvirt domain XML

最關鍵的翻譯：把 `VirtualMachineInstance.spec.domain` 翻成 libvirt XML。converter 在這：

```go
// File: pkg/virt-launcher/virtwrap/converter/converter.go
// (~300 個 Convert_* function)
// e.g.
// Convert_v1_Disk_To_api_Disk(...)        // line 87 — disk → libvirt <disk>
// Convert_v1_Interface_To_api_Interface() // network interface
// Convert_v1_Memory_To_api_Memory()       // memory + hugepages
// Convert_v1_CPU_To_api_CPU()             // cpu topology + pinning
```

這些 Convert 函式拼出一個 `*api.DomainSpec` (KubeVirt 自己的 Go struct)，最終 marshal 成 libvirt 接受的 XML。每個 disk / interface / volume hot-plug 都是一個 Convert 來回。

## CNI 與 storage 的接點

**CNI**：VMI 的 primary network 走 Pod 既有的 CNI（cilium/calico/flannel...）。Pod 起來時 kubelet 已經呼叫 CNI 配好 net namespace；virt-handler 在 sync 時把 Pod 的 network interface 透過 bridge / macvtap / passt 等模式接到 QEMU。secondary network 透過 multus annotation (`k8s.v1.cni.cncf.io/networks`)，多接幾張 interface 進 Pod，virt-launcher 看到都會綁進 QEMU。

**Storage**：DataVolume / PVC mount 進 virt-launcher Pod 的 volume 路徑；converter 把 path 翻成 libvirt `<disk>` 元素，QEMU 直接 open 那個 file (或 block device)。 ephemeral disk、container disk、emptyDir、hostPath 都各自有對應路徑與 converter 邏輯 (`pkg/ephemeral-disk/`、`pkg/container-disk/`、`pkg/empty-disk/`)。
