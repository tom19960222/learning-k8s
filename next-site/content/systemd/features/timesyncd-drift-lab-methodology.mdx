## 場景：一台機器斷網幾小時，時鐘會漂多遠、多久追得回來

先講一個會痛的真實情境。

ceph 這套分散式儲存，靠多台「監控節點」（ceph mon）互相投票來維持一致性。這些節點彼此的時鐘只要差超過 50 毫秒（`50ms`），ceph 就跳 `MON_CLOCK_SKEW` 警告；差超過 5 秒，就可能弄垮它選主用的 Paxos lease，整個 cluster 開始抖。

現在想像你在維護其中一台 node，把它的網路斷掉幾個小時。斷網期間，這台機器沒有任何「對時」的對象，只能靠自己肚子裡那顆計時零件自走——而那顆零件本來就不完美，會慢慢走快或走慢。等你把網路接回來，它的時鐘已經漂出去一截了。

問題是：**接回網路那一刻，負責對時的程式要花多久，才能把時鐘拉回 50 毫秒以內？**

而且我要的不是「跑一次看看」這種軼事，是一張**能對照預測的表**——先用原理算出「應該要多久」，再實測對照，看原理對不對。

具體要回答四個問題：

1. **失聯恢復**：跟對時來源失聯 1 小時 / 4 小時 / 24 小時、而且這台機器的計時零件本身有 0 / ±10 / ±100 / ±1000 ppm 的快慢誤差（ppm 是什麼下面會解釋），恢復連線後多久把誤差降到 50 毫秒以下？再加 ±400 / ±500 ppm 兩個邊界探針，找出 kernel 補償能力的極限。
2. **慢慢修錶的速度**：刻意把時鐘誤差卡在 399 毫秒（恰好在「跳錶門檻」下方 1 毫秒），讓 kernel 純靠「慢慢走快/走慢」追回來，要多久滑回 50 毫秒以內？
3. **重啟對時行為**：把對時程式 `systemctl restart` 之後，它是不是馬上對時？10 / 50 / 100 / 500 毫秒的偏移各要多久收斂？
4. **降低對時頻率的穩定度**：把「最長多久對一次時」這個設定（`PollIntervalMaxSec`）從預設 2048 秒壓到 256 秒，在「一個 ping 封包都不能掉、5 秒停機都不可接受」的生產級嚴苛判準下，會不會引入任何不穩定？

這頁只講**怎麼設計這套實驗**——背景知識、為什麼不必真的等一整天、誤差怎麼注入才像真的、收斂怎麼量才不會被受測程式自己騙。實驗本身的步驟、矩陣與「預測 vs 實測」的對照結果在下一篇 [四組實驗設計與結果](/systemd/features/timesyncd-drift-lab-experiments)。

設計的事實基礎來自三篇既有文章：失聯時對時程式完全不碰時鐘的 source 證明在 [斷網時 timesyncd 在做什麼](/systemd/features/timesyncd-outage-and-maintenance)；重啟路徑在 [重啟對系統時間的影響](/systemd/features/timesyncd-restart-impact)；降低對時頻率的理論推演在 [把 PollIntervalMaxSec 壓低會怎樣](/systemd/features/timesyncd-low-max-poll-simulation)。

## 背景小教室：時鐘怎麼走、為什麼會歪、NTP 是什麼

如果你已經很熟 NTP 與 kernel 的時鐘機制，可以跳到下一節。如果不熟，這節給你看懂全文需要的最小知識。一個詞一句白話、再給英文原文。

**系統時鐘（CLOCK_REALTIME）。** 你的機器有一個「現在幾點」的計時器，這就是系統時鐘，kernel 裡叫 `CLOCK_REALTIME`。它的走動由一顆石英晶體（晶振，crystal oscillator）驅動——晶振以固定頻率振盪，kernel 數振盪次數來推進時間。

**晶振與 ppm。** 沒有一顆晶振是完美的，每顆都略快或略慢一點，這個誤差用 ppm（parts per million，百萬分之幾）來量。換成有感的講法：**100ppm ≈ 一天快或慢 8.6 秒**（`100e-6 × 86400s = 8.64s`）；10ppm 約一天 0.86 秒；1000ppm 約一天 86 秒。ppm 越大，時鐘歪得越快。

**offset（偏移）。** 你的時鐘跟「真正的時間」之間的差距，就是 offset。50 毫秒的 offset 就是你的錶比標準時間快或慢 50 毫秒。整套實驗的目標數字（`< 50ms`）量的就是它。

**NTP 與 timesyncd。** 既然每台機器的時鐘都會自己歪，就需要一個機制定期跟一個「準的時間來源」校正。這個機制叫 NTP（Network Time Protocol，網路時間協定）：你的機器（client）週期性問一台時間 server「現在幾點」，算出自己差多少，再修正。systemd-timesyncd 就是 Linux 上一個輕量的 NTP client 程式（嚴格說它講的是 SNTP，NTP 的簡化版），下文簡稱 timesyncd。

**poll（對時週期）。** timesyncd 不是每秒都問 server，而是隔一段時間問一次，這個間隔叫 poll interval。它會自適應：對得準就把間隔拉長（省封包），對不準就縮短。預設最短 32 秒、最長 2048 秒。問題 4 動的就是這個「最長」。

**slew 與 step：修錶的兩種手法。** 假設你的錶慢了。修法有兩種——

- **step（跳錶，直接撥針）**：一把抓住指針，瞬間撥到正確位置。快，但時間會「不連續」：剛才還是 10:00:00，下一瞬間變 10:00:00.4，中間那 0.4 秒從來沒存在過。
- **slew（滑動，把錶調快讓它自己追上）**：不去撥針，而是把錶的走速調快一點點，讓它在接下來幾秒、幾十秒內自己慢慢追上正確時間，追上後再恢復正常速度。慢，但時間始終連續、不跳、不倒退。

對 ceph mon lease、TLS 憑證的生效時間、資料庫 timestamp 排序這種「時間絕對不能倒退」的東西，**slew 安全、step 危險**。timesyncd 的規則是：偏移小（`< 0.4s`）就 slew，偏移大就 step。這個 0.4 秒的門檻，正是問題 2 與問題 3 在探的東西。

**kernel 的 register（暫存器）。** register 是 kernel 裡記住一個設定值的格子。下文會出現幾個跟時鐘有關的格子，例如記「每個 tick 多長」的 tick register、記「目前頻率補償多少」的 freq register。「往 register 寫值」就是改那格設定，「kernel clamp 這個 register」就是 kernel 不准那格超過某個上限。

有了這些詞，正式開始。

## TL;DR：這套實驗的四個設計決策

這張表怎麼讀：左欄是一個設計決策，右欄是一句話的理由。看不懂的細節下面每節都會展開，這裡先給全貌。

| 設計決策 | 一句理由 |
|---|---|
| **失聯那段用數學算掉**，不真的等、也不做時間加速 | 失聯期間 timesyncd 完全不碰時鐘，時鐘只是以固定快慢自走，累積的偏移 = 失聯時長 × 快慢誤差，可以直接一條公式算出來、一次注入；比加速更準（不是近似）又更快（失聯 0 秒） |
| **誤差用 adjtimex 三個旋鈕注入** | 其中 tick 這個旋鈕是 timesyncd 與 kernel 都不會去動的，注入後就像真實晶振誤差一樣賴著不走，逼對時程式真的學會補償，最像真的 |
| **收斂用獨立的探針量**，不採信受測程式自報 | timesyncd 自報的偏移是它自己做決策的依據，拿它來判 timesyncd 對不對，等於請被告當法官；獨立探針直接量 server 與 client 的真實差 |
| **單機驗證靠一個永遠不會被動到的時鐘** | `CLOCK_MONOTONIC_RAW` 不受任何撥錶影響，所以同一顆 kernel 上把系統時鐘撥成什麼樣，拿它當基準的假 server 回的時間都還是真的——一台機器就能驗整套流程 |

## 為什麼不用真的等一整天：數學折疊

直覺做法是真的斷網 24 小時，再插回去看收斂。問題很明顯：這要花 24 小時，光 4 小時 + 24 小時兩個 cell 就吃掉一整天。退而求其次的「時間加速」（把虛擬機的時鐘調快 100 倍模擬時間流逝）也不好：它只是近似，還會引入它自己的假象。

更聰明的辦法，是先看清楚**失聯那段到底發生了什麼**。

[斷網時 timesyncd 在做什麼](/systemd/features/timesyncd-outage-and-maintenance) 已經從 source 證明一件事：**只要沒收到有效的對時回覆，timesyncd 完全不碰系統時鐘**。它寫時鐘的唯一入口 `manager_adjust_clock`，只有在收到、解析、通過合法性檢查的封包之後才會被呼叫；斷網期間它頂多是「等一下再重試」，時鐘則純粹以硬體晶振的速率自己走。

這代表失聯那段是**完全可預測的自走**：沒有任何回授、沒有任何決策，時鐘就以固定的快慢誤差 F 累積偏移。一條公式就能算出來：

```
累積 offset = 失聯時長 T × 頻率誤差 F
```

舉例：一顆 10ppm 的晶振漂 24 小時，累積偏移 = `10e-6 × 86400s = 0.864s`。既然這段完全沒有 timesyncd 參與、結果可以一條算式算死，就沒有理由真的去等——直接把這段「折疊」成兩個初始條件，一次注入到恢復階段的起點：

- **(a) 累積偏移**：失聯時長 × 快慢誤差算出來的數字，直接撥進 client 時鐘。
- **(b) 殘留的快慢誤差**：晶振本來就在漂，恢復後它還會繼續漂，所以這個 ppm 也要當成「持續存在的條件」一起注入（怎麼注入見下一節）。

折疊之後，實驗只跑**恢復階段**——也就是 timesyncd 重新拿到回覆、kernel 開始修錶的那段。這段是受 kernel 修錶機制約束的真實動態，本來就無法（也不該）加速，最長的 cell 也才十幾分鐘。整套實驗於是從「24 小時 × 好幾個 cell」壓成「每個 cell 幾分鐘到半小時」，而且結果不是近似：失聯那段被精確的數學取代，恢復那段是 1:1 的真實時間。

<Callout type="warning" title="折疊的前提">
折疊只在「失聯期 timesyncd 不碰時鐘」這個 source 事實成立時才正確。一旦中途有任何一個合法回覆漏進來（哪怕只是封包亂序漏進一封），自走假設就破了——所以實驗開跑前必須確認：注入前的 NTP 流量是真的被擋死的。
</Callout>

## 怎麼把時鐘弄歪：adjtimex 的三個旋鈕

要注入誤差，得先有「往時鐘塞誤差」的工具。Linux 提供 `clock_adjtime(2)` 這個 syscall，透過一個叫 `struct timex` 的結構操作時鐘。可以把它想成三個互相獨立的旋鈕，各自控制不同的東西，而且**誰會去動這個旋鈕**也不同——這個差異是選對旋鈕的關鍵。

這張表怎麼讀：左欄是要模擬的目標，中欄是用哪個旋鈕，右欄是為什麼選它。

| 模擬目標 | 用哪個旋鈕 | 為什麼選它 |
|---|---|---|
| 失聯累積的偏移 | `ADJ_SETOFFSET`（一次性把偏移加進時鐘） | 原子注入、不停錶，比「先讀時間再寫回」沒有 race |
| ±100 / ±1000 ppm 的快慢誤差 | `ADJ_TICK`（調 tick 長度，每 tick 10000 ± 1 / ± 10 µs） | tick 是 timesyncd 與 kernel 修錶機制**都不會去動**的旋鈕，注入後像真實晶振誤差一樣賴著不走，逼對時程式真的學會補償，最像真的 |
| ±10 ppm 的快慢誤差 | `ADJ_FREQUENCY`（直接寫 freq register） | tick 旋鈕一格就是 100ppm，切不出 10ppm；freq register 會被 kernel 逐步改寫，但效果等價（見下） |

**先講人話：tick 旋鈕為什麼最像真的。** 系統時鐘是靠數一個個固定長度的 timer tick 在前進。每個 tick 預設 10000 微秒（µs）。如果你把每個 tick 偷偷調長一點點，時鐘每秒數的 tick 就少一點，於是走慢；調短就走快。重點在於：timesyncd 修錶時動的是別的旋鈕，**從來不碰 tick**——所以你用 tick 灌進去的誤差會像真實晶振誤差一樣賴著不走，逼對時程式必須真的在別處累積出反向補償才壓得住。這正是我們要測的真實行為。

kernel 不准 tick 被亂調，合法範圍是預設值 ±10%。下面這段 source 就是那道防線——每個 tick 長度在 timex 驗證路徑裡被夾在 `900000/USER_HZ` 到 `1100000/USER_HZ` 之間，超出就整個拒絕（`USER_HZ` 在我們的 kernel 是 100，所以範圍是 9000 到 11000 µs）：

```c File: linux/kernel/time/timekeeping.c:2360
		    (txc->tick <  900000/USER_HZ ||
		     txc->tick > 1100000/USER_HZ))
```

換算很直接：把 tick 設成 10010µs，等於每秒多走 `10µs / 10000µs = 1000ppm`（快 1000ppm）；設 10001µs 就是 100ppm。

**±10ppm 為什麼改用 freq register 注入，這樣算不算作弊。** ±10ppm 太細，tick 旋鈕一格 100ppm 切不出來，只能改寫 freq register。但 freq register 正是 kernel 修錶會去動的旋鈕——你灌進去的值會被它慢慢吃掉，乍看像作弊。其實效果完全等價，差別只在參考系：

- **真實情境**：晶振有 +10ppm 誤差（在 tick 端），freq register 從 0 開始，kernel 學習後把它推到 −10ppm 來抵銷。
- **模擬情境**：晶振 0 誤差，freq register 注入 +10ppm，kernel 學習後把它推回 0。

兩邊 kernel 看到的「有效快慢誤差」走勢（晶振誤差 + register 補償的總和）一模一樣，收斂行為因此相同。對 ±10ppm 這種細偏移，用 freq register 注入是可接受的等價簡化；±100ppm 以上才必須用 tick 拿到完整保真度。

**偏移為什麼用 `ADJ_SETOFFSET` 而不是 `clock_settime`。** 注入累積偏移時，我們要的是「在現有時間上原子地加一個 delta」。`ADJ_SETOFFSET` 一個 syscall 就把偏移加進時鐘、不停錶。`clock_settime` 是「先讀現在時間、加 delta、再寫回」，讀和寫之間時鐘還在走，會有 race，注入量不精確。

## kernel 修錶的三種方式、三個上限——拆穿「slew 只有 500ppm」這句傳言

整套實驗的預測，都建立在拆穿一句江湖傳言上：「kernel 慢慢修錶（slew）的上限是 500ppm」。

**先講人話版。** 這句話為什麼錯？因為它把三件不同的事混成一件。kernel 其實有**三種不同的修錶方式，各有各的上限**，而那個 500ppm 只是其中一條老路的上限，偏偏 timesyncd 根本不走那條老路。三種方式是：

1. **相位滑動**——「你慢了 360 毫秒，我每秒幫你補掉剩餘的一小塊」。這是 timesyncd 修小偏移用的，**沒有 ppm 形式的上限**，補得飛快。
2. **頻率補償**——「我發現你的晶振長期偏快，我幫你長期調慢一點」。這個有 ±500ppm 的硬上限，但它管的是「能補多歪的晶振」，不是「修錶多快」。
3. **古典 adjtime() 老路**——固定每秒最多修 500 微秒，恰好 500ppm。這才是傳言 500ppm 的真出處，但 timesyncd 不走這條。

把第三條的 500ppm 安到 timesyncd 的 slew 頭上，就是這句傳言的錯誤所在。下面用一張表先給三條路的全貌，再逐一拆 source。

這張表怎麼讀：每一列是一種修錶機制，欄位依序是它的上限、source 出處、以及 timesyncd 用不用它。

| 機制 | 上限 | Source | timesyncd 用嗎 |
|---|---|---|---|
| 相位滑動（`ADJ_OFFSET` + `STA_PLL`） | **沒有 ppm 形式上限**：每秒修掉剩餘偏移的 `1/2^(SHIFT_PLL+tc)`；偏移進場時被 clamp 在 ±500ms（`MAXPHASE`）→ 有效上限約 62,500ppm、且指數衰減（32s poll 時 tc=1、時間常數 τ≈7.5s） | `linux/kernel/time/ntp.c:130`（`ntp_offset_chunk`）、`:315`（`MAXPHASE` clamp）、`linux/include/linux/timex.h:120`（`SHIFT_PLL=2`）、`:135`（`MAXPHASE`） | 是 — 修 `< 0.4s` 偏移走這條 |
| 頻率補償（`time_freq` register） | **±500ppm 硬 clamp**（`MAXFREQ`）——能補償晶振誤差的極限 | `linux/include/linux/timex.h:136`（`MAXFREQ`）、`ntp_update_offset` 結尾把 `time_freq` clamp 進 `±MAXFREQ_SCALED` | 是 — 補償持續性晶振誤差 |
| 古典 `adjtime()` 老路（`ADJ_OFFSET_SINGLESHOT`） | 固定 500µs/s = **恰好 500ppm**（`MAX_TICKADJ`；傳言的真出處） | `linux/kernel/time/ntp.c:43`（`MAX_TICKADJ`）、`second_overflow` 每秒消耗 `time_adjust` | 否 — 不走這條 |

下面逐一拆三段 source。

### 機制一：相位滑動，沒有 ppm 形式上限

**先講人話。** 你慢了 360 毫秒，kernel 不會等速慢慢補，而是每秒砍掉「剩餘偏移」的固定一塊比例。第一秒剩 360 砍掉一塊、第二秒剩得更少再砍一塊……越接近 0 補得越慢，這叫指數衰減。所以它一開始補得非常猛，跟「500ppm 上限」這種定速概念完全是兩回事。

這個機制有個數學名字叫 PLL（phase-locked loop，相位鎖定迴路）。不用管全名的數學，**只要記住它的功能：一個自動回授迴路，會盯著剩餘偏移、每秒咬掉一塊，讓時鐘平滑地滑回去**。

具體咬多少，由 `ntp_offset_chunk` 決定——每秒消耗的偏移，是剩餘量右移 `SHIFT_PLL + time_constant` 位，也就是乘上 `1/2^(SHIFT_PLL+tc)`：

```c File: linux/kernel/time/ntp.c:130
		return shift_right(offset, SHIFT_PLL + time_constant);
```

`SHIFT_PLL` 是固定常數 2：

```c File: linux/include/linux/timex.h:120
#define SHIFT_PLL	2	/* PLL frequency factor (shift) */
```

`time_constant`（簡寫 tc）由 poll 週期決定。32 秒 poll 時 timesyncd 算出的 tc = `log2(32) − 4 = 1`（slew 路徑見 [timesyncd-manager.c:247](/systemd/features/timesyncd-low-max-poll-simulation)），所以每秒消耗比例是 `1/2^(2+1) = 1/8`。剩餘量每秒乘上 7/8，這就是指數衰減。它的時間常數（衰減一個 e 倍需要多久）：

```
τ = 1 / ln(8/7) ≈ 7.5 秒
```

這是個指數，不是定速。對一個 399 毫秒的偏移，**起始那一刻**的有效修正速率是 `399ms × ln(8/7) ≈ 53ms/s`，換算成等效頻率約是 **53,000ppm**——比那句傳言的 500ppm 高出兩個數量級。也就是說，小偏移收斂之快，跟「500ppm 上限」毫無關係。

有了 τ≈7.5s，任何小偏移要滑到 50 毫秒以內的時間都能算：360 毫秒偏移約 15 秒收進 50 毫秒（`ln(360/50) × 7.5s ≈ 15s`），144 毫秒約 8 秒。這幾個算出來的數字，下一篇會拿去跟實測對照。

那這條路有沒有上限？有一個——偏移一進場就被夾進 ±500 毫秒（這個 clamp 上限叫 `MAXPHASE`）：

```c File: linux/kernel/time/ntp.c:315
	offset = clamp(offset, -MAXPHASE, MAXPHASE);
```

`MAXPHASE` = 500000000 奈秒 = 500 毫秒（`linux/include/linux/timex.h:135`）。所以相位滑動的最壞情況有效速率，是偏移頂滿 500 毫秒時的 `500ms × 1/8 = 62.5ms/s`，換算約 **62,500ppm** 等效——而且這只是第一秒，之後沿指數衰減——完全不是 500ppm。（53ms/s 是用連續式 `×ln(8/7)`、62.5ms/s 是用離散式 `×1/8`，兩者都是同一條指數的首步近似。實務上 timesyncd 對超過 0.4 秒的偏移根本不走 slew，原因見下面門檻那段。）

### 機制二：頻率補償，這才是真正的 ±500ppm 硬上限

**先講人話。** 機制一修的是「現在差了多少」，機制二修的是「你晶振長期偏快/偏慢多少」。kernel 學到你晶振的持續性誤差後，把它記在 freq register，長期幫你調速。但這個調速能力有個硬上限：±500ppm。晶振如果歪超過這個量（壞掉的 RTC、計時異常的虛擬機），kernel 頂死也補不回來。

這個 register 被硬 clamp 在 ±MAXFREQ：

```c File: linux/include/linux/timex.h:136
#define MAXFREQ 500000		/* max frequency error (ns/s) */
```

500000 ns/s = 500µs/s = **500ppm**。這是「kernel 能補償多大的晶振頻率誤差」的真極限。如果晶振本身漂超過 500ppm，freq register 頂死 clamp 也補不回來，淨漂移永遠 `≥` 0——這就是為什麼下一篇實驗 1 預測 ±1000ppm 永遠無法穩定收斂、±500ppm 只能「脆弱鎖定」。

請特別分清楚：這個 500ppm 是**頻率補償**的上限，跟機制一相位滑動的速度完全是兩回事。把它們混在一起，就生出了那句傳言。

### 機制三：古典 adjtime()，500ppm 傳言的真出處

**先講人話。** 還有一條更老的修錶路：BSD 的 `adjtime()` / 老式 `adjtimex(ADJ_OFFSET_SINGLESHOT)`。它每秒最多修 500 微秒，恰好 500ppm——傳言就是從這裡來的。但 timesyncd 根本不走這條。

它的上限常數在這：

```c File: linux/kernel/time/ntp.c:43
#define MAX_TICKADJ		500LL		/* usecs */
```

每秒 500µs = **恰好 500ppm**。這就是「kernel slew 上限 500ppm」這句傳言的出處。但 **timesyncd 不走這條老路**：它對小偏移用機制一的相位滑動（`STA_PLL`），對大偏移直接 step。把古典 `adjtime()` 的 500ppm 安到 timesyncd 頭上，就是這句傳言的全部錯誤所在。

### timesyncd 端的門檻：0.4 秒決定 slew 還是 step

kernel 提供機制，但「走哪條」是 timesyncd 在 user space 決定的。它的分流門檻是一個常數：

```c File: systemd/src/timesync/timesyncd-manager.c:52
#define NTP_MAX_ADJUST                  0.4
```

`manager_adjust_clock` 用它分流：偏移絕對值 `< 0.4s` 走 slew（機制一的相位滑動），否則直接 step（跳錶）：

```c File: systemd/src/timesync/timesyncd-manager.c:247
        if (fabs(offset) < NTP_MAX_ADJUST) {
```

這就是為什麼下一篇實驗 2 要把偏移卡在 399 毫秒與 401 毫秒、各差 1 毫秒：399 毫秒走平滑的相位滑動（τ≈7.5s），401 毫秒走不連續的跳錶——0.4 秒這道懸崖，正是這兩種行為的分界。

## 怎麼量才公平：不採信受測程式

設計量測時有個致命誘惑：直接讀 timesyncd 自報的偏移（`timedatectl show-timesync` 的 `Offset` 欄、或 journal 日誌）。不行——那個數字是 timesyncd 拿來做 slew/step 決策的**依據**，拿它來判 timesyncd 對不對，等於請被告當法官。它自報「我已經對準了」，不代表它真的對準了。

所以量測必須完全獨立於受測程式。做法是在 client 上另跑一支獨立探針 `ntp_probe.py`：用 raw socket 直接對 server 發 SNTP 查詢，每秒一次（1Hz），把每次量到的 `offset_ms, delay_ms` 記成 CSV。偏移定義為 `server_time − client_time`，純粹由 SNTP 封包的四個時戳算出，跟 timesyncd 的內部狀態完全無關。在 LAN 內往返延遲是 sub-ms 等級，對我們 50 毫秒的判準綽綽有餘（量測雜訊遠小於判準）。

**收斂判定全實驗統一**：`|offset| < 50ms` 持續 60 秒，取首次進入該狀態的時刻。為什麼要求「持續 60 秒」而不是「碰到一次就算」？因為時鐘可能剛好滑過 50 毫秒、又彈回去——要求撐滿 60 秒，才能排除這種瞬間的假合格。

（有一個例外：下一篇的實驗 1 問的是「能不能**持續鎖定**」而不是「滑過一次」，所以它的撐住時間拉長成 660 秒——60 秒收斂判定 + 600 秒穩定窗。這個延長是 L4 實測逼出來的：60 秒的撐住時間會被「超界 cell 的低檔 poll 鋸齒」騙過去，理由與實證見[實驗篇的共用協定](/systemd/features/timesyncd-drift-lab-experiments)。）

**校準：扣掉天然漂移。** client 與 server 各有自己的晶振，就算兩邊都設成 0ppm，兩顆晶振之間還是有天然的相對漂移（±幾 ppm）。如果不處理，±10ppm 那一格 cell 會被這個背景污染。所以實驗前先跑 `calibrate.sh`：停掉 timesyncd、把 freq/tick 旋鈕歸零、用探針量 30–60 分鐘的天然基線 ppm 存檔；之後 `analyze.py` 從每個 cell 的結果扣掉這個 `client_ppm`，得到真正注入造成的 `effective_ppm`。這讓 client 與 server 跑在不同實體機時，細偏移的 cell 也乾淨。

## 一台機器自我驗證的 trick：MONOTONIC_RAW = 不會被動到的碼錶

最妙的一招在這裡：整套流程不需要兩台機器就能端到端驗證一遍。

關鍵是一個特別的時鐘 `CLOCK_MONOTONIC_RAW`。前面說的撥錶手段（注入偏移、灌 ppm、跳錶）動的全是系統時鐘 `CLOCK_REALTIME`。而 `CLOCK_MONOTONIC_RAW` 是唯一**不受任何撥錶影響**的時鐘——不被 slew、不被 freq/tick 旋鈕、不被 step 撥動，純粹反映硬體計時。把它想成一個放在桌上、誰都動不了的碼錶：你怎麼亂撥牆上的鐘，這個碼錶照走不誤。

於是可以在同一台機器上跑一支 `fake_ntp_server.py`，讓它拿 `CLOCK_MONOTONIC_RAW`（加一個固定的 epoch 偏移）當作「真時」來回答 SNTP 查詢。然後把 client 的 timesyncd 指向 `NTP=127.0.0.1`。現在不管實驗怎麼把這顆 kernel 的系統時鐘撥歪（注入偏移、灌 ppm、觸發跳錶），假 server 回的時間都還是來自那個動不了的碼錶——等於在同一顆 kernel 上同時擁有「被玩壞的受測時鐘」和「乾淨的參考時鐘」。一台機器就能跑通注入、收斂量測、跳錶偵測整條路。

<Callout type="warning" title="啟動順序">
唯一注意事項：假 server 必須在**注入誤差之前**就啟動並錨定好 epoch。如果先撥系統時鐘再算 epoch 偏移，參考點就被污染了。
</Callout>

這也是交付前 L3 端到端驗證的基礎——用一台真實的 client VM（同 kernel、同 systemd 版本、同 PVE 環境）跑單機版流程，就能把前置檢查、注入、量測、收斂判定整條鏈驗過一遍，不必先開另一台 server VM。L3 實際驗證通過：假 server 被 timesyncd 接受同步（`timedatectl` 顯示 `Reference=RAWC`，正是這個 `CLOCK_MONOTONIC_RAW` 基準假 server 的識別碼），整條「注入 → 量測 → 收斂」鏈在單機上跑得通。
