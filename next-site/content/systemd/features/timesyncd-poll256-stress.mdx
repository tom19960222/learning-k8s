## 場景

前面 [drift lab 實驗篇](/systemd/features/timesyncd-drift-lab-experiments) 的實驗 4 已經回答過「把 `PollIntervalMaxSec` 從預設 2048 秒壓到 256 秒，在乾淨網路下穩不穩」——答案是穩（4 小時 soak、90 萬個 ping 0 丟包、0 step、0 lease miss）。

但對一個「連 2 個 ping 都不能掉、5 秒 downtime 都不可接受」的 ceph 生產環境，乾淨網路下的全綠還不夠。真正想知道的是：**256s 到底能扛多爛的網路？它從哪裡開始壞？** 這頁專門回答這個——收斂時間的問題前面都有了，這裡只談穩定性的「壓力上限」。

## 名詞速查

- **jitter（抖動）**：網路延遲不固定、忽快忽慢。NTP 最怕這個，因為它靠來回時間估算時鐘差。
- **netem**：Linux 內建的網路模擬工具，可以人為注入延遲、抖動、丟包來壓測。
- **offset（偏移）**：時鐘跟正確時間差多少。`< 50ms` 是 ceph 的紅線。
- **對稱 vs 不對稱整形**：netem 預設只整形「出向」封包（送出去的），造成來回路徑不對稱。要模擬真實的對稱壞網路，得用 **IFB** 這個技巧把「進來」的封包也一起整形（下面方法論會解釋為什麼這很關鍵）。
- **滾動中位數**：把一串忽高忽低的量測值，取每段的中位數，濾掉零星尖刺、留下真正的趨勢。

## 256s 相對 2048s，只改了兩件事

從原始碼看，把 `PollIntervalMaxSec` 壓到 256 只動兩個東西：

1. **問 NTP 的頻率**：封包量 ×8（每 ~4 分鐘一個 76-byte UDP）。微不足道，**不可能造成不穩**。
2. **PLL 的 time constant**：poll 越短、修錶迴路反應越快，但對單筆網路雜訊越敏感。**這是唯一講得出來的穩定性風險**——也正是這次要壓測的對象。

[把 PollIntervalMaxSec 壓低會怎樣](/systemd/features/timesyncd-low-max-poll-simulation) 那篇從理論推演過：壓低 max poll 會降低抗雜訊能力，擔心 jitter 被放大進時鐘、offset 變成 sawtooth 踩穿 50ms。這頁就是來實測這個擔憂到底成不成立、在多大的 jitter 下才成立。

## 量測方法論：兩個非做不可的修正

這次實驗最花工夫的不是跑，是**確保量到的是「時鐘穩不穩」，而不是量測工具自己的假象**。煙霧測試一開始就踩到兩個坑，都修掉了：

### 坑一：netem 只整形出向 → 路徑不對稱 → 時鐘被偏掉，但與 poll 無關

`tc netem` 掛在預設的 root qdisc 上，**只延遲送出去的封包**，回來的不延遲。這讓 NTP 的來回路徑不對稱，offset 估計被系統性偏掉約「延遲的一半」。煙霧實測：

| 注入 | probe 量到的 offset |
|---|---|
| 無 | 1.9ms |
| 只延遲出向 80ms | **42.0ms**（≈ 80÷2 的偏移） |
| 雙向對稱 80ms（IFB） | **1.9ms**（offset 不被偏） |

這個偏移很重要但**容易被誤讀**：它在任何 poll interval 都會發生（是 NTP「來回路徑對稱」這個假設破裂，不是 256s 的鍋）。如果照「只整形出向」去跑，會得到「256s 在 80ms 抖動就快爆了」的錯誤結論。所以要測「256s 的 PLL 耐受」，延遲必須**對稱**——用 IFB 把進向封包也導去整形。本實驗因此分兩種模式：

- **sym（對稱，IFB 雙向）**：來回都延遲，offset 不被偏，純粹隔離「PLL 對雜訊／樣本被餓死」的耐受——這才是 256s 真正的問題。
- **asym（不對稱，只整形出向）**：當對照，量化「路徑不對稱 → 偏移 ≈ 延遲/2」這個 poll 無關的效應，順便標出它的膝點。

### 坑二：重 jitter 下，單發探針自己就會跳 ±幾十毫秒

獨立探針是「單發 SNTP 查詢」，每筆的誤差 ≈ `(出向抖動 − 回向抖動)/2`。在 ±40ms 抖動下，**單筆 offset 讀數就能跳到 ±40ms，即使時鐘本身穩如泰山**。煙霧實測 j40（對稱 40±40ms）：raw 單筆 peak 53ms，但全程中位數只有 −1.1ms、滾動中位數 peak 6.2ms——時鐘其實準到 1ms，那 53ms 全是探針自己的封包被打到的雜訊。

所以判準改用**滾動中位數**（視窗 31 秒）：零均值的量測雜訊被濾掉，留下時鐘真正的慢變偏移。這也跟 ceph 的實際感受一致——ceph 的 timecheck 看的是持續性偏移，不會被單筆網路尖刺觸發。

## 階梯設計

全程 `PollIntervalMaxSec=256`。每階獨立判準（jitter 模式）：**offset 滾動中位數全程 `< 50ms`、0 次非預期 step、0 次 5 秒 lease miss**；ping 因 netem 注入必掉、只當參考不作判準。

這張表怎麼讀：mode 欄 sym=雙向對稱（測 PLL 耐受）、asym=只出向（測路徑不對稱偏移）。由乾淨錨點 → 不對稱延遲找膝點 → 對稱抖動加重 → 對稱丟包加重。

| # | tier | mode | 注入 | 在測什麼 | 時長 |
|---|---|---|---|---|---|
| 0 | clean | clean | 無 | 乾淨 256s 錨點 | 1.5h |
| 1–4 | a40 / a80 / a120 / a200 | asym | 出向延遲 40 / 80 / 120 / 200ms | 路徑不對稱偏移 ≈ 延遲/2 的膝點 | 各 2h |
| 5–9 | j10…j250 | sym | 對稱延遲抖動 10±10 → 250±250ms | PLL 對抖動的真實耐受 | 各 2.5h |
| 10–13 | loss10…loss75 | sym | 對稱丟包 10 / 30 / 50 / 75% | 樣本被餓死、靠 freq 補償撐 free-run | 各 2.5h |
| | **合計** | | | | **≈ 32h** |

## Source 推導預測

預測都標「機制推論」，跑完回填實測對照。

- **對稱抖動（sym jitter）→ 預測一路 PASS，可能到 250±250ms 都不破**。理由：NTP 的 offset 估計 `((T2−T1)+(T3−T4))/2` 對「對稱」延遲大致抵銷；加上 spike rejection 會丟掉雜訊樣本，不讓它污染時鐘。抖動主要灌進 `delay` 與 jitter 統計、觸發 poll 變密，但時鐘本身靠 freq 補償守住。
- **對稱丟包（sym loss）→ 預測一路 PASS，含 75%**。理由：丟包只是讓有效樣本變稀，樣本空檔時鐘靠 freq 補償 free-run；殘餘 ppm 學會後極小（`< 1ppm`），就算空檔拉長，漂移也遠不到 50ms。
- **不對稱延遲（asym）→ 預測在 a120 附近破界**。偏移 ≈ 延遲/2：a40→20ms（PASS）、a80→40ms（PASS，貼線）、a120→60ms（**FAIL**）、a200→100ms（FAIL）。**膝點 ≈ 出向延遲 100ms（偏移 50ms）**——但這條是「路徑不對稱」的鍋、**與 256s 無關**，預設 2048 也一樣會破。

**如果預測成立，這頁的結論會是**：256s 對「對稱的網路惡化（抖動、丟包）」異常耐操——因為 NTP 的 offset 估計對對稱延遲免疫、spike rejection 擋雜訊、freq 補償撐丟包；真正能把時鐘推過 50ms 的是**路徑不對稱**，而那跟 poll interval 無關。換句話說，**256s 的穩定性不是問題，網路對稱性才是**。

## 實測結果

資料源 `experiments/timesyncd/results/exp5/`（已 commit 進 repo），由 `experiments/timesyncd/run/exp5-jitter-sweep.sh` 產生；下表 rm31 / rm241 兩欄可由各 tier 的 `probe.csv` 重算，彙整在 `experiments/timesyncd/results/exp5/rolling-median-windows.txt`。client = Ubuntu 22.04 / kvm-clock VM、天然漂移 +60.9ppm；每階全程 `PollIntervalMaxSec=256`，判準 = offset 滾動中位數（視窗 31s）`< 50ms`、0 step、0 lease miss。

下表有兩個滾動中位數欄位，對比很有資訊量（獨立探針每秒量一次（1 Hz），所以 rm31 ≈ 31 秒視窗、rm241 ≈ 4 分鐘視窗）：**rm31**（31s 視窗，判準用的就是它，抓得到短暫尖峰）與 **rm241**（4 分鐘視窗，把短暫的東西平均掉、只留「持續」偏移）。兩者都低 = 穩；rm31 高但 rm241 低 = **暫態**（一下子歪、很快回來）；兩者都高 = **持續失穩**（時鐘真的被推著待在那裡）。

| 階段 | 注入 | rm31（暫態, ms） | rm241（持續, ms） | verdict |
|---|---|---|---|---|
| clean | 無 | 1.0 | 0.2 | ✅ PASS |
| a40 | 出向延遲 40ms（不對稱） | 20.0 | 10.6 | ✅ PASS |
| a80 | 出向延遲 80ms（不對稱） | 47.0 | 5.3 | ✅ PASS |
| a120 | 出向延遲 120ms（不對稱） | 74.3 | 31.6 | ❌ FAIL（暫態） |
| a200 | 出向延遲 200ms（不對稱） | 102.0 | 43.7 | ❌ FAIL（暫態） |
| j10 | 對稱抖動 10±10ms | 44.0 | 35.8 | ✅ PASS（貼線） |
| j40 | 對稱抖動 40±40ms | 68.2 | 60.5 | ❌ FAIL（持續） |
| j100 | 對稱抖動 100±100ms | 207.2 | 151.7 | ❌ FAIL（持續） |
| j160 | 對稱抖動 160±160ms | 275.4 | 236.7 | ❌ FAIL（持續） |
| j250 | 對稱抖動 250±250ms | 381.3 | 293.6 | ❌ FAIL（持續） |
| loss10 | 對稱丟包 10% | 34.0 | 23.6 | ✅ PASS |
| loss30 | 對稱丟包 30% | 11.9 | 8.4 | ✅ PASS |
| loss50 | 對稱丟包 50% | 5.1 | 3.8 | ✅ PASS |
| loss75 | 對稱丟包 75% | 60.0 | 58.5 | ❌ FAIL（見下） |

第一個收穫就藏在兩欄的對比裡：**不對稱延遲（a120/a200）的 FAIL 是暫態**——rm31 衝到 74/102，但 rm241 降回 32/44，代表那是套用 netem 瞬間的 slew 過衝（機制推論）、幾分鐘就被吸收；而**對稱抖動（j40 以上）的 FAIL 是持續**——rm31 與 rm241 都居高不下，時鐘被真正推著待在 60ms 以外。下面分軸說明。

四軸逐一解讀：

**乾淨網路：完美。** rm241 = 0.2ms，整整 1.5 小時紋風不動。256s 在乾淨網路下沒有任何穩定性問題——這跟 [drift lab 實驗篇](/systemd/features/timesyncd-drift-lab-experiments) 實驗 4 的 4 小時 soak 結論一致。

**對稱抖動：真的會把時鐘搞歪——而且打臉我的預測。** 我原本推論「對稱抖動會在 NTP 的 offset 估計裡抵銷掉，一路 PASS」。錯。實測（rm31，判準欄）隨抖動單調惡化：j10 44 → j40 68 → j100 207 → j160 275 → j250 381（ms）。注意 j10（10±10ms）只是**貼著 50ms 線下方勉強 PASS**，真正第一個 FAIL 是 j40——膝點就卡在 j10 與 j40 之間。**這不是探針雜訊**——我用多種視窗的滾動中位數交叉驗證：若是零均值的量測雜訊，視窗加大（241 樣本 = 4 分鐘）會被抹平到趨近 0；但 j40 的 rm241 仍有 60.5ms、j100 仍有 151.7ms，居高不下，代表時鐘**真的**被推到那麼遠。機制（對照原始碼）：spike 偵測是拿這筆 offset 跟前一筆比、門檻是 `3 × jitter`，而抖動一大、jitter 統計被撐高，這個門檻也跟著水漲船高——所以**穩定的對稱抖動看起來不像 spike**，那些被系統性偏掉的非 spike 樣本就直接過 `if (!spike)` 這道閘餵進時鐘修正；同時被撐高的 jitter 又把 poll 逼降到 min（小時間常數 → 每筆雜訊權重更大），雜訊就這樣被積分進時鐘。這也回答了「為什麼 spike rejection 沒擋住」——它擋的是孤立尖刺，不是這種被門檻一起抬高、整片過關的穩定偏移。這驗證了 [把 PollIntervalMaxSec 壓低會怎樣](/systemd/features/timesyncd-low-max-poll-simulation) 那篇的擔憂——但**先別急著怪 256s**，下面控制組會翻案。

而且要警告一點：前面 13 階失穩都還是平滑的 slew 漂移（0 次 step、時間不倒退），但**最極端的 j250（250±250ms，最壞情況下來回兩向都打到極值、RTT 在 0–1000ms 間狂跳）出現了 1 次 step**——某一筆樣本的量測 offset 衝破 0.4 秒門檻，timesyncd 直接跳錶。對「時間不能倒退」的系統，這是比 slew 漂移更危險的一級：在荒謬等級的抖動下，時鐘不只是歪，還可能不連續跳一下（lease 仍 0 miss，沒到弄垮的程度）。

**對稱丟包：極度耐操。** loss10/30/50 全 PASS，而且**丟得越兇越穩**（loss50 的 rm241 只有 3.8ms）。原因很漂亮：丟包不會把雜訊灌進時鐘，只是讓有效樣本變稀；樣本空檔時 kernel 靠 freq 補償平滑 free-run，反而少了抖動污染。loss75 判 FAIL，但要看細節：全程 median = −0.1ms、前後半都 ~0，時鐘其實穩在 0，只是中途有一段約 4 分鐘的局部偏移叢集（極可能是 75% 丟包下偶爾擠過來一筆 ±300ms 的野樣本、timesyncd slew 了一下又修回）。外加一個量測校準警訊：注入名目 75% 但實際只壓出約三分之一的有效丟包——ping 量到 `33.7%`（`loss75/ping.txt`）、獨立探針的查詢失敗率 `37.4%`（IFB ingress 的 loss 可能沒完全生效），所以 loss75 這格的「75%」名不副實。結論不變、但要按**實際驗證到的有效丟包**來下：在低階各檔到頂端的 ~37% 這個已驗證範圍內，**丟包對 256s 的時鐘穩定性幾乎無威脅**；名目 75% 因校準問題沒真正壓測到，不在這個保證裡。

**不對稱延遲：膝點 ~100ms，但這條跟 poll 無關。** a40/a80 PASS、a120/a200 FAIL，膝點落在出向延遲 ~100ms。但這個 FAIL 是 netem 只整形出向造成的路徑不對稱——offset 被偏約「延遲/2」，a120 的暫態 slew 因此衝到 74ms。更深一層的事實：穩態下 timesyncd 會把時鐘 slew 到「它量到的 offset = 0」，而探針走同一條不對稱路徑、也被偏一樣多，所以**穩態的真實時鐘誤差（≈延遲/2）對同路徑量測是隱形的**。重點是：這個偏移在任何 poll interval 都會發生，**不是 256s 的問題**，是 NTP「路徑對稱」假設破裂的通病。

### 控制組：抖動失穩是 256s 特有，還是任何 poll 通病？

對稱抖動把時鐘搞歪了——但這是「設成 256s」害的，還是抖動本身害的？跑同樣抖動、但 max poll 用**預設 2048** 的控制組來判（這是 14 階階梯之外另外加跑的 2 次：j40 / j100 各一次、各約 2.5h，所以不算在前面 ≈32h 的預算裡）：

| 抖動 | poll 256（rm241） | poll 2048（rm241） |
|---|---|---|
| 40±40ms | 60.5ms | 63.6ms |
| 100±100ms | 151.7ms | 197.6ms |

**判決：預設 2048 在同樣抖動下漂得跟 256 一樣多，甚至略多。** 所以抖動失穩**不是 256s 特有**。機制上也說得通：抖動會持續觸發 spike / ÷2 分支，把 poll 自動降到 min，**不管你的上限設 256 還是 2048**；上限只在網路乾淨、poll 能往上爬時才有意義，而那種情況下兩者都穩。換句話說，[低 max poll 那篇](/systemd/features/timesyncd-low-max-poll-simulation) 的「低 max poll 降低抗雜訊能力」在**強到會逼降 poll 的抖動**下不成立——因為任何 poll 設定都會被逼降。

## 結論：把 PollIntervalMaxSec 設成 256，穩定性上有問題嗎？

**沒有。** 把整份地圖收成一句：

- **乾淨網路**：256s 完美（rm 0.2ms），跟預設一樣穩。
- **網路抖動**：時鐘會被雜訊推歪——但這是**抖動**的鍋，不是 256s 的鍋；預設 2048 在同樣抖動下一樣歪（甚至略歪）。256s 沒有讓事情變差，邊際上還略好。
- **網路丟包**：256s 幾乎免疫（freq 補償撐 free-run）。
- **路徑不對稱**：會造成 ≈延遲/2 的時鐘偏移，但這是 NTP 的通病、與 poll 無關。

所以對「連 2 個 ping 都不能掉、5 秒 downtime 都不可接受」的 ceph 環境：**設 256s 是安全的**——它在乾淨網路下完美，在惡網下不比預設差。真正會傷到時鐘準度的是**網路抖動本身**，而對付抖動的槓桿不是調 max poll（沒用），是改善網路品質、或換用 filtering 更強的 discipline（如 chrony）。256s 的唯一實質代價是封包量 ×8（每 4 分鐘一個 76-byte UDP，可忽略）。

一個誠實的範圍標註：控制組只在 j40 / j100（中–強抖動）證實 256 ≈ 2048；更輕微的抖動（如 j10）下、poll 還爬得上去時，上限的差異是否會浮現，本實驗沒有對照（留待後續）。

## 指令

```bash
# 全 14 階耐受地圖（detached、斷點續跑、約 32h）
sudo ./run/exp5-jitter-sweep.sh --detach

# 部署後煙霧（clean + asym + sym 各 1 階，~90s）
sudo ./run/exp5-jitter-sweep.sh --smoke

# 隨時查進度（SSH 重連也行）
```
