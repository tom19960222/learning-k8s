## 場景

過去一週監控我家三台 ceph node 上的 systemd-timesyncd，每分鐘抓一次 `timedatectl timesync-status` 的數值，取一週統計：

| 指標 | 觀察值 |
| --- | --- |
| offset 範圍 | -898us 到 +974us |
| jitter 最大 | 980us |
| root distance | 1.876ms 到 3.944ms |
| frequency 範圍 | -12.475ppm 到 -0.936ppm |

問題：`PollIntervalMaxSec` / `PollIntervalMinSec` 該設多少？從激進到保守，並且考慮兩種狀況：

1. **一般情況** — RTC 行為穩定（如上面的數據）。
2. **RTC 突然抽一下** — 想盡快偵測並恢復。

這頁從源碼把「每個指標代表什麼」與「設定值會怎麼影響行為」串起來，最後給三組推薦設定 + 取捨分析。

前置閱讀：[poll interval adaptation](./timesyncd-poll-interval-adaptation) 講演算法、[low max poll simulation](./timesyncd-low-max-poll-simulation) 講過度激進的副作用。這頁是「實測驅動」的對應篇。

## 先解釋一下 PLL

下面會一直提到「PLL」與「PLL constant」，這裡先用日常生活比喻解釋一次，後面就不再重複。

**PLL = Phase-Locked Loop（鎖相迴路）** — 一種讓「本地時鐘」鎖定在「外部參考時鐘」上的控制方法。

### 跟車比喻

想像你開車跟在前車後面，要保持固定的安全距離：

```
       距離 = offset
       ↓
  [前車]----------[你]
   ↑               ↑
   NTP server     你的系統時間
```

每隔一段時間（= NTP poll interval），你抬頭看一下車距：

- 距離太遠 → 踩油門加速一點
- 距離太近 → 鬆油門讓車速降一點
- 不太對勁但又不確定（前車是真的減速？還是你眼花？）→ 不動，先觀察

這個「看到車距、決定加減速多少」的反應強度，就是 **PLL constant**。

### PLL constant 在做什麼

- **constant 小** = 反應很急 = 看到一點點車距變化就大力油門/煞車
  - 好處：能快速跟上前車的真實變化
  - 壞處：對「眼花」（網路雜訊、jitter）也會反應 → 你的車一直在抽搐
- **constant 大** = 反應很冷靜 = 看到車距變化先觀察、看很多次才確定
  - 好處：忽略眼花、平穩跟車
  - 壞處：前車真的緊急變道你要好一會才反應過來

在 NTP 場景：

| 比喻 | 對應 |
| --- | --- |
| 前車的位置 | NTP server 報的時間 |
| 你的車的位置 | 你的系統時間 |
| 車距 | offset |
| 抬頭看一次的間隔 | poll interval |
| 油門/煞車的操作 | `clock_adjtime()` 微調系統時間每秒走多快 |
| 反應強度設定 | PLL constant |

### 為什麼 NTP PLL constant 跟 poll interval 綁在一起

直觀想：

- 抬頭看得頻繁（短 poll）→ 每筆資料品質都還沒過濾就要用 → 必須反應慢一點（大 constant）才不會被雜訊抽搐 — 但 Linux NTP 是反過來
- 實際上 Linux 的設計：**短 poll → 小 constant（更敏感）**

為什麼？因為 NTP PLL 把「constant」定義為「時間常數的 log2」— 是「需要多少秒才完整反應一次完整 offset」的指數。短 poll 意味著你想要「快速反應系統」，所以時間常數要小。

實作上 Linux kernel 收 `constant=N` 之後，PLL 內部會把反應時間定為 `2^(N+4)` 秒。所以：

- poll=64s → constant=2 → 反應時間 = 2^6 = 64s（剛好跟 poll 一致）
- poll=2048s → constant=7 → 反應時間 = 2^11 = 2048s

設計意圖：**反應時間 = poll interval**。每次抬頭、剛好用一個 poll interval 把上次看到的偏差吃完。

### 那「constant 小 = 對 jitter 敏感」怎麼解？

回到跟車：你 1 秒看一次車距（短 poll）vs 你 30 秒看一次車距（長 poll）。

- 短 poll：每筆讀數獨立反映 1 秒內的車距變化。雜訊（眼花、儀表晃動）會直接進。
- 長 poll：你看到的是「平均」車距。雜訊在 30 秒間自然平均掉。

NTP 同理。**短 poll + 小 constant** 意味著你拿到的是「最新的瞬時讀數，且 PLL 全力相信它」。如果瞬時讀數本身有雜訊，這雜訊就直接餵給你的 system clock。

長 poll + 大 constant 則是「拿到一筆讀數，PLL 視為長期趨勢的一個小貢獻」— 自然把雜訊磨平。

### 這頁用 PLL 的地方

- **PLL constant** = `manager_adjust_clock` 算給 kernel 的 `.constant = log2i(poll) - 4`。決定 system clock 對 NTP offset 的反應強度。
- **PLL slew** = 「微調 clock 走快走慢」這個動作（對 |offset| < 0.4s）。對應跟車的踩油門/鬆油門。
- **PLL jump** = 「直接把時鐘 set 過去」（對 |offset| ≥ 0.4s）。對應「車距太大、放棄追、直接瞬移過去」 — 罕見、重啟才常見。

知道這些就夠了，下面可以看實測數據怎麼套進去。

## 每個指標來自哪裡

### offset

每筆 NTP 回應算出的 system clock 與 server time 的差距：

File: `systemd/src/timesync/timesyncd-manager.c` (line 530)

```c
offset = ((receive - origin) + (trans - dest)) / 2;
```

這是 RFC5905 的標準對稱式：`((T2−T1) + (T3−T4)) / 2`。`timedatectl timesync-status` 印出來的 `Offset` 就是最後一次量到的值。

實測 ±1ms 的意義：**`fabs(offset) < NTP_ACCURACY_SEC × 0.25 = 50ms` 永遠成立**。對 `manager_adjust_poll` 來說，這代表 Branch 3（增加 poll interval）會一路觸發。

### jitter

8 筆 sample 的 offset 對「delay 最短那筆」的 RMS 差：

File: `systemd/src/timesync/timesyncd-manager.c` (line 337)

```c
j = 0;
FOREACH_ELEMENT(sample, m->samples)
        j += pow(sample->offset - m->samples[idx_min].offset, 2);

size_t n = ELEMENTSOF(m->samples);
m->samples_jitter = sqrt(j / (n - 1));
```

jitter 980us ⇒ spike 判定門檻 `3 × jitter = 2.94ms`。

實際上 spike detection 有兩條：

File: `systemd/src/timesync/timesyncd-manager.c` (line 357)

```c
/* do not accept anything worse than the maximum possible error of the best sample */
if (fabs(offset) > m->samples[idx_min].delay)
        return true;

/* compare the difference between the current offset to the previous offset and jitter */
return fabs(offset - m->samples[idx_cur].offset) > 3 * jitter;
```

第一條：「offset 超過 8 筆 sample 裡最小 delay」也算 spike。實測 root distance 最小 1.876ms ⇒ 最小 delay 至少 ~2ms（delay 通常與 root distance 同數量級）⇒ 第一條的閾值 ~2ms。

合在一起：實測環境的 spike 門檻 ≈ 2~3ms。

### root distance

NTP server 自己回報的「我跟我的上游 server 距離有多遠」：

File: `systemd/src/timesync/timesyncd-manager.c` (line 493)

```c
root_distance = ntp_ts_short_to_d(&ntpmsg.root_delay) / 2 + ntp_ts_short_to_d(&ntpmsg.root_dispersion);
if (root_distance > (double) m->root_distance_max_usec / (double) USEC_PER_SEC) {
        log_info("Server has too large root distance. Disconnecting.");
        return manager_connect(m);
}
```

預設 `NTP_ROOT_DISTANCE_MAX_USEC = 5s`：

File: `systemd/src/timesync/timesyncd-manager.c` (line 55)

```c
/* Default of maximum acceptable root distance in microseconds. */
#define NTP_ROOT_DISTANCE_MAX_USEC      (5 * USEC_PER_SEC)
```

實測 1.876 ~ 3.944ms，遠低於 5s 上限。連用 LAN NTP 都不必擔心換 server 的問題。

### frequency

kernel 上次 `adjtimex` 回報的 PLL 校頻量：

File: `systemd/src/timesync/timesyncd-manager.c` (line 295)

```c
m->drift_freq = tmx.freq;
```

`timedatectl` 印的 `Frequency` 是 `drift_freq / 0x10000`：

File: `systemd/src/timedate/timedatectl.c` (line 534)

```c
r = table_add_cell_stringf(table, NULL, "%+.3fppm", (double) i->freq / 0x10000);
```

實測 -12.475 ~ -0.936ppm 的意義：

- **方向永遠是負的** — 本地 oscillator 比 server 慢，PLL 持續往「加快」方向校。RTC quartz 的 ppm 偏移在常溫下穩定為負屬於正常製程偏差。
- **變動範圍 11.5ppm** — 室溫變化、CPU thermal 等慢變因素造成的「真實 drift rate 變動」。每秒會累積 12.5us，在 2048s（預設 max poll）期間理論最大累積 = 25.6ms。
- **但實測 offset 仍 < 1ms** — 因為 PLL 在 11 分鐘模式持續修正 rate，所以 drift 不會累積到下次 NTP 才被發現。

## 這些數字告訴你 timesyncd 現在的行為

把實測數據套進 `manager_adjust_poll`：

File: `systemd/src/timesync/timesyncd-manager.c` (line 364)

```c
static void manager_adjust_poll(Manager *m, double offset, bool spike) {
        if (m->poll_resync) {
                m->poll_interval_usec = m->poll_interval_min_usec;
                ...
        }
        if (!spike && fabs(offset) > NTP_ACCURACY_SEC) {                  /* > 200ms */
                m->poll_interval_usec = m->poll_interval_min_usec;
                return;
        }
        if (fabs(offset) < NTP_ACCURACY_SEC * 0.25) {                     /* < 50ms */
                if (m->poll_interval_usec < m->poll_interval_max_usec)
                        m->poll_interval_usec *= 2;
                return;
        }
        if (spike || fabs(offset) > NTP_ACCURACY_SEC * 0.75) {            /* > 150ms */
                if (m->poll_interval_usec > m->poll_interval_min_usec)
                        m->poll_interval_usec /= 2;
                return;
        }
}
```

實測 offset 一直在 1ms 量級，永遠落在 Branch 3 的「< 50ms」區間。所以行為是：

- 重啟後 poll 從 `min_usec` 開始
- 連續 ×2 直到 max（六輪：32→64→128→256→512→1024→2048，累積 4032s）
- 之後一直待在 max

**結論：在我家這種穩定環境，poll interval 99.9% 的時間都等於 `PollIntervalMaxSec`。`min` 只在前 6 輪有意義。**

所以真正要決策的是 max。

### offset 永遠不會觸發 ÷2 嗎？

對。實測最大 |offset| = 974us，遠低於 Branch 4 的 0.15s 門檻。也低於 spike 判定的 2~3ms 門檻。所以正常運作期間，poll 永遠只會 ×2 或不動。

只有當 **RTC 突然抽一下** 把 offset 推到 ≥150ms 或觸發 spike 判定，才會 ÷2。

## 三組推薦設定

下面以「重啟後在第 N 輪達到穩態」、「PLL constant」、「每天 NTP packet 數量」、「spike 視窗時長」四個維度比較。穩態 poll 計算前提是 offset 持續 < 50ms（依實測就是這種情況）。

| 設定 | min | max | 達穩態輪數 | 穩態 poll | PLL constant | packet/day | spike 視窗 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 預設 | 32 | 2048 | 6 | 2048s | 7 | ~42 | 4.5 小時 |
| 保守 | 32 | 1024 | 5 | 1024s | 6 | ~84 | 2.3 小時 |
| 平衡 | 32 | 256 | 3 | 256s | 4 | ~337 | 34 分鐘 |
| 激進 | 32 | 64 | 1 | 64s | 2 | ~1350 | 8.5 分鐘 |

PLL constant 算法（[adjust_clock 那一段](./timesyncd-poll-interval-adaptation)）：

File: `systemd/src/timesync/timesyncd-manager.c` (line 252)

```c
.constant = log2i(m->poll_interval_usec / USEC_PER_SEC) - 4,
```

- 64s → log2i(64) − 4 = 6 − 4 = 2
- 256s → log2i(256) − 4 = 8 − 4 = 4
- 1024s → log2i(1024) − 4 = 10 − 4 = 6
- 2048s → log2i(2048) − 4 = 11 − 4 = 7

constant 越小、PLL 對 sample 越敏感。對 offset 持續 < 1ms 的環境，constant=2 跟 constant=7 在 steady-state 沒有實際差別 — sample noise 本身就比 PLL 反應的雜訊還小。

### 激進設定 (min=32, max=64)

```ini
[Time]
PollIntervalMinSec=32
PollIntervalMaxSec=64
```

特性：

- 每 64 秒一輪 NTP query。對任何 RTC 異常的 **偵測上限 64 秒**。
- PLL constant=2，反應快但對 jitter 敏感。實測 jitter 980us 對 PLL 噪音貢獻：constant=2 vs constant=7 大約 7-10 倍。但因為 offset 本身就 < 1ms，噪音放大 10 倍仍然 < 10ms。
- 每天 ~1350 packet 對 self-hosted NTP server 沒問題；用 `pool.ntp.org` 邊界（建議 ≥64s）。
- spike 視窗 8.5 分鐘，只能 filter 偶發 jitter。

適合場景：

- 自己跑 NTP server，packet 量無壓力。
- ceph mon clock skew 警報是高優先級監控對象，希望 ≤ 1 分鐘內偵測到。

### 平衡設定 (min=32, max=256)

```ini
[Time]
PollIntervalMinSec=32
PollIntervalMaxSec=256
```

特性：

- 每 256 秒（~4.3 分鐘）一輪 query。**比 ceph timecheck 5 分鐘略快 — 兩個 timer 對 RTC 異常的偵測敏感度相當**。
- PLL constant=4，仍然偏敏感但比激進設定多一級 noise filter。
- 每天 ~337 packet，對 pool.ntp.org 完全友善。
- spike 視窗 34 分鐘，足夠 filter 大部分網路 jitter。

適合場景：

- 標準 production 設定。
- 不確定該選哪個就選這組。

### 保守設定 (min=32, max=1024)

```ini
[Time]
PollIntervalMinSec=32
PollIntervalMaxSec=1024
```

特性：

- 每 1024 秒（~17 分鐘）一輪 query。比預設 2048 略快但不會多很多 NTP traffic。
- PLL constant=6，noise immunity 接近預設。
- 每天 ~84 packet。
- spike 視窗 2.3 小時，filter 噪音能力強。

適合場景：

- ceph clock skew 不是首要監控對象、用 NTP 服務商（如 pool.ntp.org）要省 packet。
- 想比預設快一點但不想顯著增加 NTP server 壓力。

## RTC 突然抽一下的恢復路徑

「RTC 抽一下」可以拆成幾種具體情境，每個觸發 timesyncd 不同分支。下面每個 Case 都給兩個指標：

- **系統時鐘修正完** — 從 RTC 異常那一刻起，到 timesyncd 把 offset 拉回 < 50ms（ceph 門檻以下）所需的時間。
- **ceph HEALTH_WARN 持續** — ceph 看到 `MON_CLOCK_SKEW` 警報「亮起 → 熄滅」的時間。**這個比「系統時鐘修正完」多了 0~5 分鐘的 ceph timecheck round 滯後**：ceph 每 5 分鐘做一次 timecheck round，所以警報亮起最多比實際偏差晚 5 分鐘、熄滅最多比實際修好晚 5 分鐘。

兩個指標的關係：

```
時間軸 →
[RTC 抽]----+----------[時鐘修完]----+-----
            ↑                        ↑
            可能 ≤ 5min 後 ceph     可能 ≤ 5min 後 ceph
            timecheck 看到 → 亮警報   timecheck 看到 → 熄警報

系統時鐘修正完 = (時鐘修完) − (RTC 抽)
HEALTH_WARN 持續 ≈ 系統時鐘修正完 + 0~5min（ceph round 滯後）
```

### Case A：offset 累積到 5~50ms（最常見）

例如 RTC drift rate 從 -10ppm 突然變成 -200ppm 30 秒後恢復：累積 offset = 190ppm × 30s = 5.7ms。

下一輪 NTP query：
- 5.7ms < 50ms → 仍走 Branch 3（×2 或維持 max）
- 不會觸發 spike（5.7ms < jitter×3 ~3ms？不對，5.7ms > 3ms，會觸發！）

實際走向取決於 spike 判定：5.7ms > 2.94ms ⇒ 視為 spike ⇒ Branch 4 觸發 ÷2。但 spike 那筆 sample **不會** 進 PLL：

File: `systemd/src/timesync/timesyncd-manager.c` (line 573)

```c
if (!spike) {
        /* Fix up our idea of the time. */
        dts.realtime = (usec_t) (dts.realtime + offset * USEC_PER_SEC);

        r = manager_adjust_clock(m, offset, leap_sec);
        ...
}
```

spike 被擋住、不修 clock。下一輪 poll interval 已經是 max/2，再下一輪如果 offset 還在 5ms 級會再被當 spike ÷2，直到 poll 降到讓 PLL 跟得上。

**這就是「越穩定的環境、jitter 越小、突發 5ms 越容易被誤判為 spike」的副作用。** 但這也是設計意圖：jitter 980us 的環境裡 5ms 偏差確實「不像真實」。

恢復路徑：
- 偵測延遲 = 第一次 query 看到偏差 = 最壞情況 = max poll interval
- 連續幾輪 spike → poll ÷2 → 直到偏差不再被視為 spike
- 修 clock 採 PLL slew，offset 5ms 大約 30s 內收斂

| 設定 | 偵測延遲 | 系統時鐘修正完 | HEALTH_WARN 持續 |
| --- | --- | --- | --- |
| 激進 max=64 | ≤ 64s | ~1.5min | 0（不觸發） |
| 平衡 max=256 | ≤ 256s | ~5min | 0（不觸發） |
| 保守 max=1024 | ≤ 1024s | ~17.5min | 0（不觸發） |
| 預設 max=2048 | ≤ 2048s | ~35min | 0（不觸發） |

**Case A 不觸發 HEALTH_WARN** — ceph 50ms 門檻沒被打到，警報整段都不會亮。所以 max 設多少，對 ceph 監控完全沒差。但對「系統時鐘準確度」還是有差。

### Case B：offset 累積到 50~150ms（ceph 50ms 門檻會觸發）

例如 RTC 暫時跑到 -1000ppm 60 秒：累積 = 988ppm × 60s = 59ms。

- 59ms < 200ms ⇒ 不走 Branch 2 reset
- 59ms > 50ms ⇒ 不走 Branch 3 ×2
- 59ms < 150ms ⇒ 不走 Branch 4 ÷2
- 但 59ms > 2.94ms ⇒ spike

落在 50~150ms 死區、但會被當 spike：行為跟 Case A 一樣（spike 不入 PLL、poll ÷2）。

ceph 端：59ms 偏差 > 50ms 門檻 ⇒ 下一輪 timecheck（≤ 5min）會看到 HEALTH_WARN。

實際恢復機制有個 nuance：spike 第一筆被擋（不入 PLL），第二筆 sample offset 仍 ~59ms 但 |Δoffset| 變小（RTC 已回到正常 drift）⇒ spike 第二條判定 false ⇒ 第二筆被接受 ⇒ 進 PLL slew。

所以實際路徑：

1. 第一筆異常 sample 被當 spike 擋住、poll ÷2
2. 第二筆 sample 仍 ~59ms 但 |Δ| 小 ⇒ 不再 spike ⇒ 進 PLL
3. PLL slew 59ms 大約 120s 收斂到 < 50ms（過 ceph 門檻）

| 設定 | NTP 偵測 + spike 擋一輪 + 接受 | 系統時鐘修正完 | HEALTH_WARN 持續 |
| --- | --- | --- | --- |
| 激進 max=64 | ≤ 2 × 64s = 128s | ~4min | 5~9min |
| 平衡 max=256 | ≤ 2 × 256s = 512s | ~10min | 10~15min |
| 保守 max=1024 | ≤ 2 × 1024s = 2048s | ~36min | 36~41min |
| 預設 max=2048 | ≤ 2 × 2048s = 4096s | ~70min | 70~75min |

**這裡才看出 max poll 對 ceph 監控的明顯差別。** 激進 vs 預設差了一個量級的 HEALTH_WARN 持續時間。

註：HEALTH_WARN 持續的範圍上限 = 系統時鐘修正完 + 5 分鐘（ceph timecheck round 滯後）；下限為「ceph 剛好在 RTC 抽起後立刻 timecheck 一次」的最佳情境，但實務上 ceph 不會剛好對齊，所以取近上限是比較保險的估計。

### Case C：offset > 200ms（嚴重 RTC abnormality）

例如 RTC 卡住 5 秒、累積 60ppm × 5s + 之後正常運作 1 分鐘把 offset 推到 ~250ms。或更糟的情況。

- 250ms > 200ms ⇒ Branch 2 直接 `poll_interval_usec = poll_interval_min_usec`
- 一定不是 spike（spike 判定走 `samples[idx_min].delay`，假設 delay < 5ms ⇒ 250 > 5ms ⇒ spike 第一條返回 true... wait）

實際 spike 判定第一條 `fabs(offset) > m->samples[idx_min].delay`：250ms > best delay（~2ms）⇒ true ⇒ spike。

但 Branch 2 的條件是 `!spike && fabs(offset) > NTP_ACCURACY_SEC`，因為 spike，**不走 Branch 2 reset**。

走 Branch 4: `spike || fabs(offset) > 0.15s` 觸發 ÷2。一樣是 spike 擋住、不修 clock。

下一筆 sample：offset 仍 ~250ms（沒被修）但 |Δ| 小 → spike 第二條 false → 但 spike 第一條（offset > best_delay）**仍然 true**！⇒ 仍然 spike。

問題：spike 第一條會一直把 250ms 擋著，clock 永遠不修？

仔細看：

File: `systemd/src/timesync/timesyncd-manager.c` (line 332)

```c
jitter = m->samples_jitter;
for (idx_min = idx_cur, i = 0; i < ELEMENTSOF(m->samples); i++)
        if (m->samples[i].delay > 0 && m->samples[i].delay < m->samples[idx_min].delay)
                idx_min = i;
```

`idx_min` 是 sample 裡 delay 最小的那筆。delay 跟 offset 是分開存的。即使 offset 全部 250ms，delay 仍然是網路 RTT（~2ms）。

所以 250ms > 2ms 一直成立 ⇒ 一直 spike ⇒ 一直不修。**這似乎是 timesyncd 的盲點**：累積異常 offset 又沒有大幅波動的話，spike detection 會把它擋死。

實務上會解套的條件：
- offset 繼續變大（>>250ms）讓 |Δ| 突破 3×jitter ⇒ Branch 2 reset to min（但需要 `!spike` 條件，spike 就不走）
- 連續 spike 累積到 `NTP_MAX_MISSED_REPLIES = 2` 切 server？不對，這是 missed reply 不是 spike
- 重啟 timesyncd / `systemctl restart`

實際的逃脫機制：**spike 仍然會 ÷2 poll**，所以 poll 會持續減半到 min（最快 6 輪後到 16s）。當 poll 降到 min 時，連續密集 sample，jitter 估計會被異常 offset 拉高、spike 門檻變寬鬆、最終一筆 sample 被接受。

但這個機制是 **「靠加大 jitter 估計把 spike 門檻拉高」**，不是直接修正 — 過程可能需要 8 個 sample 才能讓 jitter 統計穩定。max=2048 ⇒ 6 輪 ÷2 到 min ⇒ 1024+512+256+128+64+32 = 2016s = 34min；之後 8 個 sample × 32s = 256s = 4min 才完整刷新 jitter；總計 ~38min。

| 設定 | ÷2 到 min + jitter 刷新 | 系統時鐘修正完 | HEALTH_WARN 持續 |
| --- | --- | --- | --- |
| 激進 max=64 | 2 輪 ~96s + 128s = ~4min | ~4min | 5~9min |
| 平衡 max=256 | 3 輪 + 刷新 ~9min | ~9min | 9~14min |
| 保守 max=1024 | 5 輪 + 刷新 ~36min | ~36min | 36~41min |
| 預設 max=2048 | 6 輪 + 刷新 ~68min | ~68min | 68~73min |

對嚴重異常恢復來說，激進設定是真的差別很大。

### 三種 Case 整理

**系統時鐘修正完所需時間**（從 RTC 抽起到 offset < 50ms）：

| Case | offset 量級 | 激進 max=64 | 平衡 max=256 | 保守 max=1024 | 預設 max=2048 |
| --- | --- | --- | --- | --- | --- |
| A | 5ms | 1.5min | 5min | 17.5min | 35min |
| B | 50-150ms | 4min | 10min | 36min | 70min |
| C | >200ms | 4min | 9min | 36min | 68min |

**ceph HEALTH_WARN 持續時間**（系統時鐘修正完 + 0~5min ceph timecheck round 滯後）：

| Case | offset 量級 | 激進 max=64 | 平衡 max=256 | 保守 max=1024 | 預設 max=2048 |
| --- | --- | --- | --- | --- | --- |
| A | 5ms | 0（不觸發） | 0（不觸發） | 0（不觸發） | 0（不觸發） |
| B | 50-150ms | 5~9min | 10~15min | 36~41min | 70~75min |
| C | >200ms | 5~9min | 9~14min | 36~41min | 68~73min |

要點：

- **Case A 對 ceph 完全沒影響** — 5ms 偏差不過 50ms 門檻，ceph 永遠不會亮警報。所以這個情境只在意「系統時鐘準確度」，跟 ceph 監控無關。
- **Case B/C 的差別主要在 ÷2 escape 路徑**：B 走第二筆 sample 被接受（spike 第二條判定 false）、C 必須降到 min + 刷新 jitter 統計才能逃離。但對選 max poll 的決策來說兩種 Case 結論一致。
- **HEALTH_WARN 持續 = 系統時鐘修正完 + 0~5min** — 這 5 分鐘來自 ceph 自己的 timecheck round，跟 timesyncd 設定無關。所以「想讓 ceph 警報持續時間更短」的最後 5 分鐘永遠收不回來。

## 我的推薦

**選平衡設定 (min=32, max=256)。**

理由：

1. **配合 ceph timecheck 5 分鐘節奏** — 偵測敏感度 ≈ ceph 偵測敏感度。timesyncd 256s + ceph 300s = 一個 ceph round 內必偵測到。
2. **Case A 的 5ms 偏差在 5 分鐘內恢復** — 還不到 ceph 50ms 門檻，不會 false alert。
3. **Case B 的 50-150ms 偏差在 10 分鐘內恢復** — 跟 ceph HEALTH_WARN 持續時間相近，alert 不會卡很久。
4. **每天 337 packet** — 對 pool.ntp.org 友善，自己跑 NTP 也無壓力。
5. **重啟後 3 輪達穩態 = 4.3 分鐘** — 比預設 6 輪 67 分鐘快很多。

```ini
[Time]
NTP=pool.ntp.org
PollIntervalMinSec=32
PollIntervalMaxSec=256
RootDistanceMaxSec=1
```

`RootDistanceMaxSec=1` 比預設 5 嚴格，但根據實測 root distance 上限 3.944ms，1 秒已經非常寬鬆 — 主要是把那種「明顯不健康」的 server 直接擋掉。

## 怎麼套用 + 怎麼確認生效

```bash
sudo install -m 644 /dev/stdin /etc/systemd/timesyncd.conf <<'EOF'
[Time]
PollIntervalMinSec=32
PollIntervalMaxSec=256
EOF

# 注意：reload 不會套用新值（沒有 reload handler）
sudo systemctl restart systemd-timesyncd

# 確認生效
timedatectl timesync-status
```

預期輸出：

```
       Server: 162.159.200.123 (time.cloudflare.com)
Poll interval: 32s (min: 32s; max 4min 16s)
```

`max 4min 16s` 就是 256s。

如果 reload 沒生效會顯示舊值（[restart 本身可能跳鐘](./timesyncd-restart-impact)，需要評估時機）。

## 我什麼時候會考慮改成激進

實測一週的數據完全沒看到 Case B / C 級別的異常，所以平衡設定夠用。但下面情況我會切回激進 (max=64)：

1. **發現 ceph 偶爾出現秒等級 HEALTH_WARN 且自動恢復** — 表示有 Case B 級的偏移。要縮短偵測延遲。
2. **VM live migration 後** — KVM 的 KVM Clock 偶爾不穩。短期把 max 拉低觀察。
3. **debug 模式** — 想看每 64s 一輪的 PLL 行為。改完記得拉回。

## 邊界與容易踩坑

### freq 永遠負的 — 是壞掉嗎？

不是。`-12.475ppm ~ -0.936ppm` 表示 oscillator 比參考 server 慢 12 ppm 左右，這是常溫下 quartz 製程偏差的正常範圍（典型工業級 RTC ±20ppm）。

**警訊**是：
- freq 接近 ±500ppm — 已經是 Linux PLL 校頻能力的邊界
- freq 不穩、來回擺動超過 50ppm — 可能 thermal 異常

實測 11.5ppm 變動範圍很穩定，不用擔心。

### offset peak-to-peak 1.87ms 是好還是壞？

中等偏好。對照值：

| 環境 | typical offset peak-to-peak |
| --- | --- |
| 同機房 LAN NTP server | 50us 量級 |
| WAN NTP (RTT 5~10ms) | 1ms 量級 |
| 跨大陸 NTP (RTT 50+ms) | 5-10ms 量級 |

實測 1.87ms ⇒ 用的是 WAN NTP，RTT 大約 5-10ms。如果想壓到 50us 量級，要架本地 NTP server。

### 為什麼沒有真的把 max 拉到 32（min=max）？

[Low max poll simulation 那一頁](./timesyncd-low-max-poll-simulation) 拆過 — `min=max=32` 會讓 `manager_adjust_poll` 整支變 no-op，spike 視窗只剩 4.3 分鐘，PLL constant=1 把 jitter 直接餵 kernel。對穩定環境是把優勢做掉。

`max=64` 仍保留 32↔64 兩個檔位，並且配合 spike detection 自動 ÷2 機制有逃脫路徑。

### `PollIntervalMinSec=16` 看起來能用

```ini
PollIntervalMinSec=16
```

`timesyncd-conf.c:121` 的 check 是嚴格小於 16：

File: `systemd/src/timesync/timesyncd-conf.c` (line 121)

```c
if (m->poll_interval_min_usec < 16 * USEC_PER_SEC) {
        log_warning("Invalid PollIntervalMinSec=. Using default value.");
        m->poll_interval_min_usec = NTP_POLL_INTERVAL_MIN_USEC;
}
```

所以 `=16` 通過、`=15` 被打回 32。但這已經低於 NTPv4 RFC5905 建議的 `min poll 16s（2^4）`下限 — 不要這樣設。

### restart 後恢復穩態的時間

每個設定的「重啟到穩態」輪數 × 各輪累計時間：

| 設定 | 路徑 | 累計時間 |
| --- | --- | --- |
| 預設 max=2048 | 32→64→128→256→512→1024→2048 | 6 輪 67 分鐘 |
| 保守 max=1024 | 32→64→128→256→512→1024 | 5 輪 33 分鐘 |
| 平衡 max=256 | 32→64→128→256 | 3 輪 7.5 分鐘 |
| 激進 max=64 | 32→64 | 1 輪 1.5 分鐘 |

平衡設定 7.5 分鐘是另一個它的好處 — 重啟後快速進入穩態。
