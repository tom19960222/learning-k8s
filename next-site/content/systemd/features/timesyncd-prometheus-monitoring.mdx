## 場景

3-node bare-metal ceph cluster，每台跑一個 mon。你沒有 chrony、沒有 ntpd，只有 `systemd-timesyncd`；監控也只想走 Prometheus。你真正要回答的是兩個不同問題：

1. **每台 node 上的 timesyncd 有沒有持續 discipline clock？**
2. **三個 mon 彼此看到的 wall clock 差距有沒有超過 ceph 的 50ms 門檻？**

50ms 來自 ceph mon 的預設設定。也就是說，Prometheus 規則的最高層不是「我猜 node 偏了多少」，而是 ceph 自己的 health check。

File: `ceph/src/common/options/mon.yaml.in` (line 609)

```yaml
- name: mon_clock_drift_allowed
  type: float
  level: advanced
  desc: allowed clock drift (in seconds) between mons before issuing a health warning
  default: 0.05
```

先把結論說清楚：**node_exporter timex 是早期 proxy，systemd_exporter 是 daemon 存活檢查，ceph health 才是跨 node 權威判決。** 三者要同時存在，不要互相替代。

## 三層訊號怎麼分工

```
Prometheus scrape
│
├─ node_exporter timex collector
│  ├─ node_timex_sync_status
│  ├─ node_timex_offset_seconds
│  └─ node_timex_maxerror_seconds
│
│     意義：這台 node 的 kernel time discipline 狀態。
│     用法：早期發現單台 node 正在漂、或 kernel 已不認為 clock synchronized。
│     限制：不是跨 node 即時量測。
│
├─ prometheus-community/systemd_exporter
│  └─ systemd_unit_state{name="systemd-timesyncd.service", state="active"}
│
│     意義：disciplining daemon 還活著嗎。
│     用法：補 timex 的空窗；daemon 死掉時 kernel 狀態不一定立刻翻成 unsync。
│     限制：service active 不代表 offset 正常。
│
└─ ceph-mgr prometheus module
   └─ ceph_health_detail{name="MON_CLOCK_SKEW"}

      意義：ceph 自己做跨 mon timecheck 後的 0/1 health 判決。
      用法：真正破 50ms 時 page。
      限制：只告訴你有沒有破表，不給實際毫秒差。
```

這個分工能避免兩個常見錯誤：只看 `systemd-timesyncd.service active`，會漏掉 clock 其實已經偏；只看 `node_timex_offset_seconds`，又會把 per-node residual offset 誤當成 ceph 的跨 mon time difference。

## Prometheus alert rules（集中版）

下面這段是本頁的核心。若你用原生 Prometheus，直接放進 rules file；若你用 Prometheus Operator，把 `groups:` 底下的內容放到 `PrometheusRule.spec.groups`。

這組規則的假設是：3-node ceph mon、所有 node 指向同一組可信時間參考、ceph 採用預設 `mon_clock_drift_allowed=0.05`。`0.025` 秒不是 ceph 的硬門檻，而是「單台 node residual offset 已接近 50ms 的一半」的早期 proxy；正式上線前要用你自己的 poll interval、scrape interval、NTP source 品質調整。

```yaml
groups:
- name: timesyncd-clock.rules
  rules:
  - alert: TimesyncdServiceNotActive
    expr: sum by (instance) (systemd_unit_state{name="systemd-timesyncd.service", state="active"}) != 1
    for: 2m
    labels:
      severity: warning
      component: timesyncd
    annotations:
      summary: "systemd-timesyncd is not active on {{ $labels.instance }}"
      description: "systemd_exporter reports systemd-timesyncd.service is not active. Check systemctl status systemd-timesyncd and journalctl -u systemd-timesyncd."

  - alert: TimesyncdKernelUnsynchronized
    expr: min_over_time(node_timex_sync_status[5m]) == 0
    for: 10m
    labels:
      severity: warning
      component: timesyncd
    annotations:
      summary: "Kernel time sync status is unsynchronized on {{ $labels.instance }}"
      description: "node_exporter timex sees sync_status=0 for 10m. Check NTP reachability, RTC local-time mode, and timesyncd logs."

  - alert: TimesyncdResidualOffsetHigh
    expr: abs(node_timex_offset_seconds) > 0.025
    for: 5m
    labels:
      severity: warning
      component: timesyncd
    annotations:
      summary: "Residual clock offset is high on {{ $labels.instance }}"
      description: "node_timex_offset_seconds is above 25ms for 5m. This is an early proxy for ceph's 50ms MON_CLOCK_SKEW threshold, not a cross-node measurement."

  - alert: TimesyncdMaxErrorAtTimedatedLimit
    expr: node_timex_maxerror_seconds >= 16
    for: 2m
    labels:
      severity: warning
      component: timesyncd
    annotations:
      summary: "Kernel maxerror reached timedated unsync boundary on {{ $labels.instance }}"
      description: "systemd-timedated treats maxerror >= 16s as not synchronized. Check whether timesyncd has stopped receiving valid NTP samples."

  - alert: CephMonClockSkew
    expr: ceph_health_detail{name="MON_CLOCK_SKEW"} == 1
    labels:
      severity: page
      component: ceph
    annotations:
      summary: "Ceph reports MON_CLOCK_SKEW"
      description: "ceph-mgr prometheus reports MON_CLOCK_SKEW active. This is the authoritative cross-mon 50ms clock skew signal; run ceph health detail and ceph time-sync-status."
```

五條規則的讀法：

| Alert | 它回答什麼 | 看到時先做什麼 |
|---|---|---|
| `TimesyncdServiceNotActive` | daemon 還在跑嗎 | `systemctl status systemd-timesyncd`、`journalctl -u systemd-timesyncd` |
| `TimesyncdKernelUnsynchronized` | kernel 還認為 clock synchronized 嗎 | 看 NTP server 可達性、RTC local time、timesyncd log |
| `TimesyncdResidualOffsetHigh` | 單台 node 的 residual offset 是否偏大 | 跟其他 node 比趨勢，不要直接當跨 node 差 |
| `TimesyncdMaxErrorAtTimedatedLimit` | maxerror 是否大到 timedated 也會說不同步 | 查上一次有效 sample 距今多久 |
| `CephMonClockSkew` | ceph 是否已經判定跨 mon 破 50ms | 這條是 page；去 ceph health / time-sync-status 看量級 |

另外要單獨監控 exporter 本身：`up{job="node_exporter"} == 0`、`up{job="systemd_exporter"} == 0`、ceph-mgr prometheus scrape 是否正常。那些是 scrape health，不要混進 clock skew 語意裡。

## 為什麼 timex 對 timesyncd 有效

`systemd-timesyncd` 本身沒有 Prometheus endpoint。它真正做的是把 NTP sample 轉成 kernel time discipline 操作；node_exporter 的 timex collector 則透過 `adjtimex(2)` 讀 kernel 狀態。

timesyncd 在 `manager_adjust_clock()` 先用 `NTP_MAX_ADJUST=0.4` 秒分流：小於 0.4 秒用 slew，大於等於 0.4 秒用 jump。Prometheus timex 監控最重要的是 slew 這條，因為它把 status 設成 `STA_PLL`，並且沒有設 `STA_UNSYNC`。

File: `systemd/src/timesync/timesyncd-manager.c` (line 247)

```c
if (fabs(offset) < NTP_MAX_ADJUST) {
        tmx = (struct timex) {
                .modes = ADJ_STATUS | ADJ_NANO | ADJ_OFFSET | ADJ_TIMECONST | ADJ_MAXERROR | ADJ_ESTERROR,
                .status = STA_PLL,
                .offset = offset * NSEC_PER_SEC,
                .constant = log2i(m->poll_interval_usec / USEC_PER_SEC) - 4,
        };
```

`STA_UNSYNC` 只在 RTC 跑 local time 時被 timesyncd 主動設回去。正常的 server 環境會讓 RTC 跑 UTC，所以每次成功校正都會把 kernel 留在 synchronized 狀態。

File: `systemd/src/timesync/timesyncd-manager.c` (line 275)

```c
/* An unset STA_UNSYNC will enable the kernel's 11-minute mode, which syncs the system time
 * periodically to the RTC.
 *
 * In case the RTC runs in local time, never touch the RTC, we have no way to properly handle
 * daylight saving changes and mobile devices moving between time zones. */
if (m->rtc_local_time)
        tmx.status |= STA_UNSYNC;
```

node_exporter 的 timex collector 是預設開啟的；它輸出的 `node_timex_sync_status` 來自 `adjtimex()` 回傳值是否為 `TIME_ERROR`，`node_timex_offset_seconds` 來自 kernel timex 的 `Offset`，`node_timex_maxerror_seconds` 來自 `Maxerror`。node_exporter 官方 `docs/TIME.md` 也直接把 `sync_status` + `offset` 的組合列為監控 systemd-timesyncd 是否做事的方法。

這裡有個很容易讀錯的細節：`node_timex_offset_seconds` 不是 node_exporter 每次 scrape 主動向 NTP server 重新量測；它是 kernel 目前還在修的 residual offset。它能告訴你「這台 node 看起來正在偏」，但不能直接替你量「node A 跟 node B 差幾毫秒」。

> 這條 metric 怎麼從 `collector/timex.go` 一路被 `adjtimex(2)` 算出來（含 `STA_NANO` 決定 µs/ns 的 divisor、為什麼是 residual 而非即時量測），有一頁從三層原始碼逐行追完：[node_exporter — node_timex_offset_seconds 是怎麼算出來的](/systemd/features/node-exporter-timex-offset-source)。

## maxerror 跟 timedatectl 的關係

`node_timex_maxerror_seconds` 不是 offset，而是 kernel 對目前 clock error 上界的估計。timesyncd 成功調整時會帶 `ADJ_MAXERROR | ADJ_ESTERROR`，struct 裡沒有填值，因此會把 maxerror / esterror 歸零；之後 kernel 會讓 maxerror 隨時間變大。

這個值值得做 alert，是因為 `systemd-timedated` 的 synchronized 判斷直接看它是否小於 16 秒，而且明確忽略 `STA_UNSYNC`。

File: `systemd/src/timedate/timedated.c` (line 570)

```c
static bool ntp_synced(void) {
        struct timex txc = {};

        if (adjtimex(&txc) < 0)
                return false;

        /* Consider the system clock synchronized if the reported maximum error is smaller than the maximum
         * value (16 seconds). Ignore the STA_UNSYNC flag as it may have been set to prevent the kernel from
         * touching the RTC. */
        return txc.maxerror < 16000000;
}
```

所以 `TimesyncdMaxErrorAtTimedatedLimit` 用 `node_timex_maxerror_seconds >= 16`，不是因為 16 秒對 ceph 有特別意義，而是因為這會跟 `timedatectl` 的「System clock synchronized」語意對齊。

## ceph health 為什麼是權威判決

前面的 timex 指標都在講「單台 node 對時間參考的狀態」。ceph mon clock skew 則是 ceph 自己跨 mon 做 timecheck 後的 health check；超過 `mon_clock_drift_allowed` 預設 0.05 秒時，會進入 `MON_CLOCK_SKEW`。

ceph-mgr prometheus module 先宣告一個 `health_detail` gauge，label 是 health check name 與 severity。

File: `ceph/src/pybind/mgr/prometheus/module.py` (line 742)

```python
metrics['health_detail'] = Metric(
    'gauge',
    'health_detail',
    'healthcheck status by type (0=inactive, 1=active)',
    HEALTHCHECK_DETAIL
)
```

接著它把每個 health check 的 active 狀態設成 0 或 1。

File: `ceph/src/pybind/mgr/prometheus/module.py` (line 932)

```python
for name, info in self.health_history.healthcheck.items():
    v = 1 if info.active else 0
    self.metrics['health_detail'].set(
        v, (
            name,
            str(info.severity))
    )
```

因此 `ceph_health_detail{name="MON_CLOCK_SKEW"} == 1` 是這頁唯一的跨 mon 權威告警。不過它也是 0/1：知道「已超標」，不知道實際差了 51ms、500ms 還是 5 秒。實際量級要看 `ceph health detail` 或 `ceph time-sync-status` 的文字輸出。

## per-node offset 只能當早期 proxy

`TimesyncdResidualOffsetHigh` 用 25ms，是把 ceph 50ms 門檻拆成「單台 node 已經偏到一半」的早期警告。它成立有前提：

1. 所有 node 指向同一組可信時間參考。
2. `node_timex_offset_seconds` 足夠新鮮，代表最近有效 sample 造成的 residual offset。
3. 你把它當趨勢訊號，不把它當精密跨 node 量測。

如果 node A 對時間參考偏 `X`，node B 偏 `Y`，那 A、B 之間的差可以用 `|X - Y|` 理解；在只看絕對值時，`|X| + |Y|` 是保守上界。所以「每台 node 都穩定低於 25ms」通常代表還沒逼近 ceph 的 50ms 門檻；但只要 ceph 自己亮 `MON_CLOCK_SKEW`，就以 ceph 為準。

反過來說，如果你有不同 NTP source、某台 node 長時間連不到 server、VM snapshot restore、host resume、live migration 這些外部跳鐘情境，per-node offset 會變成「先看到怪事」的訊號，不是完整證明。這也是為什麼 alert rules 同時保留 timex、daemon active、ceph health 三層。

## Exporter 選擇

截至 2026-06-02 重新查證：

| Exporter | 角色 | 狀態 | 建議 |
|---|---|---|---|
| `prometheus/node_exporter` | timex collector 讀 kernel `adjtimex(2)` 狀態 | latest release v1.11.1，2026-04-07；timex collector 預設開啟 | **主力** |
| `prometheus-community/systemd_exporter` | `systemd_unit_state` 讀 unit active / inactive / failed | latest release v0.7.0，2025-03-14 | 建議補上 daemon 存活 |
| `dmitry-ee/time_exporter` | timesync 專屬 exporter | 最後 push 2021-11-30，6 stars | 不建議 |

node_exporter 也有 systemd collector，但它是 disabled by default，而且 metric 名稱會是 `node_systemd_unit_state`。這頁用 `prometheus-community/systemd_exporter` 是因為它專門做 systemd unit 監控，README 也明列 `systemd_unit_state` 每個 unit 會有 active / inactive / failed 等狀態。

如果你想看 timesyncd 自己量到的 jitter、spike 與 NTP timestamps，D-Bus 上確實有 `NTPMessage` property。

File: `systemd/src/timesync/timesyncd-bus.c` (line 221)

```c
SD_BUS_PROPERTY("NTPMessage", "(uuuuittayttttbtt)", property_get_ntp_message, 0, SD_BUS_VTABLE_PROPERTY_EMITS_CHANGE),
```

但是目前沒有值得依賴的 active exporter 專門 scrape 這個 property。所以 Prometheus 路線仍然回到 node_exporter timex + systemd_exporter + ceph-mgr prometheus。

## 邊界與除錯

如果 `TimesyncdServiceNotActive` 先響，先處理 service：`systemctl status systemd-timesyncd`、`journalctl -u systemd-timesyncd`，再看是不是 NTP server config、網路、或 service restart 造成。

如果 `TimesyncdKernelUnsynchronized` 或 `TimesyncdMaxErrorAtTimedatedLimit` 先響，但 service 還 active，重點不是「systemd 還活著」，而是它最近有沒有拿到有效 NTP sample。這時看 `timedatectl timesync-status`、`timedatectl status`、RTC 是否 local time、NTP server 是否 reachable。

如果只有 `TimesyncdResidualOffsetHigh` 響，先把三台 node 的 `node_timex_offset_seconds` 放在同一張圖上。單台偏大通常是早期問題；多台同時偏向同一方向，可能是共同時間來源或網路路徑的問題。這條不是 page，除非它持續擴大或跟 ceph health 一起出現。

如果 `CephMonClockSkew` 響，這就是 ceph 自己判定跨 mon 超過 50ms。此時不要再用 node_exporter 指標跟它爭論；用 `ceph health detail` / `ceph time-sync-status` 找是哪個 mon 偏，再回到那台 node 檢查 timesyncd。
