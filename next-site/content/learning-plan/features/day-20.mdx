### 3. 建最小 CephCluster（hostPath 模式）

正式生產用 raw disk；學習用直接給 hostPath 即可。kind 容器裡有 `/var/lib/rook` 可以寫。

```yaml
# 🟡 已對照官方文件 / 原始碼
apiVersion: ceph.rook.io/v1
kind: CephCluster
metadata:
  name: my-cluster
  namespace: rook-ceph
spec:
  cephVersion:
    image: quay.io/ceph/ceph:v19.2.3
  dataDirHostPath: /var/lib/rook
  mon:
    count: 1   # 學習用，正式環境至少 3
    allowMultiplePerNode: true
  mgr:
    count: 1
  dashboard:
    enabled: false
  storage:
    useAllNodes: true
    useAllDevices: false
    config:
      databaseSizeMB: "256"
      journalSizeMB: "256"
    directories:
    - path: /var/lib/rook/storage
  resources:
    mon: {requests: {cpu: "200m", memory: "512Mi"}}
    mgr: {requests: {cpu: "200m", memory: "256Mi"}}
    osd: {requests: {cpu: "200m", memory: "512Mi"}}
```

```bash
# 🟡 已對照官方文件 / 原始碼
kubectl apply -f cephcluster.yaml
```

### 5. 看 operator reconcile log

```bash
# 🟢 已驗證 (kind / 顯然正確)
```


打開 [pkg/operator/ceph/cluster/cluster_controller.go](https://github.com/rook/rook/blob/release-1.16/pkg/operator/ceph/cluster/cluster_controller.go)，找 `Reconcile()` 函式。標記主要分支：

1. 拿到 CephCluster CR；如果 spec 變了要 detect。
2. 確認 dependencies（CRD 已安裝、namespace 存在、auth secret）
3. 呼叫 mon controller → 部署 MON Pod、等到 quorum
4. 呼叫 mgr controller → 部署 MGR
5. 呼叫 osd controller → 對每個 storage 配置建 OSD prepare Job → OSD Deployment
6. 更新 `.status` 反映目前 phase

## 觀念深化

Rook 沒有重寫 ceph —— 它跑的是 **upstream ceph binary 包成 container**。Operator 只負責 lifecycle 編排：

- 把 CephCluster CR 翻譯成 Pod / Deployment / DaemonSet / ConfigMap / Secret
- 處理 keyring 生成、auth bootstrap、版本升級、failure 回收
- 提供 CRD 給 user 操作 (CephCluster、CephBlockPool、CephObjectStore、CephFilesystem)

這是典型的 **operator pattern**：domain-specific knowledge 編成 reconcile 邏輯。Rook 一個 operator 內含多個 controller（cluster、mon、mgr、osd、pool、object、file 等），各自 watch 自己的 CRD 與 owned resources。

對比 Day 18 的 ceph-csi：ceph-csi 是 **CSI driver**（提供 volume mount），Rook 是 **lifecycle operator**（管 daemon）。兩者完全互補，可以同時用 —— Rook 部署 ceph，ceph-csi 提供 PVC。
