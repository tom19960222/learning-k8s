## 動手做

### 1. 在 ceph 建立 k8s 專用 client key

ceph-csi 需要一個 ceph user 來操作 RBD pool。原則上權限最小化：只給 RBD 用、只能操作 `k8s-rbd` pool。

```bash
# 🔴 需在你環境跑後對照
ceph auth get-or-create client.k8s mon 'profile rbd' osd 'profile rbd pool=k8s-rbd'
```

```
[client.k8s]
        key = AQ...base64...==
```

把 key 與 username 記下來。

### 2. 取得 cluster fsid 與 MON 列表

```bash
# 🔴 需在你環境跑後對照
ceph fsid
```

```

```

```bash
# 🔴 需在你環境跑後對照
ceph mon dump 2>/dev/null | grep 'mon\.'
```

### 4. 寫 ceph-csi config

ceph-csi 用一個 ConfigMap 描述一或多個 ceph cluster。建立一個 namespace + 把 config 與 secret 注入：

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl create namespace ceph-csi-rbd
```

```yaml
# 🔴 需在你環境跑後對照（fsid / MON IPs / key 換成你的）
# csi-config.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: ceph-csi-config
  namespace: ceph-csi-rbd
data:
  config.json: |-
    [
      {
        "clusterID": "<your-fsid>",
        "monitors": ["10.0.0.1:6789","10.0.0.2:6789","10.0.0.3:6789"]
      }
    ]
---
apiVersion: v1
kind: Secret
metadata:
  name: csi-rbd-secret
  namespace: ceph-csi-rbd
stringData:
  userID: k8s
  userKey: AQ...base64...==
```

```bash
# 🔴 需在你環境跑後對照
kubectl apply -f csi-config.yaml
```

### 5. 安裝 ceph-csi-rbd

```bash
# 🟡 已對照官方文件 / 原始碼
helm install ceph-csi-rbd ceph-csi/ceph-csi-rbd \
  --namespace ceph-csi-rbd --version 3.13.0 \
  --set csiConfig[0].clusterID=<your-fsid> \
  --set csiConfig[0].monitors='{10.0.0.1:6789,10.0.0.2:6789,10.0.0.3:6789}'
```

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl -n ceph-csi-rbd get pods
```

### 6. 建立 StorageClass

```yaml
# 🔴 需在你環境跑後對照（fsid 換成你的）
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ceph-rbd
provisioner: rbd.csi.ceph.com
parameters:
  clusterID: <your-fsid>
  pool: k8s-rbd
  imageFeatures: layering
  csi.storage.k8s.io/provisioner-secret-name: csi-rbd-secret
  csi.storage.k8s.io/provisioner-secret-namespace: ceph-csi-rbd
  csi.storage.k8s.io/controller-expand-secret-name: csi-rbd-secret
  csi.storage.k8s.io/controller-expand-secret-namespace: ceph-csi-rbd
  csi.storage.k8s.io/node-stage-secret-name: csi-rbd-secret
  csi.storage.k8s.io/node-stage-secret-namespace: ceph-csi-rbd
reclaimPolicy: Delete
allowVolumeExpansion: true
```

### 7. 建第一個 PVC

```yaml
# 🟢 已驗證 (kind / 顯然正確)
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: ceph-test-pvc
spec:
  accessModes: [ReadWriteOnce]
  storageClassName: ceph-rbd
  resources:
    requests:
      storage: 1Gi
```

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl get pvc ceph-test-pvc -w
```

幾秒後應該變 `Bound`。

### 8. 從 ceph 端看自動建立的 image

```bash
# 🔴 需在你環境跑後對照
rbd ls k8s-rbd
```

```

```

`csi-vol-<uuid>` 就是剛剛 PVC 觸發 ceph-csi provisioner 建出來的。可以再 `rbd info k8s-rbd/csi-vol-<uuid>` 看細節。

## 觀念深化

ceph-csi 由兩部分組成：

- **controller plugin（Deployment）** — 跑在某個 node 上，負責「建立 / 刪除 volume」「擴容」等需要呼叫 ceph API 的操作。對應 PVC 的生命週期。
- **node plugin（DaemonSet）** — 每個 node 一個，負責「stage / publish 到 Pod」「unstage / unpublish」。當 Pod 排到該 node 才需要這個 plugin。

兩者都實作 CSI gRPC interface（[CSI spec](https://github.com/container-storage-interface/spec)）。kubelet 不需要懂 ceph，只跟 CSI socket 對話。所以同一個 k8s cluster 可以同時跑很多 CSI driver（NFS、AWS EBS、ceph-rbd、ceph-cephfs 各自獨立）。

更深入：

- 整個 controller-plugin 進程結構 [internal/csi-common/](https://github.com/ceph/ceph-csi/tree/v3.13.0/internal/csi-common)
- RBD provisioner 邏輯 [internal/rbd/controllerserver.go](https://github.com/ceph/ceph-csi/blob/v3.13.0/internal/rbd/controllerserver.go) 裡的 `CreateVolume()`
