## 動手做

```bash

kubectl apply -f https://github.com/kubevirt/containerized-data-importer/releases/download/${CDI_VERSION}/cdi-operator.yaml

kubectl -n cdi wait --for=condition=Available cdi cdi --timeout=5m
```

```
cdi.cdi.kubevirt.io/cdi condition met
```

### 2. 建立 DataVolume 從 URL 拉 image

```yaml
# 🟡 已對照官方文件 / 原始碼
apiVersion: cdi.kubevirt.io/v1beta1
kind: DataVolume
metadata:
  name: fedora-rootdisk
spec:
  source:
    http:
      url: "https://download.fedoraproject.org/pub/fedora/linux/releases/39/Cloud/x86_64/images/Fedora-Cloud-Base-39-1.5.x86_64.qcow2"
  pvc:
    accessModes: [ReadWriteOnce]
    resources:
      requests:
        storage: 10Gi
    storageClassName: ceph-rbd
```

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl apply -f datavolume.yaml
kubectl get dv -w
```

```
NAME              PHASE              PROGRESS   ...
fedora-rootdisk   ImportInProgress   25.00%
fedora-rootdisk   ImportInProgress   80.00%
fedora-rootdisk   Succeeded          100.0%
```

CDI 起一個 importer Pod 下載 qcow2，用 `qemu-img convert` 寫進 PVC。完成後 PVC 變 Bound。

### 4. 改 VM 用 PVC 當 rootdisk

```yaml
# 🟡 已對照官方文件 / 原始碼
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: fedora-vm
spec:
  runStrategy: Always
  template:
    metadata:
      labels: {kubevirt.io/vm: fedora-vm}
    spec:
      domain:
        cpu: {cores: 1}
        memory: {guest: 1Gi}
        devices:
          disks:
          - name: rootdisk
            disk: {bus: virtio}
          - name: cloudinit
            disk: {bus: virtio}
          interfaces:
          - name: default
            masquerade: {}
      networks:
      - name: default
        pod: {}
      volumes:
      - name: rootdisk
        persistentVolumeClaim:
          claimName: fedora-rootdisk
      - name: cloudinit
        cloudInitNoCloud:
          userData: |
            #cloud-config
            password: fedora
            chpasswd: { expire: False }
            ssh_pwauth: True
```

```bash
# 🟡 已對照官方文件 / 原始碼
virtctl stop fedora-vm 2>/dev/null
kubectl apply -f vm.yaml
kubectl get vmi -w
```

幾秒後 `Running`。

## 觀念深化

VM disk 在 KubeVirt 的層次：

```
VirtualMachine.spec.template.spec.volumes[].persistentVolumeClaim
  ↓
PVC (Day 18 的 ceph-rbd StorageClass)
  ↓ ceph-csi node plugin
node 上的 /dev/rbdX
  ↓ virt-launcher Pod 內掛載
qemu 看到的 virtio block device
  ↓
VM 內 /dev/vda
```

DataVolume 是 KubeVirt 給 PVC 加 lifecycle 的 wrapper：自動建 PVC + 跑 importer 把 image 灌進去 + 標記 status。如果你只 `kubectl get pvc` 看到 PVC、不知道哪來的，多半就是 DataVolume 建的。

`storage` size 跟 image size 不直接相等：DataVolume 會把 PVC 的 size attribute 對齊到 storage class 的最小單位（ceph-rbd 通常是 4 MiB），實際 image 比 PVC 小很多時 PVC 還是該 size，多餘的空間是 sparse。

更深入：

- [DataVolumes](https://kubevirt.io/user-guide/storage/containerized_data_importer/) — CDI 文件
- [Disks and Volumes](https://kubevirt.io/user-guide/storage/disks_and_volumes/) — VM disk 配置
- [pkg/virt-launcher/virtwrap/converter/](https://github.com/kubevirt/kubevirt/tree/v1.8.2/pkg/virt-launcher/virtwrap/converter) — VMI spec 轉 libvirt domain XML 的程式碼

### 為什麼這樣設計

CDI 把「下載／轉換 image」的工作做成 importer Pod，而不是塞進 launcher Pod 的 init container。直觀答案是 importer Pod 必須在「PVC 還沒被任何 VM 用」時就跑完，否則會跟 VM 啟動的 mount 行為打架。把這層做成獨立的 controller-driven Pod，CDI 可以自由排程 importer 到任意有 ceph-csi node 的位置、用獨立 ServiceAccount 拿 image registry credential、importer 失敗時 controller 重新拉一個新 Pod 都不影響 PVC 本身。

設計選擇背後的取捨是「lifecycle 解耦」。如果 image import 跟 VM 啟動綁在 launcher 的 init container 裡，會出三個問題：(1) VM 每次重啟都要驗證 image 完整性，浪費時間；(2) init container 跑失敗會讓整個 launcher Pod CrashLoopBackOff，VMI 的 condition 看起來會像「VM 起不來」而不是「image 還沒準備好」；(3) 多個 VM 共用同一份 base image 時沒辦法重用。DataVolume 走 controller pattern 後，import 是一次性事件、PVC 是長期資源，VM controller 看到 DataVolume `Succeeded` 才開始排 launcher Pod。這也是為什麼 KubeVirt 鼓勵「先建 base image PVC、再用 PVC clone 出每個 VM 的 root disk」。

常見誤解是「DataVolume 等於 PVC + 一段下載指令」。實際上 DataVolume 帶 lifecycle 語意：source 換成 `pvc:` 時 CDI 會走 CSI clone（不起 importer Pod、由 ceph 端做 RBD snapshot）；source 是 `blank:` 時連 importer Pod 都不需要、controller 直接建 sparse PVC。這層抽象讓上層 CR（VirtualMachine）不必區分「我這個 root disk 是怎麼來的」，只要 reference DataVolume 就好。

## 🧹 Cleanup / 回退

🔴 **警告**：刪除 VM、PVC、CDI 後，今天匯入的 fedora cloud image 會永久消失，下次重新匯入要再下載一次幾百 MiB。Day 26 / 28 / 30 都會用到 `fedora-vm` / `fedora-rootdisk`，所以**通常不需要 cleanup**。下面是「整個 reset 回 Day 23 結束狀態」用：

順序：VM → DataVolume / PVC → CDI（如果連 CDI 也想拆）。

```bash
# 1. stop VM 後再 delete（避免 launcher Pod 卡住）
# 🟡 已對照官方文件 / 原始碼
virtctl stop fedora-vm 2>/dev/null
kubectl delete vm fedora-vm --ignore-not-found
```

等到 `kubectl get vmi` 沒有 `fedora-vm`。

```bash
# 2. 刪 DataVolume（連同 PVC）
# 🟢 已驗證 (kind / 顯然正確)
kubectl delete dv fedora-rootdisk --ignore-not-found
```

刪 DataVolume 會自動連帶刪 PVC；ceph-csi 收到 PVC delete 後會把對應 RBD image 砍掉。

```bash
# 3. 確認 ceph 端 RBD image 也消失了
# 🔴 需在你環境跑後對照
rbd ls k8s-rbd
```

預期不再列 `csi-vol-<uuid>`（fedora-rootdisk 對應的那一行）。

```bash
# 4.（可選）卸 CDI — Day 28 還會用，建議留著
# 🟡 已對照官方文件 / 原始碼
kubectl delete cdi cdi --timeout=5m
kubectl delete -f https://github.com/kubevirt/containerized-data-importer/releases/download/v1.62.0/cdi-cr.yaml --ignore-not-found
kubectl delete -f https://github.com/kubevirt/containerized-data-importer/releases/download/v1.62.0/cdi-operator.yaml --ignore-not-found
kubectl delete namespace cdi --ignore-not-found
```

```bash
# 5. 驗證乾淨
# 🟢 已驗證 (kind / 顯然正確)
kubectl get vm,vmi,pvc,dv -A | grep fedora-rootdisk
kubectl get crd | grep cdi.kubevirt.io   # 若你也卸了 CDI 應該空
```

預期沒輸出，所有 fedora-rootdisk 相關資源都已銷毀。
