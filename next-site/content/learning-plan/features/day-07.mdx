## 動手做

### 1. 初始化 operator 專案

```bash
# 🟢 已驗證 (kind / 顯然正確)
mkdir cmwatcher && cd cmwatcher
kubebuilder init --domain example.com --repo example.com/cmwatcher
```

這會產生 `main.go`、`go.mod`、Makefile 等骨架。

### 2. 建立 API（CRD 定義）與 controller

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubebuilder create api \
  --group demo \
  --version v1 \
  --kind ConfigMapWatcher \
  --resource \
  --controller
```

產生：
- `api/v1/configmapwatcher_types.go` — CRD 的 Go struct
- `internal/controller/configmapwatcher_controller.go` — reconcile 邏輯

### 3. 編輯 Types：加入 spec 與 status 欄位

開啟 `api/v1/configmapwatcher_types.go`，找到 `ConfigMapWatcherSpec` 與 `ConfigMapWatcherStatus`，修改如下：

```go
// 🟢 已驗證 (kind / 顯然正確)
type ConfigMapWatcherSpec struct {
    // NamespaceSelector is the namespace to watch for ConfigMaps.
    NamespaceSelector string `json:"namespaceSelector"`
}

type ConfigMapWatcherStatus struct {
    // Count is the number of ConfigMaps found in the watched namespace.
    Count int32 `json:"count,omitempty"`
}
```

### 4. 編輯 Reconcile：List ConfigMap 並更新 status

開啟 `internal/controller/configmapwatcher_controller.go`，在 `Reconcile()` 函式裡加入以下邏輯（放在讀取 CR 物件之後）：

```go
// 🟢 已驗證 (kind / 顯然正確)
// List all ConfigMaps in the watched namespace.
cmList := &corev1.ConfigMapList{}
if err := r.List(ctx, cmList,
    client.InNamespace(watcher.Spec.NamespaceSelector),
); err != nil {
    return ctrl.Result{}, err
}

// Update status with the current count.
watcher.Status.Count = int32(len(cmList.Items))
if err := r.Status().Update(ctx, watcher); err != nil {
    return ctrl.Result{}, err
}
```

同時確認 `SetupWithManager` 裡 watches 了 ConfigMap：

```go
// 🟢 已驗證 (kind / 顯然正確)
return ctrl.NewControllerManagedBy(mgr).
    For(&demov1.ConfigMapWatcher{}).
    Watches(
        &corev1.ConfigMap{},
        handler.EnqueueRequestsFromMapFunc(func(ctx context.Context, obj client.Object) []reconcile.Request {
            // Re-enqueue all ConfigMapWatcher objects when any ConfigMap changes.
            var list demov1.ConfigMapWatcherList
            if err := mgr.GetClient().List(ctx, &list); err != nil {
                return nil
            }
            reqs := make([]reconcile.Request, len(list.Items))
            for i, item := range list.Items {
                reqs[i] = reconcile.Request{NamespacedName: types.NamespacedName{
                    Name: item.Name, Namespace: item.Namespace,
                }}
            }
            return reqs
        }),
    ).
    Complete(r)
```

### 8. Apply 一個 CR 並觀察 status

```bash
# 🟡 已對照官方文件 / 原始碼
kubectl apply -f - <<EOF
apiVersion: demo.example.com/v1
kind: ConfigMapWatcher
metadata:
  name: default-watcher
spec:
  namespaceSelector: default
EOF
```

等 1-2 秒後：

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl get configmapwatcher default-watcher -o yaml
```

```yaml
status:
  count: 2    # e.g. kube-root-ca.crt + any others in default namespace
```

建立一個新 ConfigMap，count 應自動 +1：

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl create configmap test-cm --from-literal=hello=world
kubectl get configmapwatcher default-watcher -o jsonpath='{.status.count}'
```

```
3
```

## 觀念深化

**Reconcile() 的觸發方式**：controller-runtime 的 informer 收到 add / update / delete event 後，把 object key 放進 work queue。Work queue 內建 dedup 和 rate-limit——即使 1 秒內 100 個 event 打進來，同一個 object key 也只會讓 Reconcile() 被叫一次（多餘的 dedup 掉）。

`.For()` 登記主要 watch 對象（ConfigMapWatcher CR），`.Watches()` 登記次要 watch 對象（ConfigMap）。當次要對象有變動時，透過 `MapFunc` 把事件轉換成要 reconcile 的 ConfigMapWatcher key。

整個 operator 的 Go 程式碼大約 200 行，但背後承載的是和 kube-controller-manager 完全相同的架構：shared informer factory → work queue → reconcile loop。
