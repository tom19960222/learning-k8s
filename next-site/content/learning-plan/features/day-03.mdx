## 動手做

### 1. 建立一個 Pod 並觀察排程事件

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl run scheduler-demo --image=busybox --command -- sleep 3600
```

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl get events --field-selector reason=Scheduled --sort-by=.metadata.creationTimestamp | tail -5
```

```
LAST SEEN   TYPE     REASON      OBJECT                   MESSAGE
...         Normal   Scheduled   Pod/scheduler-demo       Successfully assigned default/scheduler-demo to k8s-worker-01
```

`Successfully assigned ... to <node>` 這行由 scheduler 寫入，代表 Bind 完成。

### 2. 強制排程到指定 node（nodeSelector）

先找出你的 node 名稱：

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl get nodes -o jsonpath='{.items[*].metadata.name}'
```

建立一個帶有 `nodeSelector` 的 Pod（把 `<your-node>` 換成實際 node 名稱）：

```bash
# 🟡 已對照官方文件 / 原始碼
kubectl apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: pinned-pod
spec:
  nodeSelector:
    kubernetes.io/hostname: <your-node>
  containers:
  - name: busybox
    image: busybox
    command: ["sleep", "3600"]
EOF
```

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl get pod pinned-pod -o wide
```

```
NAME         READY   STATUS    NODE          ...
pinned-pod   1/1     Running   <your-node>   ...
```

`NODE` 欄位應完全符合你指定的 node 名稱。

### 3. 故意讓 Pod 進入 Pending——看 scheduler 的診斷

建立一個帶有不存在 label 的 nodeSelector：

```bash
# 🟡 已對照官方文件 / 原始碼
kubectl apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: unschedulable-pod
spec:
  nodeSelector:
    disktype: nvme
  containers:
  - name: busybox
    image: busybox
    command: ["sleep", "3600"]
EOF
```

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl describe pod unschedulable-pod
```

```
Events:
  Warning  FailedScheduling  ...  0/3 nodes are available: 3 node(s) didn't match Pod's node affinity/selector. preemption: 0/3 nodes are available: 3 Preemption is not helpful for scheduling.
```

這段 message 由 scheduler 寫回 API server 的 Event，告訴你每個 filter plugin 拒絕了多少 node。

### 4. 清理 Pending Pod，並查看 scheduler log

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl delete pod unschedulable-pod scheduler-demo pinned-pod
```

```bash
# 🟡 已對照官方文件 / 原始碼
kubectl logs -n kube-system \
  $(kubectl get pod -n kube-system -l component=kube-scheduler -o jsonpath='{.items[0].metadata.name}') \
  --tail=50
```

```
... "Successfully assigned" ...
... "Attempting to bind" ...
```

scheduler log 使用結構化 JSON，可以用 `| jq .msg` 過濾出可讀的訊息。

## 觀念深化

**Scheduler 是獨立 process，透過 watch 拿到 `spec.nodeName == ""` 的 Pod。** 排程流程分三階段：

1. **Filtering（predicates）**：逐一跑 filter plugin（NodeSelector、NodeUnschedulable、PodFitsResources 等），把不符合條件的 node 排除。
2. **Scoring（priorities）**：對剩餘 node 打分（LeastAllocated、PodTopologySpread 等），選出最高分。
3. **Binding**：呼叫 API server，把 `spec.nodeName` 寫進 Pod object。寫完之後，kubelet watch 到這個變更，開始真正建立 sandbox 與 container。

scheduler 本身不做任何 container 操作——它只負責「決定去哪裡」並寫一個欄位。
