## 動手做

### 1. 瀏覽所有 API resource

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl api-resources --verbs=list -o wide | head -20
```

```
NAME                 SHORTNAMES   APIVERSION   NAMESPACED   KIND
bindings                          v1           true         Binding
configmaps           cm           v1           true         ConfigMap
endpoints            ep           v1           true         Endpoints
...
deployments          deploy       apps/v1      true         Deployment
...
```

每一行都是一個 REST resource。`APIVERSION` 就是 etcd key 的路徑片段之一。

### 2. 用 explain 看欄位型別

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl explain deployment.spec.replicas
```

```
KIND:       Deployment
VERSION:    apps/v1

FIELD: replicas <integer>

DESCRIPTION:
    Replicas is the number of desired pods. Defaults to 1.
```

`kubectl explain` 的資料直接來自 API server 內嵌的 OpenAPI schema，不需要連到任何外部文件。

### 3. 在 control-plane node 找 etcd container ID

SSH 進 control-plane node 後：

```bash
# 🟡 已對照官方文件 / 原始碼
crictl ps | grep etcd
```

```
CONTAINER     IMAGE   CREATED   STATE     NAME   ...
<etcd-id>     ...     ...       Running   etcd   ...
```

記下 `<etcd-id>` 的前幾碼（後面要用）。

### 4. 進入 etcd container，列所有 key

```bash
# 🟡 已對照官方文件 / 原始碼
crictl exec -it <etcd-id> sh
```

進入 shell 後：

```bash
# 🟡 已對照官方文件 / 原始碼
etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  get /registry --prefix --keys-only | head -20
```

```
/registry/apiregistration.k8s.io/apiservices/v1.
/registry/clusterroles/system:aggregate-to-admin
/registry/configmaps/default/kube-root-ca.crt
/registry/deployments/kube-system/coredns
/registry/namespaces/default
...
```

key 的結構是 `/registry/<resource-plural>/<namespace>/<name>`（cluster-scoped resource 就沒有 namespace 片段）。

### 5. 讀一個 raw value 看 protobuf binary

挑一個熟悉的 resource，例如 `kube-root-ca.crt` ConfigMap：

```bash
# 🟡 已對照官方文件 / 原始碼
etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  get /registry/configmaps/default/kube-root-ca.crt
```

```
k8s
    v1ConfigMap...  (binary gibberish with embedded protobuf)
```

不是 JSON，是 protobuf binary。API server 在收到 `kubectl get` 請求時，把這份 binary 反序列化成物件，再序列化成 JSON 或 YAML 回給你。

## 觀念深化

**API server 是 stateless 的 façade。** 它負責：(1) 驗證身份（authn）與授權（authz）；(2) 跑 admission webhook pipeline；(3) 把物件序列化後寫進 etcd；(4) 把 watch event fan-out 給所有訂閱者（scheduler、controller-manager、kubelet 等）。它自己不存任何狀態——API server 程序重啟後，所有資料都還在 etcd 裡。

Resource 在 etcd 裡是以 **protobuf** 序列化（非 JSON），這讓讀寫速度更快、存儲更省空間。透過 `kube_apiserver_storage_objects` Prometheus metric 可以看到每種 resource 目前的 object count。

可以透過 `kubectl proxy` 直接打 REST endpoint 觀察 API server 的 HTTP 介面：`curl http://127.0.0.1:8001/api/v1/namespaces/default/configmaps`。
