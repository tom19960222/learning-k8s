## 動手做

### 1. 確認有一個 nginx Pod 在 worker node 上

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl run nginx --image=nginx
```

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl get pod nginx -o wide
```

```
NAME    READY   STATUS    NODE           ...
nginx   1/1     Running   k8s-worker-01  ...
```

記下 Pod 所在的 node 名稱（例如 `k8s-worker-01`），SSH 進去。

### 2. 在 worker node 找 sandbox（Pod sandbox）

```bash
# 🟡 已對照官方文件 / 原始碼
crictl pods --name nginx
```

```
POD ID              CREATED         STATE     NAME    NAMESPACE   ...
<sandbox-id>        ...             Ready     nginx   default     ...
```

Sandbox 對應到 Kubernetes Pod（不是 container）。一個 Pod 只有一個 sandbox。

### 3. 看 sandbox 下的 application container

```bash
# 🟡 已對照官方文件 / 原始碼
crictl ps --pod <sandbox-id>
```

```
CONTAINER           IMAGE     CREATED   STATE     NAME    ...
<container-id>      nginx     ...       Running   nginx   ...
```

這個 `<container-id>` 就是真正跑 nginx 程序的 container。

### 4. Inspect sandbox，找 network namespace 路徑

```bash
# 🟡 已對照官方文件 / 原始碼
crictl inspectp <sandbox-id> | head -60
```

```json
{
  "status": { ... },
  "info": {
    "runtimeSpec": {
      "linux": {
        "namespaces": [
          { "type": "network", "path": "/var/run/netns/cni-<uuid>" },
          ...
        ]
      }
    }
  }
}
```

記下 `"path"` 的值（例如 `/var/run/netns/cni-<uuid>`）。

### 5. 進入 network namespace，確認 eth0

先找出 sandbox 的 PID（pause process）：

```bash
# 🟡 已對照官方文件 / 原始碼
crictl inspectp <sandbox-id> | grep -i '"pid"' | head -3
```

```
"pid": <sandbox-pid>,
```

用 `nsenter` 進入那個 network namespace：

```bash
# 🟡 已對照官方文件 / 原始碼
nsenter -t <sandbox-pid> -n ip a
```

```
1: lo: <LOOPBACK,UP,LOWER_UP> ...
2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> ...
    inet 10.x.x.x/24 ...
```

`eth0` 的 IP 應與 `kubectl get pod nginx -o wide` 顯示的 Pod IP 相同。

### 6. 清理

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl delete pod nginx
```

## 觀念深化

**kubelet 透過 gRPC 跟 containerd 對話，使用的 protocol 是 CRI（Container Runtime Interface）**，定義在 `runtime.v1.RuntimeService`。CRI 把「容器執行」這件事抽象化，讓 kubelet 不需要知道底層是 containerd、CRI-O 還是其他 runtime。

一個 Pod 在 runtime 層面是：
- **1 個 sandbox container**（`pause` image）：負責持有 network / IPC / （可選）PID namespace。Sandbox 的 pause process 幾乎什麼都不做，但只要它存在，這些 namespace 就不會消失。
- **N 個 application container**：加入 sandbox 已建好的 namespace（`--network=container:<sandbox>` 的概念），共用同一個 network stack，所以它們看到相同的 eth0 與 IP。

**Sandbox 存在的關鍵原因**：當某個 application container crash 並重啟時，sandbox 還在，所以 network namespace（包含 Pod IP）保留不變。外部流量不需要因為 container 重啟而更新路由。

CRI protocol 參考：
- [cri-api proto](https://github.com/kubernetes/cri-api/blob/master/pkg/apis/runtime/v1/api.proto)
- [kubelet/cri 原始碼](https://github.com/kubernetes/kubernetes/tree/v1.36.0/pkg/kubelet/cri)
