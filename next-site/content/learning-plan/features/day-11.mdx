## 今日在 30 天計畫中的位置

Day 10 我們設定了 NetworkPolicy 與 CiliumNetworkPolicy；今天用 Hubble 把那些 allow / drop flow 視覺化。Hubble 本質上是把 eBPF event ring buffer 的資料透過 gRPC 暴露出來——這和 Day 9 的 `cilium monitor` 是相同的底層資料來源，只是 Hubble 多了跨 node 聚合與 UI。Day 12 啟用 WireGuard 加密後，你可以用 Hubble 觀察加密前的 flow（Hubble 在 WireGuard 之前 tap，所以仍能看到明文 flow metadata）。

## 動手做

### 1. Helm upgrade：啟用 Hubble Relay 與 UI

```bash
# 🟡 已對照官方文件 / 原始碼
helm upgrade cilium cilium/cilium \
  --version 1.19.3 \
  --namespace kube-system \
  --reuse-values \
  --set hubble.relay.enabled=true \
  --set hubble.ui.enabled=true
```

```
Release "cilium" has been upgraded. Happy Helming!
STATUS: deployed
REVISION: 2
```

`--reuse-values` 保留 Day 8 設定的所有 values（`kubeProxyReplacement=false` 等），只疊加新的 flag。

`Connected Nodes: 2/2` 表示 Relay 已聚合兩個 node 的資料。

### 6. 建一個 demo Pod 並觀察 flow

建立測試環境：

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl create ns hubble-demo
kubectl run -n hubble-demo server --image=nginx --labels=app=server
kubectl run -n hubble-demo client --image=curlimages/curl --labels=app=client -- sleep 3600
```

等 Pod Ready 後，在另一個 terminal 持續觀察：

```bash
# 🟡 已對照官方文件 / 原始碼
hubble observe --namespace hubble-demo --follow
```

```
TIMESTAMP             SOURCE                   DESTINATION              TYPE        VERDICT   REASON
...
```

回到原本的 terminal 觸發流量：

```bash
# 🟢 已驗證 (kind / 顯然正確)
POD_A_IP=$(kubectl get pod -n hubble-demo server -o jsonpath='{.status.podIP}')
kubectl exec -n hubble-demo client -- curl -s -o /dev/null -w "%{http_code}" http://${POD_A_IP}/ -m 5
```

observe terminal 應該出現類似：

```
2026-05-07T... hubble-demo/client:xxxxx -> hubble-demo/server:80   http-request   FORWARDED
2026-05-07T... hubble-demo/server:80   -> hubble-demo/client:xxxxx  http-response  FORWARDED
```

### 7. 開啟 Hubble UI service map

```bash
# 🔴 需在你環境跑後對照
kubectl -n kube-system port-forward service/hubble-ui 12000:80 &
```

用瀏覽器開啟 `http://localhost:12000`，切換到 `hubble-demo` namespace，你應該看到 `client` → `server` 的有向連線，帶有 FORWARDED 狀態。若先設定 NetworkPolicy 再觀察，可以看到 DROPPED flow 和 drop reason。

### 8. 清理

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl delete ns hubble-demo
```

## 觀念深化

**Hubble 的資料流程**：cilium-agent 在處理封包的 eBPF program 裡，把 flow 事件寫入 `BPF_MAP_TYPE_PERF_EVENT_ARRAY`（ring buffer）。Agent 的 user-space 部分讀取這個 buffer，把 flow 存進 in-memory circular buffer（預設 4096 個 flow per node）。Hubble Relay 透過 Unix socket 連到每個 node 的 Hubble server（cilium-agent 的 gRPC endpoint），聚合所有 node 的 flow，再暴露給 `hubble` CLI 或 Hubble UI 查詢。

Hubble 能看到的是 **flow metadata**（src/dst IP、port、verdict、reason、HTTP path），而不是 payload content。即使啟用了 WireGuard 加密（Day 12），Hubble 仍能看到 flow metadata——因為 tap 點在 WireGuard 封裝之前（node 內部的 veth hook），此時封包還是明文 Pod 流量。

**hubble observe 的篩選語法**：`--pod`、`--namespace`、`--type` 等 flag 在 Relay 端過濾，不是在 CLI 端；所以不用擔心漏掉 event。
