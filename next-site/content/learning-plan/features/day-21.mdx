## 動手做

### 1. 部署 RGW

依 Proxmox 版本的差異，方式不同。Proxmox VE 9 使用 cephadm orchestrator：

```bash
# 🔴 需在你環境跑後對照
ceph orch apply rgw default --placement="1 pve-01"
```

```
Scheduled rgw.default update...
```

如果你的環境是 Proxmox 自己包的指令：

```bash
# 🔴 需在你環境跑後對照
pveceph rgw create
```

部署完成後 RGW 會在 7480 port 監聽 HTTP（預設）：

```bash
# 🔴 需在你環境跑後對照
curl http://<rgw-host>:7480
```

```
<?xml version="1.0" encoding="UTF-8"?><ListAllMyBucketsResult ...><Owner>...
```

看到 XML 回應代表 RGW 起來了（雖然 anonymous 還沒權限做任何事）。

### 2. 建 S3 user

```bash
# 🔴 需在你環境跑後對照
radosgw-admin user create \
  --uid=k8s-app \
  --display-name="K8s App" \
  --email=devnull@example.com
```

```json
{
    "user_id": "k8s-app",
    "display_name": "K8s App",
    "keys": [
        {
            "user": "k8s-app",
            "access_key": "AKIA...",
            "secret_key": "SECRET..."
        }
    ],
    ...
}
```

把 `access_key` 與 `secret_key` 抄下來。

### 3. 從本機 aws CLI 測

```bash
# 🟡 已對照官方文件 / 原始碼
export AWS_ACCESS_KEY_ID=AKIA...
export AWS_SECRET_ACCESS_KEY=SECRET...
export ENDPOINT=http://<rgw-host>:7480

aws --endpoint-url=$ENDPOINT s3 mb s3://test-bucket --region=default
```

```
make_bucket: test-bucket
```

```bash
# 🟡 已對照官方文件 / 原始碼
echo "hello from local" > hello.txt
aws --endpoint-url=$ENDPOINT s3 cp hello.txt s3://test-bucket/
aws --endpoint-url=$ENDPOINT s3 ls s3://test-bucket/
```

```

2026-05-17 ... hello.txt
```

### 4. 從 k8s Pod 用 boto3

把 access/secret 注入成 Secret，然後跑一個 Python Pod：

```yaml
# 🟢 已驗證 (kind / 顯然正確)
apiVersion: v1
kind: Secret
metadata:
  name: rgw-creds
type: Opaque
stringData:
  access: "AKIA..."
  secret: "SECRET..."
---
apiVersion: v1
kind: Pod
metadata:
  name: s3-tester
spec:
  restartPolicy: Never
  containers:
  - name: py
    image: python:3.12-slim
    command:
    - sh
    - -c
    - |
      pip install boto3 -q
      python - <<'PY'
      import os, boto3
      s3 = boto3.client(
          's3',
          endpoint_url=os.environ['EP'],
          aws_access_key_id=os.environ['AK'],
          aws_secret_access_key=os.environ['SK'],
      )
      with open('/etc/hostname') as f:
          body = f.read()
      s3.put_object(Bucket='test-bucket', Key='hostname.txt', Body=body)
      objs = s3.list_objects_v2(Bucket='test-bucket').get('Contents', [])
      for o in objs:
          print(o['Key'], o['Size'])
      PY
    env:
    - {name: EP, value: "http://<rgw-host>:7480"}
    - {name: AK, valueFrom: {secretKeyRef: {name: rgw-creds, key: access}}}
    - {name: SK, valueFrom: {secretKeyRef: {name: rgw-creds, key: secret}}}
```

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl logs s3-tester
```

```
hello.txt 17
hostname.txt 12
```

## 觀念深化

S3 物件在 RADOS 上的存法：

```
object key (e.g. "hostname.txt")
  → head object (1 個 RADOS object，存 metadata + 第一段資料)
  → 0..N tail objects（每個 4 MiB，存剩下資料）
```

Head object 的命名是 `<bucket-marker>_<object-key>`；tail object 是 `<bucket-marker>__shadow_<hash>.<seq>`。可以用 `rados -p default.rgw.buckets.data ls | head` 看（會看到一堆 hex 名）。

RGW 自己也是 stateless：所有 bucket / user / ACL / policy metadata 都存在另外的 RADOS pool（`*.rgw.meta` / `*.rgw.log` / `*.rgw.control`）。所以 RGW 可以水平 scale — 起多個 RGW Pod 共享同一份 metadata 即可。

對比 RBD：RBD 也是把 image 拆成 4 MiB object，但完全沒有 metadata layer（RBD object 名是 deterministic 的 `rbd_data.<image-id>.<seq>`）。RGW 則需要 metadata layer 處理 bucket policy、versioning、lifecycle 等 S3 語義。
