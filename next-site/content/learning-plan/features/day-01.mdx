## 動手做

### 1. 在 kube-system namespace 找 4 個 control-plane Pod

```bash
# 🟡 已對照官方文件 / 原始碼
kubectl get pods -n kube-system | grep -E "(apiserver|etcd|scheduler|controller-manager)"
```

```
etcd-k8s-control-01                      1/1     Running   0          ...
kube-apiserver-k8s-control-01            1/1     Running   0          ...
kube-controller-manager-k8s-control-01   1/1     Running   0          ...
kube-scheduler-k8s-control-01            1/1     Running   0          ...
```

這 4 個 Pod 名稱後面帶有 node 名稱，是 static Pod 的命名慣例。

### 2. SSH 進 control-plane node，用 crictl 看 container

```bash
# 🟡 已對照官方文件 / 原始碼
crictl ps -a | head -10
```

```
CONTAINER     IMAGE     CREATED     STATE    NAME                   ...
<id>          ...       ...         Running  kube-apiserver         ...
<id>          ...       ...         Running  etcd                   ...
<id>          ...       ...         Running  kube-controller-manager...
<id>          ...       ...         Running  kube-scheduler         ...
```

`crictl` 直接跟 containerd 的 CRI socket 對話，不需要 API server 也能列出 container。這也是為什麼 static Pod 不依賴 API server 就能跑。

### 3. 看 static Pod manifest 目錄

```bash
# 🟡 已對照官方文件 / 原始碼
ls /etc/kubernetes/manifests/
```

```
etcd.yaml  kube-apiserver.yaml  kube-controller-manager.yaml  kube-scheduler.yaml
```

kubelet 在啟動時（以及每隔 20 秒）會掃這個目錄，把找到的 YAML 當成 Pod spec 直接建立，完全不經過 API server。

### 4. 看 kube-apiserver 的啟動參數

```bash
# 🟡 已對照官方文件 / 原始碼
head -30 /etc/kubernetes/manifests/kube-apiserver.yaml
```

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: kube-apiserver
  namespace: kube-system
spec:
  containers:
  - command:
    - kube-apiserver
    - --advertise-address=<control-plane-ip>
    - --etcd-servers=https://127.0.0.1:2379
    - --service-cluster-ip-range=10.96.0.0/12
    ...
```

重點參數：`--etcd-servers` 告訴 API server 去哪裡找 etcd；`--service-cluster-ip-range` 定義 ClusterIP 的 CIDR。

## 觀念深化

**Static Pod 是雞生蛋蛋生雞問題的解法。** kube-apiserver 自己也是 static Pod，所以不需要 API server 就能存在——這讓 cluster 啟動成為可能。

kubelet 啟動後會 watch `/etc/kubernetes/manifests/`（以及透過 `--pod-manifest-path` 指定的任意目錄）。每找到一個 YAML，kubelet 就在本地建立對應的 Pod 並維護其生命週期（crash 後自動重啟）。Static Pod 的 mirror Pod 會由 kubelet 同步到 API server，所以你在 `kubectl get pods -n kube-system` 看到的那 4 個 Pod，其實是「唯讀的 mirror」——直接用 `kubectl delete` 刪掉也沒用，kubelet 會立刻重建。

詳見官方文件：
- [Static Pods](https://kubernetes.io/docs/tasks/configure-pod-container/static-pod/)
- [Kubernetes Components](https://kubernetes.io/docs/concepts/overview/components/)
