## 動手做

### 1. http source（複習）

```yaml
# 🟡 已對照官方文件 / 原始碼
apiVersion: cdi.kubevirt.io/v1beta1
kind: DataVolume
metadata:
  name: ubuntu-cloud
spec:
  source:
    http:
      url: "https://cloud-images.ubuntu.com/jammy/current/jammy-server-cloudimg-amd64.img"
  pvc:
    accessModes: [ReadWriteOnce]
    resources:
      requests:
        storage: 10Gi
    storageClassName: ceph-rbd
```

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl apply -f dv-http.yaml
kubectl get dv ubuntu-cloud -w
```

### 2. PVC clone source

複製已有的 PVC 變成新 PVC（適合做 base image → instance image）：

```yaml
# 🟡 已對照官方文件 / 原始碼
apiVersion: cdi.kubevirt.io/v1beta1
kind: DataVolume
metadata:
  name: fedora-clone
spec:
  source:
    pvc:
      namespace: default
      name: fedora-rootdisk   # Day 24 建的
  pvc:
    accessModes: [ReadWriteOnce]
    resources:
      requests:
        storage: 10Gi
    storageClassName: ceph-rbd
```

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl apply -f dv-clone.yaml
kubectl get dv fedora-clone -w
```

```
NAME           PHASE                 PROGRESS   ...
fedora-clone   CSICloneInProgress    50.0%
fedora-clone   Succeeded             100.0%
```

ceph-csi 支援 CSI VolumeSource clone（用 ceph 的 RBD snapshot/copy），CDI 偵測到後會用 native clone 取代 importer Pod；速度比 http 快很多（不必跨網路傳）。

### 3. registry source（從 OCI image 匯入）

KubeVirt 提供了一些 ready-to-use containerdisk image 在 quay.io/containerdisks，CDI 可以把 container image 內的 disk 匯出到 PVC：

```yaml
# 🟡 已對照官方文件 / 原始碼
apiVersion: cdi.kubevirt.io/v1beta1
kind: DataVolume
metadata:
  name: centos-from-registry
spec:
  source:
    registry:
      url: "docker://quay.io/containerdisks/centos-stream:9"
  pvc:
    accessModes: [ReadWriteOnce]
    resources:
      requests:
        storage: 10Gi
    storageClassName: ceph-rbd
```

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl apply -f dv-registry.yaml
```

importer Pod 會用 containers/image library 拉 OCI image 後解壓 disk image 寫進 PVC。

### 4. blank source（空白磁碟）

需要一個空白 disk（例如給 VM 額外 data disk）：

```yaml
# 🟡 已對照官方文件 / 原始碼
apiVersion: cdi.kubevirt.io/v1beta1
kind: DataVolume
metadata:
  name: data-disk
spec:
  source:
    blank: {}
  pvc:
    accessModes: [ReadWriteOnce]
    resources:
      requests:
        storage: 5Gi
    storageClassName: ceph-rbd
```

不需要 importer Pod；直接建 PVC 然後 `qemu-img create -f raw` 一份 sparse 的空白 image。

### 5. 用 clone 過的 PVC 跑新 VM

```yaml
# 🟡 已對照官方文件 / 原始碼
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: fedora-clone-vm
spec:
  runStrategy: Always
  template:
    metadata: {labels: {kubevirt.io/vm: fedora-clone-vm}}
    spec:
      domain:
        cpu: {cores: 1}
        memory: {guest: 1Gi}
        devices:
          disks:
          - {name: rootdisk, disk: {bus: virtio}}
          interfaces:
          - {name: default, masquerade: {}}
      networks: [{name: default, pod: {}}]
      volumes:
      - name: rootdisk
        persistentVolumeClaim:
          claimName: fedora-clone
```

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl apply -f vm-clone.yaml
virtctl console fedora-clone-vm
```

跟 Day 23 的 fedora-vm 內容一致（包含你前面寫過的 `/root/persist.txt`），但是新 VM 獨立的 PVC。

## 觀念深化

CDI 三個元件：

- **cdi-operator**：Deployment，1 replica。負責部署/升級 CDI 自己。
- **cdi-deployment（cdi-controller + cdi-apiserver）**：watch DataVolume CR，依 source 類型決定要起 importer Pod 還是用 CSI clone / snapshot。
- **cdi-uploadproxy**：HTTPS upload endpoint，給 user 從本機 `virtctl image-upload` 上傳 image 用。

DataVolume 與 PVC 的關係：

- DataVolume 是個 CR，CDI controller 看到後幫你建 PVC（命名相同）+ 必要時起 importer Pod
- 你也可以直接寫 `kubectl get pvc`，看到的就是 DataVolume 對應的 PVC
- 在 VM YAML 也可以直接寫 `dataVolume:` 而非 `persistentVolumeClaim:`，VM 啟動時自動等 DataVolume Succeeded 才開始

PVC clone 走 CSI 的 [`spec.dataSource`](https://kubernetes.io/docs/concepts/storage/volume-pvc-datasource/)，由 storage backend 決定是 fast clone（ceph-rbd 用 RBD snapshot）還是 slow copy（fallback dd）。Ceph 的 RBD clone 是 copy-on-write，建立瞬間完成、實際 IO 在 write 時才 copy。
