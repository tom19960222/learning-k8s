### 2. PVC clone source

複製已有的 PVC 變成新 PVC（適合做 base image → instance image）：

```yaml
# 🟡 已對照官方文件 / 原始碼
apiVersion: cdi.kubevirt.io/v1beta1
kind: DataVolume
metadata:
  name: fedora-clone
spec:
  source:
    pvc:
      namespace: default
      name: fedora-rootdisk   # Day 24 建的
  pvc:
    accessModes: [ReadWriteOnce]
    resources:
      requests:
        storage: 10Gi
    storageClassName: ceph-rbd
```

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl apply -f dv-clone.yaml
kubectl get dv fedora-clone -w
```

```
NAME           PHASE                 PROGRESS   ...
fedora-clone   CSICloneInProgress    50.0%
fedora-clone   Succeeded             100.0%
```

ceph-csi 支援 CSI VolumeSource clone（用 ceph 的 RBD snapshot/copy），CDI 偵測到後會用 native clone 取代 importer Pod；速度比 http 快很多（不必跨網路傳）。

### 3. registry source（從 OCI image 匯入）

KubeVirt 提供了一些 ready-to-use containerdisk image 在 quay.io/containerdisks，CDI 可以把 container image 內的 disk 匯出到 PVC：

```yaml
# 🟡 已對照官方文件 / 原始碼
apiVersion: cdi.kubevirt.io/v1beta1
kind: DataVolume
metadata:
  name: centos-from-registry
spec:
  source:
    registry:
      url: "docker://quay.io/containerdisks/centos-stream:9"
  pvc:
    accessModes: [ReadWriteOnce]
    resources:
      requests:
        storage: 10Gi
    storageClassName: ceph-rbd
```

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl apply -f dv-registry.yaml
```

importer Pod 會用 containers/image library 拉 OCI image 後解壓 disk image 寫進 PVC。

### 4. blank source（空白磁碟）

需要一個空白 disk（例如給 VM 額外 data disk）：

```yaml
# 🟡 已對照官方文件 / 原始碼
apiVersion: cdi.kubevirt.io/v1beta1
kind: DataVolume
metadata:
  name: data-disk
spec:
  source:
    blank: {}
  pvc:
    accessModes: [ReadWriteOnce]
    resources:
      requests:
        storage: 5Gi
    storageClassName: ceph-rbd
```

不需要 importer Pod；直接建 PVC 然後 `qemu-img create -f raw` 一份 sparse 的空白 image。

### 5. 用 clone 過的 PVC 跑新 VM

```yaml
# 🟡 已對照官方文件 / 原始碼
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: fedora-clone-vm
spec:
  runStrategy: Always
  template:
    metadata: {labels: {kubevirt.io/vm: fedora-clone-vm}}
    spec:
      domain:
        cpu: {cores: 1}
        memory: {guest: 1Gi}
        devices:
          disks:
          - {name: rootdisk, disk: {bus: virtio}}
          interfaces:
          - {name: default, masquerade: {}}
      networks: [{name: default, pod: {}}]
      volumes:
      - name: rootdisk
        persistentVolumeClaim:
          claimName: fedora-clone
```

```bash

virtctl console fedora-clone-vm
```

## 觀念深化

CDI 三個元件：

- **cdi-operator**：Deployment，1 replica。負責部署/升級 CDI 自己。
- **cdi-deployment（cdi-controller + cdi-apiserver）**：watch DataVolume CR，依 source 類型決定要起 importer Pod 還是用 CSI clone / snapshot。
- **cdi-uploadproxy**：HTTPS upload endpoint，給 user 從本機 `virtctl image-upload` 上傳 image 用。

DataVolume 與 PVC 的關係：

- DataVolume 是個 CR，CDI controller 看到後幫你建 PVC（命名相同）+ 必要時起 importer Pod
- 你也可以直接寫 `kubectl get pvc`，看到的就是 DataVolume 對應的 PVC
- 在 VM YAML 也可以直接寫 `dataVolume:` 而非 `persistentVolumeClaim:`，VM 啟動時自動等 DataVolume Succeeded 才開始

PVC clone 走 CSI 的 [`spec.dataSource`](https://kubernetes.io/docs/concepts/storage/volume-pvc-datasource/)，由 storage backend 決定是 fast clone（ceph-rbd 用 RBD snapshot）還是 slow copy（fallback dd）。Ceph 的 RBD clone 是 copy-on-write，建立瞬間完成、實際 IO 在 write 時才 copy。

更深入：

- [DataVolume types](https://kubevirt.io/user-guide/storage/datavolumes/) — 所有 source 形式
- [CDI repository](https://github.com/kubevirt/containerized-data-importer) — 原始碼
- [Volume PVC dataSource](https://kubernetes.io/docs/concepts/storage/volume-pvc-datasource/) — k8s clone 規格

### 為什麼這樣設計

CDI 的 clone 流程其實有三條路：smart clone（CSI VolumeSnapshot）、host-assisted clone（importer Pod + qemu-img）、network clone（rsync over Pod-to-Pod tunnel）。直觀答案是「能用 storage backend 的 native 能力就用」。ceph-rbd 支援 CSI VolumeSnapshot 跟 RBD layered clone，建立 snapshot 是 O(1)、clone 出新 image 也是 O(1)（copy-on-write），所以 CDI controller 偵測到 source PVC 跟 target PVC 在同一個 storage class、同一個 node selector 時，會直接走 smart clone — 不需要起 importer Pod，秒回。

設計選擇背後的取捨是「正確性 vs 效能」。CSI smart clone 雖然快，但限制條件多：source / target 必須在同一個 CSI driver、同一個 storage pool、同一個 namespace 範圍（依 driver 而定）；跨 storage class（例如從 ceph-rbd clone 到 longhorn）就絕對不行。host-assisted clone 是 fallback：CDI 在 source PVC 那邊起一個 source Pod（read-only mount）、target PVC 那邊起一個 target Pod（read-write mount），中間用 unix pipe 把 qcow2 stream 過去；速度大約 = 網路頻寬，但 source / target 可以在不同 node、不同 storage class。network clone 是更老的 fallback，跨 cluster 用，現在很少用到。CDI controller 會自動依當前 storage 能力選 mode，使用者只寫一份 DataVolume YAML。

常見誤解是「smart clone 的 PVC 跟 source PVC 互相影響」。實際上 RBD layered clone 是 copy-on-write：兩個 image 共享 base block，任一邊寫入時才 copy 到自己的 layer。但這代表 source 不能被刪掉（會把 child 的 base 抽掉）；CDI 處理這個用 PVC finalizer + CDI volume cloning 的 ownership chain。如果你看到 PVC 卡在 `Terminating`，多半就是被另一個 cloned PVC 鎖住。

## 🧹 Cleanup / 回退

🔴 **警告**：今天建了 4 個 DataVolume（`ubuntu-cloud` / `fedora-clone` / `centos-from-registry` / `data-disk`）+ 1 個 VM（`fedora-clone-vm`）。若不清理，每個都佔 5–10 GiB ceph-rbd 空間，多累積幾天 lab 會把小 cluster 撐爆。Day 29 不依賴這些資源，**建議跑完 cleanup**。

順序：VM → DataVolume（會連同 PVC、importer Pod 一起收）。

```bash
# 1. stop 並 delete fedora-clone-vm
# 🟡 已對照官方文件 / 原始碼
virtctl stop fedora-clone-vm 2>/dev/null
kubectl delete vm fedora-clone-vm --ignore-not-found
```

```bash
# 2. 刪 4 個 DataVolume（連同 PVC + ceph 端 RBD image）
# 🟢 已驗證 (kind / 顯然正確)
kubectl delete dv ubuntu-cloud fedora-clone centos-from-registry data-disk --ignore-not-found
```

刪 DataVolume → CDI controller 會把對應 PVC delete → ceph-csi 把 RBD image 砍掉。整個過程通常 < 30 秒。

```bash
# 3. 看 ceph 端確認 image 真的消失
# 🔴 需在你環境跑後對照
rbd ls k8s-rbd | wc -l
```

預期數字回到 lab 開始前的水準（通常只剩 Day 24 的 `fedora-rootdisk` 對應 image）。

```bash
# 4. 驗證 PVC 也清乾淨
# 🟢 已驗證 (kind / 顯然正確)
kubectl get pvc | grep -E "(ubuntu-cloud|fedora-clone|centos-from-registry|data-disk)"
```

預期沒輸出。

如果某個 DataVolume 卡在 `Terminating` 狀態（例如 importer Pod 還在跑沒被中止），先 `kubectl delete pod importer-<dv-name> --force --grace-period=0`，再 retry step 2。
