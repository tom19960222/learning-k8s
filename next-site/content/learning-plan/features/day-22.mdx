### 4. 認識 4 個元件

```bash
# 🟢 已驗證 (kind / 顯然正確)
kubectl -n kubevirt get deploy,daemonset
```

```
NAME                          READY  ...
deployment.apps/virt-api      2/2
deployment.apps/virt-controller 2/2
deployment.apps/virt-operator 2/2

NAME                       DESIRED  CURRENT  READY ...
daemonset.apps/virt-handler  3        3        3
```

- **virt-operator**: lifecycle 編排 KubeVirt 自身（升級、重啟、卸載）
- **virt-api**: API server 擴充，新增 `VirtualMachine` / `VirtualMachineInstance` 資源；同時做 admission webhook
- **virt-controller**: VirtualMachine ↔ VMI ↔ Pod 的 reconcile，類似 Deployment Controller 對 Pod
- **virt-handler**: per-node DaemonSet，跟 libvirt/qemu 互動，類似 kubelet 對 container

### 5. 安裝 virtctl CLI（本機）

`virtctl` 是 KubeVirt 額外提供的 CLI，給 console / vnc / migrate / start / stop 用。

```bash
# 🟡 已對照官方文件 / 原始碼
curl -L -o /usr/local/bin/virtctl https://github.com/kubevirt/kubevirt/releases/download/v1.8.2/virtctl-v1.8.2-linux-amd64
chmod +x /usr/local/bin/virtctl
virtctl version --client
```

## 觀念深化

KubeVirt 把 VM 當 first-class k8s 資源：

- 從使用者角度看：寫 `VirtualMachine` YAML 就跟寫 Deployment 一樣，`kubectl get vm` 就像 `kubectl get pod`
- 從實作角度看：每個 VM 在 node 上其實是個 Pod（virt-launcher Pod），裡面跑 libvirt + qemu

四個元件對應四種職責：

| 元件 | 在哪 | 職責 | 類比 |
|---|---|---|---|
| virt-operator | cluster-wide | KubeVirt 自身 lifecycle | kube-aggregator/operator |
| virt-api | cluster-wide | 擴充 k8s API + admission | kube-apiserver subresource |
| virt-controller | cluster-wide | VM/VMI/Pod 的 reconcile | kube-controller-manager |
| virt-handler | per-node | 跟 libvirt/qemu 互動 | kubelet |

更深入：

- [KubeVirt architecture](https://kubevirt.io/user-guide/architecture/)
- 原始碼進入點：[cmd/](https://github.com/kubevirt/kubevirt/tree/v1.8.2/cmd) 下各元件的 main.go
- virt-controller 主邏輯：[pkg/virt-controller/watch/](https://github.com/kubevirt/kubevirt/tree/v1.8.2/pkg/virt-controller/watch)

### 為什麼這樣設計

KubeVirt 把整套虛擬化平台拆成 operator + apiserver-extension + controller + per-node agent 四層，並用 operator pattern 管理自身 lifecycle，而不做成單一 DaemonSet。直觀答案是「KubeVirt 自己也要升級／降級／reconfigure，而 DaemonSet 沒辦法自我管理 CRD、webhook、ServiceAccount、SCC 這些 cluster-scoped 資源」。`KubeVirt` CR 是「我想要某個 version 的 KubeVirt 在 cluster 裡跑」的宣告，virt-operator 看到這個 CR 就把對應 version 的 manifest bundle 攤開、做版本差比、滾動更新。整個流程跟 cluster-api 或 prometheus-operator 同套思路。

設計選擇背後的取捨主要在「資料平面 vs 控制平面」的隔離。virt-controller 是無狀態的 reconciler（任何 replica 接手都行，靠 leader election 決定主節點），所以做成 Deployment；virt-handler 必須 1:1 對應 node 上的 libvirt socket（VM 進程的本地 supervisor），所以做成 DaemonSet。把這兩種職責壓在同一個 binary 會讓 cluster-wide 的決策跟 per-node 的執行混在一起，後續要做 RBAC 收斂、要上 SELinux/seccomp profile、要分開 release cadence 都會卡住。virt-launcher 又再往下分一層 — 每個 VM 起一個 launcher Pod，把 libvirt + qemu 包進 Pod 裡跑，讓 k8s scheduler 直接做 VM placement。

常見誤解是「virt-handler 跟 kubelet 重疊」。實際上 kubelet 還是負責 Pod lifecycle，virt-handler 只在 launcher Pod 內透過 unix socket 跟 libvirt 講話、把 domain state 對齊到 VMI status。沒有 virt-handler，VM 還是會跑（launcher Pod 裡的 libvirt 自己就能起 qemu），但 VMI 的 phase 不會收斂、live migration 也無法被觸發。

## 🧹 Cleanup / 回退

🔴 **警告**：卸載 KubeVirt 會把所有 VirtualMachine / VirtualMachineInstance / DataVolume 等資源連同 CRD 一併消失。Day 23 之後的所有 VM lab 都建立在這套 KubeVirt 上；只有當你**確定要打掉重練**才執行下面步驟。日常情況不需要 cleanup，留著繼續用就好。

順序：先刪 workload（VM），再刪 KubeVirt CR 等 operator 走 finalizer，最後刪 operator manifest 與 namespace。

```bash
# 1. 列出並刪除所有 VM（如果 Day 23+ 已經有 VM，先 stop 再 delete）
# 🟡 已對照官方文件 / 原始碼
kubectl get vm,vmi -A
# 對每個 VM：
# virtctl stop <vm-name> -n <ns>
# kubectl delete vm <vm-name> -n <ns>
```

```bash
# 2. 刪 KubeVirt CR，operator 會 reconcile 把 virt-api / virt-controller / virt-handler 拆光
# 🟡 已對照官方文件 / 原始碼
kubectl -n kubevirt delete kv kubevirt --timeout=5m
```

預期 5 分鐘內完成。如果 finalizer 卡住（少數情況），可手動移除：

```bash
# 🟡 已對照官方文件 / 原始碼
kubectl -n kubevirt patch kv kubevirt --type=merge -p '{"metadata":{"finalizers":[]}}'
```

```bash
# 3. 刪除 operator deployment 與 CRD（用裝的時候那份 manifest 反向 apply）
# 🟡 已對照官方文件 / 原始碼
kubectl delete -f https://github.com/kubevirt/kubevirt/releases/download/v1.8.2/kubevirt-cr.yaml --ignore-not-found
kubectl delete -f https://github.com/kubevirt/kubevirt/releases/download/v1.8.2/kubevirt-operator.yaml --ignore-not-found
```

```bash
# 4. 確保 kubevirt namespace 也走光
# 🟢 已驗證 (kind / 顯然正確)
kubectl delete namespace kubevirt --ignore-not-found
```

```bash
# 5. （可選）移除本機 virtctl
# 🟡 已對照官方文件 / 原始碼
sudo rm -f /usr/local/bin/virtctl
```

```bash
# 6. 驗證乾淨
# 🟢 已驗證 (kind / 顯然正確)
kubectl get crd | grep kubevirt.io
kubectl get ns kubevirt
```

預期 `kubectl get crd | grep kubevirt.io` 沒輸出（沒有殘留 CRD），`kubectl get ns kubevirt` 回 `Error from server (NotFound)`。
