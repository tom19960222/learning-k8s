- **文章類型**：experiment-report（工程驗證報告）。它只回答一件事——**這支工具的行為驗證過了嗎、怎麼驗的、結果如何**。「工具怎麼用」不在這頁，在姊妹頁 [`incident-bundle-operator-runbook`](./incident-bundle-operator-runbook)。
- **這是日期化快照（as-of 2026-07），分三波，本頁全程標明是哪一波**：
  - **2026-06-29 健康收集（第一期歷史快照）**：第一次在真 6-host cephadm lab 上跑健康路徑，確認整條 pipeline 跑得完、6 台 node 都收到、`VERIFY PASS`。

  - 工具現行 HEAD＝commit `5673a82`（2026-07-11），也就是把 prom 驗證證據落 git 的那個 commit。
  - **2026-06-30 multi-fault** 驗的是當時的工具（multi-fault 結果 commit `2c7eac9`、auto smoke `0e3b076`，皆 2026-06-30）。那時**還沒有** `--prom-url`——該能力自 2026-07-07 的 `8310fdd`（helper 函式）起開發、同日 `aa4e5bb` 把旗標接進 `collect.sh`。所以 multi-fault 的故障注入結論綁在 2026-06-30 的碼，prom 這層它沒測到。

## 術語表

完整詞彙表在姊妹 runbook 頁；這裡只給本頁承重詞各一句話，讀本頁不必跳頁：

| 詞 | 一句話定義 |
|---|---|
| **bundle** | 最終產物，一個 `ceph-incident-YYYYMMDDTHHMMSSZ.tar.gz`，裝這次收到的所有證據。 |
| **workdir** | 打包前的暫存目錄。正常打包完會刪；**verify 失敗時會保留**讓你檢查已收到的東西。 |
| **exit code 語意** | `0`＝收齊、沒有已知收集失敗（**叢集自身故障如 OSD/MON down 被收進 bundle 仍算 0**）；`2`＝部分收集失敗、但 bundle 已產出；`1`＝用法／必要輸入錯，或 **bundle verify 失敗**（此時不打包、保留 workdir）。 |
| **redaction** | 打包前把文字 artifact 裡「明顯像 secret 的整行」換成 `[REDACTED]`。best-effort，**不是完整 DLP**。 |
| **`SKIPPED.txt`** | 某層／某台 node 沒收到時寫的一個檔，內含**具體原因**；它讓「沒收到」變成可稽核紀錄，而非靜默漏掉。 |
| **seed / runner** | seed＝在哪台 node 跑 cluster-level `ceph`；runner＝在 seed 上怎麼跑（`direct` 直接 `ceph`／`sudo` `sudo -n ceph`／`cephadm` `sudo -n cephadm shell -- ceph`），工具按連線性自動挑。 |
| **manifest** | `manifest.jsonl`，每個 artifact 一行 JSON：哪台 host、哪條指令、exit code、起訖時間。`CONTENTS.md` 是它的人類可讀版。 |
| **T3／單元層** | T3＝真機實測；單元層＝`tests/` 用 fake 指令注入故障的行為測試。 |
| **budget 截斷** | prom dump 的**整體時間預算**（`--prom-timeout`，預設 600s）用完時，dump 停止、已 dump 的 metric 保留，並在 `dump-info.txt`／該 job 的 `index.txt` 標 `TRUNCATED: budget <n>s exceeded`（`lib/collect-prometheus.sh:265-268`）。 |

## 1. 前提

### 1.1 驗證環境（真機，非模擬）

三波都在**同一座真 cephadm 叢集**上跑（不是 kind／minikube、不是 fake）：

| 項目 | 值 |
|---|---|
| Ceph | cephadm **v19.2.3**（Squid LTS） |
| 拓樸 | **6 個實體 host**：3 mon（`.166`／`.167`／`.164`）＋ 3 OSD host（`.169`／`.171`／`.174`，每台 3 顆＝共 **9 OSD**） |
| pool | `.mgr`，size 3／min_size 2 |
| 工作機 | macOS（bash 3.2、**沒有** `timeout`／`gtimeout`）——每次跑都印一行「找不到 timeout/gtimeout」警告，是設計行為；外層逾時停用，只靠 SSH `ConnectTimeout`／`ServerAlive` 把關（此為環境綁定，見 [§6](#6-侷限與誠實保留)） |
| Prometheus（僅第三波） | v3.12.0，工作機本機 port 9790，scrape 真 active mgr `192.168.18.167:9283`（`scrape_interval` 5s） |

破壞性情境（MF-2 停 OSD、MF-3 停 mon）一律**先 `ceph osd ok-to-stop` / 確認 quorum 再注入**，downtime 壓在 600s `mon_osd_down_out_interval` 內（不觸發 rebalance），每次回退後**重新確認 `HEALTH_OK` 才進下一場**。

### 1.2 判準（先於實驗定案）

每一場的 PASS **同時**要滿足兩條，缺一不算過：

1. **exit code 語意對**：實際 exit code＝該場預測的語意（見術語表 exit code 定義）。這是本報告的主軸——工具最關鍵的契約就是「把『叢集有病』和『我收集失敗』分開」。
2. **bundle 內容斷言成立**：該收的 artifact 在不在、故障證據有沒有進 bundle、redaction 有沒有守住、`SKIPPED.txt`／`environment.txt`／`manifest.jsonl` 該有的欄位對不對。

### 1.3 三波規模

三波共 **16 個可稽核斷言點**（H-1 一場整體斷言 ＋ MF 六場 ＋ PD 九條），全部對得上素材檔：`docs/superpowers/reviews/2026-06-30-lab-validation.md`（multi-fault 逐場紀錄，tracked）、`experiments/ceph-incident-bundle/PROM-VALIDATION-2026-07.md`（prom dump 逐項斷言，tracked）、`experiments/ceph-incident-bundle/README.md` §「Lab 驗證（multi-fault）」（摘要表）。**raw bundle 本體被 `.gitignore` 排除、只在本機**（見 [§6](#6-侷限與誠實保留)）。

## 4. 發現與邊界

**一句話結論**：這支工具的核心契約——**「叢集有病」≠「我收集失敗」，且部分失敗仍保住現場**——在真 6-host cephadm v19.2.3 lab 上以 exit code 對照（0 vs 2）驗證成立（multi-fault 波為 2026-06-30 版工具）；`--prom-url` 這層對真 Prometheus 的 103-metric dump 也全數過。缺口都是**環境綁定或只有單元層背書**（見 §2.2／§6），沒有機制級失敗。下一步（§7）以「在現行版重跑 multi-fault」與「Rook external 真機演練」優先。

- **exit code 語意的設計驗證（核心發現）**：MF-2／MF-3 exit **0**、MF-4／MF-5 exit **2**，兩組對照證明工具刻意把「叢集自身故障」（收進 bundle 當證據）與「收集動作失敗」（記進 `errors.log`、rc=2）分開。事故現場最常見的就是某台 node 半死不活；若因它的 `journalctl` 失敗就讓整包不產生，反而損失其他 node 的證據。這個語意是工具最重要的契約，也是本報告花最多篇幅驗的東西。
- **partial failure 仍產 bundle**：MF-4（node 不可達）、MF-5（seed 不可達）都是 exit 2 但 bundle 照樣產出、失敗寫進 `errors.log`／`summary.txt`——符合 README 契約「先保留現場、把失敗記下來」。
- **`SKIPPED.txt` 路徑是可稽核的「沒收到」**：MF-4 的 `nodes/badnode/SKIPPED.txt`、MF-6 的 rook 層 `SKIPPED`、PD-8 的 prom `SKIPPED.txt`——三個不同層級的「沒收到」都是寫一個帶**具體原因**的檔，而不是靜默漏掉。這讓判讀者能區分「這台真的沒問題」與「這台根本沒收到」。
- **redaction 守住、但不是 DLP**：MF-1 的 312 行遮蔽＋`verify_no_secret_content` 在 live config 通過，證明沒有金鑰材料 survive 進打包的 bundle。但 redaction 是 best-effort：它擋明顯 secret，**不保證**內部 IP／hostname／pool name 這類半敏感資料被清掉。PD-7 另證一個刻意設計——**metric `json.gz` 故意不 redact**（逐行掃單行大 JSON 極慢，且 regex 誤中會讓整檔變 `[REDACTED]`），只有文字 artifact 進 redaction。
- **macOS 工作機沒有 `timeout`／`gtimeout`**：三波都在這台跑，每次都印「找不到 timeout/gtimeout」警告、外層逾時停用，只靠 SSH `ConnectTimeout`／`ServerAlive` 綁住遠端呼叫。MF-5（死 seed）因此偏慢——~24 條 cluster cephadm 呼叫各自等到 `ConnectTimeout`（但**有界**，不會永久卡住）。要完整逐 node 逾時把關，需 `brew install coreutils`（提供 `gtimeout`）或改在 Linux ops 機跑。這是環境綁定、非機制缺陷。
- **SIGKILL 無法攔**：process group 被外部 `kill -9` 時 trap 攔不到，workdir 可能殘留；工具自己的中斷／失敗（EXIT／INT／TERM）由 trap 清乾淨（由一支 regression test 背書）。

## 5. tests/ 單元層背書（真機沒逐一演練的分支在這裡釘住）

真機只能演練「當下叢集會發生的情境」；很多故障分支（Prometheus 截斷、多 job、URL 帳密、Rook 各種 namespace／連線失敗、verify 黑名單）在健康 lab 上不會自然出現。這些由 `experiments/ceph-incident-bundle/tests/` 的 **7 個測試檔**用 fake `ssh`／`curl`／`kubectl`（PATH 覆蓋）＋環境變數注入故障、對產出的 bundle／檔案斷言來釘住：

## 6. 侷限與誠實保留

- **bundle 本體不在 git**：三波產出的 `.tar.gz` 都被 `.gitignore` 排除、**只在本機**（bundle 是事故證據，不該進版控）。committed 的是**驗證索引**——review 紀錄、`PROM-VALIDATION-2026-07.md`、README 摘要表。所以本頁的 T3 證據可以逐行對到 committed 的文字紀錄，但**無法**從 repo 重新解開原始 bundle 覆核；要覆核得在本機重跑收集。
- **歷史 bundle 只在本機**：PD-8（Prometheus 不可達→`SKIPPED.txt`）引的是 2026-07-07 的歷史 bundle，同樣 gitignored、只在本機。這條斷言的可稽核性弱於當輪 dump 的其他八條。
- **版本非同一份**：如「版本錨點」所述，MF 這波驗的是 2026-06-30 的碼（無 `--prom-url`），PD 這波驗的是現行版。兩波之間工具改過（主要是加 prom 層）。2026-07-10 的完整 6-node 收集把 happy-path 行為在現行版複驗過，但 **MF-2…MF-5 的故障注入本身沒有在現行版重跑**——若之後改動觸及 exit code 語意，需重驗。
- **prom dump 的 job 與樣本密度**：本輪 Prometheus 只 scrape 了 `ceph` 一個 job（預設 `ceph|node` 的 `node` job 在此 stack 不存在，`jobs_matched=ceph` 是正確行為）；多 job fan-out 只有單元層。24h 窗內也只有 stack 存活的 ~25 分鐘有樣本——dump 的是「窗內既有的所有樣本」，行為正確，但**樣本密度不代表長期運行下的檔案大小**。
- **MF-4 用縮減 inventory**：node 不可達那場的 `node_ok=2 node_failed=1` 顯示該輪用了比 6-host 小的 inventory 來隔離假 host；斷言的是「假 host 被 SKIPPED、真 node 照收」的行為，不是 6-host 規模。
- **Rook／external 拓樸沒有 T3**：本 lab 是純 cephadm、無 kubectl node，Rook 層的 remote／local kubectl、external namespace 只有單元層背書（`test-rook-collector.sh`）。真 Rook external 叢集上的行為未在真機演練。

## 7. 開放問題

- **exit code 語意在現行版的真機重驗**：MF-2…MF-5 的故障注入只在 2026-06-30 的碼跑過；現行版（含 prom 層）的完整 multi-fault 重跑尚未做。下次動到 `collect.sh` 的 rc 匯總邏輯時應補。
- **Rook external 真機 dump**：`.160` 的 single-node k0s＋Rook external 拓樸可作為 Rook 層 remote/local kubectl 的真機驗證環境，目前只有單元層。
- **prom budget 截斷的真機演練**：`TRUNCATED` 路徑目前只有 fake curl 覆蓋；要真機演練需刻意用極大時間窗或極小 `--prom-timeout` 逼出截斷。
- **redaction 對半敏感資料的邊界**：redaction 守住金鑰材料，但內部 IP／hostname／pool name／image name 未涵蓋；若要把 bundle 給外部人，仍需人工資料外洩檢查。這是設計取捨（redaction 不是 DLP），不是待修 bug，但值得記著。
