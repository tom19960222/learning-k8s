## 場景

你看 `ceph osd tree` 或 decompile CRUSH map 時，常會看到 bucket 後面寫著 `alg straw2`。下一個直覺問題是：CRUSH 不是已經把 PG 算到 OSD 了嗎，為什麼還有 straw / straw2？

關鍵是 CRUSH 不是一次把 PG 直接丟到某顆 OSD。它會沿著拓樸樹一層一層挑：root 挑 rack、rack 挑 host、host 再挑 osd。**straw / straw2 就是每一個 bucket 裡「怎麼從多個 child item 挑出一個」的演算法**。

```
PG id x + replica position r
        |
        v
crush_do_rule(rule, x)
        |
        v
rule step: take root -> chooseleaf firstn type host
        |
        v
每層 bucket 都跑一次 bucket choose
        |
        v
straw2：每個 child 抽一支依 weight 修正的籤，分數最高者勝出
```

這件事會直接影響兩個線上現象：第一，OSD weight 不同時資料大致按比例分散；第二，加 OSD、拔 OSD、reweight OSD 時，只有必要的 PG 需要重 placement，不會整個 cluster 洗牌。

## CRUSH bucket 在哪裡被挑選

`crush_do_rule()` 是 placement 的入口。傳入 rule、input `x`、weight vector 後，它會進到一般 rule 或 MSR rule；一般 replicated pool 走的是 `crush_do_rule_no_retry()`。在 rule step 遇到 `choose` / `chooseleaf` 時，才會呼叫 `crush_choose_firstn()` 或 `crush_choose_indep()`，再進一步進到每個 bucket 的 choose algorithm。

真正分派到 `straw` 或 `straw2` 的地方在 `crush_bucket_choose()`。同一個 CRUSH rule 不需要知道 bucket 用哪種 algorithm，因為 bucket 自己的 `h.alg` 決定要走哪段邏輯。

File: ceph/src/crush/mapper.c (line 368)

```c
static int crush_bucket_choose(const struct crush_bucket *in,
			       struct crush_work_bucket *work,
			       int x, int r,
			       const struct crush_choose_arg *arg,
			       int position)
{
	switch (in->alg) {
	case CRUSH_BUCKET_STRAW:
		return bucket_straw_choose(
			(const struct crush_bucket_straw *)in, x, r);
	case CRUSH_BUCKET_STRAW2:
		return bucket_straw2_choose(
			(const struct crush_bucket_straw2 *)in,
			x, r, arg, position);
	}
}
```

所以 straw / straw2 不是「另一套 CRUSH」，而是 CRUSH traversal 裡某一層 bucket 的局部選擇器。

## straw：先算好每支籤的長度

舊的 `straw` bucket 會在 CRUSH map 內同時保存兩組資料：每個 item 的 `item_weights`，以及預先算好的 `straws`。選擇時，每個 item 用同一個 input 算 hash，取低 16 bits，再乘上自己的 `straws[i]`；乘完後分數最高的 item 勝出。

File: ceph/src/crush/crush.h (line 328)

```c
struct crush_bucket_straw {
	struct crush_bucket h;
	__u32 *item_weights;   /* 16-bit fixed point */
	__u32 *straws;         /* 16-bit fixed point */
};
```

File: ceph/src/crush/mapper.c (line 208)

```c
static int bucket_straw_choose(const struct crush_bucket_straw *bucket,
			       int x, int r)
{
	for (i = 0; i < bucket->h.size; i++) {
		draw = crush_hash32_3(bucket->h.hash, x, bucket->h.items[i], r);
		draw &= 0xffff;
		draw *= bucket->straws[i];
		if (i == 0 || draw > high_draw) {
			high = i;
			high_draw = draw;
		}
	}
	return bucket->h.items[high];
}
```

這個做法的問題不是「抽籤」這個直覺錯，而是 `straws` 的預先計算很難在所有 weight 組合下都保持理想行為。`builder.c` 裡 `crush_calc_straw()` 的註解已經直接說明：這段舊邏輯是 fundamentally flawed，後面才用 `straw_calc_version` 修正部分 scaler 問題。

File: ceph/src/crush/builder.c (line 430)

```c

{
	...

		...

		...

}
```

`straw_calc_version=1` 修的是舊 straw 的 scaler 計算，不是把 `straw` 變成 `straw2`。在 v19.2.3 的 `crush.h` 裡，這個 tunable 被標成 backward compatibility，註解也寫著 `straw2 replaces it`。

## straw2：每次抽籤時才用 weight 算分數

`straw2` 的 struct 少了一個關鍵欄位：它只保存 `item_weights`，不再保存預先算好的 `straws`。也就是說，weight 是持久狀態；每次 placement 的 draw 是用 `x`、item id、replica position `r` 和 weight 即時計算。

File: ceph/src/crush/crush.h (line 341)

```c
struct crush_bucket_straw2 {
	struct crush_bucket h; /*!< generic bucket information */
	__u32 *item_weights;   /*!< 16.16 fixed point weight for each item */
};
```

`straw2` 的 source 註解把直覺寫得很清楚：如果 osd.0 weight 是 8、osd.1 weight 是 4，抽樣結果會讓 PG 數量大致按 weight 比例分布。實作上，它用 hash 產生一個 uniform random 值，再透過 `crush_ln()` 轉成 exponential distribution 的 draw；因為 `ln` 值是負的，weight 越大，除完後越接近 0，也就越容易成為最高分。

File: ceph/src/crush/mapper.c (line 315)

```c
static inline __s64 generate_exponential_distribution(int type, int x, int y,
						      int z, int weight)
{
	unsigned int u = crush_hash32_3(type, x, y, z);
	u &= 0xffff;

	__s64 ln = crush_ln(u) - 0x1000000000000ll;

	/*
	 * divide by 16.16 fixed-point weight.  note
	 * that the ln value is negative, so a larger
	 * weight means a larger (less negative) value
	 * for draw.
	 */
	return div64_s64(ln, weight);
}
```

最後 `bucket_straw2_choose()` 對 bucket 內每個 item 算一次 draw，挑最大者。weight 為 0 的 item 直接給 `S64_MIN`，等於不可能勝出。

File: ceph/src/crush/mapper.c (line 342)

```c
static int bucket_straw2_choose(const struct crush_bucket_straw2 *bucket,
				int x, int r, const struct crush_choose_arg *arg,
				int position)
{
	__u32 *weights = get_choose_arg_weights(bucket, arg, position);
	__s32 *ids = get_choose_arg_ids(bucket, arg);
	for (i = 0; i < bucket->h.size; i++) {
		if (weights[i]) {
			draw = generate_exponential_distribution(
				bucket->h.hash, x, ids[i], r, weights[i]);
		} else {
			draw = S64_MIN;
		}
		if (i == 0 || draw > high_draw) {
			high = i;
			high_draw = draw;
		}
	}
	return bucket->h.items[high];
}
```

這也是 straw2 比 modular hashing 好懂的地方：不是 `hash(x) % N`。每個 item 都有一個自己的分數，新增 item 只是新增一個競爭者；舊 item 彼此之間的分數沒有被重新排序。

## 為什麼 reweight 只搬必要的 PG

把 `straw2` 想成每個 item 都有自己的 draw：

```
item A: draw = hash(x, A, r) / weight(A)
item B: draw = hash(x, B, r) / weight(B)
item C: draw = hash(x, C, r) / weight(C)

最高分勝出
```

如果你只改 item B 的 weight，A 和 C 的 draw 沒變，A 與 C 之間誰高誰低也沒變。因此 mapping 只應該在「B 與其他 item」之間移動，不應該從 A 跑到 C。這就是 CRUSH 所謂 stable movement 在 bucket 內的核心。

Ceph 的測試直接鎖住這個行為。`straw2_reweight` 先用舊 weight 算一次 `out0`，再改某個 item 的 weight 算 `out1`；如果新結果不是被改 weight 的 item，而且舊結果也不是它，測試要求新舊結果必須相同。

## 邊界與除錯

- `straw2` 解決的是 bucket 內 item add/remove/reweight 時的資料移動範圍；它不代表 OSD utilization 會完美相同。PG 數、weight、CRUSH rule、failure domain 都仍然會影響實際分布。
- `straw_calc_version` 只影響舊 `straw` bucket 的 scaler 計算；看到 `straw_calc_version=1` 不等於所有 bucket 都是 `straw2`。
- 把 bucket 從 `straw` 改成 `straw2` 是 CRUSH map 變更，會造成 placement 重新計算。即使 source 裡提供 `set-all-straw-buckets-to-straw2`，也應先在測試 map 上估算 movement，再套到 production cluster。
- 如果你在讀 CRUSH map 時看到 `choose_args` 或 weight-set，記得它們只在 straw2 bucket 路徑被使用；`crush_bucket_choose()` 傳給 `bucket_straw2_choose()` 的 `arg` 不會進入舊 `bucket_straw_choose()`。
