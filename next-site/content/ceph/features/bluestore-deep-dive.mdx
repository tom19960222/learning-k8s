## 場景

你已經知道 Ceph 上層概念：MON 維 cluster state、OSD 真的存資料、CRUSH 決定 object 落到哪幾顆 OSD。今天的問題是 OSD **內部那一層**：

- client 寫 4 KB 跟寫 4 MB 走的路徑不一樣，差在哪？
- `block.db` 滿了出現 `BLUEFS_SPILLOVER` 警告，是誰算的、影響什麼？
- `ceph -s` 偶爾跳 `BLUESTORE_SLOW_OP_ALERT`，從 BlueStore 角度看是哪段卡？
- 為什麼 `/var/lib/ceph/osd/ceph-N/block` 是個 symlink、目錄本身又是 tmpfs？

這頁不重複講「BlueStore 是 OSD backend」這種定義（[osd-and-bluestore](./osd-and-bluestore) 已經建立過骨架了），而是把 BlueStore 內部從 device 布局、metadata schema、寫入路徑、到背景流程，逐層對到 v19.2.3 原始碼。

<Callout type="info" title="前置閱讀">
  本頁假設你已讀過 [osd-and-bluestore](./osd-and-bluestore) 的 OSD daemon 大致結構，並對 `ObjectStore` 抽象、PG、RADOS write op flow 有印象。
</Callout>

## BlueStore 在 OSD 內的位置

```
ceph-osd process
  │
  ├── Messenger          ← 收 client / peer OSD op
  ├── OpThreadPool       ← dispatch 到 PG
  ├── PG (PrimaryLogPG)  ← op handling, replication, pg log
  │
  └── ObjectStore (抽象介面)
       └── BlueStore     ← 本頁主題
            ├── BlockDevice (bdev)      ← raw block IO (libaio / io_uring)
            ├── Allocator               ← free space 管理
            ├── RocksDB (db)            ← 所有 metadata
            └── BlueFS                  ← 服務 RocksDB 的極簡 FS
```

`BlueStore` 是 `ObjectStore` 的實作之一：

File: src/os/bluestore/BlueStore.h (line 236)

```cpp
class BlueStore : public ObjectStore,
                  public md_config_obs_t {
  // ...
};
```

PG 透過 `ObjectStore::Transaction` 把寫入意圖（write、setattr、omap_setkeys、remove ...）丟給 BlueStore，BlueStore 負責把這份 transaction 同時落到 **block device 上的 object data** 與 **RocksDB 裡的 metadata**，並維持兩者原子。

## 三裝置布局：BDEV_WAL / BDEV_DB / BDEV_SLOW

BlueStore 對外可以看到一顆 disk，也可以看到三顆。三個角色都在 BlueFS 端定義：

File: src/os/bluestore/BlueFS.h (line 252-257)

```cpp
static constexpr unsigned MAX_BDEV = 5;
static constexpr unsigned BDEV_WAL  = 0;
static constexpr unsigned BDEV_DB   = 1;
static constexpr unsigned BDEV_SLOW = 2;
static constexpr unsigned BDEV_NEWWAL = 3;   // device migration target
static constexpr unsigned BDEV_NEWDB  = 4;
```

同檔案的註解講得很直接：

File: src/os/bluestore/BlueFS.h (line 516-521)

```cpp
/*
 * There are up to 3 block devices:
 *
 *  BDEV_DB   db/      - the primary db device
 *  BDEV_WAL  db.wal/  - a small, fast device, specifically for the WAL
 *  BDEV_SLOW db.slow/ - a big, slow device, to spill over to as BDEV_DB fills
 */
std::vector<BlockDevice*> bdev;
std::vector<IOContext*> ioc;
std::vector<Allocator*> alloc;
```

注意這裡的角色分派：

| 你看到的 symlink | BlueFS 內的角色 | 慣例上放的硬體 |
|---|---|---|
| `block` | **BDEV_SLOW**（同時也是 object data 主裝置） | HDD / SSD / NVMe，看 cluster 設計 |
| `block.db` | **BDEV_DB**（RocksDB SST / 部分 metadata） | NVMe / SSD（比 block 快） |
| `block.wal` | **BDEV_WAL**（RocksDB WAL） | 又比 block.db 更快或同等的 NVMe |

只給一顆 disk 時，BlueFS 把 RocksDB 的 SST 跟 WAL 都塞到 BDEV_SLOW 上（也就是 `block`），DB device 並沒消失只是「shared」。`_open_bluefs()` 在做 volume selector 時就會根據三裝置實際大小決定哪些 SST level 放哪裡：

File: src/os/bluestore/BlueStore.cc (line 7628-7647)

```cpp

} else {
  // RocksDBBlueFSVolumeSelector：根據 write_buffer_size、max_bytes_for_level_base

```

這就是「為什麼 block.db 給太小會 spillover」的程式入口 — volume selector 拿到 BDEV_DB 容量後，發現某個 SST level 放不下，就 fallback 到 BDEV_SLOW。

## OSD 目錄為什麼是 tmpfs + symlink

實體上 `block` / `block.db` / `block.wal` 都是 LVM logical volume（或 raw partition），不是檔案。OSD 目錄只是把它們以 symlink 形式擺起來，方便 ceph-osd 進程 open。`ceph-volume` 做的就是這件事：

File: src/ceph-volume/ceph_volume/util/prepare.py (line 190-206)

```python

        'mount', '-t', 'tmpfs', 'tmpfs', path

def create_osd_path(osd_id, tmpfs=False):
    path = '/var/lib/ceph/osd/%s-%s' % (conf.cluster, osd_id)
    system.mkdir_p('/var/lib/ceph/osd/%s-%s' % (conf.cluster, osd_id))

        mount_tmpfs(path)
```

File: src/ceph-volume/ceph_volume/util/prepare.py (line 304-318)

```python
def _link_device(device, device_type, osd_id):
    device_path = '/var/lib/ceph/osd/%s-%s/%s' % (
        conf.cluster,
        osd_id,
        device_type
    )
    command = ['ln', '-s', device, device_path]
    system.chown(device)
    process.run(command)
```

`device_type` 會是 `block` / `block.db` / `block.wal`。**OSD 目錄本身不存資料** — 跑在 tmpfs 上，重開機就消失，每次 `ceph-volume lvm activate` 重建。真正的東西在那幾個 symlink 指向的 block device 裡。所以你 `ls -lh /var/lib/ceph/osd/ceph-N/` 看到的容量總是很小，不必擔心。

## RocksDB key prefix：metadata 的 schema

BlueStore 把所有 metadata 都塞進同一個 RocksDB instance，用 single-letter prefix 區分用途：

File: src/os/bluestore/BlueStore.cc (line 120-131)

```cpp
const string PREFIX_SUPER        = "S";  // field -> value
const string PREFIX_STAT         = "T";  // field -> value(int64 array)
const string PREFIX_COLL         = "C";  // collection name -> cnode_t
const string PREFIX_OBJ          = "O";  // object name -> onode_t
const string PREFIX_OMAP         = "M";  // u64 + keyname -> value
const string PREFIX_PGMETA_OMAP  = "P";  // u64 + keyname -> value(for meta coll)
const string PREFIX_PERPOOL_OMAP = "m";  // s64 + u64 + keyname -> value
const string PREFIX_PERPG_OMAP   = "p";  // u64(pool) + u32(hash) + u64(id) + keyname -> value
const string PREFIX_DEFERRED     = "L";  // id -> deferred_transaction_t
const string PREFIX_ALLOC        = "B";  // u64 offset -> u64 length (freelist)
const string PREFIX_ALLOC_BITMAP = "b";  // (see BitmapFreelistManager)
const string PREFIX_SHARED_BLOB  = "X";  // u64 SB id -> shared_blob_t
```

讀這張表就知道「BlueStore 重啟時要從 RocksDB 還原什麼狀態」：

- `O`（onode）：每個 RADOS object 的 inline metadata — size、xattr、extent map 指到 block device 上哪段、checksum 種類⋯⋯ 一個 `O:` key 對到一個 object，**這是 BlueStore 最熱的 key family**。
- `M` / `m` / `p`（omap）：RADOS 的 object map，CephFS dir entry、RGW bucket index 都是這種，量大時 omap-heavy。
- `L`（deferred transaction）：小寫先放這裡，後面會講。
- `B` / `b`（freelist / bitmap）：Allocator 的 on-disk 狀態。OSD crash 後 BlueStore 用 `B` 重建 in-memory Allocator。
- `X`（shared blob）：snapshot / clone 共用的 extent，CoW 用。

`SHARED_BLOB` 是 BlueStore 對 snapshot 的關鍵：rbd snapshot 不抄一份，而是讓兩個 onode 指向同一個 shared blob，等到某邊寫時才 CoW 分裂。RocksDB 裡多一筆 `X:` 紀錄就夠了，比 FileStore 用 hard-link xfs file 乾淨得多。

## BlueFS：為什麼要再做一個 FS

RocksDB 是「LSM 引擎」，但它假設下面有 POSIX filesystem 給它 `open()` / `write()` / `fsync()` / `rename()`。BlueStore 不想用 xfs，所以做了 BlueFS — 極簡 user-space FS，只要應付 RocksDB 的呼叫模式即可：

- 沒有 hard link、沒有 rename across dir、沒有 inode permission；只有 `dir/file` 兩層。
- File 切成多個 `bluefs_extent_t`，**這些 extent 可以橫跨 BDEV_WAL / BDEV_DB / BDEV_SLOW**，由 volume selector 決定該放哪。
- BlueFS 的 metadata 跟 transaction log 寫在自己的 log file 上，重啟靠 replay。

當你看到 `BLUEFS_SPILLOVER`：

## 寫入路徑：TransContext 狀態機

OSD 把一個 `Transaction` 丟進來，BlueStore 包成 `TransContext`（簡稱 txc），用一個確定性狀態機推到 finish。狀態定義在 BlueStore.h：

File: src/os/bluestore/BlueStore.h (line 1834-1860)

```cpp
typedef enum {
  STATE_PREPARE,
  STATE_AIO_WAIT,
  STATE_IO_DONE,
  STATE_KV_QUEUED,        // queued for kv_sync_thread submission
  STATE_KV_SUBMITTED,     // submitted to kv; not yet synced
  STATE_KV_DONE,
  STATE_DEFERRED_QUEUED,  // in deferred_queue (pending or running)
  STATE_DEFERRED_CLEANUP, // remove deferred kv record
  STATE_DEFERRED_DONE,
  STATE_FINISHING,
  STATE_DONE,
} state_t;
```

主推進器 `_txc_state_proc` 是個無限迴圈 + switch，每個 state 做完事就 `set_state()` 跳下一個：

File: src/os/bluestore/BlueStore.cc (line 14149-14245)

```cpp
void BlueStore::_txc_state_proc(TransContext *txc)
{
  while (true) {
    switch (txc->get_state()) {
    case TransContext::STATE_PREPARE:
      if (txc->ioc.has_pending_aios()) {
        txc->set_state(TransContext::STATE_AIO_WAIT);
        txc->had_ios = true;
        _txc_aio_submit(txc);
        return;
      }
      // ** fall-thru **

    case TransContext::STATE_AIO_WAIT:
      // ...
      _txc_finish_io(txc);  // may trigger blocked txc's too
      return;

    case TransContext::STATE_IO_DONE:
      // ...
      txc->set_state(TransContext::STATE_KV_QUEUED);
      // push to kv_queue; kv_sync_thread 會處理
      return;

    case TransContext::STATE_KV_SUBMITTED:
      _txc_committed_kv(txc);
      // fall-thru
    case TransContext::STATE_KV_DONE:
      if (txc->deferred_txn) {
        txc->set_state(TransContext::STATE_DEFERRED_QUEUED);
        _deferred_queue(txc);
        return;
      }
      txc->set_state(TransContext::STATE_FINISHING);
      break;
    // ...
    }
  }
}
```

每段 state 都被 `BlueStoreThrottle::log_state_latency` 計時。`ceph daemon osd.N perf dump` 裡 `state_kv_queued_lat` / `state_aio_wait_lat` 這些指標就是從這裡 emit 的 — 哪段慢，直接對應到上面哪個 case label。

簡化 path：

```
client write (RADOS op)
  │
  ▼
PrimaryLogPG.do_op → 建 ObjectStore::Transaction
  │
  ▼
BlueStore::queue_transactions → 建 TransContext (txc)
  │
  ▼  STATE_PREPARE
  │   _do_write_data 把 IO 拆成 small/big、決定 deferred or direct
  │
  ▼  STATE_AIO_WAIT      (有 direct write 要等 libaio 完成)
  ▼  STATE_IO_DONE       (block device 寫好了)
  ▼  STATE_KV_QUEUED     (推到 kv_queue)
  │
  │   kv_sync_thread 拉一批 txc，組 RocksDB transaction，commit
  │
  ▼  STATE_KV_SUBMITTED  (RocksDB 寫好但還沒 sync)
  ▼  STATE_KV_DONE       (RocksDB sync 完成 → 可以回 ack)
  │
  ▼  if deferred: STATE_DEFERRED_QUEUED → 之後 batch flush
  │
  ▼  STATE_FINISHING → STATE_DONE
```

關鍵：**ack 給 client 是在 `STATE_KV_DONE` 之後**（OSD 收到 callback 才回上層）。也就是 BlueStore 要先把 metadata sync 進 RocksDB 才算「committed」。對 HDD OSD，這個 RocksDB sync 大半是發生在 BDEV_DB / BDEV_WAL 上 — 如果 DB 在 NVMe，metadata sync 快很多，這就是 separate DB device 對 HDD 的最大價值。

## Small write vs big write：min_alloc_size 是分水嶺

`_do_write_data` 是 `BlueStore::write()` 的核心。它依 `min_alloc_size` 把寫入切三段：

File: src/os/bluestore/BlueStore.cc (line 17148-17188)

```cpp
void BlueStore::_do_write_data(
  TransContext *txc, CollectionRef& c, OnodeRef& o,
  uint64_t offset, uint64_t length, bufferlist& bl,
  WriteContext *wctx)
{
  uint64_t end = offset + length;
  bufferlist::iterator p = bl.begin();

  if (offset / min_alloc_size == (end - 1) / min_alloc_size &&
      (length != min_alloc_size)) {
    // we fall within the same block
    _do_write_small(txc, c, o, offset, length, p, wctx);
  } else {
    uint64_t head_offset, head_length;
    uint64_t middle_offset, middle_length;
    uint64_t tail_offset, tail_length;

    head_offset = offset;
    head_length = p2nphase(offset, min_alloc_size);
    tail_offset = p2align(end, min_alloc_size);
    tail_length = p2phase(end, min_alloc_size);
    middle_offset = head_offset + head_length;
    middle_length = length - head_length - tail_length;

    if (head_length) _do_write_small(txc, c, o, head_offset, head_length, p, wctx);
    _do_write_big(txc, c, o, middle_offset, middle_length, p, wctx);
    if (tail_length) _do_write_small(txc, c, o, tail_offset, tail_length, p, wctx);
  }
}
```

整段邏輯就是「把 write 切成 head（unaligned）+ middle（aligned 大塊）+ tail（unaligned）」，head/tail 走 small path、middle 走 big path。

`min_alloc_size` 在 mkfs 時決定，predefault 是 4 KiB（HDD 與 SSD 都一樣）：

## Deferred write：避免 read-modify-write 的小 IO

`_do_write_small`（length < min_alloc_size）會評估能不能塞進現存 blob：如果可以，且寫入大小小於 `prefer_deferred_size`，就**不立刻寫 block device**，而是把這次 IO 先記到 RocksDB 的 `PREFIX_DEFERRED` (`L:`) 下，等之後批次 flush。

File: src/os/bluestore/BlueStore.cc (line 16170-16182)

```cpp

    dout(20) << __func__ << " deferring small 0x" << std::hex
             << b_len << std::dec << " unused write via deferred" << dendl;
    bluestore_deferred_op_t *op = _get_deferred_op(txc, bl.length());
    op->op = bluestore_deferred_op_t::OP_WRITE;
    b->get_blob().map(

        return 0;
```

`prefer_deferred_size` 的預設值 HDD 跟 SSD 完全不一樣：

File: src/common/options/global.yaml.in (line 4546-4564)


也就是說 **HDD OSD 預設把 64 KiB 以下的小寫先進 deferred queue，SSD/NVMe OSD 預設整個關掉 deferred**。理由很合理：HDD 的 seek 成本高，把多個小寫批起來再下盤可以省 IOPS；SSD 沒這問題，直接寫快多了。


`block_size_order` 通常是 12（= 4 KiB），所以每 4 KiB 一個 crc32c。

<Callout type="tip" title="`bluestore_retry_disk_reads`">
  csum mismatch 不會一次就回 EIO — 預設會 retry 3 次（`bluestore_retry_disk_reads = 3`，定義在 global.yaml.in:4471-4482），目的是處理偶發的 spurious read error。連續 retry 都不過才認定為真的壞。
</Callout>

`passive` 跟 `aggressive` 都會看 client 給的 `alloc_hint`（RBD / RGW 寫入時可以 hint）。然後在 `_do_alloc_write` 真的壓：

File: src/os/bluestore/BlueStore.cc (line 16830-16864)

```cpp
int r = c->compress(wi.bl, t, compressor_message);
uint64_t want_len_raw = wi.blob_length * crr;
uint64_t want_len = p2roundup(want_len_raw, min_alloc_size);
// ...
uint64_t result_len = p2roundup(compressed_len, min_alloc_size);
if (r == 0 && result_len <= want_len && result_len < wi.blob_length) {
  // ... 接受壓縮結果
  txc->statfs_delta.compressed() += compressed_len;
  txc->statfs_delta.compressed_original() += wi.blob_length;
  txc->statfs_delta.compressed_allocated() += result_len;
}
```

注意三件事：

1. **壓縮失敗或壓不下** (`result_len > want_len`)，BlueStore 會拒收壓縮結果、直接寫原始 data。`l_bluestore_compress_rejected_count` 這個 counter 就是抓這種情況。
2. **單位是 blob，不是 transaction**。同一個 write 可能產生多個 blob，每個獨立決定壓不壓 — `_choose_write_options` 那段註解寫得明白：「All blobs will be deferred or none will」，但 compression 是 per-blob。
3. **`statfs_delta` 三個值** (`compressed` / `compressed_original` / `compressed_allocated`) 對應到 `ceph df` 看到的壓縮統計。

## 背景：BLUESTORE_SLOW_OP_ALERT

如果 BlueStore 偵測到 op 卡太久，會主動 emit alert：

File: src/os/bluestore/BlueStore.cc (line 18901-18907)

```cpp
if (cct->_conf->bluestore_slow_ops_warn_threshold) {
  size_t qsize = _trim_slow_op_event_queue(mono_clock::now());
  if (qsize >= cct->_conf->bluestore_slow_ops_warn_threshold) {
    ostringstream ss;
    ss << "observed slow operation indications in BlueStore";

```

結合上面的 state machine：`state_aio_wait_lat`（block device）與 `state_kv_queued_lat` / `state_kv_submitted_lat`（RocksDB）哪段尖，幾乎決定你該往哪裡查。AIO wait 高 → block device 慢；KV queued 高 → RocksDB compaction 或 BDEV_DB 滿。
