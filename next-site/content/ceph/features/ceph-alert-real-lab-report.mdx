- **rule 名**：`CephClientBlocked`（client I/O 被擋）、`CephClientRisk`（catch-all 風險 check）、`CephMonQuorumLost`（quorum 生命線）、`CephMonDownScoped`／`CephOSDDaemonDownScoped`／`CephOSDHostDownScoped`（帶 `hostname`／`ceph_daemon` 的 scoped down）、`CephMetricsAbsent`／`CephExporterAllDown`（監控盲區）、`CephMgrNoStandby`（failover 能力）、`CephLowPriorityNotice`（低優先 slack-only）、`CephOSDSlowHeartbeat`／`CephOSDFlapping`／`CephMonClockSkew`／`CephOSDLatencyOutlier`／`CephDaemonSlowOps`／`CephDaemonRecentCrash`／`CephOSDNearFull`／`CephOSDBackfillFull`／`CephMonDiskLow`／`CephMonDiskCritical`／`CephPoolNearQuota`／`CephCapacityForecast`／`CephDataDamage`／`CephObjectUnfound`／`CephPGUnhealthyStates`、`Watchdog`（恆 firing 心跳）、`CephHealthError`（保留的 upstream 預設）、`CephMonQuorumLostExternal`（out-of-band quorum 探針，TD-05 產出）。每條的完整 expr 與 routing 在 [policy 頁](./ceph-alert-policy-current)。
- **promtool／amtool**：Prometheus 與 Alertmanager 的官方 CLI。`promtool test rules` 跑規則的單元測試（dev 端 Tier A）；`amtool` 驗 routing 合約與下 silence（Tier B／C）。
- **空 vector bug（dev 代號 F1）**：dev 期 promtool 抓到的規則層 bug——早期 `count(ceph_mon_quorum_status == 1) < 2` 在「全部 mon 都 = 0」時，內層 `==1` 過濾成空 vector、`count(空)` 是「無 sample」而非 0，比較算不出結果、規則靜默。用 `or vector(0)` 修好（見下一條）。§5.2 拿它跟發現二對照層級。
- **`or vector(0)`／empty-series 路徑**：PromQL 手法——左式評估不出任何 series 時 fallback 成常數 0，讓「series 整個消失」也能參與比較而觸發告警。本頁說某場「走 empty-series 路徑」，指的是 exporter 死掉、series 變空之後靠這個 fallback 讓規則 fire。
- **inhibit**：Alertmanager 的抑制機制——某條 alert firing 時**壓制**另一條相關 alert 的通知（例：quorum-lost 壓掉個別 mon-down，免得重複吵），被壓的 alert 狀態顯示 `suppressed`。
- **`predict_linear`**：PromQL 函式，對一段時間窗的序列做線性回歸、外推未來值。RL-18 用它把用量成長斜率外推 72 小時驗容量預測告警。
- **sentinel object**：測試 pool 裡刻意放的一顆「定位用」object——先查出它的 acting set（負責存它的那組 OSD），故障注入就能精準打中會影響它的那幾顆 OSD（RL-02／RL-19／RL-20 用此法）。
- **k0s／Rook external**：k0s 是輕量 Kubernetes 發行版；lab 用一台單 node k0s 跑 Rook operator、以 **external 模式**連到這座 cephadm 叢集。Rook 的 `CephCluster` 資源回報 `Connected`＋`HEALTH_OK`，被當成一路獨立的健康 gate。

<Callout type="warning" title="一個必守的讀法：「22/22」不是一次 snapshot 驗完">
  本頁最容易被誤讀的一句就是「22/22 通過」。正確解讀是——**每一場都在「它自己的最終 rules 版本」下通過，而不是拿單一 rules snapshot 一次跑綠全部**。這 22 場橫跨 **3 個 rules commit**（`5a01a77` → `efcf5bb` → `23314bc`，時間軸見 [§1](#rules-集版本演進22-場橫跨-3-個-commit)）：slow-ops 場（RL-01）的最終 PASS 用的是 `efcf5bb` 重設計後的 `CephDaemonSlowOps`（前三次即因舊版 spiky gauge 失敗）、data-damage 場（RL-19）的 PASS 用的是 `23314bc` 窗化後的 `CephDataDamage`（前兩次失敗）。所以**不要**把它壓縮成「一次驗完 27（或 29）條規則」——那是不實的。另外 **latency-outlier 場（RL-11）是 `PASS*`**：正向斷言全過，但 in-run 只復原到 `HEALTH_WARN`（BlueStore latch），`HEALTH_OK` 是跨 run 的證據。
</Callout>

## 1. 前提

### 緣起：dev 四層證明「規則邏輯對」，真機證明「ceph 的事實對不對得上規則的假設」

這套 alert 設計在 dev 端已用四層測試（promtool／amtool／live／真 ceph 的前三層）逐條驗過（見[驗證方法論頁](./ceph-alert-validation-methodology)）。但 dev 餵給 promtool 的是**手刻的合成 series**——它能精準驗 `for:` 邊界、`count()` 空 vector 語意、`unless` 去重，卻**無法驗「ceph 在那個情境到底會不會升那個 health check、升的是哪個名字、metric 在真故障當下長什麼樣」**。本頁補的就是那一半：

1. **Tier D 先導（TD-01…TD-05）**：先確認 ceph 真的匯出設計假設的 label、維護／故障實際升哪些 health check、scoped silence／去重／monmap 持久化在真機成不成立。
2. **real-lab 22 場（RL-01…RL-22）**：把每條 alert 用**現實世界會發生的真故障**做出來，驗它 firing、送達正確 receiver、且叢集能乾淨復原。

### 判定標準（先於實驗定案）

real-lab 每場的 PASS 判準由 `scenario_verify` 原始碼定義，**五條全過才算 PASS**（逐場證據見 §3，全部對 `EVIDENCE-INDEX` 的 raw 檔）：

1. 每個目標 alert 在 `prometheus-alerts-<alert>-<label>.json` 有 `state="firing"` 的 row（含 label 匹配）；
2. 每個目標在 `sink-since-checkpoint.log` 有對應 `receiver`＋`alertname` 的 JSON row；
3. 每個 negative 斷言（`assert_..._not_firing` / `assert_sink_absent`）的目標在對應 raw 檔中**不存在** firing／row；
4. 有 ceph 端斷言的場，`health-check-<CHECK>.txt` 存在且含該 check 行；
5. `recovery/ceph-ready.txt` 回 `HEALTH_OK`、`rook-cephcluster-ready.txt` `Connected`＋`HEALTH_OK`、`prometheus-up-ceph.json` `up{job="ceph"}==1`。

latency-outlier 場（RL-11）卡在第 5 條（in-run 停在 `HEALTH_WARN`）→ 記為 `PASS*`。

**Tier D 的 `for:` 是縮短版（5m→30s）**：Tier D 是為了看「ceph 升哪些 check、規則接不接得上」，不是量 `for:` 邊界（那是 dev promtool 的工作）。為了讓每個 degradation 視窗短、能快速還原，Tier D 把 `for:` 縮成 30s。所以下方 TD 場景寫的「35s firing」是**縮短版**觀測值，讀作「規則確實接上並 fire 了」，**不要**讀成 production 的告警延遲。

### receiver 鏈

`alert-sink` 是一個 python webhook，Alertmanager 依 label routing 把 alert 送到三條 receiver：`pager`（`severity=critical` 的自訂 rule＋upstream `CephHealthError`）、`slack`（warning／info 與 aggregate context）、`watchdog`（只收恆 firing 的 `Watchdog`，餵外部 dead-man switch）。每場的 receiver 證據＝`sink-since-checkpoint.log` 裡的 JSON row（含 `receiver`／`alertname`／`severity`）。完整 routing 樹在 [policy 頁](./ceph-alert-policy-current)。

## 4. 16 個真機才抓到的發現（as-of 2026-07-06，保留原敘事）

這 16 個問題，fake 測試（`promtool`、unit test）**全部綠燈**，只有接上真硬體、真 mgr、真 cephadm shell 才會現形。它們幾乎都發生在 **rule 邏輯本身沒寫錯**的前提下——這正是「每條 alert 都要真機驗證」這條鐵律的價值：fake 測試測「expr 邏輯對不對」，真機測「這條 expr 接到的資料源在真故障當下長什麼樣子」，兩者是不同的驗證面。發現編號沿用舊頁（發現 01…16），括號內是對應的本頁場景。

<Callout type="info" title="這 16 個發現與 §2.3 ledger 缺口的對應">
  發現 01（crash 打錯行程＋sidecar 壞）對應 **G-3（crash 手動代送）**；發現 02（spiky gauge）與發現 14（data-integrity 窗化）是 **G-4（rules 橫跨 3 commit）**兩次修正的起因；發現 08（BlueStore 24h latch）是 latency-outlier 場只到 `HEALTH_WARN`（**G-1，復原證據跨 run**）的根因；發現 11／16（mgr 凍結、inhibit 不可觀察）對應 **G-2（synthetic inhibit）**並在 [§5.2 發現二](#52-發現二cephmonquorumlost-對真-quorum-loss-是盲的架構級規則層修不了) 有機制級展開。
</Callout>

**01 · daemon crash 打錯行程，而且這個 lab 的 crash sidecar 本身是壞的（RL-14 daemon-crash）**
- 症狀：`kill -SEGV` 打 systemd `MainPID`，OSD 乾淨重啟但沒產生 crash report，`RECENT_CRASH` 永遠不出現。
- 根因：`MainPID` 是 `conmon`（容器監控器）不是 ceph-osd；容器內 PID 1 是 `podman-init`、不轉發 `SIGSEGV`。更深一層：這個 lab 的 `ceph-crash` sidecar 根本沒有 keyring（`client.crash` auth 不存在），物理上 post 不了 crash。
- 修法：對 host 上 ceph-osd 真實 PID 送 `SIGSEGV` 讓真 crash meta 寫進 spool，再經 seed 節點 admin 執行 `ceph crash post`（真 crash 資料，只繞過壞掉的傳輸 daemon）。`RECENT_CRASH` 真的亮、alert 真 fire。

**02 · `CephDaemonSlowOps` 的指標太 spiky，`for:` 永遠湊不滿（RL-01 slow-ops · rule 重設計）**
- 症狀：製造了真實且持續的 slow-ops，cluster 級 `CephClientBlocked` 正常 fire，但 per-daemon `CephDaemonSlowOps` 逾時不 fire。
- 根因：對照 TSDB 發現 `ceph_daemon_health_metrics{type="SLOW_OPS"}` 峰值 30、min 0，整個視窗只有約 28.8% 時間 &gt;0——它是「當下超過 complaint-time 的 op 數」snapshot gauge，本質 spiky。原本 `for:` 要求連續多分鐘為真，在這種指標上永遠不可能滿足。
- 修法：改用 `max_over_time(...[5m]) &gt; 0` 搭配較短 `for:`——語意變成「該 daemon 近 5 分鐘內有過 slow op」。這是 rules 第二版 `efcf5bb` 的內容（§1 版本演進表），slow-ops 場前三次即因舊版失敗。

**03 · 節流太硬會讓 OSD 直接 flap；要「持續」的 slow-ops 得靠並發（RL-01 slow-ops）**
- 症狀：同樣的限速參數，一次讓 `SLOW_OPS` 穩定持續，一次卻爆發式高變異。
- 根因：把節流調更硬，被限速的 OSD 連 housekeeping I/O（心跳、metadata）都被卡住，直接 flap down——調硬換不到「持續」，只換到 OSD 掉線。
- 修法：保持 flap-safe 的節流強度，改用高並發寫入把 op queue 持續撐滿——queue 不排空，`SLOW_OPS` 就連續，也不增加 flap 風險。

**04 · cephadm shell 的 stderr banner 污染 JSON，`jq` 直接 parse error（RL-15 osd-flapping · RL-19 data-damage · RL-20 object-unfound）**
- 症狀：osd-flapping 的 down/up 輪詢永遠逾時。
- 根因：擷取指令用 `2>&1` 把 cephadm banner（`Inferring fsid…`）併進要被 `jq` 解析的 JSON，這幾行不以 `#` 開頭、既有過濾濾不掉，`jq` 回 `Invalid numeric literal`。fake 測試回乾淨 JSON，測不到。
- 修法：JSON 解析改純 stdout 重導（banner 走 stderr 到旁路 log）。同一修正一次補掉尚未真機跑的 data-damage、object-unfound 兩場同款問題。

**05 · flapping 太快，5 秒 scrape 抓不到中間狀態，`changes()` 只數到 2（RL-15 osd-flapping）**
- 症狀：`ceph osd tree` 確認 4 次 up/down transition 都完成，但 `CephOSDFlapping` 不 fire。
- 根因：`ceph osd tree`（mon 視角）比 mgr exporter 快；某個 up 只維持 4 秒，5 秒 scrape 沒把它記成獨立樣本，`changes()` 只數到 2。
- 修法：每次 transition 改成等 Prometheus 的 `ceph_osd_up` 真的反映該值之後再前進，確保 4 個狀態各被 scrape 成獨立樣本。

**06 · pool 超過 quota 要「持續的寫入壓力」才會升 `POOL_FULL`（RL-17 pool-quota）**
- 症狀：`CephPoolNearQuota` 正常 fire，但寫過 quota 後 `POOL_FULL` 超過 6 分鐘不出現。
- 根因：pool 超過 `max_bytes` 確實會升 `POOL_FULL`，但只有 client 持續嘗試寫入（寫到被 block）時 mon 才 flag；腳本原本剛超過就停手。
- 修法：過 quota 後改用 `timeout 30 rados put`（bounded，逾時 exit code 當「pool 已滿」訊號），持續寫並輪詢 `POOL_FULL`。

**07 · 想寫滿容量太慢；改用動態 ratio 貼合真實使用率（RL-16 capacity-ladder）**
- 症狀：想把某顆 OSD 寫到特定水位才調閾值，跑多輪只累積很小的量。
- 根因：叢集幾乎全空、低並發短時 bench 太慢；固定 ratio 對這麼低使用率本來就不會 trip。
- 修法：改成高並發寫到「最滿 OSD 到達某使用率」，再動態量測該使用率、把 nearfull/backfillfull/full 三級 ratio 設成它的某比例——自動保證低於實際使用率且三級有序。

**08 · BlueStore slow-op 是 24 小時的 latch，用 restart OSD 清除會弄壞 cephadm 的 inventory（RL-01 slow-ops · RL-11 latency-outlier · RL-16 capacity-ladder）**
- 症狀：限速類場景結束後留下 `BLUESTORE_SLOW_OP_ALERT`（多顆 OSD）卡住收尾要求的 `HEALTH_OK` gate；舊清理方式靠 restart OSD，卻讓 cephadm 把 daemon 狀態標成 `unknown`。
- 根因：這警告有 24 小時 latch；rolling restart 太粗暴、還讓 orchestrator 追蹤狀態不同步。
- 修法：暫時把 `bluestore_slow_ops_warn_lifetime` 調很短，讓警告在幾十秒內老化清除後再還原，不 restart 任何 OSD。抽成共用 helper。**注意 latency-outlier 場（RL-11）是這個 helper 補進去之前跑的**——所以它 in-run 停在 `HEALTH_WARN`（復原證據跨 run，§2.3）。

**09 · rollback 移除網路延遲時，自己把自己那條 SSH 連線弄斷（RL-12 net-slow-heartbeat）**
- 症狀：`CephOSDSlowHeartbeat` 正常 fire、`netem` 也確認移除、叢集回 `HEALTH_OK`，但腳本回報失敗。
- 根因：rollback 移除延遲的指令，是透過那條「正被延遲 1200ms 的介面」下 SSH 執行；delete 那一刻的網路變化擾亂了當下這條 SSH，連線異常斷開回傳非零，儘管 host 端已成功清除。
- 修法：rollback 的成功判斷改成事後用一條全新 SSH 連線輪詢「延遲確實不見了」，而不是相信會自我擾亂的刪除指令退出碼。

**10 · mgr failover 之後，standby 要幾十秒才重新註冊（RL-08 mgr-failover）**
- 症狀：`ceph mgr fail` 後立刻查 mgr 狀態，standby 欄位是空的，判定失敗。
- 根因：被降級的 mgr 需要數秒到數十秒才重新以 standby 身分註冊；腳本只查一次沒重試。
- 修法：改用有時間預算的輪詢，反覆檢查 standby 清單直到非空。順帶真機驗證 multi-mgr scrape——failover 過程 metrics 沒有中斷。

**11 · 真 quorum loss 時，mgr exporter 的遙測會「凍結」在舊世界（RL-03 mon-quorum-lost · 早期發現）**
- 症狀：真實停掉多台 mon 後，`ceph quorum_status` 已無法完成，但 Prometheus 查到的 `ceph_mon_quorum_status` 加總仍是原本數字。
- 根因：不是 PromQL 寫錯，而是**監控資料源還在回報舊世界**——單一 mgr exporter 視角在 quorum 崩潰時 stale。`CephMonQuorumLost` 最後靠 `or vector(0)` 的 empty-series path（exporter 也死、series 變空）才觸發。
- 修法：這正是後來補 `CephMetricsAbsent` 與 multi-mgr scrape 的理由。**Tier D 的真 quorum loss 先導場（TD-04）把這個機制看得更清楚，完整展開在 [§5.2](#52-發現二cephmonquorumlost-對真-quorum-loss-是盲的架構級規則層修不了)。**

**12 · 部署腳本只改 configmap，Prometheus 不會自動重載 rule（harness gap）**
- 症狀：改完 rule 的 expr、重新部署監控，Prometheus 卻還在跑舊 expr。
- 根因：套用設定變更在 Deployment spec 沒變時不會重啟 pod；configmap 改了，運行中的 Prometheus 不會自動重讀掛載的 rule 檔。
- 修法：任何 rule 變更後手動重啟 Prometheus deployment；已記入待辦，準備把部署腳本硬化成 apply 後自動 reload。

**13 · capacity-forecast 六輪纏鬥：真正的兇手是 `rados bench` 覆寫同名 object（RL-18 capacity-forecast）**
- 症狀：`CephCapacityForecast` 連續五輪不 fire——`predict_linear` 趨勢預測始終爬不到 85% 門檻。
- 層層剝開：表面像「寫入吞吐不夠」，換三個假設（PG 數太少／並發不夠／複寫倍數吃掉成長）都沒解決。
- 真根因：一次連續跑 10 分鐘的受控測試量到明顯高很多的成長率，但場景原本分輪跑、每輪重來。差別在——**`rados bench` 的分輪迴圈每輪都用固定 run 名稱，第二輪起就覆寫掉前一輪的同名 object**，pool 填滿第一輪後不再成長；連續跑的單一 bench 沒這問題。
- 修法：改成連續寫入（不分輪、不重複用同一組 run 名稱）。成長率立刻回高位、projection 越門檻、alert 真 fire。這是整輪最耗工的一次除錯（本場九次重試，§2.3），而 rule 本身從頭到尾都對——它正確地拒絕在不夠陡的趨勢上 fire。

**14 · data-damage 三層疊：health_detail 閃爍＋objectstore 的 remove 被 recovery 治好＋`HEALTH_ERR` 也閃爍（RL-19 data-damage）**
- 症狀：`CephDataDamage` 連續兩輪不 fire。
- 第一層（指標閃爍）：`objectstore-tool` 弄壞副本、deep-scrub 也抓到了，但 mgr 匯出的 `ceph_health_detail` 在整個視窗只有極少數 scrape 是 1——跟發現 02 同類 spiky snapshot gauge。**修法**：改窗化 `max_over_time(...[5m]) &gt; 0`（同一修正順便套 `CephObjectUnfound`）。這是 rules 第三版 `23314bc`（§1 版本演進表）。
- 第二層（recovery race）：窗化後仍不穩，因為用 `remove` 讓副本**缺少** object，OSD 重啟後 PG recovery 自動補回、跟 deep-scrub 形成競態。**修法**：改用 `set-bytes` **損毀內容**而非刪除——內容錯的副本 recovery 不會自動治好，能確定性觸發 scrub mismatch。
- 第三層（`HEALTH_ERR` 也閃爍）：`CephDataDamage` 終於 fire，但額外斷言的 `CephHealthError` 逾時——scrub-error 的 `HEALTH_ERR` 隨底層指標在 ERR 與 WARN 間跳。而 `CephHealthError` 早被別場（穩定的 `OSD_FULL`，即 capacity-ladder 場）驗過。**修法**：移除本場重複且不穩定的 `CephHealthError` 斷言。

**15 · `CephObjectUnfound` 預先套用窗化修正，省下一輪失敗（RL-20 object-unfound）**
- 判斷：修發現 14 時認出 `OBJECT_UNFOUND` 跟 `PG_DAMAGED`／`OSD_SCRUB_ERRORS` 同屬 mgr health_detail 的資料完整性類檢查，判斷同樣會閃爍，於是同一修正裡把 `CephObjectUnfound` 一併窗化。
- 結果：本場一次通過，省掉原本預期的一輪失敗。這是「把真機發現抽象成一整類問題、而非逐案修」的價值。

**16 · 真 quorum loss 當下，inhibit 觀察不到——exporter 凍結讓兩條 alert 永遠不重疊（RL-03 mon-quorum-lost）**
- 症狀：`CephMonQuorumLost` fire 並送到 pager，但「應該 inhibit `CephMonDownScoped`」的斷言逾時。
- 根因：這是發現 11 的延伸。真 quorum loss 時 mgr exporter 先凍結、再死掉；`CephMonQuorumLost` 只能靠 exporter 死後的 empty-series path 觸發，但那時 `ceph_mon_quorum_status` 已整個消失、`CephMonDownScoped` 無資料可算，兩條 alert 在真故障中永遠不會時間重疊，inhibit 無從被「觀察」。
- 修法：inhibit 設定本身沒問題——改成直接送兩筆 synthetic alert 到 Alertmanager，確認 `CephMonDownScoped` 立刻 suppressed、指向 `CephMonQuorumLost`（synthetic 驗證的保留，§2.3）。把「在真故障中觀察 inhibit」換成「驗證 Alertmanager 設定本身」，主 alert 仍由真 quorum loss 完整驗證。

## 5. 兩個機制級盲區（Tier D 先導揭露）

Tier D 兩個發現跟上面 16 個「量測方法／harness」層級的問題不同——它們是**設計假設 vs ceph 真實行為對不上**的機制級 bug，dev 四層全綠也抓不到。

### 5.1 發現一：OSD_FLAGS ≠ OSDMAP_FLAGS（整台維護先導場抓到，已修）

設計頁 `CephClientRisk` 的排除清單與 `CephLowPriorityNotice` 的 name 清單，當初寫的是 `OSDMAP_FLAGS`。意圖完全正確——「維護時自動設的 `noout` 會升一個 flag check，要排出 pager，否則例行維護吵醒人」。**意圖對，但 check 的名字寫錯了。**

真機分得很清楚（`ceph health detail` 證實）：

| 動作 | 升的 health check |
|---|---|
| `maintenance enter` 對 host **subtree** 設 `noout` | **`OSD_FLAGS`**（`host ceph-lab-osd-03 has flags noout`，per-subtree）|
| cluster-wide `ceph osd set noout` | `OSDMAP_FLAGS`（整個 osdmap 的 flag）|

設計排除的是後者，但 `maintenance enter` 升的是前者。結果：`OSD_FLAGS` 不在排除清單 → `CephClientRisk` 的 `name!~"..."` catch-all 把它接走 → `CephClientRisk{OSD_FLAGS}` 在整台維護先導場（TD-02）的例行維護中 **firing**——**這套設計唯一要防的失敗模式被觸發了**：維護一台 OSD node 這種最日常的操作照樣 page oncall。

<Callout type="tip" title="這個 miss 的教訓：測試與設計同源，就驗不出設計的假設錯誤">
  dev 測試的 fixture 是照著設計頁的 health-check 名字手刻的。當設計把名字寫錯（`OSDMAP_FLAGS`），測試也跟著用同一個錯名，於是「測試驗證設計」變成「設計自己跟自己對帳」——名字錯得一致、照樣全綠。這類錯誤**結構上**只有引入一個獨立事實來源（真 ceph，或 ceph 原始碼當 oracle）才抓得到。
</Callout>

**修法（已套用到設計與 `rules/ceph-stability-first.yml`）**：排除清單與 `CephLowPriorityNotice` 都改成同時列 `OSD_FLAGS|OSDMAP_FLAGS`——subtree noout 升 `OSD_FLAGS`、cluster-wide noout 升 `OSDMAP_FLAGS`，兩種都要排。現行規則見 [policy 頁](./ceph-alert-policy-current)。

### 5.2 發現二：CephMonQuorumLost 對真 quorum loss 是盲的（架構級，規則層修不了）

這個比發現一更深，也比 dev 抓到的空 vector bug 更深。空 vector bug 是 dev 用 promtool 發現的**規則層** PromQL bug（`count(空 vector)` 語意，`or vector(0)` 修好）。真 quorum loss 先導場（TD-04）揭露的是更底層的問題：**就算空 vector bug 修對了，`CephMonQuorumLost` 在真 quorum loss 時還是不會 fire——原因不在 PromQL，在 metric 來源。**

機制：`CephMonQuorumLost` 與 `CephMonDownScoped` 讀的 `ceph_mon_quorum_status` 是 **mgr 匯出**的，而 mgr 要先有 mon quorum 才能更新 cluster 視圖。先導場停掉 2/3 mon、quorum 一失守，mgr 就**凍結在最後已知狀態**（三台都＝1），但它的 prometheus HTTP endpoint 仍持續吐這份 stale 資料——scrape 還成功（`up{job="ceph"}=1`，連 `CephExporterAllDown` 都不 fire；原始 observations 記錄用的是拆分前的舊名 `CephExporterDown`，v2 拆分後「全 target down」的語意由 `CephExporterAllDown` 承接），`count(...==1)` 永遠是 stale 的 3，`3 < 2` 不成立，整個 90 秒視窗**沒有任何 alert**。

| | 空 vector bug（dev promtool 抓到）| 發現二（真機 quorum loss 先導場抓到）|
|---|---|---|
| 觸發最壞情況靜默的原因 | PromQL `count(空 vector)` 語意 | mgr metric 來源凍結 |
| 屬於哪一層 | 規則層 | 架構層（metric source）|
| 規則層修得了嗎 | 修得了（`or vector(0)`）| **修不了**——mgr 凍結時 count 仍是 stale 的 3，根本走不到空 vector |
| 哪一層能抓到 | dev promtool（合成 series）| 只有真 ceph（mgr 真的會凍結）|

也就是說「用叢集自己的 mgr 匯出的 metric 偵測 quorum loss」有**先天盲區**：graceful 的單台下線（單 mon 維護先導場——quorum 還在、mgr 能更新）測得到、scoped silence 也接得上；真正的 quorum 失守測不到，因為偵測者本身依賴它要偵測的東西。real-lab 的 mon-quorum-lost 場（發現 11／16）是同一機制的另一面——正因為凍結，inhibit 只能 synthetic 驗。

**緩解（out-of-band 探針先導場 TD-05）**：實作 `CephMonQuorumLostExternal`，讀一支**叢集外**的 TCP 探針（直連每台 mon 的 msgr2 port `3300`，匯出 `ceph_mon_tcp_up`，完全不經 mgr）。把 mgr-based 與 probe-based 兩條規則載進同一個 Prometheus、再跑一次停 2/3 mon：

| quorum 失守後全程 | `CephMonQuorumLost`（mgr 來源）| `CephMonQuorumLostExternal`（探針來源）|
|---|---|---|
| count | `3`（凍結）| `1`（正確）|
| alert | **不 fire**（盲，重現前一場）| **firing**（縮短版 `for:30s` 下約 35s 起）|

同一個 Prometheus、同一刻、同一場真 quorum loss——mgr-based 沉默，out-of-band 探針正確 fire。限制要誠實講：探針偵測「mon port 可達」不是「mon 在 quorum」，能抓 mon-down，抓不到「所有 mon 都活著卻網路分區」。這個盲區的完整拆解與其它偵測面（blackbox 探針、metric staleness 告警）在專頁 [Ceph — CephMonQuorumLost 的盲區](./mon-quorum-detection-blind-spot)。

## 6. v1 → v2 規則演進（最終規則指向 policy 頁）

最初這系列只驗了 3 個場景（v1 phase，2026-07-04：`slow-ops`／`pg-availability`／`mon-quorum`，索引在 `EVIDENCE-SUMMARY-2026-07-04.md`）。那批的教訓催出 v2 的幾個結構改動：

- **`CephClientBlocked` 擴大 catch 面**：把 `OSD_FULL`／`POOL_FULL` 併進 client I/O 生命線（capacity-ladder／pool-quota 兩場驗過）；`POOL_APP_NOT_ENABLED` 這種測試 pool 易觸發、不代表 client blocked 的 check 降級進 `CephLowPriorityNotice`。
- **`CephExporterDown` 拆分級**：`CephExporterAllDown`（全 target down 才 page）與 `CephExporterTargetDown`（單 target down 走 slack）。
- **新增監控盲區與 coverage**：`CephMetricsAbsent`、`Watchdog`、整組 `ceph-production-coverage`（網路／clock skew／容量／資料完整性／failover 能力）。
- **routing 從 alertname regex 改成 label 導向**：v1 用 alertname regex，新增 rule 時易漏；v2 改成 `severity`＋`source` label 導向，新 rule 自動落到對的 receiver。
- **MON quorum 補 out-of-band 觀測面**：發現二的緩解（`CephMonQuorumLostExternal`）。

<Callout type="warning" title="為什麼本頁不內嵌完整規則 YAML（一個刻意的設計決定）">
  舊 `prometheus-alert-real-lab-findings` 頁把一整份 v2 規則 YAML 內嵌在報告裡——這正是它被 archive 的原因之一：規則會隨 `experiments/ceph-alert-rules/rules/` 演進，內嵌快照必然過時、還會跟現況打架。本頁把「現在該部署什麼」交給唯一 single source of truth——[policy 頁](./ceph-alert-policy-current)（規則 YAML 逐字取自 rules 目錄、有防漂移 guard 綁定，現行 29 條）；本頁只負責「這些規則在真機被怎麼驗的」。所以本頁全程只給 rule 名與語意，不貼 expr。
</Callout>

## 7. 完整總結

**按研究問題收束**：

1. **這套 alert 設計搬上真 ceph，核心機制守不守得住？** 守得住。scoped label join、整台 down 的去重、`amtool` silence 鎖定、stopped mon 的 monmap 持久化（Tier D 前三場），以及 22 場真故障注入的 firing＋receiver 送達＋乾淨復原——都在真 cephadm v19.2.3 上簽收。dev 的「IF…THEN」拿到了真 ceph 的 IF。
2. **有沒有 dev 抓不到的 bug？** 有兩個機制級的。發現一（`OSD_FLAGS ≠ OSDMAP_FLAGS`，排除清單名字錯 → 例行維護 page）已修；發現二（`CephMonQuorumLost` 依賴 mgr metric、真失守時凍結）是架構級盲區、規則層修不了，只能靠 out-of-band（探針緩解已在真機證明有效）。
3. **「22/22」該怎麼引用？** 引用時**必帶 rules 版本 caveat**：每場在其最終 rules 版本下通過（rules 橫跨 3 個 commit），latency-outlier 場是 `PASS*`。**不要**寫成「一次 snapshot 驗完 27／29 條」。

**一句話版本**：同一座 6-host 真 lab 上，Tier D 5 場先簽收「ceph 事實對得上規則假設」（並抓到兩個機制盲區），real-lab 22 場再逐條把 alert 用真故障做 firing——全過，但「全過」的正確讀法是「每場在其最終 rules 版本下過、latency-outlier 場是 PASS*」。

**接下來做什麼（按優先序，每條掛依據）**：

1. **部署規則以 [policy 頁](./ceph-alert-policy-current) 的現行 29 條為準**，不要用任何舊頁內嵌的 YAML 快照（§6）。
2. **MON quorum 一定要配 out-of-band 生命線**：`CephMonQuorumLostExternal` 或等價的 blackbox／metric-staleness 偵測（發現二的緩解），因為 mgr-based 規則對真失守盲。
3. **維護前確認排除清單含 `OSD_FLAGS|OSDMAP_FLAGS` 兩者**（發現一），否則例行 `maintenance enter` 會 page。
4. **`BLUESTORE_SLOW_OP_ALERT` 當盤後追查、不當 pager**，清 latch＝縮短 `bluestore_slow_ops_warn_lifetime` 或 restart OSD 且叢集沉降後一次一顆（發現 08）。

## 8. 侷限、事故與開放問題

**侷限（機制級 vs 環境綁定）**：

- **發現二是機制級盲區**：mgr-based quorum 偵測對真失守盲，換環境也一樣（架構層）。
- **Tier D 的 firing 秒數是 `for:` 縮短版**（30s）：「35s firing」讀作「規則接上並 fire」，不是 production 告警延遲（數值級／設定綁定）。
- **復原證據跨 run**：latency-outlier 場的 `HEALTH_OK` 是跨 run 證據，非本場 in-run；BlueStore latch 清除 helper 當時尚未接進本場（環境／時序綁定）。
- **inhibit 是 synthetic 驗設定**：mon-quorum-lost 場驗的是 Alertmanager 設定本身，不是真 fault 的抑制行為——因為發現二的凍結讓真 fault 永遠做不出兩 alert 重疊。
- **crash 傳輸層手動代送**：daemon-crash 場的 sidecar 壞掉、由 harness 代送 crash meta，fault 本身真實。
- **rules 版本橫跨 3 commit**：更早兩段的 rules↔run 對應是 commit 時間軸推定，無法排除當時 working tree 有未提交差異（誠實限制）。

**研究過程的事故（誠實記錄＋教訓）**：

- **slow-ops 場三連敗**：spiky gauge（發現 02）讓 `CephDaemonSlowOps` 抓不到，`efcf5bb` 改 windowed max 後一次通過。
- **capacity-forecast 場九次失敗**（本組最難搞）：五個短命 run 在 pool setup 早段中止＋兩次「分輪迴圈覆寫同名 object」（發現 13），改連續 stream 後才過。
- **osd-flapping 場兩次失敗**：翻轉太快 scrape 抓不到（發現 05），改成等 Prometheus 反映後再前進。
- **net-slow-heartbeat 場 rollback 自斷 SSH**（發現 09，in-run 事件、ledger 無失敗重試）：改成事後用新連線輪詢驗延遲已清。
- **清 latch 的打地鼠風險**（發現 08）：restart 引發的 peering IO 在慢碟上會產生新的 &gt;5s op、讓別的 OSD 重新 latch，要沉降後一次一顆。

**開放問題（超出 scope／待後續）**：

- **out-of-band 探針抓不到網路分區**（探針場的既知限制）：探針只驗「mon port 可達」，「所有 mon 活著卻分區」要對每台真跑 `quorum_status`——見 [mon-quorum 盲區專頁](./mon-quorum-detection-blind-spot)。
- **rules 版本快照未逐次留存**：更早兩段 redeploy 沒留 manifest，未來 harness 應每次 redeploy dump 一份 rendered rules 快照。
- **部署腳本 configmap 改動不自動 reload Prometheus**（發現 12）：已記待辦，尚未硬化成 apply 後自動 reload。
