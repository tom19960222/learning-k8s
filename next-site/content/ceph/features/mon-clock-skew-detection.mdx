## 場景

3-node ceph cluster，3 顆 bare-metal、各跑一個 mon。某天 `mon.2` 上的 chrony 指向錯誤的 NTP server，system clock 開始漂出去。當漂幅超過 50ms 那一瞬間：

> **`ceph -s` 多久會冒出 `HEALTH_WARN clock skew detected on mon.2`？**

答案不是「立刻」也不是「等下次心跳」。MON 自己有獨立的 timecheck 機制，它跟 ntpd、paxos lease 都無關。下面要追的是：從你 watch 著 stopwatch、看到 mon.2 的 clock 慢慢漂過 50ms 那一秒開始，到 `ceph -s` 那行字出現之前，**程式碼跑了哪些東西、每一段花多少時間**。

## TL;DR：直接看時間

| 情境 | 平均 | 最壞 |
|---|---|---|
| cluster 健康（從沒偵測到 skew） | **~150 秒**（≈ 2.5 分鐘） | **~305 秒**（≈ 5 分 5 秒） |
| cluster 已在 skew 狀態，又漂大一些 | **~17 秒** | **~35 秒** |
| `mon_timecheck_interval` 調成 60s | ~30 秒 | ~65 秒 |
| 跨 WAN 部署 RTT > 50ms | **可能永不報**（latency 吸收 abs_delta） | — |

「最壞」是「漂超過 50ms 那一刻剛好錯過某輪 timecheck、要等下一輪」的情況。下面會把這個 305 秒拆給你看。

## 完整流程一張圖

```
T0 ┃ mon.2 的 system clock 漂超過 50ms
   ┃
   ┃ ① 等下一輪 timecheck round 觸發
   ┃    healthy cluster: 0 ~ 300s（mon_timecheck_interval, 平均 150s）
   ┃    已在 skew:        0 ~ 30s × N（mon_timecheck_skew_interval × rounds_since_clean）
   ┃
T1 ┃ leader 的 timer 觸發 timecheck_start_round()
   ┃ ├─ timecheck_acks = 1 (自己 ack 自己)
   ┃ ├─ timecheck_skews[self] = 0.0
   ┃ └─ 對每個 peon 送 MTimeCheck2(OP_PING)，戳記 timecheck_waiting[peon] = now
   ┃
   ┃ ② 網路：PING 到 peon
   ┃    1 × RTT/2（同 DC 通常 < 5ms）
   ┃
T2 ┃ peon 收 PING → handle_timecheck_peon() → 回 PONG，戳上自己的 timestamp
   ┃
   ┃ ③ 網路：PONG 回 leader
   ┃    1 × RTT/2
   ┃
T3 ┃ leader 收 PONG → handle_timecheck_leader()
   ┃ ├─ latency = ceph_clock_now() − timecheck_waiting[other]  ← 約等於 RTT
   ┃ ├─ EWMA 平滑：avg_latency = 0.8·old + 0.2·latency
   ┃ ├─ delta     = m->timestamp − ceph_clock_now()
   ┃ ├─ abs_delta = |delta|
   ┃ ├─ skew_bound = abs_delta − latency  (clip to 0; 帶正負號)
   ┃ ├─ if timecheck_has_skew(skew_bound):  status = HEALTH_WARN
   ┃ └─ timecheck_skews[other] = skew_bound   ← 寫入！
   ┃
   ┃ ④ 等下一次 Monitor::tick()
   ┃    0 ~ 5s（mon_tick_interval = 5）
   ┃
T4 ┃ Monitor::tick() 跑：iterate paxos_service → HealthMonitor::tick()
   ┃ ├─ check_leader_health()
   ┃ │   └─ check_for_clock_skew(&next)
   ┃ │       └─ 讀 mon.timecheck_skews，若有 |skew| > 0.05 → next.add("MON_CLOCK_SKEW", HEALTH_WARN, ...)
   ┃ ├─ if next != leader_checks: leader_checks = next; changed = true
   ┃ └─ if changed: propose_pending()
   ┃
   ┃ ⑤ paxos commit
   ┃    1 paxos round + local write (典型 < 10ms)
   ┃
T5 ┃ ceph -s 透過 mon command 走 HealthMonitor::get_health_status() →
   ┃ gather_all_health_checks() → 看見 MON_CLOCK_SKEW → 印 HEALTH_WARN
```

下面把每一段的程式碼與時間貢獻拆開來看。

## 三個 timer × 一個門檻

| 參數 | 預設 | 用途 | 出處 |
|---|---|---|---|
| `mon_clock_drift_allowed` | `0.05` (50ms) | 偏移門檻 | `common/options/mon.yaml.in:609` |
| `mon_timecheck_interval` | `5_min` (300s) | 健康時 leader 啟動下輪 round 的間隔 | `common/options/mon.yaml.in:628` |
| `mon_timecheck_skew_interval` | `30` (30s) | 已偵測到 skew 時、下輪間隔的基數 | `common/options/mon.yaml.in:639` |
| `mon_tick_interval` | `5` (5s) | `Monitor::tick()` 自我 re-arm 的週期 | `common/options/mon.yaml.in:1314` |

警告路徑有兩條，**完全獨立**：

1. **timecheck round**（主要）— leader 主動 ping/pong，產生 `MON_CLOCK_SKEW` health check，這是 `ceph -s` 顯示的那條。
2. **Paxos `warn_on_future_time`**（次要）— 每處理一條未來時間的 lease/ack 訊息就可能寫一行 cluster log；不上 health check。

主路徑佔本頁 90% 篇幅，次要路徑留到文末。

## 參考點：leader 是「the eye of the storm」

機制本質是 **leader-driven ping/pong + Smoothed RTT**。註解講得很清楚：

File: `ceph/src/mon/Monitor.h` (line 525)

```cpp
/**
 * @defgroup Monitor_h_TimeCheck Monitor Clock Drift Early Warning System
 *
 *  - Leader sends out a 'PING' message to each other monitor in the quorum.
 *    The message is timestamped with the leader's current time. The leader's
 *    current time is recorded in a map, associated with each peon's instance.
 *  - The peon replies to the leader with a timestamped 'PONG' message.
 *  - The leader calculates a delta between the peon's timestamp and its
 *    current time and stashes it.
 *  - The leader also calculates the time it took to receive the 'PONG'
 *    since the 'PING' was sent, and stashes an approximate latency estimate.
 *  - Once all the quorum members have pong'ed, the leader will share the
 *    clock skew and latency maps with all the monitors in the quorum.
 */
```

關鍵設計：**leader 的 skew 永遠是 0**，所有 peon 的偏移都是相對 leader 算的。

File: `ceph/src/mon/Monitor.cc` (line 5042)

```cpp
// we are at the eye of the storm; the point of reference
timecheck_skews[rank] = 0.0;
timecheck_latencies[rank] = 0.0;
```

所以**leader 自己的 clock 漂走** → 其他 peon 全部會被標 skew。**peon 漂走** → 只標那一顆。`ceph time-sync-status` 看不出「誰才是真的錯」，需要對照外部 NTP source。

## 第 1 段：等下一輪 timecheck round（0 ~ 300s）

這是延遲的大頭。每一輪結束時 leader 排下一輪：

File: `ceph/src/mon/Monitor.cc` (line 4926)

```cpp
void Monitor::timecheck_reset_event()
{
  if (timecheck_event) {
    timer.cancel_event(timecheck_event);
    timecheck_event = NULL;
  }

  double delay =
    cct->_conf->mon_timecheck_skew_interval * timecheck_rounds_since_clean;

  if (delay <= 0 || delay > cct->_conf->mon_timecheck_interval) {
    delay = cct->_conf->mon_timecheck_interval;
  }

  timecheck_event = timer.add_event_after(
    delay,
    new C_MonContext{this, [this](int) {
        timecheck_start_round();
      }});
}
```

`timecheck_rounds_since_clean` 是「連續幾輪都偵測到 skew」的計數器。決定 delay 的公式：

| 連續偵測到 skew 的第 N 輪 | 下一輪間隔 |
|---|---|
| 0（cluster 健康） | 300s = `mon_timecheck_interval` |
| 1 | 30s = `mon_timecheck_skew_interval × 1` |
| 2 | 60s |
| 3 | 90s |
| ... | `30 × N`，但 **capped at `mon_timecheck_interval` = 300s** |
| 10 起 | 300s（封頂） |

當 skew 消失，`timecheck_rounds_since_clean` 重置回 0：

File: `ceph/src/mon/Monitor.cc` (line 4960)

```cpp
bool found_skew = false;
for (auto& p : timecheck_skews) {
  double abs_skew;
  if (timecheck_has_skew(p.second, &abs_skew)) {
    found_skew = true;
  }
}

if (found_skew) {
  ++timecheck_rounds_since_clean;
  timecheck_reset_event();
} else if (timecheck_rounds_since_clean > 0) {
  // make sure the skews are really gone and not just a transient success
  timecheck_rounds_since_clean = 1;
  timecheck_reset_event();
  timecheck_rounds_since_clean = 0;
}
```

第二個 `else if` 分支是 hysteresis：偵測到 skew 消失時不是馬上回去 300s 節奏，而是再用 30s 跑一輪確認，再回 300s。

**所以「等下一輪」的時間範圍**：

- 健康 cluster：均勻分布在 `[0, 300]` 秒，平均 150s、最壞 300s
- 已知 skew：均勻分布在 `[0, 30s × N]`，最少時平均 15s、最壞 30s

leader 收 PONG：

File: `ceph/src/mon/Monitor.cc` (line 5101)

```cpp
utime_t curr_time = ceph_clock_now();

ceph_assert(timecheck_waiting.count(other) > 0);
utime_t timecheck_sent = timecheck_waiting[other];
timecheck_waiting.erase(other);
if (curr_time < timecheck_sent) {
  // our clock was readjusted -- drop everything until it all makes sense.
  timecheck_cancel_round();
  return;
}

/* update peer latencies */
double latency = (double)(curr_time - timecheck_sent);

if (timecheck_latencies.count(other) == 0)
  timecheck_latencies[other] = latency;
else {
  double avg_latency = ((timecheck_latencies[other]*0.8)+(latency*0.2));
  timecheck_latencies[other] = avg_latency;
}
```

`latency` 是 EWMA（80% 舊值 + 20% 新值），網路抖動不會單一一輪就翻轉判斷。

接著算 skew_bound：

File: `ceph/src/mon/Monitor.cc` (line 5164)

```cpp
double delta = ((double) m->timestamp) - ((double) curr_time);
double abs_delta = (delta > 0 ? delta : -delta);
double skew_bound = abs_delta - latency;
if (skew_bound < 0)
  skew_bound = 0;
else if (delta < 0)
  skew_bound = -skew_bound;
```

公式翻譯：

| 變數 | 意義 |
|---|---|
| `delta` | peon 戳的 timestamp − leader 收到 PONG 的時間（正：peon 快；負：peon 慢） |
| `latency` | leader 從 PING 送出到 PONG 收到的時間（EWMA 抹平） |
| `abs_delta − latency` | 扣掉「能用 RTT 解釋」的部分後剩下的「真實偏移下界」 |

| 情況 | skew_bound |
|---|---|
| `abs_delta ≤ latency`（外觀的偏移可以完全用 RTT 解釋） | `0`（不算 skew） |
| `abs_delta > latency` 且 peon 快 | `+ (abs_delta − latency)` |
| `abs_delta > latency` 且 peon 慢 | `− (abs_delta − latency)` |

判定就一行：

File: `ceph/src/mon/Monitor.h` (line 592)

```cpp
bool timecheck_has_skew(const double skew_bound, double *abs) const {
  double abs_skew = std::fabs(skew_bound);
  if (abs)
    *abs = abs_skew;
  return (abs_skew > g_conf()->mon_clock_drift_allowed);
}
```

**嚴格大於 0.05 才算 skew**。剛好 50.0ms 不報。

**注意 RTT 吸收效應**：latency 是被扣掉的，所以 RTT 越高、偵測門檻越寬。假設網路對稱（單向 = R/2），偏移 N、RTT R，則 `abs_delta = |N − R/2|`、`skew_bound = max(0, |N − R/2| − R)`。要超過 50ms 門檻需要 `|N − R/2| > R + 0.05`：

- RTT 1ms：peon 漂 ≥ ~51ms 就會報。
- RTT 10ms：peon 慢 ≥ 55ms、peon 快 ≥ 65ms 才會報（因為 abs_delta 在兩方向不對稱）。
- RTT 100ms：peon 慢需要 ≥ 100ms 才會報，peon 快需要 ≥ 200ms。**真實漂 70ms 不會被偵測到**。

跨 WAN 部署不要把 mon 拉太遠，否則偵測窗口會被 latency 整片吃掉。

## 第 3 段：寫入 timecheck_skews[] + 排下一輪（同步）

關鍵的一行（就在第 2 段那段程式碼的結尾）：

File: `ceph/src/mon/Monitor.cc` (line 5182)

```cpp
timecheck_skews[other] = skew_bound;

timecheck_acks++;
if (timecheck_acks == quorum.size()) {
  // everyone has acked, so bump the round to finish it.
  timecheck_finish_round();
}
```

兩個重點：

1. **每收到一個 PONG，那顆 peon 的 skew 就立刻寫入 map**，不用等整輪結束。所以若 5 顆 peon 中只有 1 顆漂走、那一顆的 PONG 一收到、map 就更新。
2. 全部 PONG 收齊才呼 `timecheck_finish_round()`，做兩件事：

File: `ceph/src/mon/Monitor.cc` (line 4878)

```cpp
void Monitor::timecheck_finish_round(bool success)
{
  timecheck_round ++;
  timecheck_round_start = utime_t();

  if (success) {
    ceph_assert(timecheck_waiting.empty());
    ceph_assert(timecheck_acks == quorum.size());
    timecheck_report();          // 把這輪結果廣播給每個 peon
    timecheck_check_skews();     // 決定下一輪節奏（rounds_since_clean ± 與 timer 排程）
    return;
  }
}
```

`timecheck_report()` 把 skew + latency map 廣播給 peon — 是為了讓 peon 上的 `ceph time-sync-status` 看得到資料，**不會** 影響 `ceph -s` 的判定（那個是 leader 自己 health check 機制做的）。

`timecheck_check_skews()` 才會呼 `timecheck_reset_event()` 排下一輪。

## 第 4 段：Monitor::tick → HealthMonitor::tick → 變成 health check

```cpp
{
  if (!mon.timecheck_skews.empty()) {
    list<string> warns;
    list<string> details;
    for (auto& i : mon.timecheck_skews) {
      double skew = i.second;
      double latency = mon.timecheck_latencies[i.first];
      string name = mon.monmap->get_name(i.first);

      health_status_t tcstatus = mon.timecheck_status(tcss, skew, latency);
      if (tcstatus != HEALTH_OK) {
        warns.push_back(name);
        // ... 累積 detail
      }
    }
    if (!warns.empty()) {

      d.detail.swap(details);
    }
  }
}
```

```cpp
                                          const double skew_bound,
                                          const double latency)
{

  ceph_assert(latency >= 0);

  double abs_skew;
  if (timecheck_has_skew(skew_bound, &abs_skew)) {
    status = HEALTH_WARN;

```

```cpp
    return false;        // 待目前 round 結束後再排
  }
}
```

paxos commit 完成後，每一個 mon（包括 leader 自己）都會跑 `update_from_paxos` 把剛 commit 進來的狀態同步到 in-memory 結構：

File: `ceph/src/mon/HealthMonitor.cc` (line 87)

```cpp

{
  version = get_last_committed();
  load_health();
  // ... 還會 decode quorum_checks, leader_checks, mutes ...
}
```

`load_health()` 是定義在 PaxosService 上的：

```cpp

{

  mon.store->get("health", service_name, bl);
  if (bl.length()) {
    auto p = bl.cbegin();

```

注意這個 `health_checks` field 是 `PaxosService` 的 base class member，**跟 `HealthMonitor::leader_checks` 是不同的東西**：

- `leader_checks`：HealthMonitor 在 leader tick 算出來、未提交的 pending state。
- `health_checks`：base class member，存的是「已經 paxos commit 過的」health check 結果。

`ceph -s` 走的是 mon command 路徑：

## 把五段加起來：avg / max

| 段 | 延遲下界 | 延遲上界 | 平均 | 備註 |
|---|---|---|---|---|
| ① 等下一輪 timecheck 觸發（healthy） | 0 | 300s | 150s | 均勻分布假設 |
| ② PING/PONG/處理 | RTT | RTT + few ms | RTT + ~1ms | 同 DC ≈ 5ms |
| ③ 寫入 `timecheck_skews[]` | — | — | — | 同步發生在 ② 內 |
| ④ 等下一次 `Monitor::tick()` | 0 | 5s | 2.5s | 5s self-armed |
| ⑤ paxos commit | ~ms | tens of ms | ~5ms | healthy cluster |

**Cluster 健康時 從 drift 到 `ceph -s` 顯示 `HEALTH_WARN`**：

- 平均 ≈ 150 + 0.005 + 2.5 + 0.005 ≈ **152.5 秒**（≈ 2.5 分鐘）
- 最壞 ≈ 300 + 0.1 + 5 + 0.05 ≈ **305 秒**（≈ 5 分 5 秒）

**Cluster 已在 skew 狀態（rounds_since_clean ≥ 1，第 N 輪間隔 = 30s × N，capped 300s）**：

- 第一輪 skew 之後馬上又漂大些 → 平均 17.5s、最壞 35s
- N 輪後 → 平均 ≈ 15·N + 2.5s、最壞 ≈ 30·N + 5s（N ≤ 10）

`mon_timecheck_interval` 預設 5 分鐘很長，loaded cluster 上把它調到 60 秒是常見做法（trade-off 是每分鐘 quorum 內就多一輪 ping/pong）：

```bash
ceph config set mon mon_timecheck_interval 60
# 此後最壞延遲 ≈ 60 + 5 + commit ≈ 65 秒
```

## 重要邊界與例外

### 真實偏移被 RTT 吃掉

參見第 2 段「RTT 吸收效應」那段：實質偵測門檻是 `|N − R/2| > R + mon_clock_drift_allowed`，且兩方向不對稱（peon 慢、peon 快需要的最小偏移不一樣）。Ceph 預設假設 mon 是 cluster-internal RTT、不為跨 WAN 部署最佳化。`mon_clock_drift_allowed` 也不能小於合理 RTT，否則 false positive 不停。

### 第一輪 latency 沒被 EWMA 平滑

File: `ceph/src/mon/Monitor.cc` (line 5139)

```cpp
if (timecheck_latencies.count(other) == 0)
  timecheck_latencies[other] = latency;        // ← 第一輪直接吃，沒平滑
else {
  double avg_latency = ((timecheck_latencies[other]*0.8)+(latency*0.2));
  timecheck_latencies[other] = avg_latency;
}
```

剛 election 完第一輪如果 RTT 抖動，會記到一個歪掉的 latency。後續幾輪會慢慢拉回（每輪向真實 RTT 收斂 20%）。所以「剛 election 完一段時間數字怪」是正常。

### leader 自己漂走

`timecheck_skews[rank] = 0.0` 永遠把 leader 自己當 0。leader 自己漂 → 其他每顆 peon 都被算成漂同樣大小、方向相反。`ceph -s` 會說「mon.peon1, mon.peon2 clock skew」，看不出真正原因是 leader。需要對照外部 NTP source。

### clock 被往回調

leader 在 `handle_timecheck_leader` 開頭擋掉 `curr_time < timecheck_sent` 的 case：

File: `ceph/src/mon/Monitor.cc` (line 5127)


如果在 mon 上手動 `date -s` 往回調，會看到 round 一直被 cancel、出現連續 skew detection。

### 單 mon cluster 不會跑

### 剛 election 完馬上跑第一輪

很多人以為要等 5 分鐘第一輪才跑，其實 `timecheck_start()` 在 election 結束時就被呼叫，它直接跑 `timecheck_start_round()`：

## 第二條警告路徑：Paxos `warn_on_future_time`

完全獨立於 timecheck — Paxos 每處理一條 lease 或 lease ack message 都會檢查 timestamp：

File: `ceph/src/mon/Paxos.cc` (line 1025)

```cpp
void Paxos::warn_on_future_time(utime_t t, entity_name_t from)
{
  utime_t now = ceph_clock_now();
  if (t > now) {
    utime_t diff = t - now;
    if (diff > g_conf()->mon_clock_drift_allowed) {
      utime_t warn_diff = now - last_clock_drift_warn;
      if (warn_diff >
          pow(g_conf()->mon_clock_drift_warn_backoff, clock_drift_warned)) {
        mon.clog->warn() << "message from " << from << " was stamped " << diff
                         << "s in the future, clocks not synchronized";
        last_clock_drift_warn = ceph_clock_now();
        ++clock_drift_warned;
      }
    }
  }
}
```

特色：

- **單向偵測**：只看「對方戳了未來時間」(`t > now`)；對方戳過去時間不報。
- **指數 backoff**：`warn_backoff^clock_drift_warned` 秒，預設 backoff = 5，所以警告間隔是 1 秒、5 秒、25 秒、125 秒…
- **不上 health check**：只寫 cluster log (`mon.clog->warn()`)，`ceph log last cluster | grep "future"` 才看得到。
- **完全 reactive**：靠 lease/ack 訊息流量，沒有獨立 timer。

呼叫點是 lease 與 lease ack 處理（同檔 line 1115 / 1188）。lease 每秒級觸發，所以這條**通常比 timecheck round 先發出警告** — 但因為不掛 health check，自動化監控系統如果只看 `ceph -s` JSON 不會抓到。

<Callout type="warning" title="NTP 不在 ceph 自己的責任範圍">
  ceph 只**偵測** clock 偏移、不**修正**它。要避免警告，operator 必須在 mon node 上跑 ntpd / chrony / systemd-timesyncd，並保證每台 mon 都指向同一個 stratum source。kubernetes/rook 部署時若使用 host network 跑 mon、host clock 通常由 node 的 kubelet host 決定；KubeVirt VM 環境裡 VM clock 由 hypervisor 給，跨 hypervisor live migrate 後可能瞬間漂。
</Callout>

<Callout type="tip" title="想加快偵測？調這兩個 knob">
  - `ceph config set mon mon_timecheck_interval 60` — 把正常輪降到 60 秒，最壞延遲 ≈ 65 秒。loaded cluster 上會增加 mon work load。
  - `ceph config set mon mon_clock_drift_allowed 0.02` — 門檻降到 20ms。RTT ~1ms 的內網可接受；跨 WAN 不要這樣設，會 false positive。
</Callout>
