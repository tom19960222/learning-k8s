## 術語表

先對齊詞彙，後面不再解釋：

| 詞 | 意思 |
|---|---|
| **工作機 / 跳板機（jump host）** | 你實際敲 `collect.sh` 的那台機器。它經 SSH 連到每台 Ceph node；自己不必是 cluster 成員。 |
| **inventory** | 一個 shell 檔，列出要收哪些 node（`alias=host`）、SSH 帳號、seed 等。`collect.sh` 用 `source` 讀它。 |
| **seed** | 「在哪一台 node 上跑 cluster-level `ceph` 指令」的來源 node。`cephadm` 與 `auto` 模式都可用 `--seed USER@HOST` 或 inventory 的 `SEED_HOST` 釘死；沒釘時 `auto` 會自動挑第一台連得上的。 |
| **runner** | 在 seed 上真正執行 `ceph` 的方式：`direct`（直接 `ceph`）、`sudo`（`sudo -n ceph`）、`cephadm`（`sudo -n cephadm shell -- ceph`）。工具會自動挑最快能連上的那個。 |
| **bundle** | 最終產物，一個 `ceph-incident-YYYYMMDDTHHMMSSZ.tar.gz`，裝著這次收到的所有證據。 |
| **workdir** | 打包前的暫存目錄（`results/tmp.<時間>.<pid>/`）。正常情況打包完會刪掉；verify 失敗時會**保留**讓你檢查。 |
| **redaction** | 打包前把文字檔裡「明顯像 secret 的整行」換成 `[REDACTED]`。是 best-effort，不是完整 DLP。 |
| **known_hosts** | SSH 記錄「這台 host 的公鑰指紋」的檔案（通常 `~/.ssh/known_hosts`）。工具用 `BatchMode=yes` 不互動，所以第一次連新 host 前這裡要先有 key。 |
| **BatchMode** | SSH 的非互動模式。連線需要打密碼或回答 yes/no 時，`BatchMode=yes` 會直接失敗而不是卡住等人。 |
| **manifest** | `manifest.jsonl`，每個收到的 artifact 一行 JSON：哪台 host、哪條指令、exit code、起訖時間。 |
| **ssh-debug** | SSH 連線失敗時工具自動補跑的 `ssh -vvv` 診斷 log，落在 bundle 的 `ssh-debug/`。 |

## 一句話定位：先保留現場，再判讀

Ceph 出問題時，第一個直覺常是立刻找 repair command、restart daemon、搬 OSD、改 flag。這些也許最後要做，但一開始就動手修，最容易把真正有價值的線索抹掉：`ceph health detail` 當下亮了哪些 check、OSD/MON/MGR 的 map 與 quorum、每台 node 在同一個時間窗內的 CPU/RAM/disk/kernel log/systemd journal/Ceph log/time sync。這些一旦動過就回不去了。

所以這頁的目標不是「怎麼修 Ceph」，而是把事故第一步固定成一句話：

> **先保留現場，再判讀；先產出 bundle，再決定要不要修。**

工具入口只有一個，從 repo root 跑：

```bash
bash experiments/ceph-incident-bundle/run/collect.sh ...
```

它是 **read-only** 收集工具：不會 restart、delete、repair、scrub，也不會改任何 Ceph flag。跑錯頂多是沒收到某些東西，不會把 cluster 弄壞。

## 前置需求：SSH 要能不互動連上每台 node

工具所有 SSH 連線都帶 `BatchMode=yes`（`lib/common.sh:35`），意思是**不會**停下來問密碼或問「要接受這台 host key 嗎」。所以跑之前要確認兩件事：

1. **能用 key 登入**：`--ssh-key` 指的私鑰能登入每台 node 的 `SSH_USER`，且**免密碼**（工具不會幫你打密碼）。
2. **known_hosts 已有每台 host 的 key**：預設工具帶 `StrictHostKeyChecking=accept-new`（`lib/common.sh:43`），第一次連新 host 會自動接受並寫進 `known_hosts`。但如果你的環境把這關掉、或用 `--no-trust-ssh-host-key`，就必須事先把 host key 準備好，否則每台都會以 `Host key verification failed` 失敗、被標 SKIPPED。

**進場前先手動測一台**：

```bash
ssh -i .ssh/id_ed25519 -o BatchMode=yes ikaros@192.168.18.166 hostname
```

- **預期輸出**：印出該 node 的 hostname，exit 0。
- **失敗時看哪裡**：如果卡住等密碼或噴 `Permission denied`，是 key/帳號問題；如果噴 `Host key verification failed`，先補 known_hosts（見下）。工具不會幫你修 SSH。

需要一次補齊所有 host key 時：

```bash
# 對 inventory 裡每台 host 先抓 host key（擇一）
ssh-keyscan -H 192.168.18.166 192.168.18.167 192.168.18.164 \
  192.168.18.169 192.168.18.171 192.168.18.174 >> ~/.ssh/known_hosts
```

<Callout type="warning" title="新跳板機最常見的失敗就是 known_hosts">
從一台**沒跑過**的工作機第一次收集，最容易踩的坑不是 Ceph、不是權限，而是 `known_hosts` 還沒有目標 node 的 key。症狀是每台 node 都 `node <alias> collector exited 255` / `Host key verification failed`、整批 SKIPPED、exit 2。先手動 ssh 一台或 `ssh-keyscan` 一次就解決。
</Callout>

## 步驟 1：填 inventory

inventory 是一個 shell 檔。lab 範例在 `experiments/ceph-incident-bundle/inventory/ceph-lab.example.env`：

```bash
SSH_USER="ikaros"
SEED_HOST="192.168.18.166"
HOSTS=(
  "monitor01=192.168.18.166"
  "mon02=192.168.18.167"
  "mon03=192.168.18.164"
  "osd01=192.168.18.169"
  "osd02=192.168.18.171"
  "osd03=192.168.18.174"
)
```

欄位意思：

| 欄位 | 必填？ | 說明 |
|---|---|---|
| `SSH_USER` | 建議填 | 登入每台 node 的 Linux 帳號。`HOSTS` 裡若已寫成 `alias=user@host` 則以那個為準。 |
| `SEED_HOST` | **選填** | 手動指定 cluster-level `ceph` 要在哪台跑。**不填**時 `auto` 會自動挑第一台「`ceph -s` 連得上」的 node（`run/collect.sh:463`、README §「如何填 inventory」）。 |
| `ROOK_NAMESPACE` | 選填 | Rook 的 namespace，未填預設 `rook-ceph`（`run/collect.sh:464`）。 |
| `ROOK_OPERATOR_NAMESPACE` | 選填 | Rook operator 所在 namespace，未填預設 `rook-ceph`（`run/collect.sh:465`）。external 拓樸常用（見下）。 |
| `HOSTS` | **必填** | `alias=host` 清單。`alias` 會變成 bundle 裡 `nodes/<alias>/` 的資料夾名稱。 |

- **進場條件**：檔案語法是合法 bash（`HOSTS` 是陣列）。
- **失敗時看哪裡**：`HOSTS` 沒定義 → `inventory must define HOSTS`（exit 1）；`HOSTS` 是空的 → `inventory HOSTS is empty`（exit 1）。個別項目寫壞（例如少了 `=`）**不會中止**：那一項被記進 `errors.log`（`skipped malformed HOSTS entry: ...`）、rc 變 2，其餘照收（`run/collect.sh:499-504`）。

## 步驟 2：執行收集

### 最省事：`--mode auto`（預設）

不指定 `--mode` 就是 `auto`（`run/collect.sh:328`）。`auto` 會逐台 node 經 SSH **偵測能力**再分層收集：

```bash
bash experiments/ceph-incident-bundle/run/collect.sh \
  --inventory experiments/ceph-incident-bundle/inventory/ceph-lab.example.env \
  --ssh-key .ssh/id_ed25519 \
  --since 24h
```

`auto` 的實際行為（`run/collect.sh:167-234`）：

- **cluster-ceph 層**：挑**第一台「`ceph -s` 真的連得上」**的 node 收。注意「連得上」不是「有 `ceph` 這支指令」——工具會實際試連（`ceph_runner_probe`，`run/collect.sh:95-105`），連不上就換下一個候選。
- **cluster-rook 層**：由 `--kube-mode` 決定（預設 `remote`），挑**第一台有 `kubectl` 指令**的 node，在該 node 上經 SSH 跑 `kubectl`（`run/collect.sh:183-184`、`lib/collect-cluster-rook.sh:149`）。
- **node 層**：一律**每台都收**，跟 mode 無關。
- 兩層都有來源就都收，各收一次；某層完全沒來源時只寫一個 `SKIPPED.txt`，**不會**讓整包失敗——`auto` 只有在**兩層都沒收到**時才算硬失敗（`run/collect.sh:229-233`）。

**cephadm runner 三階擇優**：收 cluster-ceph 時，工具不再固定走 `cephadm shell`，而是依連線性自動挑最快的（`lib/collect-cluster-cephadm.sh:10-20`、`run/collect.sh:109-113`）：

1. `direct` → 直接 `ceph`（最快，不必每條指令起一個 container）
2. `sudo` → `sudo -n ceph`
3. `cephadm` → `sudo -n cephadm shell -- ceph`（最保險，但每條指令起一個 container）

選到哪個會記在進度輸出與 `environment.txt` 的 `ceph_runner=`（`run/collect.sh:240`）。**即使你用 `--seed` 釘死來源 node，runner 仍是按連線性挑**（`run/collect.sh:155-159`）——`--seed` 只釘「在哪台跑」，不釘「怎麼跑」。

- **預期輸出**：最後一行 `bundle: .../ceph-incident-<時間>.tar.gz`，exit 0。
- **失敗時看哪裡**：探測 SSH 失敗的 node 會記進 `errors.log`（`capability probe failed for ...`）並補一份 `ssh-debug/` log，**不會**被誤當成「這台沒有該能力」而靜默忽略（`run/collect.sh:84-89`）。

<Callout type="tip" title="大型 inventory 建議跳過探測">
`auto` 的探測是**逐台序列 SSH**：純 cephadm 叢集仍會為了 rook 層把每台掃一遍；node 沒回應時會逐台等到 `ConnectTimeout`。node 很多時這段很慢。若你已知是 cephadm 叢集、也知道哪台 mon 健康，直接 `--mode cephadm --seed USER@HOST` 跳過探測（`README §auto 的限制`）。
</Callout>

### 釘死單層：`--mode cephadm` / `--mode rook`

只想收 ceph 層、且指定 seed：

```bash
bash experiments/ceph-incident-bundle/run/collect.sh \
  --inventory experiments/ceph-incident-bundle/inventory/ceph-lab.example.env \
  --ssh-key .ssh/id_ed25519 \
  --seed ikaros@192.168.18.166 \
  --mode cephadm \
  --since 24h
```

`--mode rook` 則只收 rook 層（在第一台有 `kubectl` 的 node 上跑）。差別在**硬失敗判定**：明確 `--mode cephadm` 時若沒有任何 cephadm-capable node，是 exit 2；`auto` 則寬容（`run/collect.sh:223-233`）。

### Rook：remote（預設）還是 local kubectl

Rook 層的 `kubectl` 跑在哪裡由 `--kube-mode` 決定（`run/collect.sh:331` 預設 `remote`）：

- **`remote`（預設）**：在第一台有 `kubectl` 的 **inventory node** 上、經 SSH 跑 `kubectl`（kubectl/kubeconfig 在 node 上時用這個）。
- **`local`**：在**你的工作機本機**跑 `kubectl`（kubectl/kubeconfig 在工作機、不在 node 上時用這個）。此時 `environment.txt` 會記 `rook_source=local`（`run/collect.sh:205-207`，`tests/test-collect.sh:531` 斷言背書）。

兩種都可加 `--kube-context CTX` 指定 context（`run/collect.sh:361-368`）。

```bash
# kubeconfig 在工作機本機、要用某個 context
bash experiments/ceph-incident-bundle/run/collect.sh \
  --inventory inventory/rook.env --ssh-key .ssh/id_ed25519 \
  --mode rook --kube-mode local --kube-context my-cluster --since 24h
```

### external ceph + rook：一份 inventory 混收

external 拓樸（Ceph 在 cluster 外、k8s 用 Rook external CephCluster 接）可以把 **external ceph 主機和有 `kubectl` 的 k8s node 列進同一份 `HOSTS`**（README §external）。`auto` 會：ceph 層從 ceph 主機收、rook 層在 k8s node 上跑 kubectl 收。當**資料 CR 在一個 namespace、operator 在另一個 namespace** 時，兩個 namespace 都用 inventory 變數指定——`ROOK_NAMESPACE`（資料 CR）與 `ROOK_OPERATOR_NAMESPACE`（operator）：

```bash
SSH_USER="ikaros"
ROOK_NAMESPACE="rook-ceph-external"    # 資料 CephCluster / CephBlockPool 在這
ROOK_OPERATOR_NAMESPACE="rook-ceph"    # operator Pod 在這
HOSTS=(
  "mon01=10.0.0.1"   # external ceph（有 cephadm）
  "osd01=10.0.0.2"   # external ceph
  "k8s1=10.0.0.9"    # k8s node（有 kubectl）
)
```

```bash
bash experiments/ceph-incident-bundle/run/collect.sh \
  --inventory inventory/external.env --ssh-key .ssh/id_ed25519 \
  --mode rook --kube-context my-cluster
```

`ROOK_OPERATOR_NAMESPACE` 不填時預設 `rook-ceph`（`run/collect.sh:465` 寫死字面值）——**不會**自動繼承資料 namespace，所以資料 ns 非 `rook-ceph`、operator 又不在 `rook-ceph` 時，必須明確設 `ROOK_OPERATOR_NAMESPACE`；operator log 就從這個 operator namespace 找。（`lib/collect-cluster-rook.sh:141` 有「不填等於資料 ns」的 collector 內部 fallback，但 `collect.sh` 入口永遠傳非空值，走 `collect.sh` 不會觸發它。）這個「資料 ns 與 operator ns 分離」的組合由 `tests/test-rook-collector.sh:210`（`--namespace rook-ceph-external --operator-namespace rook-ceph`）背書。注意 `--operator-namespace` **不是** `collect.sh` 的 CLI flag，只透過 inventory 的 `ROOK_OPERATOR_NAMESPACE` 設定。

### 選用：`--prom-url` 把 Prometheus metrics 也 dump 進來

給 `--prom-url` 時，工具會在收完 cluster 證據後，從那台 Prometheus 把「執行當下往回 `--since` 窗」內、job 名稱符合 `--prom-job-regex`（預設 `ceph|node`）的每個 metric 各打一次 `query_range`，原始 JSON 逐一 gzip 存進 `cluster/prometheus/<job>/<metric>.json.gz`（`lib/collect-prometheus.sh`）。不給 `--prom-url` 就完全不碰 Prometheus。

```bash
bash experiments/ceph-incident-bundle/run/collect.sh \
  --inventory experiments/ceph-incident-bundle/inventory/ceph-lab.example.env \
  --ssh-key .ssh/id_ed25519 --mode cephadm --since 24h \
  --prom-url http://192.168.18.166:9095
```

四個相關參數（全部 `run/collect.sh:42-46`）：

| 參數 | 預設 | 說明 |
|---|---|---|
| `--prom-url URL` | 無（不給就不碰 Prometheus） | Prometheus base URL。要從**工作機**直接可達，不走 SSH tunnel。 |
| `--prom-job-regex RE` | `ceph\|node` | scrape-job 過濾。只 dump job 名稱符合的。 |
| `--prom-step SECONDS` | `max(15, ceil(window/10000))` | `query_range` 的 step。自動值避開 Prometheus 每 series 11,000 點上限（`lib/collect-prometheus.sh:43-48`）。 |
| `--prom-timeout SECONDS` | `600` | 整段 metrics dump 的總時間預算。超時會截斷。 |

- **前置**：工作機要有 `curl` 與 `python3`。缺任一 → 寫 `cluster/prometheus/SKIPPED.txt` + exit 2（`lib/collect-prometheus.sh:64-72,154-157`）。
- **超時**：dump 超過 `--prom-timeout` 會截斷，在 `dump-info.txt` 與 job 的 `index.txt` 標 `TRUNCATED: budget <n>s exceeded`（`lib/collect-prometheus.sh:265-272`）。
- **exit code 語意不變**：dump 失敗或截斷 → exit 2（partial），bundle 照樣產出。
- **安全界線**：`<job>/<metric>.json.gz` 是純數值 time series，**刻意不做** redaction（逐行 redact 單行大 JSON 極慢，且 regex 誤中會讓整檔變 `[REDACTED]`）；`dump-info.txt`／`index.txt`／`buildinfo.json`／`targets.json` 照常 redact（`lib/bundle.sh:169-181`）。URL 內嵌的 `user:pass@` 寫進任何 artifact 前會遮成 `user:***@`（`lib/collect-prometheus.sh:51-60`）。

這一層已於 2026-07-10 對真 Prometheus（v3.12.0 + 真 cephadm v19.2.3 lab，103 metrics 全數 dump 成功）驗證過，逐項證據見 `experiments/ceph-incident-bundle/PROM-VALIDATION-2026-07.md`。

| `--prom-timeout SECONDS` | `600` | prom dump 總時間預算。 |
| `--timeout SECONDS` | `20` | **單一指令 / SSH 連線**的逾時。 |
| `--node-timeout SECONDS` | `600` | **單一 node 整輪收集**的逾時。 |
| `--skip-logs` | 關 | 跳過整個 Ceph log 收集（`/var/log/ceph` listing 與 log 複製都不做）。 |
| `--trust-ssh-host-key` | 開 | 自動接受新 host key（`StrictHostKeyChecking=accept-new`）。 |
| `--no-trust-ssh-host-key` | — | 改回 OpenSSH 預設 host-key 檢查行為。 |
| `--redact` | 開 | 打包前 redact 明顯 secret。 |
| `--no-redact` | — | 保留原始內容、不做 redaction。 |
| `--quiet` | 關 | 不印進度到 stderr（`bundle:` 與錯誤照舊）。 |
| `--keep-workdir` | 關 | 保留打包前的暫存 workdir 供 debug。 |
| `--help` | — | 印說明。 |

### 兩層 timeout 不要搞混

`--timeout`（預設 20）是**一條指令**的逾時；`--node-timeout`（預設 600）是**一台 node 整輪**的逾時（`run/collect.sh:47-48`）。兩者分開，是為了「慢或大的 node 不會被單指令逾時誤殺」（`run/collect.sh:271-274`）。另有兩個細節：

- **heavy timeout**：`dmesg` 與 ceph journal 在高負載下可能很大，工具給它們至少 120s（不會被 20s 誤砍，`lib/collect-node.sh:344-349`）。
- **`# TRUNCATED` 標記**：被逾時砍掉（exit 124/137）的指令，輸出末尾會標 `# TRUNCATED: command timed out after <n>s`，讓判讀者知道是截斷不是自然結束（`lib/common.sh:267-269`）。

<Callout type="warning" title="工作機沒有 timeout/gtimeout 時外層逾時會停用">
預設 macOS 沒有 GNU `timeout`。此時開頭會印一行 `WARNING: no 'timeout'/'gtimeout' found ...`（`run/collect.sh:472-474`），外層逾時停用，只靠 SSH `ConnectTimeout`/`ServerAlive` 把關——一台真的卡死的 node 可能拖比較久。要完整把關可 `brew install coreutils`（提供 `gtimeout`），或改在 Linux ops 機上跑。
</Callout>

## 進度與 stdout 契約

執行時進度會印到 **stderr**（探測每台 node、cluster ceph 的逐條指令 `[k/24]`、每台 node 收集、redact/verify/packaging）。**stdout 只會有最後一行 `bundle: <path>`**（`README §進度顯示`、`tests/test-collect.sh:577` 背書）。這讓 script 可以乾淨地抓路徑：

```bash
# 安靜模式，只取 bundle 路徑（cron / 腳本用）
BUNDLE=$(bash experiments/ceph-incident-bundle/run/collect.sh \
  --inventory inv.env --ssh-key key --since 24h --quiet | sed 's/^bundle: //')
```

`--quiet` 抑制進度，但 `bundle:` 與錯誤訊息照舊（`run/collect.sh:417-420`、`lib/common.sh:112-115`）。

## exit code 怎麼判讀

工具只用三種 exit code（`run/collect.sh` usage `:62-63`），讓非 Ceph 人也能快速決定下一步：

| exit code | 意義 | 下一步 |
|---|---|---|
| `0` | 收集完成，沒有已知失敗，bundle 已產生並雙層驗證通過。 | 保留 `.tar.gz`，交給判讀的人。 |
| `2` | 有部分指令或部分 node 失敗，但 bundle 已產生。 | **先不要丟掉 bundle**；看 `summary.txt` 與 `errors.log`，必要時針對失敗 node 補跑。 |
| `1` | 使用方式、必要輸入或 bundle 驗證失敗。 | 檢查 inventory、SSH key、mode；若是 verify 失敗，看保留下來的 workdir。 |

<Callout type="tip" title="叢集故障本身不算收集失敗">
關鍵語意（`README §exit code`）：**OSD down、MON down、PG stuck 這類「叢集本身的故障」會被完整收進 bundle，不算收集失敗，exit 仍是 `0`**。工具在乎的是「證據有沒有收到」，不是「叢集健不健康」。exit 2 代表的是「某些**收集動作**失敗」（例如一台 node SSH 不通），不是「Ceph 有問題」。這個設計刻意把「部分失敗」和「完全不能用」分開——事故現場最常見的就是某台 node 半死不活，不能因為它的 `journalctl` 失敗就讓其他 node 的證據也拿不到。
</Callout>

## bundle 結構

產物是一個 gzip tarball，解開後主要長這樣：

```text
README-FIRST.txt        # 打開 bundle 先讀這份
CONTENTS.md             # 人類可讀目錄：每個檔是什麼 + 產生它的完整指令與 exit code
summary.txt             # 本次摘要：mode/seed/node 成功失敗數/final status
environment.txt         # 收集時間、mode、seed、git commit、ceph_source/ceph_runner/rook_source（+prom 欄位）
manifest.jsonl          # 每個 artifact 一行 JSON（machine-readable）
errors.log              # 非零 exit、SSH 失敗、部分失敗
redactions.log          # 每個檔遮蔽了幾行
ssh-debug/              # （有 SSH 連線失敗時才有）ssh -vvv 診斷 log
cluster/
  ceph/
    json/               # 20 個 JSON 狀態
    text/               # 4 個 text 狀態
  rook/                 # Rook 模式時的 kubectl 收集
  prometheus/           # （有給 --prom-url 才有）<job>/<metric>.json.gz + dump-info.txt
nodes/
  monitor01/            # 每台 node 一個資料夾（名字＝inventory 的 alias）
  mon02/
  ...
```

六個入口檔用途：

| 檔案 | 用途 |
|---|---|
| `README-FIRST.txt` | 給打開 bundle 的人先讀，指出從哪裡開始（`lib/bundle.sh:31-41`）。 |
| `CONTENTS.md` | **人類可讀目錄**：分 cluster 一段、每台 node 一段，逐檔列出「這是什麼 + 產生它的完整指令 + exit code」，直接從 manifest 產生、永遠與實際收到的一致（`lib/bundle.sh:108-137`、`tests/test-collect.sh:250-255` 背書）。想知道「某個檔是哪條指令跑出來的」看這份最快。 |
| `summary.txt` | 本次收集摘要：建立時間、mode、seed、node 成功/失敗數、final status（`lib/bundle.sh:56-68`）。 |
| `environment.txt` | 收集時間、mode、seed、git commit，以及 auto 選到的 `ceph_source`/`ceph_runner`/`rook_source`；有 `--prom-url` 再加 `prom_url`/`prom_jobs`（`run/collect.sh:238-242`、`lib/collect-prometheus.sh:320-323`）。 |
| `manifest.jsonl` | 每個 artifact 的 host、collector、command、exit code、起訖時間。追查時的機器可讀索引（`CONTENTS.md` 是它的可讀版）。 |
| `errors.log` | 非零 exit code、SSH 失敗、部分失敗紀錄。 |

三份收集清單（逐項對過現行碼）：

### cephadm cluster-level（`lib/collect-cluster-cephadm.sh`）

在選定 runner 上收 **20 個 JSON**（`:142-163`）：`status`、`health-detail`、`versions`、`df-detail`、`osd-tree`、`osd-df`、`osd-dump`、`osd-perf`、`osd-blocked-by`、`pg-stat`、`pg-dump`、`pg-dump-stuck`、`mon-dump`、`quorum-status`、`mgr-dump`、`orch-host-ls`、`orch-ps`、`orch-device-ls-wide`、`config-dump`、`crash-ls`，外加 **4 個 text 版**方便人直接看：`status.txt`、`health-detail.txt`、`osd-tree.txt`、`orch-ps.txt`（`:165-170`）。若 `crash ls` 有資料，最多取 **10 筆** crash id 跑 `ceph crash info`，並把 id 轉成安全檔名避免奇怪字元影響 tar 路徑（`:63,80-88`）。

### node-level（`lib/collect-node.sh`）

每台 node 收（`:351-431`）：基本系統（`hostname`/`uname -a`/`uptime`）、資源（`free -h`、可用時 `iostat -xz 1 3`）、storage（`df -hT`/`lsblk`、可用時 `pvs`/`vgs`/`lvs`）、網路（`ip addr show`）、kernel（`dmesg -T`）、systemd（`systemctl --failed`、`journalctl -u 'ceph*'`）、time sync、container（可用時 `podman ps -a`/`docker ps -a`）、cephadm（可用時 `cephadm ls`）、Ceph log（`/var/log/ceph` listing 與可讀 log）、以及安全列出 `/var/lib/ceph` 並複製設定檔（**排除 keyring/private_key/.ssh**，`:263,291`）。

time sync 把 `systemd-timesyncd` 放在一等公民：若 node 用 timesyncd，會收 `timedatectl status`、`timedatectl show-timesync --all`、`timedatectl timesync-status`、`systemctl status systemd-timesyncd`、`journalctl -u systemd-timesyncd`，以及 `/etc/systemd/timesyncd.conf` 與 `.conf.d/*.conf`（`:395-408`；設定檔複製＝`collect_timesyncd_config`，`:149-182`）。這裡刻意同時收 Ceph 與 Linux——Ceph 的 `HEALTH_WARN` 只告訴你 cluster 覺得哪裡不對，真正原因常在 disk、kernel、time sync、container runtime 或 systemd。

### Rook cluster-level（`lib/collect-cluster-rook.sh`）

收（`:173-205`）：`kubectl get pods -o wide`、`get events --sort-by=.lastTimestamp`、`CephCluster`/`CephBlockPool`/`CephFilesystem`/`CephObjectStore` 的 YAML、operator log、toolbox Pod 存在時的 `ceph status`。namespace 不存在或 kubectl 連不上時寫 `SKIPPED.txt` 並帶上具體原因（`:164-169`），不把整包當失敗（`auto` 下）。

## 失敗處理

工具是 read-only，所以「回退」的重點不是「怎麼還原改動」（沒有改動），而是**「沒收成功怎麼辦」**。

### verify 失敗：workdir 被保留、不打包

工具打包前後各驗一次（verify → tar → verify）。**verify 失敗時不會產出可分享的 `.tar.gz`，而是保留 workdir 並印出路徑**，讓已收集的證據不會因驗證失敗被刪掉（`run/collect.sh:577-606`）：

- **打包前 verify 失敗**：`VERIFY FAILED: workdir kept at <path> (not packaged) — review errors.log`，exit 1，workdir 留著。
- **打包後 verify 失敗**：`rm` 掉那個 tar、保留 workdir，印 `VERIFY FAILED on packaged bundle; removed it, workdir kept at <path>`，exit 1。

去 workdir 看 `errors.log`。verify 會擋兩類東西（`lib/verify-bundle.sh`）：

1. **檔名黑名單**（`:27`）：路徑含 `keyring`、`.ssh`、`id_ed25519`、`private_key`，或副檔名 `*.pem`/`*.key`/`*.crt`/`*.pfx`/`*.p12`。
2. **內容層級**（`:35-44`，defense-in-depth）：檔案內容仍殘留 `-----BEGIN ... PRIVATE KEY-----` 區塊、或 `key = <base64 20+字元>` 這種金鑰材料。

### Ctrl-C：立即中止、清乾淨、exit 130

收集中途按 Ctrl-C（或收到 SIGTERM）會**立即停止**，清掉暫存 workdir，exit 130（`lib/bundle.sh:156-161`）。沒有這個 handler 的話，bash 會清完當前的 EXIT trap 後繼續跑下一台 node，變成「按了 Ctrl-C 卻停不下來」。

### `ssh-debug/` 怎麼判讀

SSH 連線出問題（exit 255＝連線失敗、124/137＝逾時被砍）時，工具會自動補跑一次 `ssh -vvv -o LogLevel=DEBUG3 <target> true` 並把完整輸出落在 `ssh-debug/<label>-<target>.log`（`lib/common.sh:57-93`）。觸發點有三：capability 探測失敗（`run/collect.sh:88`）、node 收集 SSH 失敗（`run/collect.sh:290-292`）、cluster ceph SSH 失敗（`lib/collect-cluster-cephadm.sh:39-41`）。`tests/test-collect.sh:46-47` 斷言 archive 內確實有這份 log。判讀時往 log 底部看 `Permission denied`（key/帳號）、`Host key verification failed`（known_hosts）、`Connection timed out`（網路/host 掛了）等關鍵字。

### 其他常見失敗

| 症狀 | 意義 / 處理 |
|---|---|
| `missing inventory` / `missing ssh key`（exit 1） | `--inventory` / `--ssh-key` 路徑不存在或不可讀。 |
| `node <alias> collector exited 255`（→ exit 2） | SSH 連線、帳號、key、**known_hosts** 或 sudo 權限問題。新跳板機最常見是 known_hosts 還沒 key。 |
| 某台 node `SKIPPED.txt`：`node archive ... is incomplete (no manifest.jsonl)` | 該 node 串回的 archive 缺自己的 manifest＝傳輸被截斷／中斷，工具**不把它算成功**（`run/collect.sh:304-307`）。針對這台補跑。 |
| exit code `2` | 先不要重跑覆蓋判讀脈絡，先保留 `.tar.gz`，看 `errors.log` 決定要不要針對失敗 node 補跑。 |

## 安全界線

事故工具最怕「看起來方便、實際上改了現場或洩了 secret」。這套工具的界線：

- **read-only**：不會 restart、delete、repair、scrub，也不改任何 Ceph flag（`README §安全界線`）。
- **redaction（預設開）**：遮蔽文字 artifact 中——含 `password`/`secret`/`token`/`keyring`/`private key` 的整行、Ceph 金鑰材料（`key = AQB..==` 與 base64 區塊）、整段多行 PEM private key block；並會把 `*.gz` 解壓後遮蔽再壓回（`lib/common.sh:164-224`）。**但這不是完整 DLP**。用 `--no-redact` 則保留原始內容。
- **verify 雙層把關**：`verify-bundle.sh` 以**檔名**（keyring/.ssh/id_ed25519/private_key/`*.pem`/`*.key`/`*.crt`/`*.pfx`/`*.p12`）與**內容**（殘留的 PRIVATE KEY block / `key = <base64>`）兩道擋，但仍不能保證內容完全沒有敏感資料。
- **`--skip-logs`** 可降低 log 量與敏感內容風險，但會少掉事故判讀常需要的時間序列。
- **分享前仍要人工檢查**：內部 IP、hostname、帳號名稱、mount path、pool name、image name 都可能是敏感資訊。要把 bundle 丟給外部人士前，走你自己的資料外洩檢查流程。

換句話說：這套工具會盡力避免打包明顯 secret，但它不是資安掃描器。

<Callout type="tip" title="bundle 是證據、不要 commit 進 repo">
產生的 `.tar.gz` 是事故證據，不該進版控。`experiments/ceph-incident-bundle/results/*.tar.gz` 與 `*.tar.gz` 已被 `.gitignore` 排除。
</Callout>

## 結論：事故第一步變成固定肌肉記憶

這套流程把 Ceph 事故處理的第一步，從「誰在線上、誰知道該跑什麼 command」改成「先跑一個固定 script，把當下證據打包」。

對只懂基本 Linux 的操作者來說，責任很小：確認 SSH、填 inventory、跑 `collect.sh`、看 exit code、把 bundle 保留下來。對後續判讀者來說，責任也更清楚：從同一包裡看 Ceph 狀態、node 狀態、kernel/systemd/log、manifest 與 errors，而不是追問「你剛剛有沒有順手跑某某 command」。

最重要的是：這套工具不假裝會修 Ceph。它只做事故現場最該先做、也最容易被忘記的事——**把證據留住。**
