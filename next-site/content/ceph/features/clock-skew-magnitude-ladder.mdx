## 場景

[10ms 到 5s 那一頁](./cluster-time-skew-impact) 把毫秒級的差距攤開過了，結論是「沒有全域的 50ms 規則」。這頁把刻度往上拉——**51ms、401ms、1000ms、1 分鐘、1 小時、1 天**——問每跨過一個量級，多壞掉什麼。

答案是：cluster 裡有**三條互相獨立的線**，各自有自己的門檻、自己的受害者、自己的時鐘來源。先看表，再逐條拆。

## 一張表看完

預設值、3-mon cluster，沒手動改 `mon_clock_drift_allowed` / `mon_lease`。穩態 skew（固定 offset）下：

| 差距 | 線 1：MON ↔ MON | 線 2：MON ↔ OSD（cephx） | 線 3：OSD ↔ OSD |
|---|---|---|---|
| **51ms** | `MON_CLOCK_SKEW` WARN | 無感 | 無門檻 |
| **401ms** | 同上 WARN，lease 仍正常 | 無感 | 無門檻 |
| **1000ms** | 同上 WARN，lease 仍正常 | 無感 | 無門檻 |
| **1 min** | WARN **＋ lease 崩 → election churn** | 無感（≪ 1hr TTL） | 無門檻 |
| **1 hr** | 同 1 min | **進風險區**（到 `auth_service_ticket_ttl` 等級） | 無門檻 |
| **1 day** | 同 1 min | **OSD 起不來 `exit(1)`**（OSD 比 mon 快時） | 無門檻 |

三句話讀法：

- **線 1（MON↔MON）** 有兩個量級門檻：50ms 報警、5s 真正崩。
- **線 2（MON↔OSD）** 量級門檻在 1hr（`auth_service_ticket_ttl`）等級，**但它還有一個跟量級無關的觸發**：mon 時鐘「跳回」那一刻會讓在線 OSD 大量掉線——這個不在上表（它不是穩態 skew），是線 2 的主菜。
- **線 3（OSD↔OSD）** 整欄空白：OSD 之間走 mono clock，wall-clock skew 完全不影響。

## 心智模型：誰用 wall clock、誰用 mono clock

三條線會不會被 skew 弄爆，只取決於兩個問題：**它用哪種時鐘**，以及**到期時間是「絕對時間點」還是「相對時長」**。

| | 用哪種時鐘 | 到期算法 | 對 skew |
|---|---|---|---|
| 線 1 MON↔MON lease | `real_clock`（wall clock） | leader 蓋「leader_now + 5s」，peon 拿自己 wall clock 比 | **不免疫** |
| 線 2 MON↔OSD cephx | `ceph_clock_now()`（wall clock） | **mon 蓋絕對 expiration**，OSD 拿自己 wall clock 比 | **不免疫（且雙向）** |
| 線 3 OSD↔OSD | `mono_clock` | 用 heartbeat 帶的 mono delta 換算 | **免疫** |
| （對照）client ticket | wall clock | **client 自己蓋 `now + validity`**，自己檢查 | **免疫（同機抵消）** |

關鍵差別只有一個：**到期時間是「誰蓋的、誰檢查的」**。

- 同一台機器蓋、同一台檢查（client ticket、線 3 的 delta）→ skew 在減法裡抵消 → 免疫。
- 一台蓋、另一台檢查（線 1 lease、線 2 cephx）→ 兩台時鐘直接比大小 → skew 一旦到門檻就爆。

線 2 最特別：**蓋章的永遠是 mon**（發證方），所以 mon 的時鐘錯誤會污染所有 OSD 手上的票。這就是為什麼線 2 是雙向的，下面會看到。

## 線 1 — MON ↔ MON：50ms 報警、5s lease 崩

### 50ms：偵測線

leader 對 quorum 內每個 peon（quorum 內非 leader 的 mon）跑 timecheck（ping/pong），算出 `skew_bound = |delta| − latency`，再跟 `mon_clock_drift_allowed` 比。判定是**嚴格大於**：

File: `ceph/src/mon/Monitor.h` (line 592)

```cpp
bool timecheck_has_skew(const double skew_bound, double *abs) const {
  double abs_skew = std::fabs(skew_bound);
  if (abs)
    *abs = abs_skew;
  return (abs_skew > g_conf()->mon_clock_drift_allowed);
}
```

`mon_clock_drift_allowed` 預設 `0.05`（50ms，`mon.yaml.in:609`）。所以 **51ms** 剛好踩過門檻（同 DC latency 通常 `< 1ms`，`skew_bound` 仍 `> 50ms`），下一輪 timecheck 後報 `MON_CLOCK_SKEW`。而 `401ms` 到 `1 day` 全都遠在門檻之上，**報的是同一條 WARN，沒有分級**：

File: `ceph/src/mon/HealthMonitor.cc` (line 862)

```cpp
auto& d = checks->add("MON_CLOCK_SKEW", HEALTH_WARN, ss.str(), details.size());
d.detail.swap(details);
```

整份 source 裡 `MON_CLOCK_SKEW` 只 add 過這一次，severity 寫死 `HEALTH_WARN`。**它永遠不會自己升成 `HEALTH_ERR`**——1 天的 skew 跟 51ms 在 `ceph -s` 的顏色上一模一樣。`ceph -s` 多久才看到這條 WARN，拆在 [MON clock skew 偵測那一頁](./mon-clock-skew-detection)。

### 5s：lease 崩（真正出事的地方）

報警歸報警，MON↔MON **真正崩**是在 5s。原因：Paxos lease 的到期時間用 `ceph::real_clock`（wall clock），不是 monotonic。

File: `ceph/src/mon/Paxos.h` (line 405)

```cpp
ceph::real_clock::time_point lease_expire;
```

leader 每次 renew lease，把「**自己 wall clock 的** now + `mon_lease`」當到期點送出去：

File: `ceph/src/mon/Paxos.cc` (line 979)

```cpp
lease_expire = ceph::real_clock::now();
lease_expire += ceph::make_timespan(g_conf()->mon_lease);
```

`mon_lease` 預設 `5` 秒（`mon.yaml.in:379`）。peon 收到後，拿 leader 送來的到期時間跟**自己的** wall clock 比：

File: `ceph/src/mon/Paxos.cc` (line 1118)

```cpp
if (auto new_expire = lease->lease_timestamp.to_real_time();
    lease_expire < new_expire) {
  lease_expire = new_expire;
  auto now = ceph::real_clock::now();
  if (lease_expire < now) {
    auto diff = now - lease_expire;
    derr << "lease_expire from " << lease->get_source_inst() << " is " << diff
         << " seconds in the past; mons are probably laggy (or possibly clocks are too skewed)" << dendl;
  }
}
```

分界就在這裡：lease 訊息帶的是 leader 時間軸上的「now + 5s」。如果**這個 peon 的 wall clock 比 leader 快超過 5 秒**，`lease_expire` 在 peon 自己的時間軸上**已經是過去式**——`lease_expire < now` 成立，印 `clocks are too skewed`，lease 一拿到就等於過期。

到期的 lease 過不了寫入閘：

File: `ceph/src/mon/Paxos.cc` (line 1514)

```cpp
bool Paxos::is_lease_valid()
{
  return ((mon.get_quorum().size() == 1)
	  || (ceph::real_clock::now() < lease_expire));
}
```

`is_lease_valid()` 回 false → `is_writeable()` / `is_readable()` 跟著垮 → peon 的 `lease_timeout` 觸發 → `mon.bootstrap()` → 退出 quorum 重選。**所以 1min / 1hr / 1day 的 MON↔MON skew 做的是同一件事**：那台快超過 5s 的 mon 守不住 lease，反覆 bootstrap、反覆 election。3-mon cluster 裡只有 1 台這樣時，另外 2 台同步的 mon 仍能撐住 quorum=2（cluster 帶 WARN + 一台 mon 一直 flap）；2 台以上 skew 就掉進 quorum loss。timeout → bootstrap → election 那條鏈拆在 [mon quorum loss 那一頁](./mon-quorum-loss-impact)。

### 附帶：訊息被戳「在未來」洗 log

只要 skew `> 50ms`，skew mon 每次送出 lease / lease-ack 訊息，都會被收方判定「戳在未來」，寫一行 cluster log（同一個 `mon_clock_drift_allowed` 門檻 + 指數 backoff）：

File: `ceph/src/mon/Paxos.cc` (line 1025)

```cpp
void Paxos::warn_on_future_time(utime_t t, entity_name_t from)
{
  utime_t now = ceph_clock_now();
  if (t > now) {
    utime_t diff = t - now;
    if (diff > g_conf()->mon_clock_drift_allowed) {
      // ... backoff 後 ...
      mon.clog->warn() << "message from " << from << " was stamped " << diff
                       << "s in the future, clocks not synchronized";
```

1 天的 skew 在這裡印 `stamped 86400s in the future`。只是 cluster log，不是 health check。

## 線 2 — MON ↔ OSD：cephx 到期由 mon 蓋，雙向都會爆

MON↔OSD 之間有兩條跟時間有關的路徑，命運相反：**beacon 那條免疫，cephx 那條不免疫**。

### 不出事的那半：beacon 走 mono clock

OSD 多久送一次 beacon、MON 多久沒收到 beacon 就 mark down——OSD 端用 monotonic clock 決定多久送一次 beacon；MON 端量的是「自己收到兩次 beacon 的間隔」（用 MON 自己的時鐘相減），兩邊都跟 OSD↔MON 的 wall-clock 差無關：

File: `ceph/src/osd/OSD.cc` (line 6445)

```cpp
const auto now = ceph::coarse_mono_clock::now();
{
  std::lock_guard l{min_last_epoch_clean_lock};
  const auto elapsed = now - last_sent_beacon;
  if (std::chrono::duration_cast<std::chrono::seconds>(elapsed).count() >
    cct->_conf->osd_beacon_report_interval) {
    need_send_beacon = true;
  }
}
```

MON 端 mark down 的門檻是 `max(mon_osd_report_timeout, 2 × osd_beacon_report_interval) = max(15min, 10min) = 15 分鐘`，量的是 MON 收到兩次 beacon 的*間隔*。**OSD 時鐘偏 1 天，只要它還在送 beacon，這條不會踢它。**（`mon_osd_report_timeout` 預設 `15_min`、`osd_beacon_report_interval` 預設 `5_min`。）

### 出事的那半：cephx 的到期是「mon 蓋的絕對時間戳」

每個 OSD 都要持有一份 mon 簽發、會過期、由 mon 定期重簽的 rotating service key。它不是單一 key，是一組 previous / current / next；`current()` 取中間那把：

```cpp
// File: ceph/src/auth/Auth.h (line 255)
#define KEY_ROTATE_NUM 3
```

```cpp
// File: ceph/src/auth/Auth.h (line 336)
const ExpiringCryptoKey& current() const {
  auto p = secrets.begin();
  ++p;
  return p->second;
}
```

mon 產生 key 時，用**自己的 wall clock** 蓋一個**絕對** expiration（`now + ttl`，service key 的 ttl 是 `auth_service_ticket_ttl`，預設 1 小時）：

File: `ceph/src/auth/cephx/CephxKeyServer.cc` (line 180)

```cpp
utime_t now = ceph_clock_now();
double ttl = service_id == CEPH_ENTITY_TYPE_AUTH ? cct->_conf->auth_mon_ticket_ttl : cct->_conf->auth_service_ticket_ttl;
while (r.need_new_secrets(now)) {
  ExpiringCryptoKey ek;
  // ...
  ek.expiration += ttl;
```

OSD 收到後，拿**自己的 wall clock** 判斷夠不夠新——`current().expiration <= now` 就算「該換了」：

File: `ceph/src/auth/Auth.h` (line 324)

```cpp
bool need_new_secrets(const utime_t& now) const {
  return secrets.size() < KEY_ROTATE_NUM || current().expiration <= now;
}
```

**mon 蓋、OSD 比——兩台時鐘直接對撞。** 兩台差到 TTL（1hr）等級就出事，而且 mon 與 OSD 哪邊快，爆法不同。下面兩個方向分開講。

### 方向 A — OSD 比 mon 快：OSD 起不來

OSD 週期檢查 rotating key 時，`now` 取自己的時鐘，cutoff 再往前抓 30 秒：

File: `ceph/src/mon/MonClient.cc` (line 1103)

```cpp
utime_t now = ceph_clock_now();
utime_t cutoff = now;
cutoff -= std::min(30.0, cct->_conf->auth_service_ticket_ttl / 4.0);
utime_t issued_at_lower_bound = now;
issued_at_lower_bound -= cct->_conf->auth_service_ticket_ttl;
if (!rotating_secrets->need_new_secrets(cutoff)) {
  // have uptodate secrets
  return 0;
}
```

如果 OSD 領先 mon 到 1 小時等級，mon 能給的最新 secret 在 OSD 的 `now − 30s` cutoff 下都嫌太舊，`need_new_secrets(cutoff)` 一直回 true，OSD 不停要求 renew 卻永遠拿不到夠新的。source 自己就標出這個徵兆：

File: `ceph/src/mon/MonClient.cc` (line 1115)

```cpp
if (!rotating_secrets->need_new_secrets() &&
    rotating_secrets->need_new_secrets(issued_at_lower_bound)) {
  // the key has expired before it has been issued?
  lderr(cct) << __func__ << " possible clock skew, rotating keys expired way too early"
             << " (before " << issued_at_lower_bound << ")" << dendl;
}
```

OSD 啟動時這條會直接擋住上線。`wait_auth_rotating` 等不到夠新的 key 就逾時（`rotating_keys_bootstrap_timeout` 30 秒），重試到 `max_rotating_auth_attempts`（預設 10）就放棄、`exit(1)`：

File: `ceph/src/osd/OSD.cc` (line 4068)

```cpp
while (monc->wait_auth_rotating(rotating_auth_timeout) < 0) {
  derr << "unable to obtain rotating service keys; retrying" << dendl;
  ++rotating_auth_attempts;
  if (rotating_auth_attempts > g_conf()->max_rotating_auth_attempts) {
      derr << __func__ << " wait_auth_rotating timed out"
           <<" -- maybe I have a clock skew against the monitors?" << dendl;
      exit(1);
  }
}
```

**方向 A 一句話**：OSD node 的 wall clock 比 mon 快上「天」級 → 啟動時 `exit(1)`，硬失敗。修法不是調 ceph，是把 OSD node 時鐘拉回跟 mon 一致（day 級偏差通常是 RTC 壞、時區設成 localtime、或根本沒在跑 timesyncd/chrony）。

### 方向 B — mon 先快、再跳回：在線 OSD 大量掉線（最痛）

實務上更常見、破壞力更大的不是方向 A 的穩態 skew，而是 **mon 時鐘先錯、再被 NTP 校回** 的那一刻。因為 mon 是發證方，它的時鐘錯誤會污染所有 OSD 手上的票，而**修正時鐘才是引爆點**。三步：

**Step 1 — mon 時鐘 +1 年，簽出「未來」的 secret。** mon rotate 時用自己（快了一年的）`ceph_clock_now()` 蓋 expiration，所以每把 secret 的到期都在約一年後：

File: `ceph/src/auth/cephx/CephxKeyServer.cc` (line 187)

```cpp
ek.expiration = now;            // now = mon 快了一年的 wall clock
// ... 或對 next key：
ek.expiration = std::max(next_ttl, r.next().expiration);
ek.expiration += ttl;
```

OSD 收下這批 secret/ticket，自己時間是對的，`have_key()` 檢查 `ceph_clock_now() < expires`（真實 now `<` 真實 now+1年）為真，**全部視為有效**——所以 mon 時鐘錯的當下 cluster 表面正常，沒人掉線：

File: `ceph/src/auth/cephx/CephxProtocol.cc` (line 215)

```cpp
bool CephXTicketHandler::have_key() {
  if (have_key_flag) {
    have_key_flag = ceph_clock_now() < expires;
  }
  return have_key_flag;
}
```

**Step 2 — mon 校時跳回，把「未來」那批 secret 擠掉。** mon 時鐘回到正確值後重新 rotate；`add()` 永遠只保留 prev/current/next 三把，舊的直接 erase。新一輪用正確時間蓋的 secret 進來，那批 +1 年的就被擠出視窗：

File: `ceph/src/auth/Auth.h` (line 316)

```cpp
while (secrets.size() > KEY_ROTATE_NUM)   // KEY_ROTATE_NUM = 3
  secrets.erase(secrets.begin());
```

**Step 3 — OSD 還抱著舊 `secret_id`，驗證直接查不到。** OSD 之間（及 OSD-client）互驗走 `verify_authorizer`：authorizer（一端連線時出示的憑證）裡帶著 ticket 的 `secret_id`，收方拿它去 mon keyserver 查那把 secret。一旦那個 `secret_id` 被 Step 2 擠掉，查找失敗 → 認證失敗 → 連線被拒：

File: `ceph/src/auth/cephx/CephxProtocol.cc` (line 459)

```cpp
if (!keys.get_service_secret(service_id, ticket.secret_id, service_secret)) {
  ldout(cct, 0) << "verify_authorizer could not get service secret for service "
    << ceph_entity_type_name(service_id) << " secret_id=" << ticket.secret_id << dendl;
  return false;
}
```

File: `ceph/src/auth/cephx/CephxKeyServer.cc` (line 85)

```cpp
ldout(cct, 10) << "get_service_secret service " << ceph_entity_type_name(service_id)
         << " secret " << secret_id << " not found" << dendl;
return false;
```

更糟的是 OSD **不會主動換掉**那批被作廢的 secret：它用 `current().expiration <= cutoff`（`cutoff = OSD_now − 30s`）判斷要不要 renew，而手上 secret 的 expiration 是 +1 年的絕對值，在 OSD 看來「還很新」→ 不 renew → 繼續拿已被 mon 作廢的 secret 互驗 → 全部失敗。表現就是：**校時跳回的那一刻，heartbeat / replication 的 authorizer 驗證成片失敗 → OSD 互判失聯 → 大量 mark down / 掉線。**

<Callout type="warning" title="方向 A 與方向 B 是兩種不同的爆法">
  - **方向 A（OSD 比 mon 快，穩態）**：OSD 起不來，硬失敗點是啟動的 `wait_auth_rotating` → `exit(1)`。
  - **方向 B（mon 先快、再跳回，transition）**：在線 OSD 大量掉線，失敗點是 `verify_authorizer` 查不到被 rotate 作廢的 `secret_id`。穩態時也許都沒事，**跳變才引爆**。

  共同根因都是「cephx 的 expiration 是絕對 wall-clock 時間戳、由 mon 蓋」。所以排查 mon 時鐘問題時，**對一台時間錯很多的 mon「校時跳回」本身就是高風險動作**：寧可挑維護窗、分批讓 mon 收斂，也不要在尖峰一步跳回。若已大量掉線，通常要逐台確認 OSD 重新拿到有效 rotating key（必要時 restart OSD 強制重新 auth）。
</Callout>

## 線 3 — OSD ↔ OSD：走 mono clock，skew 免疫

OSD 之間有兩條跟時間有關的路徑，**兩條都不看 wall clock**。

第一條，OSD service 的「現在」用 monotonic clock：

File: `ceph/src/osd/OSD.cc` (line 331)

```cpp
ceph::signedspan OSDService::get_mnow() const
{
  return ceph::mono_clock::now() - osd->startup_time;
}
```

heartbeat deadline、read lease 的 `readable_until` 全建在 `get_mnow()` 上。mono clock 不受 NTP 校正、不受 wall-clock skew 影響，所以兩台 OSD 的 wall clock 差多少，對 heartbeat 存活判斷沒有作用。

第二條，跨 OSD 的 read lease 要把對方時間軸換算成自己的，但用的是 heartbeat 帶的 **mono timestamp 差（delta）**，不是 wall clock：

File: `ceph/src/osd/PeeringState.h` (line 119)

```cpp
// we maintain an upper and lower bound on the delta between our local
// mono_clock time (minus the startup_time) to the peer OSD's mono_clock
// time (minus its startup_time).
//
// delta is (remote_clock_time - local_clock_time), so that
// local_time + delta -> peer_time, and peer_time - delta -> local_time.
```

這個 delta 是兩台 OSD mono_clock 起跑點的差，**跟 wall-clock skew 是兩回事**——你把某台 OSD 的 `date` 調快一天，這個 delta 不會跟著變。所以 OSD↔OSD 在 51ms → 1 天的任何刻度，wall-clock skew 都不觸發任何健康或 lease 問題。（OSD↔OSD 真正會 WARN 的是 heartbeat *latency* `> ~1s`，那是「訊息來回多久」不是「時鐘差多少」，拆在 [10ms 到 5s 那一頁](./cluster-time-skew-impact)。）

## 容易搞混：client ticket 不會因 skew 過期

`auth_service_ticket_ttl`(1hr) 這個數字在 cephx 裡被**兩條不同路徑**用到，skew 對它們的影響相反。線 2 講的是 **daemon rotating secret**（mon 蓋、會被 skew 弄爆）；這裡是另一條 **client service ticket**（client 自己蓋、對 skew 免疫），很容易混。

client（librbd / kernel rbd 等）的 service ticket，到期是用**自己一台機器的 wall clock 量的 duration**，收到時記 `expires = 自己的 now + validity`：

File: `ceph/src/auth/cephx/CephxProtocol.cc` (line 199)

```cpp
expires = ceph_clock_now();
expires += msg_a.validity;
renew_after = expires;
renew_after -= ((double)msg_a.validity.sec() / 4);
```

檢查到期也是同一台 client 的 wall clock（`have_key()`，`CephxProtocol.cc:215`）。`expires` 和檢查它的時鐘在**同一台機器**，固定 skew offset 在減法裡抵消——**client 跟誰差多少，對它自己這張 ticket 的壽命沒有影響**。驗 ticket 那端（OSD）的 `cephx_verify_authorizer()` 也沒有 wall-clock 到期檢查，靠的是 `secret_id` 解不解得開（`CephxProtocol.cc:422`）。

| | 誰蓋到期 | 誰檢查 | 對 skew |
|---|---|---|---|
| **client service ticket** | client 自己 `now + validity`（相對 duration） | client 自己 | **免疫**（同機抵消） |
| **daemon rotating secret** | **mon** `now + ttl`（絕對時間點） | OSD | **不免疫**（跨兩台時鐘比） |

所以「skew 1 小時讓 *client* I/O 斷」是誤解，但「skew 弄垮 *OSD daemon*」是真的。`auth_service_ticket_ttl`(1hr) 在「mon **掛掉**」（非 skew）情境下讓 client renew 不到而斷，那是另一回事，拆在 [mon quorum loss 那一頁](./mon-quorum-loss-impact)。

## 排查：看到 skew 怎麼定位

先 `ceph -s` 確認有沒有 `MON_CLOCK_SKEW`（確認真的有 skew），再依「誰在出事」分流——三條線的徵兆與 log 完全不同：

| 現象 | 是哪條線 | 去哪看 |
|---|---|---|
| mon 反覆 flap / election | 線 1，skew `> 5s` | mon log：`clocks are too skewed`、反覆 `calling new election` |
| OSD 起不來、一直重啟 | 線 2 方向 A，OSD 比 mon 快到 1hr+ | OSD log：`unable to obtain rotating service keys`、`maybe I have a clock skew against the monitors`、`rotating keys expired way too early` |
| 一台 mon 校時後，OSD 突然大量掉線 | 線 2 方向 B，mon 跳回 | OSD log：`verify_authorizer could not get service secret`；對照 mon 剛做過 NTP 大幅校時 |
| OSD 之間 heartbeat 慢、read lease 進 `laggy` | 不是 skew | 這是 latency 不是時鐘差，看 [10ms 到 5s 那一頁](./cluster-time-skew-impact) |

兩個常見誤判：毫秒到分鐘級的單台 OSD 偏移**不會**觸發任何 health warning（線 3 走 mono clock）；client I/O 斷掉**通常不是** ticket 因 skew 過期（client ticket 免疫，見上節）。

## 原始碼路徑一覽

| 線 | 入口 | 門檻 / 效果 |
|---|---|---|
| MON-MON 偵測 | `Monitor::timecheck_has_skew()` → `HealthMonitor` | `skew_bound > 50ms` → `MON_CLOCK_SKEW` WARN，不升 ERR |
| MON-MON lease | `Paxos::extend_lease()` / `handle_lease()` / `is_lease_valid()`（`real_clock`） | skew `> mon_lease`(5s) → lease 過期 → election churn |
| MON-MON log | `Paxos::warn_on_future_time()` | `> 50ms` 寫 cluster log（非 health check） |
| MON-OSD beacon | `OSD::send_beacon()`（mono）/ `OSDMonitor` 15min 沉默偵測 | 無 clock-skew 門檻（量間隔不量 wall-clock 差） |
| MON-OSD 方向 A | `KeyServer::_rotate_secret()` / `need_new_secrets(cutoff)` / `OSD::init() wait_auth_rotating` | OSD 比 mon 快到 1hr 等級 → secret 太舊 → 啟動重試 10 次 `exit(1)` |

| OSD-OSD | `OSDService::get_mnow()`（mono）/ `HeartbeatStamps` delta | 無 clock-skew 門檻 |
