## 場景

你的 pool 原本 `pg_num=128 / pgp_num=128`，object 一直加、`ceph osd df` 看到分布不均，pg-autoscale 又因為某些 quota 估計問題沒幫你調。你下了 `ceph osd pool set mypool pg_num 512`，等了幾分鐘，`ceph osd pool get mypool pg_num` 回 `pg_num: 132` — 不是 512、也不是還沒變、是某個中間數。

為什麼？兩個原因：v19.2.3 上的 ceph 並不是「下一筆 osdmap epoch 直接把 pg_num 變到 512」，而是 **target 與 actual 解耦**：使用者只設 target，mgr 每個 tick 拿著一束 throttle 規則小步推進；同時 `pg_num` 與 `pgp_num` 是兩個獨立旋鈕，前者控制「actual PG 數量與 object hash 落到哪個 PG」，後者控制「CRUSH placement seed 的展開範圍」。下面從原始碼層一路追到 OSD 端的 split / merge。

## pg_pool_t 的 5 個 PG-relevant 欄位

第一件事是搞清楚原始碼層真實有幾個欄位。`pg_pool_t` 的定義裡這 5 個欄位才是 truth source：

File: `ceph/src/osd/osd_types.h` (line 1458)

```cpp
private:
  __u32 pg_num = 0, pgp_num = 0;  ///< number of pgs
  __u32 pg_num_pending = 0;       ///< pg_num we are about to merge down to
  __u32 pg_num_target = 0;        ///< pg_num we should converge toward
  __u32 pgp_num_target = 0;       ///< pgp_num we should converge toward
```

| 欄位 | 意義 | 誰會寫 |
|---|---|---|
| `pg_num` | 實際存在的 PG 數量。每次 +1 = 一次 split；每次 −1 = 一次 merge | mgr → mon（via `pg_num_actual`） |
| `pgp_num` | CRUSH placement seed 的範圍。`pg_num > pgp_num` 時，object 仍會用 `pg_num` 落到 actual PG，但多個 actual PG 會暫時映到相同 placement seed，讓跨 OSD 搬移延後發生 | mgr → mon（via `pgp_num_actual`） |
| `pg_num_pending` | merge 進行中的中間狀態。下次只會減 1，所以 `pg_num = N` 想降到 `N-1` 時，`pg_num_pending` 會先變 `N-1`，等 OSD 完成 merge handshake 後 `pg_num` 才跟著降 | mon merge handshake |
| `pg_num_target` | 使用者打算把 `pg_num` 推到哪個數字 | mon（直接由 `ceph osd pool set pg_num N` 觸發） |
| `pgp_num_target` | 使用者打算把 `pgp_num` 推到哪個數字 | mon |

`pg_num` 與 `pgp_num` 都被 `private:` 保護 — 不能直接賦值，只能透過 setter。setter 的關鍵在它們同時還會更新 `last_change`，所以**每次 actual 移動都會產生新的 osdmap epoch、形成 client 端的新 interval**。

```cpp
// File: ceph/src/osd/osd_types.cc (line 1806)
pg_t pg_pool_t::raw_pg_to_pg(pg_t pg) const
{
  ps_t pps = raw_pg_to_pps(pg);
  unsigned rem = pps % pg_num;
  return pg_t(rem, pg.pool());
}

// File: ceph/src/osd/osd_types.cc (line 1817)
ps_t pg_pool_t::raw_pg_to_pps(pg_t pg) const
{
  return pg.ps() % pgp_num;
}
```

所以後面所有「先 split、再 placement 跟上」的語意都要這樣讀：`pg_num` 變大後，child PG 可以開始擁有 object range；只是 `pgp_num` 還沒追上時，CRUSH 仍用舊 seed 範圍決定 OSD set，跨 OSD 的 data movement 會被壓住。

## 只調 pg_num：物理 split / merge

當你只把 `pg_num` 從 128 改成 256（沒動 pgp_num）：

| 步驟 | 欄位變化 | 對 client 的可見效果 |
|---|---|---|
| `ceph osd pool set pool pg_num 256` | `pg_num_target = 256`（若 pgp_num_target == pg_num_target，連帶也設）；`pg_num` 仍 128 | 無 |
| 下一次 mgr `adjust_pgs()` tick | mgr 呼 `pg_num_actual <步長>`，monitor 把 `pg_num` 從 128 加到例如 132（取 min(left, max_jump - pg_gap, target - current)） | 多 4 個 empty PG (id 128~131) 開始 split |
| OSD 收到新 osdmap | `OSD::split_pgs()` 把舊 PG 的物件按新 split bits 分到 child PG | 部分讀寫會看到 `peering` 短暫卡頓 |
| 重複 tick 直到 `pg_num == pg_num_target == 256` | `pg_num` 每幾秒加一束 | object-to-PG 範圍已變細，但 CRUSH placement seed 仍受 `pgp_num` 限制 |

**重點**：純粹調 `pg_num` 不是「client 完全看不到新 PG」。client 端 actual PG 已經用新的 `pg_num`；只是 placement seed 還是 `pgp_num`，所以新 split 出來的 PG 會先沿用舊 placement 空間，跨 OSD 搬移要等 `pgp_num` 往前推才大量發生。這也是為什麼 split 的衝擊常小於 placement movement，但不是零衝擊。

merge 的時序差一點：

File: `ceph/src/mgr/DaemonServer.cc` (line 2830)


merge 是兩步握手：

1. mgr 呼 `pg_num_actual = N-1` → monitor 寫入 `pg_num_pending = N-1`（**`pg_num` 還是 N**）。merge source 是 `pg_t(N-1, pool)`，target 是它的 parent
2. 兩顆 PG 必須都 `active+clean`、`acting` set 完全一致、沒有 `upmap`，否則卡住等下一個 tick
3. OSD 完成 merge 準備後送 `MOSDPGReadyToMerge`，monitor 在 `prepare_pg_ready_to_merge` 真正把 `pg_num` 減 1（呼 `dec_pg_num()`）

ok 條件還有 **`pg_num == pgp_num` 會擋住 merge** — 必須先把 `pgp_num` 降下來，才能繼續減 `pg_num`。換句話說，merge 比 split 慢得多、而且強制走 `pgp_num` → `pg_num` 的順序。

`estmax = pg_num × min(max_misplaced - current, max_misplaced/2)`。換成數字：`pg_num = 1024`、`target_max_misplaced_ratio = 0.05`、目前 `misplaced_ratio = 0`：

一次最多把 `pgp_num` 拉 26 個 placement seed。也就是說「**單次 tick 最多讓 26 個 PG seed 進入新的 CRUSH placement 空間**」。每次 tick 之後 misplaced 上升，room 縮小，步距會自動縮小，直到 recovery/backfill 把舊位置的 object 慢慢搬完、misplaced 回到 0，再跑下一輪。

`mgr_tick_period` 預設 **2 秒**。所以實際 schedule：

| Option | 預設 | 影響 |
|---|---|---|
| `mgr_tick_period` | 2s | adjust_pgs() 的呼叫頻率 |
| `mon_osd_max_creating_pgs` | 1024 | 整個 cluster 每輪能同時開始 split 出去的 PG 數量 |
| `mgr_max_pg_num_change` | 128 | 單一 pool 在 single tick 內 pg_num 的最大跳幅（split 時被 `min(left, max_jump - pg_gap, target - current)` 限制） |
| `target_max_misplaced_ratio` | 0.05 | misplaced 超過這比例就停 pgp_num 推進；單步 pgp_num 跳幅也以此為基數 |
| `mgr_debug_aggressive_pg_num_changes` | false | dev knob：略過所有 throttle |

split 那邊的步距：

File: `ceph/src/mgr/DaemonServer.cc` (line 2944)

```cpp
unsigned add = std::min(
  std::min(left, max_jump - pg_gap),
  p.get_pg_num_target() - p.get_pg_num());
unsigned target = p.get_pg_num() + add;
left -= add;
```

`pg_gap = pg_num - pgp_num`。如果 gap 已經達到 `mgr_max_pg_num_change`，這輪不再 split 新 PG — 必須等 pgp_num 跟上來。所以一個典型的「pg_num=128 → 512」變化 timeline 大概像這樣：

```
t=0    : 命令送進去；pg_num_target=512、pgp_num_target=512、pg_num=128、pgp_num=128
t=2s   : 第一輪 tick；pg_num += min(left, 128, 384) = 128 → pg_num=256, gap=128
t=4s   : 第二輪 tick；pg_num gap 已達 max_jump → 不動 pg_num；先推 pgp_num += estmax
... pgp_num 慢慢追上來、有資料搬移 → misplaced 增加 → 後續 tick defer 一些
t=Ns   : pgp_num 推到 256，gap 歸 0 → 下一輪 tick 又可以推 pg_num
... 反覆直到 pg_num=pgp_num=512
```

實務上「`pg_num 128 → 512`」對 100GB 級 pool 常見會落在 10~30 分鐘；對 50TB 級 pool 可能要小時級。這是現場估算，不是 source 保證；真實時間取決於 object 數、OSD 數、backfill throttle、目前 misplaced ratio 與 device latency。

mon 把 `pg_num` 改大後，OSD 收到新 osdmap，發現自己 host 的 PG 對應到的「split bits」變了：

split 本身主要是 **OSD 本機** 的 ObjectStore collection 拆分：父 PG 的 object，按更精細的 bit 範圍重新分到 child PG 的 collection。這就是為什麼 split 本身對 client I/O 衝擊通常小於 placement movement。**跨 OSD 的搬移是 `pgp_num` 變大之後，CRUSH seed 展開、某些 PG 對應到不同 acting set 時才開始的**。

merge 反向更複雜 — 兩個 PG 在不同 OSD 上時，必須先讓它們 acting set 對齊（透過 upmap 或等 CRUSH 自然對齊），再做 source PG → target PG 的本地 merge。這就是 `adjust_pgs()` 裡那一大段 `source_acting == target_acting` 檢查的由來。

## 邊界與除錯

幾個會讓 operator 卡住的常見點：

**為什麼 `pg_num` 一直卡在中間數字**：看 `ceph osd pool ls detail` 確認 `pg_num_target` 與 `pg_num` 是否一致。如果不一致：

- `ceph -s` 有 inactive/degraded PG → adjust_pgs() defer pgp_num，間接卡住 pg_num（gap 達上限）
- `mgr -- log` 會看到 "max_pg_num_change" 字樣，表示在等 pgp_num 追上
- pool 上有 `FLAG_CREATING` → 初始 PG 還沒建完，adjust_pgs() 跳過這個 pool

**為什麼降 pg_num 一直不動**：

- `pgp_num` 仍 == `pg_num` 時 merge 直接 block — 必須先降 `pgp_num`
- 任何 PG 不在 `active+clean`、merge target / source 在不同 OSD acting set、有 upmap → 全擋下，每 tick 都會列在 mgr log

**為什麼 cache pool 拒絕 split**：

**用 `pg_num_actual` 直接改 `pg_num`**：理論上可以，但你會直接旁路 mgr 的 throttle、可能瞬間觸發大量 split。`mgr_debug_aggressive_pg_num_changes` 也是同樣的危險，命名就標明 dev only。

<Callout type="warning" title="先動 pg_num 再讓 pgp_num 跟上 — 不是手動分兩步">
  你不需要手動跑「先 set pg_num，等收斂後再 set pgp_num」。預設情境下 `pg_num_target == pgp_num_target` 時，set pg_num 已經同步把 pgp_num_target 也設好，mgr 會自己安排「先 split actual PG、再讓 CRUSH placement seed 跟上」的順序。**需要手動分開的場景是你刻意要做 pre-split**：先把 PG 數量增加，讓 object-to-PG range 變細，但暫時延後跨 OSD placement movement。這時才會先把 `pgp_num` 鎖在舊值，再 `set pg_num`。
</Callout>

<Callout type="tip" title="想加速 / 觀察當下狀態">
  - `ceph osd pool ls detail` — 看每個 pool 的 5 個欄位（`pg_num`, `pgp_num`, `pg_num_target`, `pgp_num_target`, `pg_num_pending`）
  - `ceph osd pool autoscale-status` — autoscaler 對每個 pool 的「理想 pg_num」估算
  - 想暫時加速：`ceph config set global target_max_misplaced_ratio 0.10`（5% → 10%）；想看每步詳細決策：`ceph tell mgr config set debug_mgr 10/10` 看 adjust_pgs 的 dout
  - 真的卡住：先用 `ceph osd pool ls detail` 找出哪個欄位卡住、再對照 `ceph -s` 看是不是有 inactive PG / health warning 把 adjust_pgs() defer 了
</Callout>
