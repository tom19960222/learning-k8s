## 場景

你 KubeVirt VM 跑著，guest OS 看到一顆 50 GiB 的硬碟，背後是 Ceph 給的 RBD volume。真正出事時，表面錯誤常不是「Ceph 壞了」，而是 PVC 卡在 provisioning、node plugin map 失敗、kernel `rbd` module 不支援某個 image feature，或你以為 50 GiB 已經吃掉 50 GiB raw space。

這頁把路徑從底往上接起來：RBD image 在 RADOS pool 裡到底有哪些 object、librbd 怎麼把 offset 切成 object op、哪些 feature 會牽動 mapping client，最後 ceph-csi 與 rook 各自負責哪一段。

## RBD = RADOS Block Device

RBD 不是真的預先配置一整段連續 block device，而是把 block volume 映射成一組 RADOS object。對一個 format 2 image 來說，最重要的是三類 object：`rbd_id.<name>` 幫 image name 找 image id，`rbd_header.<image_id>` 存 metadata，`rbd_data.<image_id>.<object_no>` 存真正的資料 chunk。

```
你看到的 RBD image                         RADOS pool 內存的東西
┌─────────────────────┐                   ┌───────────────────────────────┐
│  /dev/rbd0 (50 GiB) │                   │  rbd_id.<name>                │  ← name → image_id
│  block 0  ─────────┼──────────────────>│                               │
│  block 1            │                   │  rbd_header.<image_id>        │  ← metadata object
│  ...                │                   │  └─ image size, stripe info,  │
│  chunk 12,799       │                   │     snapshot list, features   │
└─────────────────────┘                   │                               │
                                          │  rbd_data.<image_id>.0000     │  ← data chunk 0 (4 MiB)
                                          │  rbd_data.<image_id>.0001     │  ← data chunk 1
                                          │  ...                          │
                                          │  rbd_data.<image_id>.31ff     │  ← chunk 12799
                                          │                               │
                                          │  thin provision: 只有寫過    │
                                          │  的 chunk 才真的有 object    │
                                          └───────────────────────────────┘
```

Ceph source 裡把這些 prefix 寫成常數。這也是你在 `rados -p <pool> ls` 看到 RBD object 時，為什麼名字不是原本 PVC 名稱，而是一串 `rbd_data...` 的原因。

File: ceph/src/include/rbd_types.h (line 28)

```cpp
#define RBD_HEADER_PREFIX      "rbd_header."
#define RBD_OBJECT_MAP_PREFIX  "rbd_object_map."
#define RBD_DATA_PREFIX        "rbd_data."
#define RBD_ID_PREFIX          "rbd_id."
```

預設 chunk 大小來自 `rbd_default_order = 22`，也就是 `2^22 = 4 MiB`。所以 50 GiB image 邏輯上會有 `50 * 1024 / 4 = 12800` 個 chunk，最後一個 object number 是 `12799`，十六進位是 `31ff`。這裡說「邏輯上」很重要：RBD 是 thin-provisioned，只有寫過的 chunk 才會真的在 RADOS 落物件，raw space 再乘上 pool replication 或 EC overhead 才是實際占用。

File: ceph/src/common/options/rbd.yaml.in (line 481)

```yaml

  level: advanced
  desc: default order (data block object size) for new images
```

## librbd：把 block op 切成 object op

librbd 是 user-space C++ library，提供「對某個 RBD image 做 block read/write」的 API。它把 IO request 翻成 RADOS object op：

```
guest OS write at offset 100 MiB, length 8 KiB
              │
              ▼
guest kernel → /dev/rbd0
              │
              ▼
kernel rbd module 或 rbd-nbd + librbd
   ├─ chunk_index = 100 MiB / 4 MiB = 25
   ├─ chunk_offset = 100 MiB % 4 MiB = 0
   ├─ object_name = "rbd_data.<image_id>.0019"
   │
   ▼
librados → Objecter → primary OSD (用 CRUSH 算)
   │
   ▼
RADOS object 8 KiB 寫進該 chunk's offset 0
```

真正切 object 的地方在 `ImageRequest`：先把 image-level extents 丟給 `area_to_object_extents()`，後者再呼叫 generic `Striper::file_to_extents()`。所以跨 4 MiB 邊界的 8 KiB write，不是「塞進同一個大 object」，而是被拆成兩個 object extents，各自走 librados / Objecter / CRUSH / primary OSD。

File: ceph/src/librbd/io/ImageRequest.cc (line 438)

```cpp
LightweightObjectExtents object_extents;

  if (extent.second == 0) {

                                   this->m_image_area, clip_len,
```


## librbd 用到的 server-side class extension

librbd 有些操作（create、snapshot、object map maintenance、watch/notify）需要 OSD 那邊配合。Ceph 用 **class extension** 機制：在 OSD 內 load 一個 `.so`，提供 client 可呼叫的 RPC。

File: ceph/src/cls/rbd/cls_rbd.cc (line 8235)

```cpp
cls_register_cxx_method(h_class, "get_object_prefix",
                        CLS_METHOD_RD,
                        get_object_prefix, &h_get_object_prefix);
```

client 端 librbd 透過 `librados` 對 OSD 發 `EXEC` op 呼叫這些 class method。class 的 code 在 OSD 那邊跑，操作 header object 的 omap metadata。這點解釋了兩個常見現象：

- RBD metadata 不是集中在 MON；MON 給你 map 與 auth，RBD 自己的 image metadata 在 pool 裡，由 OSD class method 改。
- `rbd ls`、`rbd info`、`rbd snap create` 這些看起來像 control plane 的動作，底層仍會對 RADOS object 發 request。

## snapshot / clone：copy-on-write 透過 RADOS 的 object snapshot

RBD snapshot **不立刻 copy data**：

1. 對 metadata object 加一個 snapshot ID 進 snapshot list
2. 後續 write to chunk → 在 OSD 那邊先 snapshot 那個 RADOS object 再 write (CoW，由 RADOS 提供的 object snapshot operation 處理)
3. 沒被 write 的 chunk 永遠不需要 copy

clone：基於 parent snapshot 建一個新 image。新 image 的 chunk read：

- 如果新 image 自己有 → 讀新的
- 如果沒 → 讀 parent snapshot 的對應 chunk

flatten：把 parent dependency 切斷（把所有 parent chunk copy 到新 image），讓新 image 獨立。

整段 CoW + clone 邏輯都在 librbd + OSD class extension 完成，沒有中央 metadata。

RBD feature 之間也不是任意組合。`object-map` 與 `fast-diff` 互相綁住，且需要 `exclusive-lock`；`journaling` 也需要 `exclusive-lock`。這不是 docs 裡的建議，而是 enable/disable operation 的硬檢查。

## ceph-csi：CSI 只負責把 Kubernetes request 翻成 RBD 動作

ceph-csi 不在 Ceph 本 repo，在 [github.com/ceph/ceph-csi](https://github.com/ceph/ceph-csi)。它是 k8s 的 CSI driver，做兩件事：

```
┌──────────── ceph-csi ────────────────────────────┐
│                                                   │
│  controller plugin (Deployment, 1 replica)       │
│   ├─ CreateVolume   → 對 Ceph 建 RBD image       │
│   ├─ DeleteVolume   → 對 Ceph 砍 RBD image       │
│   ├─ CreateSnapshot → cls_rbd.snap_create        │
│   └─ ControllerExpandVolume → 對 Ceph 改 size    │
│                                                   │
│  node plugin (DaemonSet, 每 node 一個)           │
│   ├─ NodeStageVolume    → map RBD image 成本機   │
│   │                       block device (kernel    │
│   │                       rbd 或 rbd-nbd)         │
│   ├─ NodePublishVolume  → bind-mount 給 Pod      │
│   ├─ NodeUnpublishVolume                          │
│   └─ NodeUnstageVolume  → unmap                  │
└───────────────────────────────────────────────────┘
```

把這段對回 Kubernetes 物件，最小 StorageClass 通常會長這樣。這不是本頁驗證過的 lab 指令；真正環境要依你的 Ceph fsid、pool、secret namespace、mounter 與 host kernel 能力調整。

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ceph-rbd
provisioner: rbd.csi.ceph.com
parameters:
  clusterID: ceph-fsid
  pool: kubernetes
  imageFeatures: layering
  csi.storage.k8s.io/provisioner-secret-name: csi-rbd-secret
  csi.storage.k8s.io/provisioner-secret-namespace: rook-ceph
  csi.storage.k8s.io/node-stage-secret-name: csi-rbd-secret
  csi.storage.k8s.io/node-stage-secret-namespace: rook-ceph
allowVolumeExpansion: true
reclaimPolicy: Delete
mountOptions:
  - discard
```

PVC bind 的整段可以切成兩個時段：controller plugin 建 image，node plugin 在特定 node 上 map/mount。

```
1. user 寫 PVC yaml, storageClassName: ceph-rbd
        │
        ▼
   k8s PV controller 看到 PVC
        │
        ▼
2. external-provisioner sidecar (跟 ceph-csi controller 一起跑)
   ├─ 對 ceph-csi controller plugin 發 CreateVolume gRPC
        │
        ▼
3. ceph-csi controller plugin
   ├─ 讀 StorageClass 的 pool / imageFeatures / secrets
   ├─ rbd create — 對 Ceph 建 image
   └─ 回 volume handle (含 image id, pool name)
        │
        ▼
4. k8s PV controller 建 PV 物件 with that handle, bind 給 PVC
        │
        ▼
5. user Pod schedule 到 node X, kubelet 看到 Pod 要 mount PV
        │
        ▼
6. kubelet 對 ceph-csi node plugin (on node X) 發 NodeStage + NodePublish
   ├─ NodeStage: map RBD image 成 /dev/rbdN (用 kernel rbd module 或 rbd-nbd)
   └─ NodePublish: bind-mount /dev/rbdN 進 Pod 的 volume path
        │
        ▼
   Pod 看到 mounted filesystem，可以 read/write
```

注意這裡有一個責任邊界：ceph-csi controller plugin 能決定 image 怎麼建立，但真正 map 成 `/dev/rbdN` 的是 node plugin 所在的 host。`imageFeatures`、`volumeMode`、`mounter`、host kernel 版本與是否載入 `nbd` module，會一起決定「PVC 能不能被 Pod 用起來」。

## `imageFeatures` / `mounter`：最常被漏掉的邊界

`map RBD image 成 /dev/rbdN` 主要走兩種：

- **kernel `rbd` module**：由 kernel 實作 RBD client path，性能好但需要 host kernel 支援對應 feature
- **rbd-nbd** (user-space)：用 nbd 介面，librbd 在 user-space 跑；性能略差但兼容性好

現場經驗裡，`rbd-nbd` 常被拿來繞過 host kernel `rbd` module 不支援某些 RBD image feature 的問題，但它也多了一個 user-space process / nbd device lifecycle。你在 StorageClass 裡只看到一行 `imageFeatures: layering`，背後其實是在替「所有可能 mount 這個 PVC 的 node」選共同可支援的 feature set。

常見判讀方式：

| 現象 | 優先看哪裡 | 這頁對應原理 |
|---|---|---|
| PVC 長不出 PV | ceph-csi controller log、Ceph auth caps、pool / namespace / secret | controller plugin 還沒成功 `rbd create` |
| PV 已 bind，Pod 卡 `FailedMount` | node plugin log、host kernel `rbd` / `nbd` module、`mounter` | image 已存在，但 node 不能 map |
| 開了 `journaling` 後某些 node map 失敗 | `imageFeatures` 與 host client support | `journaling` 需要 `exclusive-lock`，而 mapping client 不一定支援 |
| VM live migration 或 RWX block 需求不如預期 | KubeVirt access mode、ceph-csi feature matrix、RBD feature | Ceph image feature 與 Kubernetes access mode 是兩層契約 |

所以在 KubeVirt / Rook lab 裡，初學階段我會先用最保守的 `imageFeatures: layering` 跑通基本 PVC，再逐步加入 `exclusive-lock`、`object-map`、`fast-diff`、`journaling` 等 feature。每加一個 feature，都要同時問兩件事：Ceph/libRBD source 支不支援這個 dependency？你的 worker node mapping client 支不支援？

## rook：把整套 Ceph 當 k8s CRD 管

rook 也不在 Ceph 本 repo，在 [github.com/rook/rook](https://github.com/rook/rook)。它是 operator，把 Ceph cluster 部署/維護的整個生命週期收進 k8s CRD：

```
CRD                    Operator 對應動作
─────────────────────  ─────────────────────────────────────────
CephCluster            起 ceph-mon Deployment + ceph-osd DaemonSet
                       (用 hostPath 或 PVC 給 OSD 用)
                       管 cephadm-style 的 daemon lifecycle

CephBlockPool          對 Ceph create pool (replicated 或 EC)
                       設 CRUSH rule、failure domain

CephObjectStore        deploy radosgw + create RGW user/pool

CephFilesystem         deploy MDS

CephClient             create Ceph user + secret in k8s
```

rook 的 reconcile loop 觀察這些 CRD，呼 Ceph CLI / mgmt API 同步狀態。它跟 ceph-csi 是「兩個獨立 component」：

- rook 起 Ceph daemon 為 k8s Pod
- ceph-csi 提供 PVC backend
- 用 `StorageClass` 把兩者串起來

實務最常見的部署：用 helm/operator 一鍵裝 rook + ceph-csi → 給 KubeVirt 一個 ceph-rbd 的 StorageClass → KubeVirt VM 拿 PVC 就能跑。但除錯時不要把兩者混在一起：

- `CephCluster` / MON / OSD Pod 起不來，是 rook 或 Ceph daemon 層。
- PVC create / delete / expand 出錯，是 ceph-csi controller plugin 加 Ceph auth/pool 層。
- Pod mount / unmount 出錯，是 ceph-csi node plugin 加 host kernel / `rbd-nbd` 層。
- VM guest OS 內 I/O 慢，是 KubeVirt / QEMU / host mapping client / librbd / OSD queue / BlueStore 多層一起看。

## 整段對齊 lab Day 18

你的 KubeVirt VM 啟動：

```
1. VMI yaml 帶 dataVolumeTemplate, storageClassName: ceph-rbd
2. CDI / k8s 對 ceph-csi 要 PVC
3. ceph-csi 對 Ceph cluster create RBD image (50 GiB thin)
4. PVC bind, virt-launcher Pod schedule 到 worker-N
5. worker-N 的 ceph-csi node plugin 把 RBD image map 成 /dev/rbdN
6. virt-launcher 內 libvirt 把 /dev/rbdN 當 disk 餵給 QEMU
7. QEMU 跑 guest OS, 看到 /dev/sda
8. guest 寫 disk → QEMU → /dev/rbdN
        → kernel rbd 或 rbd-nbd + librbd
        → CRUSH 算 PG → 寫 primary OSD → BlueStore 落 raw block
```

如果這條 path 卡住，先把問題分成「image 是否已建立」與「node 是否能 map」。前者看 ceph-csi controller log 與 `rbd ls/info`；後者看 node plugin log、host kernel module、`rbd device list`、`nbd` device，以及 KubeVirt virt-launcher Pod 的 volume path。不要一看到 VM I/O error 就直接跳到 OSD；也不要一看到 PVC error 就先調 mClock。

整條 path 跨：kubevirt 的 [virt-handler-and-launcher](/kubevirt/features/virt-handler-and-launcher) → ceph-csi → librbd → Ceph 的 [crush-and-placement](/ceph/features/crush-and-placement) → [osd-and-bluestore](/ceph/features/osd-and-bluestore)。

## 收穫檢查

- 你建一個 50 GiB RBD image 但只寫了 1 GiB，Ceph 那邊實際佔多少 raw space？(別忘了 replication factor)
- snapshot 是不是立刻 copy data？
- ceph-csi 的 controller plugin 跟 node plugin 各自做什麼？為什麼 map 失敗通常在 node plugin 才爆？
- `imageFeatures: layering,exclusive-lock,object-map,fast-diff` 看起來很好，為什麼不一定適合每個 worker node？
- rook 跟 ceph-csi 的職責怎麼分？兩個都不裝行不行？
