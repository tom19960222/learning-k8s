## 場景

某天 KubeVirt VM 上的應用變慢，你 ssh 進 OSD node，`grep` OSD log 看到一整排：

```
bluestore(/var/lib/ceph/osd/ceph-7) log_latency slow operation observed for _txc_committed_kv, latency = 8.21s
```

BlueStore 明明在抱怨慢操作，可是你 Prometheus 上盯著的 `ceph_healthcheck_slow_ops` 一直是 0，`SLOW_OPS` 這個 health check 也沒亮，pager 一聲都沒響。你開始懷疑自己的 alert 是不是寫錯了。

**它沒寫錯。** 「BlueStore 喊慢」與「SLOW_OPS 亮」量的是**不同層、不同門檻、不同時間特性**的東西——它們幾乎不重疊，這是機制上的必然，不是設定問題。這頁把三條訊號路徑從原始碼追到底，再用真機注入證明 SLOW_OPS 對哪幾類事件結構性失明。

先講一句總綱：v19.2.3 裡跟「op 慢」相關的可匯出訊號其實有**三條互相獨立的路徑**——

| 路徑 | 量在哪一層 | 門檻 | 時間特性 |
|---|---|---|---|
| `SLOW_OPS`（two metrics） | OSD/MON 的 `OpRequest`（RADOS op 生命週期） | in-flight 且 age > `osd_op_complaint_time`（**30s**） | 即時快照，op 完成即消失 |
| `BLUESTORE_SLOW_OP_ALERT` | BlueStore 單一內部操作 | 單次 latency > `bluestore_log_op_age`（**5s**） | latch，事件後黏約 24h |
| 4 個 `slow_*_count` counter | BlueStore 單一內部操作 | 同上 5s | 累積 counter，不歸零 |

後兩條同源（都在 BlueStore 的 `log_latency`），跟第一條完全是兩套機制。逐條拆。

## 路徑一：兩個 SLOW_OPS metric 的來源鏈

`ceph_healthcheck_slow_ops`（cluster 級）與 `ceph_daemon_health_metrics{type="SLOW_OPS"}`（per-daemon）這兩個 metric，一路上溯都會回到同一個地方：OSD/MON 的 `OpTracker`。

```
client / peer / sub op
   └─ 包成 OpRequest，註冊進 OSD（或 MON）的 OpTracker in-flight registry
            │  每個 OSD tick：OSD::get_health_metrics()
            │  數「還在 in-flight 且 age > osd_op_complaint_time(30s)」的 op
            ▼
   DaemonHealthMetric{ SLOW_OPS, n1 = slow 計數, n2 = oldest 已卡秒數 }
            │  mgrc.update_daemon_health() → active mgr
            ▼
   mgr DaemonServer 把所有 osd + mon 的 SLOW_OPS metric 累加
            │  任一 daemon n1>0 或 n2>0 → 升 health check
            ▼
   health check "SLOW_OPS" (HEALTH_WARN)
     summary = "{n1} slow ops, oldest one blocked for {n2} sec, daemons [...] have slow ops."
            │
   ┌────────┴──────────────────────────────────┐
   ▼                                            ▼
 mgr/prometheus 解析 summary 第一個數字         逐 daemon dump n1
 ceph_healthcheck_slow_ops = n1（cluster 總數）  ceph_daemon_health_metrics{
   不 active 時 = 0                                type="SLOW_OPS", ceph_daemon="osd.N"}
```

## 路徑二＆三：BLUESTORE_SLOW_OP_ALERT 與 4 個 slow counter（同一條 log_latency）

場景裡那行 `log_latency slow operation observed` 是另一套完全獨立的機制。BlueStore 內部每個關鍵操作（onode 讀取、txc commit、kv sync、omap 迭代…）完成時都會過 `log_latency`，門檻是 `bluestore_log_op_age`（預設 **5 秒**）：

File: `ceph/src/os/bluestore/BlueStore.cc` (line 18465)

```cpp
void BlueStore::log_latency(
  const char* name, int idx, const ceph::timespan& l,
  double lat_threshold, const char* info, int idx2)
{
  logger->tinc(idx, l);
  if (lat_threshold > 0.0 &&
      l >= make_timespan(lat_threshold)) {
    dout(0) << __func__ << " slow operation observed for " << name
      << ", latency = " << l << info << dendl;
    _add_slow_op_event();
    if (idx2 > l_bluestore_first && idx2 < l_bluestore_last) {
      logger->inc(idx2);
    }
  }
}
```

這 11 行裡藏了**三件事**，也就是三個獨立的下游訊號：

1. `dout(0) ... slow operation observed`——你在 log 看到的那行字。
2. `_add_slow_op_event()`——推進 `BLUESTORE_SLOW_OP_ALERT` 的事件 queue（路徑二）。
3. `if (idx2 ...) logger->inc(idx2)`——**只有帶 idx2 的呼叫點**會 inc 一個專用 slow counter（路徑三）。

`log_latency_fn`（line 18487）是同構的第二個版本，差別只在 info 字串用 lambda 產生。兩個函式在 `BlueStore.cc` 裡一共被呼叫 **31 次**（口徑：`grep -cE 'log_latency(_fn)?\(' BlueStore.cc` 得 33，扣掉 line 18465／18487 兩個函式定義本身 = 31 個呼叫點）。這 31 個呼叫點都會過 `_add_slow_op_event()`，但只有一小部分帶 idx2——這個不對稱是路徑三一個盲區的根源（見下面「omap／scrub 盲區」）。

### 路徑二：BLUESTORE_SLOW_OP_ALERT（latch 型 health 警告）

`_add_slow_op_event()` 把事件推進一個有時效的 queue：

File: `ceph/src/os/bluestore/BlueStore.cc` (line 18455)

```cpp
void BlueStore::_add_slow_op_event() {
  if (!cct->_conf->bluestore_slow_ops_warn_threshold) {
    return;
  }
  std::lock_guard lock(qlock);
  auto cur_time = mono_clock::now();
  slow_op_event_queue.push(cur_time);
  _trim_slow_op_event_queue(cur_time);
}
```

（順帶一提：`bluestore_slow_ops_warn_threshold` 設 0 等於**關掉**這整條偵測，見上面 line 18456 的 early return。）queue 的 trim 邏輯決定了它的 latch 特性——事件保留 `bluestore_slow_ops_warn_lifetime`：

File: `ceph/src/os/bluestore/BlueStore.cc` (line 18444)

```cpp
size_t BlueStore::_trim_slow_op_event_queue(mono_clock::time_point cur_time) {
  ceph_assert(ceph_mutex_is_locked(qlock));
  auto warn_duration = std::chrono::seconds(cct->_conf->bluestore_slow_ops_warn_lifetime);
  while (!slow_op_event_queue.empty() &&
    ((slow_op_event_queue.front() < cur_time - warn_duration) ||
      (slow_op_event_queue.size() > cct->_conf->bluestore_slow_ops_warn_threshold))) {
      slow_op_event_queue.pop();
  }
  return slow_op_event_queue.size();
}
```

然後在 `_log_alerts` 裡，只要時效內事件數 `>= bluestore_slow_ops_warn_threshold` 就升 `BLUESTORE_SLOW_OP_ALERT`：

</Callout>

```cpp
  PerfCountersBuilder::PRIO_USEFUL);
```

這 4 個 counter 被打中的呼叫點只有 **6 個帶 idx2 的 `log_latency` 呼叫 + 1 個直接 inc**（原始碼，逐一開檔驗過）：

| counter | 打中它的呼叫點 | 路徑 |
|---|---|---|
| `slow_committed_kv_count` | `BlueStore.cc:14471`（`log_latency_fn`，idx2） | 寫：txc 從 start 到 committed &gt;5s |
| `slow_read_wait_aio_count` | `BlueStore.cc:12751`、`13113`（`log_latency_fn`，idx2） | 讀：等 aio 完成 |
| `slow_read_onode_meta_count` | `BlueStore.cc:12368`、`12708`、`13080`（`log_latency`，idx2） | 讀：讀 onode metadata |
| `slow_aio_wait_count` | `BlueStore.cc:14175`（不經 log_latency，直接 `logger->inc`） | 寫：txc AIO wait 段 |

`ceph_bluestore_txc_commit_lat`（`BlueStore.cc:6217`，`PRIO_CRITICAL`）與 `kv_sync_lat`（line 6252）這類 time-avg counter 是另一回事——那是均值 counter、不是「超過 5s 就 +1」的 slow counter，別搞混。這 4 個 slow counter 才是「超過門檻計次」的離散訊號：ceph-exporter 以 `promethize` 把 daemon perf counter 換成 `ceph_` 前綴系列（`ceph/src/exporter/util.cc` line 54），在 lab 的 exporter :9926 確認匯出成 `ceph_bluestore_slow_{aio_wait,committed_kv,read_onode_meta,read_wait_aio}_count{ceph_daemon="osd.N"}`（實機實驗）。它們帶 `ceph_daemon` label、累積不歸零、5s 門檻——正是下一節那些 SLOW_OPS 盲區裡**唯一還看得到**的訊號。

## SLOW_OPS 的三層盲區（真機實證）

把路徑一攤開，`SLOW_OPS` 的 30s + 即時快照特性造成三類結構性失明。每一類都在 lab 上注入驗過，數字如下（實機實驗；harness 見 `experiments/ceph-slow-ops-detection/`）。

### 第 1 層：暫態盲區

op 齡 &lt;30s 的單發卡頓，`OpTracker` 路徑物理上不可見——op 完成得比門檻還快。

- **單 OSD 卡頓注入**：`dmsetup suspend` 讓單 OSD 卡 8s（寫負載）。整個觀察窗 `ceph_daemon_health_metrics{type="SLOW_OPS"}` 與 `ceph_health_detail{name="SLOW_OPS"}` **兩路全程 0**。
- 同一事件，路徑三的 slow counter 在**卡頓開始 +14s** 就可判真（counter 首升樣本晚於 suspend 結束——見下面「完成才記帳」邊界）。

一句話：8s 的 BlueStore 卡頓對 SLOW_OPS 是零覆蓋，對 slow counter 是 14 秒可見。（實機實驗）

### 第 2 層：持續性 sub-30s 劣化盲區

比暫態更陰險：**每個 op 都慢、但每個都在 30s 內完成**。這正是 SSD 劣化／firmware 問題最常見的表現形態。

- **持續慢盤注入**：cgroup `io.max` 對 osd.0 持續節流 150 秒。`ceph_osd_op_w_latency` 的區間均值峰值衝到 **15.2s**、`ceph_osd_commit_latency_ms` 峰值 15196ms——client 痛了兩分半。
- 同一窗內 `ceph_daemon_health_metrics{type="SLOW_OPS"}` 與 `ceph_health_detail{name="SLOW_OPS"}` **全程 0**，因為沒有任何單一 op 撐過 30s。

「持續劣化」這一整類，SLOW_OPS 看不到。（實機實驗）

### 第 3 層：採樣漏失盲區

就算 op 真的破了 30s，per-daemon 那個 metric 是**快照 gauge**——op 完成即歸零，可見窗只有「op 齡減 30s」那短短幾秒，還得穿過 mgr module 的 15s cache + scrape interval 才能被 Prometheus 抓到。

- 一次 max latency **44.7s** 的真事件：mon 端 `ceph_health_detail{name="SLOW_OPS"}` 只抓到 **2 個非零樣本**（撐不過 `for: 1m` 這類等待條件），per-daemon `ceph_daemon_health_metrics` 則**整段漏掉、0 個樣本**。
- 結論：**44 秒的真 slow op，兩條吃 SLOW_OPS 的既有 alert 全部沉默。**（實機實驗）

<Callout type="warning" title="三層合起來看">
  30s 門檻（第 1、2 層）+ 快照式採樣（第 3 層）疊加，讓 SLOW_OPS 對「短」與「持續但都 &lt;30s」兩類事件系統性失明，對「破 30s 但短命」的事件則靠運氣採樣。路徑三的 slow counter（累積、5s 門檻、事後仍在）是唯一不怕採樣漏失的訊號——這是它存在的理由。（實機實驗）
</Callout>

## 定位陷阱：SLOW_OPS 會「怪錯人」

就算 SLOW_OPS 亮了，它指的 `ceph_daemon` label 也可能把 oncall 帶去查錯機器。

- **持續慢盤注入的實測**：卡住的是 **osd.0**（在 node osd-01 上）。但 `ceph_daemon_health_metrics{type="SLOW_OPS"}` 非零的是 **osd.4 / osd.6 / osd.7 / osd.8**（osd-02、osd-03 上的 primary）；osd.0 自己全程 0。

機制：client op 記在 **primary** 的 op tracker 上，primary 等被卡的 **replica**（osd.0）回 ack 等到超齡——於是超齡的計數掛在別台 node 的 primary 頭上。oncall 看到 SLOW_OPS 亮 osd.4 就去查 osd.4 的碟，方向完全錯。

分工要記牢（實機實驗）：

- **slow counter 是 device-local 訊號、指向元兇**（osd.0 自己的 counter 會動）。
- **SLOW_OPS 是 client-impact 訊號、指向受害者**（哪些 primary 的 client op 被拖累）。
- 兩者一起看：`slow_*_count` 亮 osd.0 + SLOW_OPS 亮一堆別台 primary = **osd.0 的 device 出事、且已波及 client**。

## 訊號的邊界（連 slow counter 也有盲區）

路徑三補了 SLOW_OPS 的三層盲區，但它不是萬能。四個必須知道的邊界：

| 邊界 | 內容 | 錨點 |
|---|---|---|
| **完成才記帳** | slow counter 只在 op「完成且量到 &gt;5s」時 inc。op **永久卡死**（真 hang）時 counter 全程不動，只有 `SLOW_OPS`（op 一直在飛、齡會破 30s）看得到。單 OSD 卡頓注入實測 counter 首升樣本晚於 suspend 結束——卡頓期間 counter 全平。所以 counter 與 SLOW_OPS **互補、缺一不可**。 | `BlueStore.cc:18473`（log 前先 `tinc`，inc 在完成路徑） |
| **omap／scrub 盲區** | 4 個 counter 只覆蓋上面那 6+1 個呼叫點。31 個 `log_latency` 呼叫點裡帶 idx2 的僅 6 個，其餘 25 個（omap iterator、csum、decompress、kv_flush/commit/sync、submit/throttle…）**只打 log + `_add_slow_op_event()`、不 inc counter**。純 omap／scrub 類 slow 事件因此是 counter 盲區，只有 `BLUESTORE_SLOW_OP_ALERT`（health 路徑）看得到。（寫路徑靠 txc 全程量測 `slow_committed_kv_count` 有傳遞性覆蓋。） | `BlueStore.cc:18480`（`_add_slow_op_event` 無條件呼叫）vs `18481` 的 idx2 條件 |
| **idle 盲區** | 「無 IO 即無偵測」是結構性盲區。device 卡住但當下沒有 IO 經過（idle OSD）時，counter 與 op-tracker **都無訊號**。實測：idle OSD 卡 15s，4 counter 增量 0、SLOW_OPS 0、全部靜默。緩解只能靠低頻 synthetic IO（如每分鐘一次小寫入），超出 Prometheus-only 範圍。 | idle 注入實測（實機實驗） |
| **版本依賴** | 4 個 slow counter 是 **Squid v19.2.0+ 才有**（Reef 全系列沒有）；`BLUESTORE_SLOW_OP_ALERT` 是 **v19.2.1+／Reef v18.2.6+**。生產若還在 Reef，路徑三不可用，只剩 SLOW_OPS 加上 device 層 metric 兜底。本頁機制錨在 v19.2.3，用前務必先確認你的版本。 | GitHub 逐版原始碼比對（文件） |

還有一個 false positive（誤報）心理預期要先建立（實機實驗）：slow counter 亮 **≠ 硬體壞**。純過載（recovery、backfill、深佇列）也會把單段 op 推過 5s——lab 上 16 併發純 bench（無注入）就讓 9/9 OSD 的 slow counter 全增、bench max latency 44.7s。counter 的語意是「這顆 OSD 有 op 超過 5s」的**症狀**，不是硬體定罪；真要歸因到 firmware／RAID 卡，得看「同一 node 多顆 OSD 同窗一起亮」的聚集指紋 + 持續性。這類分級與具體規則屬於實驗頁。

## 一句話關於 alert 規則

上面每條訊號要**怎麼組成規則**（表達式、`for` / `keep_firing_for`、severity、真機量到的偵測延遲：slow counter +14s、同 node 聚集 +20s、加速版 SLOW_OPS +45~60s vs 舊規則 ~120s），全部在 [slow ops 快速偵測實驗頁](./slow-ops-fast-detection-lab)——本頁只負責讓你懂「為什麼需要那些規則」。
