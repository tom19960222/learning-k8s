## 場景

凌晨三點：某 OSD 的硬碟 SMART 不可逆地壞了，oncall 進去要換碟。換碟流程是教科書答案：

```
ceph orch host maintenance enter cephnode-3   # 把整個 host 進維護
# 換碟
ceph orch host maintenance exit cephnode-3
```

可是 Prometheus 從 `ceph orch host maintenance enter` 那一刻起就開始炸：

```
[FIRING] CephHealthWarning   (HEALTH_WARN 持續 15 分鐘)
[FIRING] CephOSDDown         (OSD 持續 down 5 分鐘)
[FIRING] CephOSDHostDown     (整個 host 5 分鐘)
[FIRING] CephPGsUnclean      (PGs unclean 15 分鐘)
```

每一條都是 stock `ceph-mixin/prometheus_alerts.yml` 預設規則。它們**完全正確**——cluster 真的是 HEALTH_WARN、OSD 真的 down、PG 真的 unclean——但這些都是「我自己叫它變這樣的」。Pagerduty 從凌晨三點響到五點，oncall 被吵醒、又得手動 silence。

問題的本質：**stock 規則只看「現在是不是 WARN」，不看「我有沒有預期它是 WARN」。**

這頁從原始碼層拆三件事：

1. ceph mgr/prometheus 模組丟了哪些 metric、`ceph_health_detail` 的資料模型長怎樣。
2. 一個正當的維護動作（`ceph orch host maintenance enter`）在原始碼裡實際點亮哪些 health check。
3. 重新設計成 8 條 rule：預設把所有會危及 client 的訊號 page，OSD down 另走帶 `hostname` / `ceph_daemon` 的早期警告，只留 3 個低優先提示例外；維護期不靠 PromQL 猜測，改用人工 `amtool silence`。

也就是 `ceph orch host maintenance enter` 同時會：

1. 丟出 `HOST_IN_MAINTENANCE` health check（HEALTH_WARN）。
2. 對該 host 的 crush bucket 下 `osd set-group noout`，連帶丟出 `OSD_FLAGS` health check（HEALTH_WARN）。
3. systemd 停掉這台所有 ceph daemon，過 5 分鐘 OSD 會被 mon 標 down，丟出 `OSD_DOWN`（以及 host 層級的 `OSD_HOST_DOWN`）。

**一個正當的維護動作，會同時讓五~六條 health check 都進 HEALTH_WARN**。stock alert rules 不知道這些是同一個原因引發的——這就是警報風暴的根。

## 設計理念

stock `ceph-mixin/prometheus_alerts.yml` 的哲學是「全部丟出來、oncall 自己判斷」，正式環境就是 alert fatigue 的元兇——它不看「我有沒有預期現在是 WARN」。

舊版我曾用一條 `ceph:maintenance_active` recording rule + 每條規則掛 `unless on()` 去自動 gate 維護期。問題是：規則在「猜」維護狀態，猜錯就漏 page，看不懂也不敢改。

新設計把「現在是不是維護」這件事從 PromQL 移回人身上，只留一句心法：

> **`ceph_health_detail` 噴出來的，預設全部 page；只有 3 個低優先提示例外（info，不 page）。維護前人工下一條 `amtool silence`，到期自動解除。**

規則永遠只回答「cluster 有沒有壞」，不回答「這個壞是不是我預期的」——後者交給維護 SOP。

**為什麼 catch-all 比逐條列名更完整**：`ceph_health_detail` 是「每個 active 的 health check 各自 set 一條 =1」（`module.py:932`，label 只有 `('name','severity')` 見 `module.py:113`）。所以 `name!~"<白名單>"` 的排除法會自動接住沒列到的冷門 check、以及**未來 ceph 版本新增的 check**。舊的 4 層列了約 20 個固定名字，新 check 會被靜默漏接——對「穩定性第一」是反指標。

## 8 條 rule

```yaml
groups:
  - name: ceph-stability-first
    rules:
      # 1) cluster 進 HEALTH_ERR —— 任何 ERR 級 check 的總開關（永不 silence）
      - alert: CephHealthError
        expr: ceph_health_status == 2
        for: 1m
        labels: { severity: critical }
        annotations:
          summary: "Ceph HEALTH_ERR"
          runbook: "ceph health detail; ceph -s"

      # 2) client IO 已被擋 —— 最痛訊號，1 分鐘內叫（永不 silence）
      - alert: CephClientBlocked
        expr: ceph_health_detail{name=~"PG_AVAILABILITY|SLOW_OPS"} == 1
        for: 1m
        labels: { severity: critical }
        annotations:
          summary: "Ceph client IO blocked: {{ $labels.name }}"

      # 3) 整台 OSD node down —— 早期冗餘下降，保留 hostname 可精準 silence
      - alert: CephOSDNodeDown
        expr: |
          sum by (hostname) (
            (1 - ceph_osd_up)
              * on(ceph_daemon) group_left(hostname) ceph_osd_metadata
          )
          ==
          count by (hostname) (ceph_osd_metadata)
        for: 5m
        labels: { severity: critical }
        annotations:
          summary: "Ceph OSD node down: {{ $labels.hostname }}"

      # 4) 單顆 OSD down —— 早期冗餘下降，保留 ceph_daemon / hostname 可精準 silence
      - alert: CephOSDDown
        expr: |
          ((1 - ceph_osd_up)
            * on(ceph_daemon) group_left(hostname) ceph_osd_metadata) > 0
        for: 5m
        labels: { severity: critical }
        annotations:
          summary: "Ceph OSD down: {{ $labels.ceph_daemon }} on {{ $labels.hostname }}"

      # 5) 主規則：其餘所有會危及 client 的 WARN+ check，逐 name page
      - alert: CephClientRisk
        expr: |
          ceph_health_detail{
            name!~"PG_AVAILABILITY|SLOW_OPS|OSD_DOWN|OSD_HOST_DOWN|HOST_IN_MAINTENANCE|OBJECT_MISPLACED|PG_SLOW_SNAP_TRIMMING"
          } == 1
        for: 5m
        labels: { severity: critical }
        annotations:
          summary: "Ceph client-risk check active: {{ $labels.name }}"

      # 6) quorum 失守 —— 補 MON_DOWN 0/1 看不出「掛幾台」的盲點（永不 silence）
      - alert: CephMonQuorumLost
        expr: count(ceph_mon_quorum_status == 1) < 2   # 3-mon 多數 = 2
        for: 1m
        labels: { severity: critical }

      # 7) 抓不到 ceph metrics 本身就是盲區（永不 silence）
      - alert: CephExporterDown
        expr: up{job="ceph"} == 0
        for: 5m
        labels: { severity: critical }

      # 8) 低優先提示 —— 只進 Slack，不 page
      - alert: CephLowPriorityNotice
        expr: |
          ceph_health_detail{
            name=~"HOST_IN_MAINTENANCE|OBJECT_MISPLACED|PG_SLOW_SNAP_TRIMMING"
          } == 1
        for: 30m
        labels: { severity: info }
```

`CephOSDNodeDown` / `CephOSDDown` 故意不走 `ceph_health_detail`，而是用 `ceph_osd_up` join `ceph_osd_metadata`，因為這條路徑保留 `hostname` 與 `ceph_daemon`。維護 node A 時只 silence `hostname="node-a"`，node B 掛掉仍會 page。

`ceph_health_detail` 仍由三組 name 集合分流：rule 2（client 已被擋）、rule 5（其餘 client-risk）、rule 8（低優先提示）。`OSD_DOWN` / `OSD_HOST_DOWN` 從 rule 5 排除，是因為它們已由 rule 3 / 4 用更細 label 接住；其他未列到的 health check 仍會被 `CephClientRisk` catch-all 接住。

`CephMonQuorumLost` 的 `< 2` 是刻意針對本頁場景的 **3-mon cluster**。Prometheus module 輸出的 `ceph_mon_quorum_status` 只有 `ceph_daemon` label，每個 mon 一條 0/1；source 沒有額外輸出「預期 quorum 大小」這個 metric。5-mon cluster 要改成 `< 3`，7-mon cluster 要改成 `< 4`，或在自己的 recording rule 裡從 inventory / 靜態設定帶入 expected mon count。

```python
# File: ceph/src/pybind/mgr/prometheus/module.py (line 641)

    'gauge',
    'mon_quorum_status',
    'Monitors in quorum',
    ('ceph_daemon',)
```

## 維護時怎麼 silence

topology 前提（這套環境）：**mon node 只跑 mon+mgr；osd node 只跑 osd**，兩者不重疊。因此各場景的 silence 名單互不交叉。OSD down 相關 alert 用 `hostname` / `ceph_daemon` 精準 silence；`PG_DEGRADED` 與 flags 這類 cluster-wide health check 才掛在 `CephClientRisk` 上用 `name=~` 限定。

```bash
# A. 維護 mon/mgr node（整台）—— 只會噴 MON_DOWN
amtool silence add alertname="CephClientRisk" 'name="MON_DOWN"' \
  --duration=2h -c "maint mon node X"
#   若維護的是 active mgr，先 `ceph mgr fail <active-mgr>` 切走 → 不噴 MGR_DOWN。
#   CephExporterDown / CephMonQuorumLost 絕不 silence。

# B. 維護 osd node（整台）
amtool silence add 'alertname=~"CephOSDNodeDown|CephOSDDown"' hostname="osd-host-Y" \
  --duration=2h -c "maint osd host Y"
amtool silence add alertname="CephClientRisk" \
  'name=~"PG_DEGRADED|OSDMAP_FLAGS|OSD_FLAGS"' \
  --duration=2h -c "maint osd host Y"

# C. 維護單顆 osd（換硬碟）
amtool silence add alertname="CephOSDDown" ceph_daemon="osd.N" \
  --duration=1h -c "replace osd.N"
amtool silence add alertname="CephClientRisk" \
  'name=~"PG_DEGRADED|OSD_FLAGS"' \
  --duration=1h -c "replace osd.N"
```

如果某個 host 只有一顆 OSD，單顆 OSD 維護也會等同整台 OSD node down，這時把 `CephOSDNodeDown` 的 `hostname` silence 一併加上。

**為什麼這樣 silence 是安全的**：維護 node A 時，`CephOSDNodeDown` / `CephOSDDown` 只 silence `hostname="node-a"` 或 `ceph_daemon="osd.N"`，node B 掛掉仍會 page。真正「client 被擋」的 `PG_AVAILABILITY` / `SLOW_OPS`（→ `CephClientBlocked`）、任何 ERR 升級（→ `CephHealthError`）、quorum 失守（→ `CephMonQuorumLost`）、抓取面盲區（→ `CephExporterDown`）都**不在 silence list**，維護窗內若真的出事照樣 page。`OBJECT_MISPLACED` 因已走低優先提示，維護時不必再 silence——少一個要記的名字。

### 如果是一台 node 非預期掛掉

**OSD node 掛掉**：不要先 silence。`CephOSDNodeDown` / `CephOSDDown` 是「冗餘下降、需要人處理」，不是低優先提示。先用這條 PromQL 找出實際 down 的 OSD 與 host：

```promql
((1 - ceph_osd_up) * on(ceph_daemon) group_left(hostname) ceph_osd_metadata) > 0
```

如果同時沒有 `CephClientBlocked`、`CephHealthError`、`CephMonQuorumLost`，代表 client I/O 尚未被明確擋住，但 cluster 已少一個 failure domain；處理方向是修 host、網路、systemd 或 OSD daemon，並盯著 recovery。只有在你已確認原因、把它轉成「預期中的維修窗口」後，才套用上面的 B silence，而且 duration 要短。`PG_AVAILABILITY` / `SLOW_OPS` / HEALTH_ERR 任何一個出現，就不要把它當成警報雜訊。

**mon node 掛掉**：3-mon cluster 掉 1 台時，`MON_DOWN` 會落在 `CephClientRisk`；`CephMonQuorumLost` 不會響，因為 quorum 還有 2 台。這不是可忽略狀態，而是「只剩一次 failure buffer」。先用這條找出 out-of-quorum 的 mon：

```promql
(ceph_mon_quorum_status == 0)
  + on(ceph_daemon) group_left(hostname) (ceph_mon_metadata * 0)
```

如果是計畫內維護，先跑 `ceph orch host ok-to-stop <hostname>`（或直接 `ceph mon ok-to-stop <mon-id>`），active mgr 在該 node 上時先 `ceph mgr fail <active-mgr>` 讓它切走，再套用 A silence。`CephMonQuorumLost` / `CephExporterDown` 一律不要 silence；它們響就代表不是「少一台但還能服務」，而是 quorum 或觀測面已經失守。

### 為什麼 OSD down 要做 per-host / per-OSD silence

`ceph_health_detail` 只有 `('name','severity')` 兩個 label（`module.py:113`），無法只靜音某台的 `OSD_DOWN`。`amtool silence ... name="OSD_DOWN"` 是整 cluster 靜音，這不是 Alertmanager 的限制，而是 Ceph prometheus module 沒把 per-node 維度帶到 health_detail。

但 OSD 狀態本身不必受這個限制：`ceph_osd_up` 每顆 OSD 一條、帶 `ceph_daemon`，join 上 `ceph_osd_metadata` 後可以取得 `hostname`：

```promql
((1 - ceph_osd_up) * on(ceph_daemon) group_left(hostname) ceph_osd_metadata) > 0
```

所以 OSD down 這兩條改成專用 rule：整台 host 用 `CephOSDNodeDown{hostname="..."}`，單顆 daemon 用 `CephOSDDown{ceph_daemon="...",hostname="..."}`。下游 check（例如 `PG_DEGRADED`）仍然只有 cluster 級，只能用短 duration 的 `CephClientRisk{name="PG_DEGRADED"}` silence；這是 ceph metric schema 的邊界。

## 驗證（promtool）

規則邏輯用 Prometheus 內建 `promtool test rules` 驗，不需要真 ceph。promtool **不模擬 alertmanager silence**，所以這裡只證明「該 fire 時有 fire」、「該分流的有分流」；silence 行為靠上面的 `amtool` SOP。

```yaml
# tests.yml
rule_files: [ ./ceph-alerts.yml ]
evaluation_interval: 30s
tests:
  - interval: 30s
    name: "client IO blocked 與低優先提示分流"
    input_series:
      - series: 'ceph_health_detail{name="PG_AVAILABILITY"}'
        values: '1x10'
      - series: 'ceph_health_detail{name="OBJECT_MISPLACED"}'
        values: '1x70'
      - series: 'ceph_osd_up{ceph_daemon="osd.1"}'
        values: '0x20'
      - series: 'ceph_osd_metadata{ceph_daemon="osd.1",hostname="osd-a"}'
        values: '1x20'
    alert_rule_test:
      - eval_time: 2m            # PG_AVAILABILITY for:1m 應已 fire
        alertname: CephClientBlocked
        exp_alerts:
          - exp_labels: { severity: critical, name: "PG_AVAILABILITY" }
      - eval_time: 6m            # osd.1 down for:5m → CephOSDDown fire
        alertname: CephOSDDown
        exp_alerts:
          - exp_labels: { severity: critical, ceph_daemon: "osd.1", hostname: "osd-a" }
      - eval_time: 6m            # 這個 host 只有 osd.1，整台 OSD node down 也應 fire
        alertname: CephOSDNodeDown
        exp_alerts:
          - exp_labels: { severity: critical, hostname: "osd-a" }
      - eval_time: 2m            # OBJECT_MISPLACED 在白名單 → CephClientRisk 不該 fire
        alertname: CephClientRisk
        exp_alerts: []
      - eval_time: 31m           # OBJECT_MISPLACED for:30m → CephLowPriorityNotice fire
        alertname: CephLowPriorityNotice
        exp_alerts:
          - exp_labels: { severity: info, name: "OBJECT_MISPLACED" }
```

```bash
promtool check rules ceph-alerts.yml   # lint
promtool test rules tests.yml          # 預期 SUCCESS
```

## 邊界與除錯

### exporter / mgr 掛掉時

`ceph:maintenance_active` 那套舊 gate 依賴 health_detail series 存在；mgr/prometheus 掛掉時 series 直接消失，gate 行為很難推理。新設計不靠它：`CephExporterDown`（`up{job="ceph"} == 0`，永不 silence）會在抓不到 ceph metrics 時直接 page，這正是「盲區」最該被叫醒的時候。

### 忘記 unset noout / flag

不再需要特製的「flag 設超過 12h」規則。維護的 silence 帶 `--duration` 會自動到期；到期後若 `noout` 等 flag 還在，`OSDMAP_FLAGS` / `OSD_FLAGS` 會讓 `CephClientRisk` 在 5 分鐘後重新 page——比舊的 12h timeout 更早提醒「你換完碟回家睡覺忘了 `unset noout`」。

### 單顆 OSD noout（`ceph osd add-noout osd.5`）

這個 case 不會觸發 `HOST_IN_MAINTENANCE`，也不一定會觸發 `CephOSDDown`，但會觸發 `OSD_FLAGS` → 走 `CephClientRisk`。只 silence 場景 C 的 `CephClientRisk{name=~"PG_DEGRADED|OSD_FLAGS"}` 那行即可；silence 到期 flag 還在就會再 page，剛好兜底「per-osd noout 忘記 `rm-noout`」。

### alertmanager inhibit（選配）

若想雙保險，可在 alertmanager 對 `CephLowPriorityNotice` 的 `HOST_IN_MAINTENANCE` 設 inhibit，把同一台維護期間的低優先提示再壓一層。這是選配，不是主路徑——主路徑就是上面的 `amtool silence`。
