## 場景

你已經啟用 ceph 內建的 `ceph-mixin/prometheus_alerts.yml`。這代表 `CephHealthError`、`CephOSDDown`、`CephOSDHostDown`、`CephMonDownQuorumAtRisk`、`CephPGUnavailableBlockingIO`、`CephMgrPrometheusModuleInactive` 這些預設 rule 都在。

這時不應該再寫一整套 6 條或 8 條 rule 取代它。預設 rule 已經含有 ceph 專案維護的 health check 名稱、severity、annotation query、documentation link；全部重寫只會把維護成本搬到自己身上。

真正的問題比較窄：**預設 OSD down 類 alert 可以在 annotation 裡查出哪台 host / 哪顆 OSD，但 alert 本身沒有 `hostname` / `ceph_daemon` label，Alertmanager 不能只 silence 維護中的 node A。** 如果你為了維護 node A 而 silence cluster-wide `CephOSDDown` / `CephOSDHostDown`，node B 同時掛掉的早期 warning 也會被吃掉。

所以這頁重新設計成：

1. **保留 ceph 預設 alert rules。**
2. **調整 routing：預設 OSD down 類 rule 不當 pager 主訊號，改當 Slack / ticket / context。**
3. **只新增兩條 scoped page rule：`CephOSDHostDownScoped` 與 `CephOSDDaemonDownScoped`。**
4. **維護時只 silence scoped rule 的 `hostname` 或 `ceph_daemon`，不做 cluster-wide OSD down silence。**

## 預設 rules 先當主體

內建 rule 檔在 `ceph/monitoring/ceph-mixin/prometheus_alerts.yml`。先看幾條應該保留的主體規則：cluster error、MON quorum、PG unavailable、mgr/prometheus module down，這些是 ceph upstream 已經幫你分好的核心訊號。

File: `ceph/monitoring/ceph-mixin/prometheus_alerts.yml` (line 4)

```yaml
- alert: "CephHealthError"
  expr: "ceph_health_status == 2"
  for: "5m"
  labels:
    severity: "critical"
    type: "ceph_default"
```

File: `ceph/monitoring/ceph-mixin/prometheus_alerts.yml` (line 25)

```yaml
- alert: "CephMonDownQuorumAtRisk"
  expr: |
    (
      (ceph_health_detail{name="MON_DOWN"} == 1) * on() (
        count(ceph_mon_quorum_status == 1) == bool (floor(count(ceph_mon_metadata) / 2) + 1)
      )
    ) == 1
  for: "30s"
```

File: `ceph/monitoring/ceph-mixin/prometheus_alerts.yml` (line 396)

```yaml
- alert: "CephPGUnavailableBlockingIO"
  expr: "((ceph_health_detail{name=\"PG_AVAILABILITY\"} == 1) - scalar(ceph_health_detail{name=\"OSD_DOWN\"})) == 1"
  for: "1m"
  labels:
    severity: "critical"
    type: "ceph_default"
```

File: `ceph/monitoring/ceph-mixin/prometheus_alerts.yml` (line 342)

```yaml
- alert: "CephMgrPrometheusModuleInactive"
  expr: "up{job=\"ceph\"} == 0"
  for: "1m"
  labels:
    severity: "critical"
    type: "ceph_default"
```

這些 rule 的設計不是「雜訊」。`CephHealthError` 是總開關；`CephMonDownQuorumAtRisk` 用 `ceph_mon_metadata` 算多數 quorum；`CephPGUnavailableBlockingIO` 指向 client I/O 已被擋；`CephMgrPrometheusModuleInactive` 代表觀測面盲掉。這些應該保留 page 路由，維護時也不要 silence。

建議 routing 分成三層：

| 層級 | 例子 | 路由 |
|---|---|---|
| 立即 page | `CephHealthError`、`CephMonDownQuorumAtRisk`、`CephPGUnavailableBlockingIO`、`CephPGsInactive`、`CephPGsDamaged`、`CephMgrPrometheusModuleInactive`、`CephSlowOps`、`CephDaemonSlowOps` | Pager |
| scoped OSD page | `CephOSDHostDownScoped`、`CephOSDDaemonDownScoped` | Pager，維護時可用 `hostname` / `ceph_daemon` 精準 silence |
| context / ticket | 預設 `CephHealthWarning`、預設 `CephOSDDown`、預設 `CephOSDHostDown`、`CephPGsUnclean`、`CephPGsHighPerOSD`、scrub 類 warning | Slack / ticket，不直接 page |

重點是：**不要刪預設 rule，只是不要讓無法精準 silence 的 OSD down 預設 rule 成為 pager 主訊號。**

## 為什麼預設 OSD down 不夠

預設 `CephOSDHostDown` / `CephOSDDown` 已經很接近你要的東西，但它們的 alert expression 仍然只看 `ceph_health_detail`。`ceph_health_detail` 的 label 只有 `name` 與 `severity`，沒有 host / daemon 維度。

File: `ceph/src/pybind/mgr/prometheus/module.py` (line 113)

```python
HEALTHCHECK_DETAIL = ('name', 'severity')
```

mgr/prometheus module 會把每個 health check 依 name 寫進 `health_detail`：

File: `ceph/src/pybind/mgr/prometheus/module.py` (line 931)

```python
self.health_history.check(health)
for name, info in self.health_history.healthcheck.items():
    v = 1 if info.active else 0
    self.metrics['health_detail'].set(
        v, (
            name,
            str(info.severity))
    )
```

所以預設 OSD host / OSD daemon rule 的 expr 長這樣：

File: `ceph/monitoring/ceph-mixin/prometheus_alerts.yml` (line 95)

```yaml
- alert: "CephOSDHostDown"
  expr: "ceph_health_detail{name=\"OSD_HOST_DOWN\"} == 1"
  for: "5m"
```

File: `ceph/monitoring/ceph-mixin/prometheus_alerts.yml` (line 105)

```yaml
- alert: "CephOSDDown"
  expr: "ceph_health_detail{name=\"OSD_DOWN\"} == 1"
  for: "5m"
```

這兩條的 annotation 會用 query 顯示哪台 host / 哪顆 OSD down，但 annotation query 的結果不是 alert label。Alertmanager matcher 只能 match alert labels，不能 match annotation 裡動態 query 出來的 host。

預設 rule 裡其實已經暗示另一條路：它的 annotation query 會 join `ceph_osd_up` 與 `ceph_osd_metadata`。這組 metric 本身就有足夠 label。

File: `ceph/src/pybind/mgr/prometheus/module.py` (line 88)

```python
OSD_METADATA = ('back_iface', 'ceph_daemon', 'cluster_addr', 'device_class',
                'front_iface', 'hostname', 'objectstore', 'public_addr',
                'ceph_version')
```

File: `ceph/src/pybind/mgr/prometheus/module.py` (line 689)

```python
metrics['osd_metadata'] = Metric(
    'untyped',
    'osd_metadata',
    'OSD Metadata',
    OSD_METADATA
)
```

因此補強策略很明確：**OSD down page 不要從 `ceph_health_detail` 取，改從 `ceph_osd_up` join `ceph_osd_metadata` 取。**

## 補兩條 scoped page rule

新增 rule 不要跟預設 `CephOSDDown` / `CephOSDHostDown` 撞名。撞名會讓 routing、silence、runbook 都變得難讀。這裡用 `Scoped` suffix 表示「這是同一個風險的可精準 silence page 版本」。

```yaml
groups:
  - name: ceph-scoped-osd-availability
    rules:
      - alert: CephOSDHostDownScoped
        expr: |
          count by (hostname) (
            (
              ceph_osd_up
                * on (ceph_daemon) group_left(hostname) ceph_osd_metadata
            ) == 0
          )
          ==
          count by (hostname) (ceph_osd_metadata)
        for: 5m
        labels:
          severity: critical
          source: ceph_scoped
        annotations:
          summary: "Ceph OSD host down: {{ $labels.hostname }}"
          runbook: "ceph osd tree; ceph health detail; check host power/network/systemd"

      - alert: CephOSDDaemonDownScoped
        expr: |
          (
            ceph_osd_up
              * on (ceph_daemon) group_left(hostname) ceph_osd_metadata
          ) == 0
          unless on (hostname)
          (
            count by (hostname) (
              (
                ceph_osd_up
                  * on (ceph_daemon) group_left(hostname) ceph_osd_metadata
              ) == 0
            )
            ==
            count by (hostname) (ceph_osd_metadata)
          )
        for: 5m
        labels:
          severity: critical
          source: ceph_scoped
        annotations:
          summary: "Ceph OSD down: {{ $labels.ceph_daemon }} on {{ $labels.hostname }}"
          runbook: "ceph osd tree; ceph osd ok-to-stop {{ $labels.ceph_daemon }}; inspect daemon log"
```

`CephOSDHostDownScoped` 是整台 OSD host 的 OSD 全部 down 才叫。`CephOSDDaemonDownScoped` 是單顆 OSD down 才叫，並用 `unless on (hostname)` 避免整台 host down 時每顆 OSD 都各叫一次。

這兩條不是取代所有 ceph-mixin rule，只是取代「page 路由上的 OSD down 主訊號」。預設 `CephOSDDown` / `CephOSDHostDown` 繼續保留，作為 Slack / ticket / Dashboard context。

## Alertmanager routing

在 routing 上，把 OSD down 的 page 權交給 scoped rule。預設 OSD down 類 rule 留著，但不要送 Pager。

```yaml
route:
  receiver: slack-ceph
  routes:
    - receiver: pager-ceph
      matchers:
        - alertname=~"CephOSDHostDownScoped|CephOSDDaemonDownScoped"

    - receiver: pager-ceph
      matchers:
        - alertname=~"CephHealthError|CephMonDownQuorumAtRisk|CephPGUnavailableBlockingIO|CephPGsInactive|CephPGsDamaged|CephPGRecoveryAtRisk|CephPGBackfillAtRisk|CephMgrPrometheusModuleInactive|CephSlowOps|CephDaemonSlowOps"

    - receiver: slack-ceph
      matchers:
        - type="ceph_default"
        - alertname=~"CephOSDDown|CephOSDHostDown|CephOSDDownHigh|CephHealthWarning|CephPGsUnclean"
```

這個 routing 的核心取捨：

- `CephOSDHostDownScoped` / `CephOSDDaemonDownScoped` 負責可靠性 page，因為它們有 `hostname` / `ceph_daemon`。
- 預設 `CephOSDDown` / `CephOSDHostDown` 不 page，因為它們不能精準 silence。
- 預設 critical client / quorum / exporter 類 rule 照常 page。

如果你把所有 `severity="critical"` 都直接送 Pager，要記得排除預設 `CephOSDDownHigh`。它是 aggregate percentage rule，沒有維護目標的 label；page 會回到同一個 cluster-wide silence 問題。

## 維護時怎麼 silence

### 維護整台 OSD node

進維護前先確認停這台 host 不會讓 PG 變 inactive。`ceph orch host maintenance enter` 內部也會跑 host ok-to-stop 檢查，但手動先跑一次比較適合 oncall SOP。

```bash
ceph orch host ok-to-stop osd-host-a

amtool silence add \
  'alertname=~"CephOSDHostDownScoped|CephOSDDaemonDownScoped"' \
  hostname="osd-host-a" \
  --duration=2h \
  -c "planned maintenance osd-host-a"

ceph orch host maintenance enter osd-host-a
```

這樣只會 silence `hostname="osd-host-a"`。如果維護期間 `osd-host-b` 掛掉，`CephOSDHostDownScoped{hostname="osd-host-b"}` 仍然會 page。

### 維護單顆 OSD

單顆 OSD 先跑 `ok-to-stop`。silence 也只掛單顆 daemon，不要 silence 整個 `OSD_DOWN` health check name。

```bash
ceph osd ok-to-stop 5

amtool silence add \
  alertname="CephOSDDaemonDownScoped" \
  ceph_daemon="osd.5" \
  --duration=1h \
  -c "replace osd.5"
```

如果 host 上只有一顆 OSD，停它時也會滿足「整台 OSD host down」。這種小 cluster 需要同時 silence host：

```bash
amtool silence add \
  alertname="CephOSDHostDownScoped" \
  hostname="osd-host-a" \
  --duration=1h \
  -c "single-osd host maintenance"
```

### 不要 silence 的東西

維護期間仍然不要 silence：

- `CephHealthError`
- `CephMonDownQuorumAtRisk`
- `CephPGUnavailableBlockingIO`
- `CephPGsInactive`
- `CephSlowOps` / `CephDaemonSlowOps`
- `CephMgrPrometheusModuleInactive`

這些代表 client I/O、quorum、或觀測面已經出事。它們不該被「我正在維護一台 OSD node」掩蓋。

## 預設 HealthWarning 怎麼處理

`CephHealthWarning` 是 `ceph_health_status == 1` 的總開關。它有價值，但不適合當 pager 主訊號，因為維護、rebalance、scrub backlog、單顆 OSD down、clock skew、flags 都會讓 cluster 進 WARN。

建議路由：

- `CephHealthWarning`：Slack / ticket，保留上下文。
- `CephHealthError`：Pager。
- 具體 default critical rule：Pager。
- OSD down 早期 warning：用 scoped add-on rule pager。

這樣你不會失去預設 rule 的完整性，也不會因為維護一台 host 而把第二台 host 的 OSD down early warning silence 掉。

## 驗證（promtool）

這兩條 scoped rule 可以用 `promtool test rules` 驗，不需要真 ceph cluster。測試重點是：整台 host down 時只 fire host alert；單顆 OSD down 時 fire daemon alert。

```yaml
rule_files: [ ./ceph-scoped-osd.yml ]
evaluation_interval: 30s
tests:
  - interval: 30s
    name: "scoped OSD host and daemon down"
    input_series:
      - series: 'ceph_osd_up{ceph_daemon="osd.1"}'
        values: '0x20'
      - series: 'ceph_osd_metadata{ceph_daemon="osd.1",hostname="osd-a"}'
        values: '1x20'
      - series: 'ceph_osd_up{ceph_daemon="osd.2"}'
        values: '0x20'
      - series: 'ceph_osd_metadata{ceph_daemon="osd.2",hostname="osd-a"}'
        values: '1x20'
      - series: 'ceph_osd_up{ceph_daemon="osd.3"}'
        values: '0x20'
      - series: 'ceph_osd_metadata{ceph_daemon="osd.3",hostname="osd-b"}'
        values: '1x20'
      - series: 'ceph_osd_up{ceph_daemon="osd.4"}'
        values: '1x20'
      - series: 'ceph_osd_metadata{ceph_daemon="osd.4",hostname="osd-b"}'
        values: '1x20'
    alert_rule_test:
      - eval_time: 6m
        alertname: CephOSDHostDownScoped
        exp_alerts:
          - exp_labels: { severity: critical, source: ceph_scoped, hostname: "osd-a" }
      - eval_time: 6m
        alertname: CephOSDDaemonDownScoped
        exp_alerts:
          - exp_labels: { severity: critical, source: ceph_scoped, ceph_daemon: "osd.3", hostname: "osd-b" }
```

## 邊界與除錯

### 預設 OSD alert 還會出現

預設 `CephOSDDown` / `CephOSDHostDown` 仍然會 fire，因為我們沒有刪 rule。差別是 routing 不讓它們直接 page。它們可以留在 Slack、ticket、Dashboard，提供跟 upstream 文件一致的上下文。

### scoped rule 依賴 `ceph_osd_metadata`

`CephOSDHostDownScoped` / `CephOSDDaemonDownScoped` 需要 `ceph_osd_metadata`。如果 mgr/prometheus module 掛掉，預設 `CephMgrPrometheusModuleInactive` 會 page；這是比 OSD scoped alert 更上層的觀測面故障。

### PG 類 alert 仍然是 cluster / pool 級

`CephPGsUnclean`、`CephPGUnavailableBlockingIO`、`PG_DEGRADED` 類 health check 本來就不是 per-host label。你可以讓 non-blocking 的 `CephPGsUnclean` 走 Slack，但 `CephPGUnavailableBlockingIO` 必須 page。它代表 placement group 已不能服務 I/O，不是維護雜訊。

### MON node 維護

本頁沒有新增 scoped MON rule，因為預設 `CephMonDownQuorumAtRisk` 已經用 quorum majority 做關鍵判斷。對 3-mon cluster 來說，掉 1 台就已經只剩一次 failure buffer；可靠性優先時，計畫內 MON 維護可以短暫 silence `CephMonDown` 這種 warning，但不要 silence `CephMonDownQuorumAtRisk`，除非你明確接受維護期間失去 MON early warning。
