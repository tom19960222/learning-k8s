## 場景

這頁的前提是：你已經套用 ceph v19.2.3 內建的 `ceph-mixin/prometheus_alerts.yml`。也就是 `CephHealthError`、`CephHealthWarning`、`CephOSDDown`、`CephOSDHostDown`、`CephMonDown`、`CephMonDownQuorumAtRisk`、`CephPGUnavailableBlockingIO`、`CephMgrPrometheusModuleInactive` 這些預設 rule 都在。

因此這頁不是「自己重寫一整套 Ceph alert」。比較好的做法是把預設 rule 留著，然後明確分工：

1. `CephHealthError` 用預設 rule，照樣 page。
2. `ceph_health_detail` 只適合做「哪個 health check active」的穩定性 page，不適合用來做 host / daemon 維護 silence。
3. OSD / MON down 改用 scoped rule，alert label 裡要有 `hostname` 或 `ceph_daemon`，維護時才能只 silence 目標。
4. 維護期間可以短時間 silence aggregate context alert，但不能 silence 會代表 client I/O blocked、quorum lost、或 metrics blind spot 的 alert。

下面的設計是把兩個方向合併：一組 `ceph-stability-first` rules 負責 client 風險，一組 `ceph-scoped-availability` rules 負責 MON / OSD 維度。

## 預設 rules 怎麼保留

預設 `CephHealthError` 保留，因為它是 cluster 已經進 `HEALTH_ERR` 的總開關。這種 alert 沒有維護目標 label，但語意夠重：它不是「哪台 node 維護中」的雜訊。

File: `ceph/monitoring/ceph-mixin/prometheus_alerts.yml` (line 4)

```yaml
- alert: "CephHealthError"
  expr: "ceph_health_status == 2"
  for: "5m"
  labels:
    severity: "critical"
    type: "ceph_default"
```

預設 `CephHealthWarning` 也保留，但不進 Pager。它只是 `ceph_health_status == 1`，任何 maintenance、rebalance、scrub backlog、OSD down、MON down 都可能讓它 active。

File: `ceph/monitoring/ceph-mixin/prometheus_alerts.yml` (line 14)

```yaml
- alert: "CephHealthWarning"
  expr: "ceph_health_status == 1"
  for: "15m"
  labels:
    severity: "warning"
    type: "ceph_default"
```

`ceph_health_detail` 也要理解清楚。mgr/prometheus module 只把 `name` 與 `severity` 放進 `health_detail`，沒有 host / daemon label。

File: `ceph/src/pybind/mgr/prometheus/module.py` (line 113)

```python
HEALTHCHECK_DETAIL = ('name', 'severity')
```

File: `ceph/src/pybind/mgr/prometheus/module.py` (line 931)

```python
self.health_history.check(health)
for name, info in self.health_history.healthcheck.items():
    v = 1 if info.active else 0
    self.metrics['health_detail'].set(
        v, (
            name,
            str(info.severity))
    )
```

這解釋了為什麼 `CephClientBlocked` / `CephClientRisk` 可以逐 `name` page，但 OSD / MON 維護不能只靠 `ceph_health_detail`。`MON_DOWN`、`OSD_DOWN`、`OSD_HOST_DOWN` 都會變成 cluster-wide health check name；如果拿它們直接 page，維護時很難只 silence 目標 node。

## 合併後的 alert rules

這份 rule set 假設你的 MON 是 3-mon quorum，因此 `CephMonQuorumLost` 用 `count(...) < 2`。如果未來改成 5-mon 或更多，這條要改成動態 majority。

我也對你的基底做一個必要修正：`CephClientRisk` 要排除 `MON_DOWN`。否則 MON node 維護時，`CephMonDownScoped` 雖然可以精準 silence，`CephClientRisk{name="MON_DOWN"}` 還是會 cluster-wide page，等於把維護噪音繞回來。

```yaml
groups:
  - name: ceph-stability-first
    rules:
      - alert: CephClientBlocked
        expr: ceph_health_detail{name=~"PG_AVAILABILITY|SLOW_OPS"} == 1
        for: 1m
        labels:
          severity: critical
          source: ceph_stability
        annotations:
          summary: "Ceph client I/O blocked: {{ $labels.name }}"

      - alert: CephClientRisk
        expr: |
          ceph_health_detail{
            name!~"PG_AVAILABILITY|SLOW_OPS|OSD_DOWN|OSD_HOST_DOWN|MON_DOWN|HOST_IN_MAINTENANCE|OBJECT_MISPLACED|PG_SLOW_SNAP_TRIMMING"
          } == 1
        for: 5m
        labels:
          severity: critical
          source: ceph_stability
        annotations:
          summary: "Ceph client-risk check active: {{ $labels.name }}"

      - alert: CephMonQuorumLost
        expr: count(ceph_mon_quorum_status == 1) < 2
        for: 1m
        labels:
          severity: critical
          source: ceph_stability
        annotations:
          summary: "Ceph MON quorum lost or below 3-mon majority"

      - alert: CephExporterDown
        expr: up{job="ceph"} == 0
        for: 5m
        labels:
          severity: critical
          source: ceph_stability
        annotations:
          summary: "Ceph Prometheus exporter is down"

      - alert: CephLowPriorityNotice
        expr: |
          ceph_health_detail{
            name=~"HOST_IN_MAINTENANCE|OBJECT_MISPLACED|PG_SLOW_SNAP_TRIMMING"
          } == 1
        for: 30m
        labels:
          severity: info
          source: ceph_stability
        annotations:
          summary: "Ceph low-priority notice: {{ $labels.name }}"

  - name: ceph-scoped-availability
    rules:
      - record: ceph:osd_up:with_hostname
        expr: |
          ceph_osd_up
            * on (ceph_daemon) group_left(hostname) ceph_osd_metadata

      - record: ceph:osd_host_down:scoped
        expr: |
          count by (hostname) (ceph:osd_up:with_hostname == 0)
          ==
          count by (hostname) (ceph:osd_up:with_hostname)

      - alert: CephOSDHostDownScoped
        expr: ceph:osd_host_down:scoped
        for: 5m
        labels:
          severity: critical
          source: ceph_scoped
        annotations:
          summary: "Ceph OSD host down: {{ $labels.hostname }}"
          runbook: "ceph osd tree; ceph health detail; check host power/network/systemd"

      - alert: CephOSDDaemonDownScoped
        expr: |
          (ceph:osd_up:with_hostname == 0)
          unless on (hostname) ceph:osd_host_down:scoped
        for: 5m
        labels:
          severity: critical
          source: ceph_scoped
        annotations:
          summary: "Ceph OSD down: {{ $labels.ceph_daemon }} on {{ $labels.hostname }}"
          runbook: "ceph osd tree; ceph osd ok-to-stop {{ $labels.ceph_daemon }}; inspect daemon log"

      - alert: CephMonDownScoped
        expr: |
          (
            (1 - ceph_mon_quorum_status)
              * on (ceph_daemon) group_left(hostname) ceph_mon_metadata
          ) == 1
        for: 30s
        labels:
          severity: critical
          source: ceph_scoped
        annotations:
          summary: "Ceph MON down or out of quorum: {{ $labels.ceph_daemon }} on {{ $labels.hostname }}"
          runbook: "ceph quorum_status; ceph health detail; inspect mon log and host network"
```

<details>
<summary>為什麼 OSD / MON down 要另外做 scoped rules</summary>

預設 `CephOSDDown` 與 `CephOSDHostDown` 的 expr 只看 `ceph_health_detail`：

```yaml
- alert: "CephOSDHostDown"
  expr: "ceph_health_detail{name=\"OSD_HOST_DOWN\"} == 1"

- alert: "CephOSDDown"
  expr: "ceph_health_detail{name=\"OSD_DOWN\"} == 1"
```

它們的 annotation query 可以列出哪台 host / 哪顆 OSD，但是 annotation 不是 alert label，Alertmanager silence matcher 不能 match annotation 裡查出來的 host。

OSD metadata 本身有 `ceph_daemon` 與 `hostname`：

File: `ceph/src/pybind/mgr/prometheus/module.py` (line 88)

```python
OSD_METADATA = ('back_iface', 'ceph_daemon', 'cluster_addr', 'device_class',
                'front_iface', 'hostname', 'objectstore', 'public_addr',
                'ceph_version')
```

MON metric 也有同樣的 label 來源：

File: `ceph/src/pybind/mgr/prometheus/module.py` (line 77)

```python
MON_METADATA = ('ceph_daemon', 'hostname',
                'public_addr', 'rank', 'ceph_version')
```

File: `ceph/src/pybind/mgr/prometheus/module.py` (line 641)

```python
metrics['mon_quorum_status'] = Metric(
    'gauge',
    'mon_quorum_status',
    'Monitors in quorum',
    ('ceph_daemon',)
)
```

所以 OSD / MON down 的 pager 主訊號要從 metadata metric 裡拿 label，而不是直接從 `ceph_health_detail` 拿。

</details>

<details>
<summary>OSD daemon down query 怎麼讀</summary>

`ceph:osd_up:with_hostname` 是 recording rule，把 `ceph_osd_up` 的 0/1 狀態接上 `ceph_osd_metadata` 的 `hostname`。

```promql
ceph_osd_up
  * on (ceph_daemon) group_left(hostname) ceph_osd_metadata
```

`ceph:osd_host_down:scoped` 用同一台 host 上「down 的 OSD 數量」等於「全部 OSD 數量」判斷整台 OSD host down。

`CephOSDDaemonDownScoped` 剩下兩句：

```promql
(ceph:osd_up:with_hostname == 0)
unless on (hostname) ceph:osd_host_down:scoped
```

第一句找單顆 OSD down，第二句排除已經由 host alert 接手的 hostname。效果是：只有 `osd.5` down 時 page `CephOSDDaemonDownScoped{ceph_daemon="osd.5"}`；整台 `osd-host-a` 掛掉時只 page `CephOSDHostDownScoped{hostname="osd-host-a"}`，不會每顆 OSD 各叫一次。

</details>

## Alertmanager routing

因為預設 ceph-mixin rules 仍然存在，routing 不能再寫成「所有 `severity=critical` 都送 Pager」。那會讓預設 `CephMonDownQuorumAtRisk`、`CephOSDDownHigh` 這類 aggregate alert 在維護期間繼續叫。

比較清楚的 routing 是：

```yaml
route:
  receiver: slack-ceph
  routes:
    - receiver: pager-ceph
      matchers:
        - type="ceph_default"
        - alertname="CephHealthError"

    - receiver: pager-ceph
      matchers:
        - alertname=~"CephClientBlocked|CephClientRisk|CephMonQuorumLost|CephExporterDown|CephOSDHostDownScoped|CephOSDDaemonDownScoped|CephMonDownScoped"

    - receiver: slack-ceph
      matchers:
        - alertname="CephLowPriorityNotice"

    - receiver: slack-ceph
      matchers:
        - type="ceph_default"
```

這樣預設 rule 還是會被保留在 Slack / ticket / Dashboard 裡，但 Pager 的主訊號來自：

- 預設 `CephHealthError`
- `ceph-stability-first`
- `ceph-scoped-availability`

## 維護時怎麼 silence

### 維護 MON node

MON 維護前先確認停這台 host 或這個 mon 不會立刻打掉可用性。`mon ok-to-stop` 是 Ceph 原生 monitor command；`orch host ok-to-stop` 是 host 級檢查。

File: `ceph/src/mon/MonCommands.h` (line 250)

```cpp
COMMAND("mon ok-to-stop "
    "name=ids,type=CephString,n=N",
    "check whether mon(s) can be safely stopped without reducing immediate "
    "availability",
    "mon", "r")
```

File: `ceph/src/pybind/mgr/orchestrator/module.py` (line 782)

```python
@_cli_write_command('orch host ok-to-stop')
def _host_ok_to_stop(self, hostname: str) -> HandleCommandResult:
    completion = self.host_ok_to_stop(hostname)
```

假設 `mon.a` 跑在 `mon-host-a`：

```bash
ceph orch host ok-to-stop mon-host-a
ceph mon ok-to-stop a
ceph quorum_status

amtool silence add \
  alertname="CephMonDownScoped" \
  hostname="mon-host-a" \
  ceph_daemon="mon.a" \
  --duration=1h \
  -c "planned maintenance mon.a on mon-host-a"

ceph orch host maintenance enter mon-host-a
```

如果 Slack / ticket 不想在維護期間一直收到預設 aggregate context，可以短時間 silence 這些 context alert：

```bash
amtool silence add \
  'alertname=~"CephHealthWarning|CephMonDown|CephMonDownQuorumAtRisk|CephLowPriorityNotice"' \
  --duration=1h \
  -c "mute aggregate ceph context during mon.a maintenance"
```

不要 silence `CephMonQuorumLost`。如果 3-mon cluster 在維護 `mon.a` 時又掉 `mon.b`，`count(ceph_mon_quorum_status == 1) < 2` 會 page，這就是你要保留的生命線。

### 維護整台 OSD node

整台 OSD node 維護前先跑 host ok-to-stop。這會比直接進 maintenance 更適合 oncall SOP，因為你會先看到停這台 host 是否讓 PG 風險上升。

```bash
ceph orch host ok-to-stop osd-host-a

amtool silence add \
  'alertname=~"CephOSDHostDownScoped|CephOSDDaemonDownScoped"' \
  hostname="osd-host-a" \
  --duration=2h \
  -c "planned maintenance osd-host-a"

ceph orch host maintenance enter osd-host-a
```

這只會 silence `hostname="osd-host-a"`。如果維護期間 `osd-host-b` 掛掉，`CephOSDHostDownScoped{hostname="osd-host-b"}` 還是會 page。

如果 Slack / ticket 也想降噪，可以 silence aggregate context：

```bash
amtool silence add \
  'alertname=~"CephHealthWarning|CephLowPriorityNotice|CephOSDDown|CephOSDHostDown|CephOSDDownHigh"' \
  --duration=2h \
  -c "mute aggregate ceph context during osd-host-a maintenance"
```

不要 silence `CephClientBlocked`、`CephClientRisk`、`CephHealthError`。如果維護真的讓 client I/O 或更高層 health check 出問題，那就不是噪音。

### 維護單顆 OSD

單顆 OSD 先跑 `ok-to-stop`，再只 silence 該 daemon。

```bash
ceph osd ok-to-stop 5

amtool silence add \
  alertname="CephOSDDaemonDownScoped" \
  ceph_daemon="osd.5" \
  --duration=1h \
  -c "replace osd.5"
```

如果這台 host 上只有一顆 OSD，停它會同時符合「整台 OSD host down」。這種小 cluster 要多加 host silence：

```bash
amtool silence add \
  alertname="CephOSDHostDownScoped" \
  hostname="osd-host-a" \
  --duration=1h \
  -c "single-osd host maintenance"
```

## 發出 alert 時怎麼處理

### `CephClientBlocked`

這代表 client I/O 已經被 `PG_AVAILABILITY` 或 `SLOW_OPS` 影響。不要 silence。

先做：

```bash
ceph -s
ceph health detail
ceph pg dump_stuck inactive
ceph pg dump_stuck unclean
ceph osd tree
```

如果是 `PG_AVAILABILITY`，優先找 inactive / unavailable PG、down OSD、full / backfillfull 狀態。這類 alert 的目標不是「先分類」，而是先恢復 client 可用性。

如果是 `SLOW_OPS`，先看 slow op 是卡在 MON、OSD，還是整體 I/O 壓力。接著看 OSD latency、磁碟、網路、recovery / backfill 壓力。

### `CephClientRisk`

這是「沒有被低優先或 scoped rule 接走的 health check」。處理方式是以 `$labels.name` 為入口：

```bash
ceph health detail
ceph -s
```

如果 `$labels.name` 是明確的 client 風險，照 health detail 的 check 名稱往下查。如果你發現某個 check 其實只是維護期間常態噪音，不要直接 silence 整條 `CephClientRisk`；應該把那個 health check name 移到排除清單或 `CephLowPriorityNotice`，讓規則語意保持乾淨。

### `CephMonQuorumLost`

這是 MON quorum 低於 3-mon majority。不要 silence，不要等它自己恢復。

```bash
ceph quorum_status
ceph health detail
ceph orch ps --daemon_type mon
```

先確認是否還能拿到 quorum。如果 quorum 已經失守，優先恢復 MON host 的網路、時間同步、磁碟空間與 mon daemon。不要同時重啟多台 mon；先讓一台已知健康的 mon 回到 quorum。

### `CephExporterDown`

這代表 Prometheus 抓不到 Ceph metrics。它本身不是 client I/O failure，但它會讓所有 Prometheus alert 變成盲區，所以不要 silence。

```bash
ceph -s
ceph mgr services
ceph mgr module ls
```

如果 mgr/prometheus module 沒啟用，修正 module。若 active mgr 掛掉，先恢復 mgr。這段期間不要只看 Prometheus，改用 `ceph -s` / `ceph health detail` 做人工確認。

### `CephOSDHostDownScoped`

這是整台 OSD host 的 OSD 都 down。先判斷是 planned maintenance、host power、網路、systemd，還是 cephadm 問題。

```bash
ceph osd tree
ceph orch ps --host osd-host-a
ceph health detail
```

如果不是計畫內維護，不要先做 cluster-wide silence。先恢復 host 或確認是否需要讓資料恢復路徑接手。

### `CephOSDDaemonDownScoped`

這是單顆 OSD down，不是整台 host down。

```bash
ceph osd tree
ceph osd ok-to-stop <id>
ceph health detail
```

接著查該 OSD 的 daemon log、device、BlueStore 狀態、是否被 mark out。這條 alert 有 `ceph_daemon` 與 `hostname`，所以維護或更換硬碟時可以只 silence 該 OSD。

### `CephMonDownScoped`

這是某個 mon 不在 quorum。先看 quorum，再看該 host：

```bash
ceph quorum_status
ceph health detail
ceph orch ps --daemon_type mon
```

如果是計畫內維護，確認 silence 是否只 match 目標 `ceph_daemon` / `hostname`。如果不是計畫內維護，先恢復 mon host 的網路、時間、磁碟，再看 mon log。

### `CephLowPriorityNotice`

這只進 Slack / ticket，不 page。典型例子是 `HOST_IN_MAINTENANCE`、`OBJECT_MISPLACED`、`PG_SLOW_SNAP_TRIMMING`。

處理方式是記錄、追蹤、看是否超過預期時間。它不應該把人叫醒，但也不應該完全消失。

## 邊界

### 3-mon hard-code

`CephMonQuorumLost` 用的是 3-mon majority：

```promql
count(ceph_mon_quorum_status == 1) < 2
```

如果 cluster 不是 3-mon，改成：

```promql
count(ceph_mon_quorum_status == 1) < (floor(count(ceph_mon_metadata) / 2) + 1)
```

### 預設 rules 仍然會出現

預設 `CephOSDDown`、`CephOSDHostDown`、`CephMonDown`、`CephMonDownQuorumAtRisk` 仍然會 fire。差別是 routing 不讓它們當 pager 主訊號。它們留在 Slack / ticket / Dashboard，提供跟 upstream 文件一致的 context。

### `CephClientRisk` 不要變成垃圾桶

`CephClientRisk` 的重點是「其餘會危及 client 的 health check」。如果某個 check 長期不是 client 風險，就把它移出這條 rule；不要讓 oncall 被一條過大的 catch-all rule 訓練到無視 Pager。
