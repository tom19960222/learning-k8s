## 場景

假設你已經套用 ceph v19.2.3 內建的 `ceph-mixin/prometheus_alerts.yml`——`CephHealthError`、`CephHealthWarning`、`CephOSDDown`、`CephOSDHostDown`、`CephMonDown`、`CephMonDownQuorumAtRisk`、`CephPGUnavailableBlockingIO`、`CephMgrPrometheusModuleInactive` 這些預設 rule 都在。

預設 rule 拿來「看 cluster 健不健康」夠用，但拿來**當 oncall 的 pager 訊號**有個痛點：維護一台 node 時，那些 cluster-wide 的 aggregate alert（`CephOSDHostDown`、`CephMonDown`…）會一起叫，而你**沒辦法只 silence 維護中的那台**。結果不是被維護噪音吵，就是粗暴地把整類 alert 全 silence、連真故障一起蓋掉。

這頁不是重寫一整套 alert，而是在預設 rule **之上加一層**：保留預設 rule 當 context，另外補一組「維護友善、可精準 silence」的 rule，並把三件事分清楚——哪些該 page、哪些只進 Slack、哪些維護時可以安全 silence。

## 核心問題：`ceph_health_detail` 沒有 host label

整頁的設計都繞著一個事實轉，先講清楚：**mgr/prometheus module 匯出的 `ceph_health_detail` 只帶 `name` 與 `severity`，沒有 host / daemon label。**

File: `ceph/src/pybind/mgr/prometheus/module.py` (line 113)

```python
HEALTHCHECK_DETAIL = ('name', 'severity')
```

File: `ceph/src/pybind/mgr/prometheus/module.py` (line 931)

```python
self.health_history.check(health)
for name, info in self.health_history.healthcheck.items():
    v = 1 if info.active else 0
    self.metrics['health_detail'].set(
        v, (
            name,
            str(info.severity))
    )
```

所以 `MON_DOWN`、`OSD_DOWN`、`OSD_HOST_DOWN` 在 `ceph_health_detail` 裡都只是一個 cluster-wide 的 `name`，**查不到是哪台 host**。預設 `CephOSDHostDown` / `CephOSDDown` 的 expr 也正是直接看這個：

```yaml
- alert: "CephOSDHostDown"
  expr: "ceph_health_detail{name=\"OSD_HOST_DOWN\"} == 1"

- alert: "CephOSDDown"
  expr: "ceph_health_detail{name=\"OSD_DOWN\"} == 1"
```

它們的 annotation 雖然能用 query 列出哪台 host，但 **annotation 不是 alert label，Alertmanager 的 silence matcher 不能 match annotation**。所以維護一台 OSD host 時，沒辦法用 `hostname=...` 去 silence 這條 alert——這就是痛點的根源。

好消息是 host / daemon 資訊本來就在**另一個** metric 裡。OSD / MON 的 metadata metric 帶 `ceph_daemon` 與 `hostname`：

```python

                'front_iface', 'hostname', 'objectstore', 'public_addr',
                'ceph_version')
```

File: `ceph/src/pybind/mgr/prometheus/module.py` (line 77)

```python
MON_METADATA = ('ceph_daemon', 'hostname',
                'public_addr', 'rank', 'ceph_version')
```

File: `ceph/src/pybind/mgr/prometheus/module.py` (line 641)

```python

    'gauge',
    'mon_quorum_status',
    'Monitors in quorum',
    ('ceph_daemon',)
)
```

**結論：OSD / MON down 的 pager 主訊號要從 metadata metric 拿 label（才有 `hostname` / `ceph_daemon` 可以 silence），而不是直接從 `ceph_health_detail` 拿。** 這就是下面「scoped rule」的全部動機。

## 三種角色：每條 alert 該扮演什麼

加這一層之前，先定清楚每條 alert 屬於哪一類——這決定它進 routing 的哪條路、維護時能不能 silence：

| 角色 | 意思 | 維護時能 silence 嗎 |
|---|---|---|
| **Page 主訊號** | client I/O blocked、quorum lost、metrics 盲區、scoped 的 host/daemon down | 只有 scoped 的能按目標 silence；client/quorum/exporter 類**不可** silence |
| **Slack / ticket context** | 預設 aggregate rule、低優先 notice | 可以短時間 silence 降噪 |
| **維護可 silence** | scoped host/daemon down（帶 `hostname`/`ceph_daemon`） | 維護時只 silence 目標那台 |

對應地，這層補兩組核心 rule：`ceph-stability-first`（負責 client 風險，幾乎都不可 silence）與 `ceph-scoped-availability`（負責 MON/OSD 維度，帶 label 可精準 silence）。真機故障注入後，又把「早期風險 / coverage」獨立成 `ceph-production-coverage`，避免 `CephClientRisk` 變成垃圾桶；完整修訂理由見 [Ceph - 真機故障注入後的 alert 韌性修訂](./prometheus-alert-real-lab-findings)。

## `ceph-stability-first`：client 風險，多數不可 silence

這組的共通點是「真的出事才會 active」，所以多數是 page 主訊號、維護時也不該 silence。

```yaml
groups:
  - name: ceph-stability-first
    rules:
      - alert: CephClientBlocked
        expr: ceph_health_detail{name=~"PG_AVAILABILITY|SLOW_OPS|OSD_FULL|POOL_FULL"} == 1
        for: 1m
        labels:
          severity: critical
          source: ceph_stability
        annotations:
          summary: "Ceph client I/O blocked: {{ $labels.name }}"

      - alert: CephClientRisk
        expr: |
          ceph_health_detail{
            name!~"PG_AVAILABILITY|SLOW_OPS|OSD_FULL|POOL_FULL|OSD_DOWN|OSD_HOST_DOWN|MON_DOWN|HOST_IN_MAINTENANCE|OBJECT_MISPLACED|PG_SLOW_SNAP_TRIMMING|PG_DEGRADED|OSDMAP_FLAGS|POOL_APP_NOT_ENABLED|POOL_NEARFULL|OSD_SLOW_PING_TIME_BACK|OSD_SLOW_PING_TIME_FRONT|MON_CLOCK_SKEW|RECENT_CRASH|OSD_NEARFULL|OSD_BACKFILLFULL|MON_DISK_LOW|MON_DISK_CRIT|PG_DAMAGED|OSD_SCRUB_ERRORS|OBJECT_UNFOUND"
          } == 1
        for: 5m
        labels:
          severity: critical
          source: ceph_stability
        annotations:
          summary: "Ceph client-risk check active: {{ $labels.name }}"

      - alert: CephMonQuorumLost
        expr: (count(ceph_mon_quorum_status == 1) or vector(0)) < 2
        for: 1m
        labels:
          severity: critical
          source: ceph_stability
        annotations:
          summary: "Ceph MON quorum lost or below 3-mon majority"

      - alert: CephExporterAllDown
        expr: (count(up{job="ceph"} == 1) or vector(0)) == 0
        for: 5m
        labels:
          severity: critical
          source: ceph_stability
        annotations:
          summary: "All Ceph Prometheus exporter targets are down"

      - alert: CephExporterTargetDown
        expr: up{job="ceph"} == 0
        for: 15m
        labels:
          severity: warning
          source: ceph_stability
        annotations:
          summary: "Ceph exporter target {{ $labels.instance }} is down"

      - alert: CephLowPriorityNotice
        expr: |
          ceph_health_detail{
            name=~"HOST_IN_MAINTENANCE|OBJECT_MISPLACED|PG_SLOW_SNAP_TRIMMING|PG_DEGRADED|OSDMAP_FLAGS|POOL_APP_NOT_ENABLED|POOL_NEARFULL"
          } == 1
        for: 30m
        labels:
          severity: info
          source: ceph_stability
        annotations:
          summary: "Ceph low-priority notice: {{ $labels.name }}"
```

兩個設計重點：

- **`CephClientBlocked` vs `CephClientRisk` 的分工**：前者只盯 `PG_AVAILABILITY` / `SLOW_OPS` / `OSD_FULL` / `POOL_FULL`（client I/O 已經被擋或容量已到硬擋），1 分鐘就 page；後者是「其餘還沒被專屬 rule 接住、但會危及 client 的 health check」的 catch-all，5 分鐘 page。
- **`CephClientRisk` 的 `name!~` 排除清單**很關鍵：它排掉了 `OSD_DOWN` / `OSD_HOST_DOWN` / `MON_DOWN`（這些交給下面 scoped rule，否則維護時會 cluster-wide 繞回來叫）、低優先 notice（`HOST_IN_MAINTENANCE` / `OBJECT_MISPLACED` / `PG_SLOW_SNAP_TRIMMING` / `PG_DEGRADED` / `OSDMAP_FLAGS` / `POOL_APP_NOT_ENABLED` / `POOL_NEARFULL`），以及已經由 `ceph-production-coverage` 專責的網路、容量、clock skew、crash、資料完整性訊號。把 `MON_DOWN` 排掉特別重要：否則 MON 維護時 `CephMonDownScoped` 雖能精準 silence，`CephClientRisk{name="MON_DOWN"}` 還是會 cluster-wide page。
- **特別注意 `PG_DEGRADED` / `OSDMAP_FLAGS` —— 這兩個是「換硬碟一定會叫」的元兇**。停一顆 OSD 換硬碟時，它上面的 PG 副本掉一份 → ceph 升 `PG_DEGRADED`；如果維護前設了 `noout` 旗標，會升 `OSDMAP_FLAGS`。`CephClientRisk` 是 `name!~` 的 catch-all，**任何不在排除清單裡的 health check active 滿 5 分鐘都 page**——所以早期版本只 silence `CephOSDDaemonDownScoped` 是不夠的，`CephClientRisk{name="PG_DEGRADED"}` 還是會在 5 分鐘後叫。把這兩個排除掉、歸到 Slack-only 的 `CephLowPriorityNotice` 是對的：`PG_DEGRADED` 在 ceph 內部就是 `HEALTH_WARN`（`PGMap.cc:2578`，`sev = HEALTH_WARN`），而且 upstream ceph-mixin 預設**沒有任何 rule 直接 page 它**（degraded 只讓 `CephHealthWarning` 這條 Slack-only 的 warning 亮）。換句話說，把 degraded 排出 pager 不是「降級一個該叫的訊號」，是把它修回 ceph 預設本來的等級——真正擋到 I/O 的 `PG_AVAILABILITY` 仍由 `CephClientBlocked` page，生命線不動。
- **`CephMonQuorumLost` 寫死 3-mon majority**（`(count(...) or vector(0)) < 2`）。`or vector(0)` 不是裝飾——少了它，全部 mon 掛掉時 `ceph_mon_quorum_status == 1` 過濾成空 vector、`count(空)` 在 PromQL 是**無 sample**（不是純量 0），`< 2` 算不出結果反而**不 fire**，剛好在 quorum 全失守的最壞情況靜默。非 3-mon 的改法見文末邊界。

<Callout type="warning" title="為什麼 CephMonQuorumLost 一定要 or vector(0)（promtool 抓到的真 bug）">
  早期版本寫的是 `count(ceph_mon_quorum_status == 1) < 2`，看起來合理：在 quorum 的 mon 數少於 2 就叫。但 PromQL 的 `count()` 是對「存在的 series」做聚合——當**所有** mon 都 down、`ceph_mon_quorum_status` 全部 = 0 時，`== 1` 過濾後一條都不剩，`count(空 vector)` 回傳的是**無 sample**而非 `0`，於是 `無 sample < 2` 整個運算式沒有結果，alert **不會 fire**。

  也就是這條「quorum 失守生命線」在它最該叫的情況（一台不剩）反而靜默；只有「還剩 1 台」（count=1）時才叫。本頁的開發環境測試計劃用 `promtool test rules` 餵「3 台全 0」就重現了 `got:[]`。

  修法是標準 idiom `(count(...) or vector(0)) < 2`：左邊為空時 `or vector(0)` 退回一條值為 0 的 series，`0 < 2` 成立照樣 page。已驗證 healthy 不 fire、掉 2 台 fire、**掉 3 台也 fire**。對應的驗證見 [Ceph - 在開發環境驗證這套 alert](./prometheus-alert-testing)。

  **一個誠實的 trade-off**：`or vector(0)` 分不出「3 台 mon 都回報 quorum_status=0」與「完全沒有 `ceph_mon_quorum_status` 這個 metric」（例如 mgr 剛重啟、module 還沒匯出），兩者它都會 fire。多數情況這可接受——連 quorum metric 都沒有本來就代表你對 quorum 盲了，值得叫；而 `CephMetricsAbsent` / `CephExporterAllDown` 是「整個 Ceph metrics 管線抓不到」的主訊號。若你想讓這條只在「真的有 mon 回報但不足額」時 fire，用文末 **metadata-anchored 動態版**：它的門檻 `floor(count(ceph_mon_metadata)/2)+1` 在 metadata 缺席時也變空，`0 < 空` 不成立 → 不會被 metric 缺席誤觸（測試計劃頁有寫死版 fire、動態版不 fire 的對照）。
</Callout>

## `ceph-scoped-availability`：帶 label，維護可精準 silence

這組把上一節的洞見落實：用 `ceph_osd_metadata` / `ceph_mon_metadata` 把 down 狀態接上 `hostname` / `ceph_daemon`，alert label 裡就有 host 可以 silence。

```yaml
  - name: ceph-scoped-availability
    rules:
      - record: ceph:osd_up:with_hostname
        expr: |
          ceph_osd_up
            * on (ceph_daemon) group_left(hostname) ceph_osd_metadata

      - record: ceph:osd_host_down:scoped
        expr: |
          count by (hostname) (ceph:osd_up:with_hostname == 0)
          ==
          count by (hostname) (ceph:osd_up:with_hostname)

      - alert: CephOSDHostDownScoped
        expr: ceph:osd_host_down:scoped
        for: 5m
        labels:
          severity: critical
          source: ceph_scoped
        annotations:
          summary: "Ceph OSD host down: {{ $labels.hostname }}"
          runbook: "ceph osd tree; ceph health detail; check host power/network/systemd"

      - alert: CephOSDDaemonDownScoped
        expr: |
          (ceph:osd_up:with_hostname == 0)
          unless on (hostname) ceph:osd_host_down:scoped
        for: 5m
        labels:
          severity: critical
          source: ceph_scoped
        annotations:
          summary: "Ceph OSD down: {{ $labels.ceph_daemon }} on {{ $labels.hostname }}"
          runbook: "ceph osd tree; ceph osd ok-to-stop {{ $labels.ceph_daemon }}; inspect daemon log"

      - alert: CephMonDownScoped
        expr: |
          (
            (1 - ceph_mon_quorum_status)
              * on (ceph_daemon) group_left(hostname) ceph_mon_metadata
          ) == 1
        for: 30s
        labels:
          severity: critical
          source: ceph_scoped
        annotations:
          summary: "Ceph MON down or out of quorum: {{ $labels.ceph_daemon }} on {{ $labels.hostname }}"
          runbook: "ceph quorum_status; ceph health detail; inspect mon log and host network"
```

query 怎麼讀：

- `ceph:osd_up:with_hostname` 是 recording rule，把 `ceph_osd_up` 的 0/1 狀態接上 `ceph_osd_metadata` 的 `hostname`。
- `ceph:osd_host_down:scoped` 用「同一台 host 上 down 的 OSD 數」等於「該 host 全部 OSD 數」判斷整台 OSD host down。
- `CephOSDDaemonDownScoped` 的 `unless on (hostname) ceph:osd_host_down:scoped`：第一句找單顆 OSD down，第二句排除已經由 host alert 接手的 hostname。效果是 **只有 `osd.5` down 時 page `CephOSDDaemonDownScoped{ceph_daemon="osd.5"}`；整台 `osd-host-a` 掛掉時只 page `CephOSDHostDownScoped{hostname="osd-host-a"}`，不會每顆 OSD 各叫一次。**

預設 `CephHealthError` 則直接保留當 page 總開關——它是 cluster 已進 `HEALTH_ERR` 的訊號，沒有維護目標 label，但語意夠重，不是「哪台 node 維護中」的雜訊：

File: `ceph/monitoring/ceph-mixin/prometheus_alerts.yml` (line 4)

```yaml
- alert: "CephHealthError"
  expr: "ceph_health_status == 2"
  for: "5m"
  labels:
    severity: "critical"
    type: "ceph_default"
```

預設 `CephHealthWarning`（`ceph_health_status == 1`）保留但**不 page**——任何 maintenance、rebalance、scrub backlog、OSD/MON down 都會讓它 active，當不了精準訊號，只放 Slack。

## Alertmanager routing：把角色映射到 receiver

因為預設 ceph-mixin rules 仍在，routing 不能寫成「所有 `severity=critical` 都送 Pager」——那會讓預設 `CephMonDownQuorumAtRisk`、`CephOSDDownHigh` 這類 aggregate alert 在維護期間繼續叫。把上面三種角色映射到 receiver：

```yaml
route:
  receiver: slack-ceph
  group_by: ['alertname', 'name', 'hostname', 'ceph_daemon']
  routes:
    - receiver: watchdog-ceph
      matchers:
        - alertname="Watchdog"

    - receiver: pager-ceph
      matchers:
        - severity="critical"
        - source=~"ceph_stability|ceph_scoped|ceph_coverage"

    - receiver: pager-ceph
      matchers:
        - type="ceph_default"
        - alertname="CephHealthError"

    - receiver: slack-ceph
      matchers:
        - source=~"ceph_stability|ceph_scoped|ceph_coverage"

    - receiver: slack-ceph
      matchers:
        - type="ceph_default"
```

效果：Pager 的主訊號只來自**自訂 critical rule + 預設 `CephHealthError`**；warning / info 級自訂 rule 與預設 aggregate rule 仍保留在 Slack / ticket / Dashboard 當 context，跟 upstream 文件一致，但不吵 pager。這裡刻意改成 label 導向，而不是 alertname regex：新增 `ceph-production-coverage` rule 時，只要 severity/source 正確，routing 就不會漏。

## 維護時怎麼 silence

scoped rule 帶 `hostname` / `ceph_daemon`，所以維護可以只 silence 目標那台。三種粒度：

### 維護 MON node

先確認停這台 mon / host 不會立刻打掉可用性。`mon ok-to-stop` 是 Ceph 原生 monitor command；`orch host ok-to-stop` 是 host 級檢查。

File: `ceph/src/mon/MonCommands.h` (line 250)

```cpp
COMMAND("mon ok-to-stop "
    "name=ids,type=CephString,n=N",
    "check whether mon(s) can be safely stopped without reducing immediate "
    "availability",
    "mon", "r")
```

File: `ceph/src/pybind/mgr/orchestrator/module.py` (line 782)

```python
@_cli_write_command('orch host ok-to-stop')
def _host_ok_to_stop(self, hostname: str) -> HandleCommandResult:
    completion = self.host_ok_to_stop(hostname)
```

假設 `mon.a` 跑在 `mon-host-a`：

```bash
ceph orch host ok-to-stop mon-host-a
ceph mon ok-to-stop a
ceph quorum_status

amtool silence add \
  alertname="CephMonDownScoped" \
  hostname="mon-host-a" \
  ceph_daemon="mon.a" \
  --duration=1h \
  -c "planned maintenance mon.a on mon-host-a"

ceph orch host maintenance enter mon-host-a
```

若不想在維護期間一直收到預設 aggregate context，可以短時間 silence 它們：

```bash
amtool silence add \
  'alertname=~"CephHealthWarning|CephMonDown|CephMonDownQuorumAtRisk|CephLowPriorityNotice"' \
  --duration=1h \
  -c "mute aggregate ceph context during mon.a maintenance"
```

**不要 silence `CephMonQuorumLost`。** 如果 3-mon cluster 在維護 `mon.a` 時又掉 `mon.b`，`count(ceph_mon_quorum_status == 1) < 2` 會 page——這是維護期間要保留的生命線。

<Callout type="info" title="維護 mon 會升哪些 health check，上面的 silence 蓋得完整嗎">
  `ceph orch host maintenance enter mon-host-a` 之後實際會升的是 `MON_DOWN`（`HealthMonitor.cc:820`，`HEALTH_WARN`）與 `HOST_IN_MAINTENANCE`（`cephadm/module.py:2126`，`HEALTH_WARN`）。對照三條 pager 訊號：

  - `CephMonDownScoped{hostname="mon-host-a"}` —— 已被上面第一條 silence 蓋掉。它能精準 match 是因為 `mon_quorum_status` / `mon_metadata` 是**對 monmap 裡每一個 mon** 匯出的（`prometheus/module.py:1021` iterate `monmap['mons']`），停掉的 `mon.a` 仍在 monmap，所以 `mon_quorum_status{ceph_daemon="mon.a"}=0` 與帶 `hostname` 的 metadata 都還在、silence matcher 接得到。
  - `CephClientRisk{name="MON_DOWN"}` —— **不會叫**，因為 `MON_DOWN` 在排除清單裡（這就是當初把 `MON_DOWN` 排掉的原因）。
  - `HOST_IN_MAINTENANCE` —— 不 page，它在 `CephLowPriorityNotice`（Slack-only）。

  所以維護單台 mon，**上面那條 scoped silence 就完整了**，不用再多 silence。`CephMonQuorumLost` 維護 1 台時 `count==2` 不觸發，保留它當第二台也掉時的生命線。
</Callout>

### 維護整台 OSD node

先跑 host ok-to-stop（比直接進 maintenance 更適合 SOP，因為會先看到停這台是否讓 PG 風險上升）：

```bash
ceph orch host ok-to-stop osd-host-a

amtool silence add \
  'alertname=~"CephOSDHostDownScoped|CephOSDDaemonDownScoped"' \
  hostname="osd-host-a" \
  --duration=2h \
  -c "planned maintenance osd-host-a"

ceph orch host maintenance enter osd-host-a
```

這只 silence `hostname="osd-host-a"`；維護期間 `osd-host-b` 掛掉，`CephOSDHostDownScoped{hostname="osd-host-b"}` 照樣 page。若 Slack / ticket 也要降噪：

```bash
amtool silence add \
  'alertname=~"CephHealthWarning|CephLowPriorityNotice|CephOSDDown|CephOSDHostDown|CephOSDDownHigh"' \
  --duration=2h \
  -c "mute aggregate ceph context during osd-host-a maintenance"
```

**不要 silence `CephClientBlocked`、`CephClientRisk`、`CephHealthError`。** 如果維護真的讓 client I/O 或更高層 health check 出問題，那就不是噪音。

<Callout type="info" title="維護整台 OSD node 會升哪些 health check，上面的 silence 蓋得完整嗎">
  `ceph orch host maintenance enter osd-host-a` 對 OSD host 會做兩件有 health check 副作用的事：停掉該 host 全部 OSD，並**自動對該 host subtree 設 `noout`**（`cephadm/module.py:2131` 的 `osd set-group noout`，避免維護期間資料搬移）。所以實際升起的是：

  - `OSD_HOST_DOWN` / `OSD_DOWN`（`OSDMap.cc:7316`/`7342`，`HEALTH_WARN`）—— 已被上面 `CephOSDHostDownScoped|CephOSDDaemonDownScoped{hostname="osd-host-a"}` 蓋掉；`CephClientRisk` 排除清單也有這兩個 name。
  - **`OSDMAP_FLAGS`**（`OSDMap.cc:7493`，因為自動設了 `noout`）與 **`PG_DEGRADED`**（副本掉一份）—— 這兩個**不在** scoped silence 裡，但它們被歸進 Slack-only 的 `CephLowPriorityNotice`、且在 `CephClientRisk` 排除清單裡，所以**不會 page**，維護時不用特別 silence。（這就是為什麼排除清單一定要有 `OSDMAP_FLAGS`：`noout` 是 maintenance enter 自動設的，不設它這條一定漏。）

  唯一仍會 page 的是真出事：若維護讓 PG 掉到 `min_size` 以下變 `PG_AVAILABILITY`，`CephClientBlocked` 會叫——那時不該 silence。前面的 `ceph orch host ok-to-stop osd-host-a` 通過就是在確認不會走到這一步。
</Callout>

### 維護單顆 OSD

先 `ok-to-stop`，再只 silence 該 daemon：

```bash
ceph osd ok-to-stop 5

amtool silence add \
  alertname="CephOSDDaemonDownScoped" \
  ceph_daemon="osd.5" \
  --duration=1h \
  -c "replace osd.5"
```

如果這台 host 上只有一顆 OSD，停它會同時符合「整台 OSD host down」，要多加 host silence：

```bash
amtool silence add \
  alertname="CephOSDHostDownScoped" \
  hostname="osd-host-a" \
  --duration=1h \
  -c "single-osd host maintenance"
```

<Callout type="tip" title="換硬碟為什麼上面這樣就夠了（不用再 silence PG_DEGRADED）">
  停一顆 OSD 換硬碟，除了 `OSD_DOWN`，ceph 一定會升 `PG_DEGRADED`（副本掉一份），設了 `noout` 還會升 `OSDMAP_FLAGS`。這兩個**不是**用上面的 `CephOSDDaemonDownScoped` silence 蓋掉的——它們是被歸進 Slack-only 的 `CephLowPriorityNotice`（見前面 rule 設計），本來就不 page，所以維護時不用特別為它們加 silence。

  你只需要 silence `CephOSDDaemonDownScoped{ceph_daemon="osd.5"}` 這一條 pager 訊號。**唯一仍會 page 的情況是真的出事**：如果換硬碟讓某些 PG 掉到 `min_size` 以下變成 `PG_AVAILABILITY`（client I/O 被擋），`CephClientBlocked` 會 page——這時不該 silence，那是你要知道的。先 `ceph osd ok-to-stop 5` 通過就是在確認不會走到這一步。
</Callout>

## 發出 alert 時怎麼處理

每條 page 主訊號的第一步動作：

### `CephClientBlocked`

client I/O 已被 `PG_AVAILABILITY` / `SLOW_OPS` 影響，或容量已到硬擋（`OSD_FULL` / `POOL_FULL`）。**不要 silence。** 目標不是先分類，是先恢復 client 可用性。

```bash
ceph -s
ceph health detail
ceph pg dump_stuck inactive
ceph pg dump_stuck unclean
ceph osd tree
ceph df
```

`PG_AVAILABILITY` → 優先找 inactive / unavailable PG、down OSD、full / backfillfull。`SLOW_OPS` → 先看 slow op 卡在 MON、OSD 還是整體 I/O 壓力，再看 OSD latency、磁碟、網路、recovery / backfill。`OSD_FULL` / `POOL_FULL` → 立刻用 `ceph df` 找出寫滿的 OSD 或 pool，擴充容量或清資料；這是硬擋，不會自己恢復。

### `CephClientRisk`

「沒被低優先或 scoped rule 接走的 health check」。以 `$labels.name` 為入口：

```bash
ceph health detail
ceph -s
```

照 health detail 的 check 名稱往下查。**如果發現某個 check 其實只是維護期間常態噪音，不要直接 silence 整條 `CephClientRisk`**；把那個 health check name 移到排除清單或 `CephLowPriorityNotice`，讓規則語意保持乾淨。

### `CephMonQuorumLost`

MON quorum 低於 3-mon majority。**不要 silence，不要等它自己恢復。**

```bash
ceph quorum_status
ceph health detail
ceph orch ps --daemon_type mon
```

先確認是否還能拿到 quorum。若已失守，優先恢復 MON host 的網路、時間同步、磁碟空間與 mon daemon。**不要同時重啟多台 mon**；先讓一台已知健康的 mon 回 quorum。

### `CephExporterAllDown`

Prometheus 抓不到任何 Ceph exporter target。它本身不是 client I/O failure，但會讓所有 Prometheus alert 變盲區，所以**不要 silence**。

```bash
ceph -s
ceph mgr services
ceph mgr module ls
```

mgr/prometheus module 沒啟用就修 module；active mgr 掛掉就先恢復 mgr。這段期間改用 `ceph -s` / `ceph health detail` 人工確認，別只看 Prometheus。

### `CephExporterTargetDown`

單一 Ceph exporter target 抓不到，進 Slack / ticket。多 mgr scrape 之後，單點 exporter down 不一定等於全盲；先查對應 `instance` 是 planned maintenance、standby mgr、網路，還是 scrape config 問題。若同時 `CephMetricsAbsent` 或 `CephExporterAllDown` firing，就把它當監控生命線事故處理。

### `CephOSDHostDownScoped`

整台 OSD host 的 OSD 都 down。先判斷是 planned maintenance、host power、網路、systemd 還是 cephadm 問題：

```bash
ceph osd tree
ceph orch ps --host osd-host-a
ceph health detail
```

不是計畫內維護就**不要先 cluster-wide silence**；先恢復 host 或確認是否要讓資料恢復路徑接手。

### `CephOSDDaemonDownScoped`

單顆 OSD down，不是整台 host。

```bash
ceph osd tree
ceph osd ok-to-stop <id>
ceph health detail
```

查該 OSD 的 daemon log、device、BlueStore 狀態、是否被 mark out。這條帶 `ceph_daemon` 與 `hostname`，維護或換硬碟時可只 silence 該 OSD。

### `CephMonDownScoped`

某個 mon 不在 quorum。先看 quorum，再看該 host：

```bash
ceph quorum_status
ceph health detail
ceph orch ps --daemon_type mon
```

計畫內維護就確認 silence 是否只 match 目標 `ceph_daemon` / `hostname`；不是的話先恢復 mon host 的網路、時間、磁碟，再看 mon log。

### `CephLowPriorityNotice`

只進 Slack / ticket，不 page。典型是 `HOST_IN_MAINTENANCE`、`OBJECT_MISPLACED`、`PG_SLOW_SNAP_TRIMMING`、`POOL_APP_NOT_ENABLED`。記錄、追蹤、看是否超過預期時間——不該把人叫醒，但也不該完全消失。

## 邊界

### 3-mon hard-code

`CephMonQuorumLost` 用的是 3-mon majority（`or vector(0)` 的原因見前面 callout）：

```promql
(count(ceph_mon_quorum_status == 1) or vector(0)) < 2
```

非 3-mon 的 cluster 改成動態 majority（一樣要保留 `or vector(0)`，否則全失守時 `count(空)` 一樣不 fire）：

```promql
(count(ceph_mon_quorum_status == 1) or vector(0)) < (floor(count(ceph_mon_metadata) / 2) + 1)
```

### 預設 rules 仍然會出現

預設 `CephOSDDown`、`CephOSDHostDown`、`CephMonDown`、`CephMonDownQuorumAtRisk` 仍會 fire——差別只是 routing 不讓它們當 pager 主訊號。它們留在 Slack / ticket / Dashboard，提供跟 upstream 文件一致的 context。

### `CephClientRisk` 不要變成垃圾桶

`CephClientRisk` 的重點是「其餘會危及 client 的 health check」。如果某個 check 長期不是 client 風險，就把它移出這條 rule——不要讓 oncall 被一條過大的 catch-all 訓練到無視 Pager。

### host 由 partial-down 轉 full-down 時的計時

一台 OSD host 可能先掉一顆、幾分鐘後整台才 down（同 host 兩顆碟先後壞）。這時 `ceph:osd_host_down:scoped` 是在「整台變 full-down 那刻」才第一次出現，所以 `CephOSDHostDownScoped` 的 `for: 5m` 是**從整台 down 起算**，不是從第一顆 down 起算；而那段 partial 期間單顆 down 的 `CephOSDDaemonDownScoped` 還沒滿自己的 5m 就被 `unless` 接手，**整段轉換不會雙重 page**。這個邊界由測試計劃用 promtool 實測確認（非推論）。

### Prometheus alert 的共同盲區：metric 缺席

`up{job="ceph"} == 0`（`CephExporterTargetDown`）與所有 `ceph_health_detail` 系列規則都要求**對應的 series 存在**才會 fire。如果某個 metric 從沒被抓過（scrape config 打錯、target 沒註冊），規則靜默、毫無 pager。v2 以 `CephMetricsAbsent` 補 `ceph_health_status` 缺席，並以 `CephExporterAllDown` 捕捉整個 Ceph scrape job 全盲；更多真機發現與 coverage rule 見 [Ceph - 真機故障注入後的 alert 韌性修訂](./prometheus-alert-real-lab-findings)。
