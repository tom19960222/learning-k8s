## 場景：這一頁是 v2——v1 為什麼整個重寫

這頁原本（v1）是一份 KubeVirt 軸的實驗計畫：拿 [參數目錄](./rbd-io-tuning-catalog)裡 🔴 / 🟡 的旋鈕，在 KubeVirt VMI 上一個一個量。動手前照規矩把每個「假設」拿回 pinned source 驗證，結果**計畫本身的三個前提被原始碼推翻**（詳見下一節）。加上一個環境判斷——KubeVirt 只是 control plane，不在 IO data path 上，它對效能的唯一影響是「幫你把 QEMU 旋鈕設成什麼預設值」——所以 v2 改成：**在實際使用中的 Proxmox VE + Ceph cluster 上做，只動自有測試 VM 與測試 image 的參數，完全不動 ceph**。QEMU / krbd 層的旋鈕結論兩邊通用，KubeVirt 特有的部分只剩「control plane 選了哪些預設」，那個用讀原始碼就能回答，不用實驗。

這頁要回答你三個問題：**總共要做哪些實驗**（見「實驗總覽」一張表）、**每個實驗的變因是什麼、預期會有什麼結果**（每個實驗一節，變因唯一、預期標明是機制推論）、然後**結果邊做邊回填**（每節帶一張空的回填表，跑完填 verdict）。完整的研究方法紀錄（hypothesis backlog、evidence 規則）在 repo 的 `experiments/rbd-io-perf/HYPOTHESES.md`。

## v1 被原始碼推翻的三個前提

這三條全部是先寫了計畫、再對 pinned source 查證時抓到的。留在這裡當教訓：**「baseline 幻覺」——你以為的最樸素起點，其實系統早就幫你調過了**。

| # | v1 的前提 | 原始碼說的 | 後果 |
|---|---|---|---|
| 1 | 「VMI `cache` 留空 = 不開 O_DIRECT、不觸發 auto 推導」 | KubeVirt 對 block device 後端在 cache 留空時**自動設 `cache=none`**，接著再推 `io=native` | v1 實驗 1 的 baseline 與變體是同一組設定，量不出任何差異 |
| 2 | baseline VMI 一顆 datadisk 就能跑 | 該 VMI 沒有 boot volume，**開不起來**；補上 boot 盤後資料盤是 `/dev/vdb` | v1 所有 fio 指令的 `--filename=/dev/vda` 都指錯裝置 |
| 3 | 「不設 queues 的 baseline 是單 virtqueue」 | QEMU 的 virtio-blk `num-queues` 預設是 AUTO，realize 時**自動解析成 vCPU 數**；libvirt 在 queues 未設時不輸出該屬性，預設直通 | v1 實驗 3（blockMultiQueue）的 baseline 與變體很可能又是同一組設定 |

前提 1 的證據——cache 留空時 converter 的真實行為：

```go
// File: kubevirt/pkg/virt-launcher/virtwrap/converter/converter.go
	// if user did not set a cache mode and fs supports direct I/O then set cache = 'none'
	// else set cache = 'writethrough
	if mode == "" && supportDirectIO {
		mode = v1.CacheNone
	} else if mode == "" && !supportDirectIO {
		mode = v1.CacheWriteThrough
	}
```

（`SetDriverCacheMode` 約 L371-429；`manager.go` 約 L838-842 對每顆 disk 無條件先跑它、再跑 `SetOptimalIOMode`——所以留空 ⇒ `cache=none` ⇒ `io=native` 一路自動。）

前提 3 的證據——QEMU 端 num-queues 預設值與 AUTO 的解析：

```c
// File: qemu/hw/virtio/virtio-blk-pci.c
    if (conf->num_queues == VIRTIO_BLK_AUTO_NUM_QUEUES) {
        conf->num_queues = virtio_pci_optimal_num_queues(0);
    }
```

（屬性預設值在 `qemu/hw/block/virtio-blk.c` 約 L1997-1998 的 `DEFINE_PROP_UINT16("num-queues", ..., VIRTIO_BLK_AUTO_NUM_QUEUES)`；`virtio_pci_optimal_num_queues` 回傳 vCPU 數。libvirt 端 `qemu_command.c` 約 L1692 `"p:num-queues", disk->queues` 的 `p:` 修飾詞在 queues 未設（=0）時不輸出屬性。）

<Callout type="warning" title="教訓：先驗 baseline，再談變體">
  三條錯誤同一個型：**變體實驗的 baseline 沒有先對原始碼驗證過**。v2 的每個實驗因此都把「生效驗證」放在量測之前——先證明 baseline 與變體真的是兩組不同設定、而且改動真的到達 QEMU / kernel，才開始跑 fio。
</Callout>

## 新環境與操作邊界

- **環境**：一座實際在用的 Proxmox VE cluster（內建 Ceph）。版本（pve-qemu / kernel / ceph）、OSD 拓樸（是否 hyperconverged）、媒體、網路、pool 用量——全部由 E-00 pre-flight 盤點後回填到文末的 environment snapshot，**這頁所有數字都以那個 snapshot 為準**。
- **只動自有資源**：自建測試 VM、自建測試 image（含建檔參數）、自有 image 的 `rbd image-meta conf_*` 覆寫、自己手動 `rbd map`、專屬測試用 storage id（`krbd=1`，測完刪）。
- **三條底線**：不改任何 ceph cluster / daemon 設定、不重啟 ceph、不讓其他 VM down（實驗負載造成的「變慢」可接受）。因此 v1 裡的 `osd_op_num_shards` 與 mClock profile 變體實驗**全部剪除**——mClock 只在 pre-flight read-only 記錄，供解讀天花板用。
- **courtesy guardrails**：每輪監控，出現**相對基線新增**的 `HEALTH_ERR` / 持續 slow ops、pool 用量逼近 nearfull、（krbd 時）節點可用記憶體低於門檻——立即中止並清理。生產叢集的 `HEALTH_WARN` 可能常態存在，所以判斷用差分不用絕對值。

## 兩條 datapath 軸：krbd vs librbd

[datapath 頁](./rbd-io-datapath)的主軸是 krbd（host kernel map 出 `/dev/rbdX`）。但 **PVE 掛 RBD 的預設是 librbd**——storage 設定 `krbd=0` 時，QEMU 用內建的 rbd driver（`qemu/block/rbd.c`）在 user space 直連 OSD，**沒有 `/dev/rbdX`、沒有 host iostat 邊界**。兩軸差異：

| | krbd 軸（`krbd=1`） | librbd 軸（PVE 預設） |
|---|---|---|
| datapath | QEMU → `/dev/rbdX` → host kernel krbd → libceph → OSD | QEMU 內 librbd → OSD（不經 host block layer） |
| host 端觀測 | `iostat -x /dev/rbdX`（datapath §⑤ 邊界） | 無 block device；用 QMP `query-blockstats` + `rbd perf image iostat` |
| cache 旋鈕控制的東西 | host page cache（O_DIRECT 與否） | **librbd 的 `rbd_cache`** |
| 專屬旋鈕 | map options（queue_depth / alloc_size / rxbounce …） | `rbd_cache` 等 librbd 選項（可用 per-image `conf_*` 覆寫） |

librbd 軸的 cache 機制有原始碼直接證據——QEMU 依 cache mode 直接設 `rbd_cache`：

```c
// File: qemu/block/rbd.c
    if (flags & BDRV_O_NOCACHE) {
        rados_conf_set(*cluster, "rbd_cache", "false");
    } else {
        rados_conf_set(*cluster, "rbd_cache", "true");
    }
```

（約 L955-963；呼叫端約 L1121 以 `!(flags & BDRV_O_NOCACHE)` 傳入。所以 `cache=none` 在 librbd 軸關掉的是 librbd cache，跟 krbd 軸繞過 host page cache 是**兩個不同機制**——實驗 E-05 的預期因此分軸寫。）

<Callout type="info" title="krbd 能不能用，本身是第一個要驗的事">
  這座 PVE 沒調過 krbd。E-01 會用三關驗證：自建 throwaway image 手動 `rbd map` 讀寫、image features 相容性、`krbd=1` 專屬 storage id 掛盤到測試 VM。全過 → 雙軸都測（krbd vs librbd 對打是本計畫的頭牌實驗）；不過 → librbd 單軸，krbd 相關實驗（E-10 / E-12 / E-14）降級成 host 層對照或剪除，結果如實記錄在 snapshot。
</Callout>

## 實驗方法論 v2

六條規則，貫穿每一個實驗：

1. **一個實驗只動一個變因**，其餘全部固定；E-07 之後的實驗若疊在某變體上，會明寫比較基準是誰。
2. **prediction 先行**：每個實驗的「預期」在跑之前寫死（就是本頁各節的預期欄），跑完由機器比對，verdict 三態——`confirmed`（差異方向與預期一致且超出噪音帶）、`violated`（方向相反或該有差異卻沒有）、`indistinguishable`（差異落在噪音帶內，這個環境量不出來）。**violated 是有價值的結果，不准事後改口說「其實這是預期的」**。
3. **n≥3 + 噪音帶**：baseline 先連跑多輪算出變異係數（CoV），差異要超過噪音帶才能宣稱方向。生產叢集噪音大，必要時升到 n≥5。
4. **A/B 交錯執行**：baseline 輪與變體輪交錯跑（A/B/A/B…），不做「先全 A 再全 B」——生產叢集的背景負載隨時間變，區塊式執行會把別人的尖峰時段誤判成旋鈕效果。每輪同時記下當時的叢集併發負載（`ceph -s` 的 client io）。
5. **生效驗證是量測的前置條件**：改完參數先斷言 `qm config`、QEMU 命令列（`/proc/<pid>/cmdline`）、krbd 軸再加 `rbd showmapped` / sysfs 讀值，全對才開始跑 fio。
6. **量測有效性守則**：測試盤建立後先全盤 pre-fill（thin image 沒寫過的 extent 讀起來是假數字）；fio 帶 `--ramp_time` 丟暖身段；guest / host 觀測窗與 fio 量測窗對齊；一輪中若 ceph 出現 recovery / backfill 該輪作廢延後重跑，scrub 與他人負載則記錄下來靠交錯與重複吸收。

## 共用 fio 矩陣與要收的 metric

矩陣 8 個 pattern，每個實驗都跑同一組（特例會在該實驗註明）：

```bash
# File: fio 矩陣 v2 — guest 內對測試資料盤（virtio-blk，/dev/vdb）跑
# 前置：該盤已 pre-fill（fio --rw=write --bs=1M 寫滿一遍）
# {randread, randwrite} × {qd 1, 8, 32}：
fio --name=rr-4k-qd$QD --filename=/dev/vdb --direct=1 --rw=randread \
    --bs=4k --iodepth=$QD --numjobs=1 --ioengine=libaio \
    --ramp_time=15 --runtime=60 --time_based --randseed=8675309 \
    --group_reporting --output-format=json
# （randwrite 同參數，--rw=randwrite）

# seqread / seqwrite 1M（readahead / striping 的主場）：
fio --name=seq{read,write}-1m --filename=/dev/vdb --direct=1 \
    --rw={read,write} --bs=1M --iodepth=16 --numjobs=1 --ioengine=libaio \
    --ramp_time=15 --runtime=60 --time_based --group_reporting --output-format=json
```

v1 的矩陣沒有 seqread——整條 readahead 讀路徑等於沒測過，v2 補上（E-13 的主場）。

每輪收的 metric：

- **fio JSON**：IOPS、`clat` p50 / p99、BW。
- **guest `/dev/vdb` iostat**：`await` / `aqu-sz`——guest 端邊界。
- **host 端邊界**：krbd 軸收 `iostat -x /dev/rbdX`；librbd 軸沒有 host block device，改收 QMP `query-blockstats`（QEMU 自己量的 per-disk 延遲）與 `rbd perf image iostat`（ceph mgr 端 read-only）。
- **QEMU thread CPU**：`/proc/<pid>/task/<tid>/schedstat` 前後差分（不用 top 瞬時值），另記 host loadavg。
- **dmesg 增量**（krbd 軸）：盯 `bad crc` / socket closed——rxbounce 問題的訊號（E-14）。
- **`ceph -s` 前後**：recovery / scrub / client io，判 tainted 與記錄併發負載。

## 各實驗協定

### E-01：krbd 可行性三關

1. 自建 throwaway image → 手動 `rbd map` → 讀寫 → `rbd unmap`。
2. `rbd info` 檢查 PVE 建檔的預設 features 與 kernel rbd 相容性（不相容時 map 會報錯或要求 disable feature——只對自有測試 image 處理，不碰別人的）。
3. storage.cfg 加專屬測試 storage id（同 pool、`krbd=1`）→ 掛一顆盤到測試 VM → 確認 host 出現 `/dev/rbdX` 且 VM 可讀寫。

### E-02：host 層天花板（完整協定）

- **變因**：host 直接對手動 map 的 `/dev/rbdX` 跑矩陣，fio `ioengine=libaio` vs `io_uring` 兩變體。VM 完全不參與。
- **怎麼設**：PVE node 上 `rbd map` 自建測試 image（pre-fill 後）→ 對 `/dev/rbdN` 跑矩陣兩輪（兩種 engine）→ `rbd unmap`。librbd 天花板：若 node 的 fio 帶 rbd engine（`fio --enghelp`）則加跑 `ioengine=rbd`，沒有就略過並記錄。
- **預期（機制推論）**：這組數字是「不含任何虛擬化層」的 ceph 側天花板——後續所有 VM 實驗的分母。libaio vs io_uring 在高 QD 差異小（都是非同步批次提交），qd1 io_uring 系統呼叫開銷較低、略優。若這裡的 4k randwrite qd1 延遲已經數 ms 級，虛擬化層旋鈕的效果空間就很小——先知道這件事再解讀後面所有實驗。
- **生效驗證**：`rbd showmapped` 對到正確 image；fio JSON 的 `ioengine` 欄位。
- **回填**：

|---|---|---|---|---|---|

### E-03：VM baseline（完整協定）

- **變因**：無——PVE 預設全收（依 PVE 文件預設 `aio=io_uring`、cache=none；**以 pre-flight 讀到的實際值為準**），唯一固定選擇：測試資料盤走 `virtio` bus（virtio-blk，guest 內 `/dev/vdb`），對齊 datapath 頁的軸；boot 盤與資料盤分開，fio 只打資料盤。
- **怎麼設**：cloud image + PVE cloud-init 建測試 VM（4 vCPU / 4GiB，fio 定版）；資料盤 16G、pre-fill。連跑 n≥5 輪矩陣，算每個 pattern 的 CoV → **噪音帶**。

### E-04：krbd vs librbd（頭牌，完整協定）

- **變因**：同規格測試 image、同一台 VM，唯一差別是掛法——librbd（`krbd=0` storage）vs krbd（`krbd=1` 專屬 storage id）。
- **怎麼設**：兩個 storage id 指同一個 pool；資料盤各建一顆（同大小、同 features、都 pre-fill）；A/B 交錯跑矩陣。
- **預期（機制推論）**：兩軸 datapath 結構不同（librbd 全在 QEMU process 內 user-space 直連 OSD；krbd 多過一層 host block layer + kernel messenger），延遲與 CPU 分佈**應可測出差異，但方向誠實不預設**——librbd 少一層 block queue 但多 user-space messenger 開銷，krbd 反之；哪邊贏由環境決定。這是 KubeVirt + ceph-csi 做不到的同機對照，也是「PVE 該用哪個」的直接答案。
- **生效驗證**：librbd 變體 QEMU cmdline 應見 rbd 協定的 blockdev、host 無 `/dev/rbdX`；krbd 變體 `rbd showmapped` 有對應 device。
- **回填**：

| pattern | librbd IOPS / p99 | krbd IOPS / p99 | Δ vs 噪音帶 | QEMU CPU（兩軸） | verdict |
|---|---|---|---|---|---|
| rr 4k qd1 | | | | | |
| rr 4k qd32 | | | | | |
| rw 4k qd32 | | | | | |
| seqwrite 1M | | | | | |

### E-05：cache 模式（完整協定）

- **變因**：`qm set <vmid> --virtio1 ...,cache=X`——none / writethrough / writeback 三方（v1 只測 none 對「留空」，而那兩個是同一組設定；directsync 列選配）。
- **預期（機制推論，分軸）**：
  - **krbd 軸**：cache 控制 QEMU 開 `/dev/rbdX` 的方式。`writethrough` 下 guest 看到 write cache off、每個 write 等同帶 flush——4k randwrite 預期三者中最慢；`writeback` 走 host page cache、小寫可被吸收合併，但 fsync 密集時優勢消失；`none`（O_DIRECT）繞過 page cache、行為最穩定。
  - **librbd 軸**：cache 控制的是 `rbd_cache`（見上方 `block/rbd.c` 錨點），host page cache 根本不在路徑上。`writeback`（rbd_cache=true）對小寫的吸收效果、與 `none`（rbd_cache=false）的差距，機制上預期比 krbd 軸的 page cache 效果更直接。
  - v1 那條「buffered + fsync=1 拉開差異」的 job 已撤——機制上可疑（fsync=1 逼每寫落盤，兩種 cache 反而趨同）；v2 改成兩條對照：`direct=0` 無 fsync（看吸收效果）與 `direct=0 --fsync=1`（看 durability 成本），預期前者才拉得開。
- **生效驗證**：QEMU cmdline 的 cache 相關旗標；librbd 軸可另以 `rbd image-meta` 與 QMP 確認（優先序見 E-15）。
- **回填**（每軸一份）：

| 軸 | 變體 | rw 4k qd1 p99 | rw 4k qd32 IOPS | buffered 無 fsync IOPS | buffered fsync=1 p99 | verdict |
|---|---|---|---|---|---|---|
| krbd | none | | | | | |
| krbd | writethrough | | | | | |
| krbd | writeback | | | | | |
| librbd | none | | | | | |
| librbd | writethrough | | | | | |
| librbd | writeback | | | | | |

### E-06：aio 模式（完整協定）

- **變因**：`qm set <vmid> --virtio1 ...,aio=X`——io_uring（PVE 預設）/ native / threads。PVE 直接暴露這顆旋鈕，v1 需要 hook sidecar 的問題不存在了。
- **預期（機制推論）**：三種 host 提交模型（datapath §④ `raw_co_prw` 的三個分支）。高 QD 下 `threads`（thread pool 同步 pread/pwrite）預期落後 `native` / `io_uring`（非同步批次）；qd1 三者接近（單流延遲由 ceph 側主導）。與 E-02 的 host 層對照可以互相印證——如果 host 層 libaio vs io_uring 沒差，VM 層也不該有差。
- **生效驗證**：QEMU cmdline `aio=` 值；`native` 需要 O_DIRECT（cache=none），維持 baseline 的 cache 設定。
- **回填**：

| 變體 | rr 4k qd32 IOPS / p99 | rw 4k qd32 IOPS / p99 | qd1 p99 | QEMU CPU | verdict |
|---|---|---|---|---|---|
| io_uring（預設） | | | | | |
| native | | | | | |
| threads | | | | | |

### E-07～E-15：精簡協定

每條格式：**變因 → 預期（機制推論）→ 生效驗證 → 回填欄**。

**E-07 IOThread**：`qm set` disk `iothread=1` vs 0。→ 把這顆盤的 IO 處理搬離 QEMU main loop；單盤單流預期差異有限、高 QD 降 p99（main loop 不再兼職）。→ QEMU cmdline 出現 iothread 物件、cmdline 中 disk 綁定該 iothread。→ 回填：qd32 rr/rw IOPS、p99、main loop vs IOThread CPU 分佈、verdict：＿＿

**E-08 queues 反向對照**：`qm set --args` 強制 `num-queues=1` vs 預設 auto（=vCPU 數）。→ 前提 3 的實測面：多 job 高 QD 下強制單 queue 預期明顯變差（所有 vCPU 擠一條 vring）；單流 qd1 預期沒差。特例：矩陣外加 `--numjobs=4` 的 rr/rw。→ guest `ls /sys/block/vdb/mq/` 目錄數 1 vs vCPU 數。→ 回填：numjobs=4 rr IOPS（1q vs auto）、guest mq 數、verdict：＿＿

**E-09 image layout**：`rbd create --object-size {4M,16M}`、`--stripe-unit 64K --stripe-count 16` 三顆 image 各掛各測（建檔期參數不能改既有 image，逐顆建、測完刪）。→ striping 預期幫 seqwrite/seqread 1M（一個大 IO 打散到多 OSD）、4k random 幫助有限；object-size 4M vs 16M 方向不預設。→ `rbd info` 的 order / stripe_unit / stripe_count。→ 回填：seq 1M BW 三變體、rr 4k qd32、verdict：＿＿

**E-10 queue_depth**：手動 `rbd map -o queue_depth={64,128,256}` + 以 raw path 掛給 VM。→ krbd 的 blk-mq tag 深度；但 pinned kernel 顯示 `nr_hw_queues = num_present_cpus()`（`rbd.c` 約 L4962），queue_depth 是 per-hctx 深度——實際 in-flight 上限與 v1 敘述不同，**精確預測等 T1 重推完才寫**；觀測聚焦高 QD 下 host `aqu-sz` 上得去否。→ `/sys/block/rbdN/queue/nr_requests` 讀值 + 同時記 scheduler（讀值語意受 scheduler 影響）。→ 回填：qd32 rw IOPS / host aqu-sz（64/128/256）、verdict：＿＿

**E-11 guest scheduler**：guest `echo none > /sys/block/vdb/queue/scheduler` vs 預設。→ RBD 底層已排程，guest 再排一次通常多餘；預期 none 降 guest CPU 與 p99，幅度情境性。→ `cat .../scheduler` 中括號在 `[none]`。→ 回填：rw 4k qd32 p99、guest CPU、verdict：＿＿

**E-12 alloc_size**：手動 map options `alloc_size`（power of 2、≤ object size）。→ 只影響 io_min/io_opt 對齊與 discard 粒度；一般 read/write **預期 indistinguishable**（這是少數預測「沒差」的實驗——若量出顯著差異反而是 violated，值得追）；discard 場景（guest `fstrim`）才預期有感。→ `/sys/block/rbdN/queue/minimum_io_size`。→ 回填：rr/rw 4k qd32、fstrim 時長、verdict：＿＿

**E-13 readahead**：guest 與 host（krbd 軸）`read_ahead_kb` 各掃低/預設/高三檔。→ seqread 1M 吞吐預期對 readahead 敏感（預讀命中），random 不受影響；guest 與 host 兩層 readahead 的疊加行為是開放問題。→ sysfs 讀值。→ 回填：seqread BW × 檔位矩陣、verdict：＿＿

**E-14 rxbounce**：手動 map options `rxbounce` on/off（kernel 6.8 支援已驗：`net/ceph/ceph_common.c` 約 L299）。→ guest 在 read 飛行中改寫 buffer 會讓 krbd 收包 CRC 驗證失敗、messenger 連線重建、p99 尖峰；**條件預測**——若全程 dmesg 見 `bad crc`：rxbounce=on 消除尖峰（confirmed）；若從未出現：兩者 indistinguishable 且「此 workload 不觸發」本身就是結論。→ dmesg 增量 + map options 回讀。→ 回填：bad crc 出現？＿＿；rr 4k qd32 p99（on/off）、verdict：＿＿

**E-15 librbd cache 覆寫**：自有 image `rbd image-meta set conf_rbd_cache false/true`。→ per-image 覆寫與 QEMU `rados_conf_set` 的優先序**先 T1/T2 釐清再寫預測**（backlog H-028 Notes）——這一條的價值是確立 librbd 軸上「per-image 調 cache 而不動 VM 設定」是否可行。→ QMP + 行為對照（小寫吸收有無）。→ 回填：優先序結論：＿＿；verdict：＿＿

## 邊界：什麼情況這些實驗白做

v1 的這節大部分仍然成立，v2 補三條：

- **E-02 的天花板先看**：如果 host 層 4k randwrite qd1 延遲已經數 ms，虛擬化層旋鈕（E-05～E-08）大概全部 indistinguishable——這不是實驗失敗，是這個環境的真實答案（瓶頸在 ceph 側，調 VM 參數沒用）。
- **生產叢集的別人負載**：交錯執行 + 併發負載記錄能吸收大部分，但若某輪 `ceph -s` 顯示 recovery/backfill，該輪一定作廢重跑——recovery 對延遲的干擾遠超過任何旋鈕效果。
- **mClock 天花板**：ceph v19 預設 mclock_scheduler，QoS profile 可能封頂 client IOPS。我們不動它（read-only 記錄），但解讀高 QD 實驗時要拿它對照——天花板可能是 mClock 不是你調的旋鈕。
