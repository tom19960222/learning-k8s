## 場景

CSI 比 CRI/CNI 更容易被誤解，因為它橫跨三個世界：

1. Kubernetes API：`PersistentVolume`、`PersistentVolumeClaim`、`StorageClass`、`CSIDriver`、`CSINode`、`VolumeAttachment`。
2. CSI gRPC contract：`Identity`、`Controller`、`Node` 三組 service。
3. Linux storage/mount 現實：block device、filesystem、NFS、bind mount、mount namespace、mount propagation。

文章的主軸是「Kubernetes 不提供真正的儲存系統」。Kubernetes 只把 Pod 對 storage 的需求轉成 API object 與 CSI RPC，真正的耐久性、快照、複寫、效能、HA、備份，都由儲存後端與 CSI driver 負責。

## 先建立全局心智模型

```
PVC + StorageClass
  │
  ▼
PV controller 標記 external provisioner
  │
  ▼
external-provisioner -> CSI Controller CreateVolume
  │
  ▼
建立 PV，PV 綁定 PVC
  │
  ▼
Pod scheduled 到某個 node
  │
  ├─ attach/detach controller 建 VolumeAttachment
  │   └─ external-attacher -> CSI ControllerPublishVolume
  │
  └─ kubelet volume manager
      ├─ node-driver-registrar / plugin watcher 找到 CSI socket
      ├─ CSI NodeStageVolume (如果 driver 支援)
      └─ CSI NodePublishVolume -> Pod volume target path
```

這條鏈路裡，Kubernetes API object 是「意圖與狀態」，CSI RPC 是「對儲存 driver 的命令」，Linux mount 是「Pod 最終能看到檔案或 block device 的原因」。

## 用現代 Kubernetes 修正舊文章脈絡

這組文章寫於 2019 年，概念仍然很有用，但幾個地方需要更新：

- 文章範例使用 Kubernetes 1.15 與 `storage.k8s.io/v1beta1`；本專案對照的 Kubernetes checkout 是 `v1.36.0`，`CSIDriver`、`CSINode`、`VolumeAttachment` 都已在 `storage.k8s.io/v1`。
- 文章說 CSI persistent volume 不能直接在 Pod 內引用，對「持久 volume」仍然正確：正常路徑是 PV/PVC。現代 Kubernetes 另有 inline CSI ephemeral volume，但需要 driver 宣告支援 `Ephemeral` lifecycle，語義與 PV/PVC 不同。
- CSI migration 在文章中還是 alpha 脈絡；在本專案 checkout 裡，`pkg/volume/csimigration/plugin_manager.go` 已直接把 AWS EBS、GCE PD、Azure Disk/File、Cinder、vSphere、Portworx 等 migration 視為 enabled/complete。
- 舊範例的 sidecar image、API version 與 driver manifest 不應直接照抄；學的是架構與資料流，實作時要以你要部署的 driver 官方 manifest 為準。
- `MountPropagation` 不是 CSI 專有功能，而是 Kubernetes/CRI/Linux mount namespace 的能力；CSI Node Pod 常需要它，是因為 driver 在 container 裡做 mount，結果必須傳回 host。
