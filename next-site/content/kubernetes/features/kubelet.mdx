## 場景

scheduler 寫好 `spec.nodeName=worker-02`，apiserver 通知 worker-02 上的 kubelet。幾秒後 Pod Running。worker node 上發生什麼？kubelet 怎麼跟 containerd 對話？為什麼每個 Pod 會多一個叫 `pause` 的 container？

## kubelet 是個 daemon

kubelet 由 systemd 啟動 (不是 Pod)，因為它是「啟動其他 Pod 的人」。進入點：

```go
// File: kubernetes/cmd/kubelet/kubelet.go
package main

import "k8s.io/kubernetes/cmd/kubelet/app"

func main() {
    command := app.NewKubeletCommand()
    code := cli.Run(command)
    os.Exit(code)
}
```

啟動後 kubelet 同時做幾件事：

1. **Watch 自己 node 上要跑的 Pod** (從 apiserver 加 static Pod 加 manifest URL)
2. **Watch 自己 node 上實際跑的 Pod 狀態** (透過 CRI gRPC 跟 container runtime 對話)
3. **持續 reconcile**：把實際狀態推向期望狀態 (建 sandbox、起 container、清理死掉的)
4. **回報 status**：每 10 秒把 node 與 Pod 狀態 PATCH 回 apiserver

## syncPod 主迴圈

對每個應該存在的 Pod，kubelet 跑：

```go
// File: kubernetes/pkg/kubelet/kubelet.go
// (line 2019)
func (kl *Kubelet) SyncPod(ctx context.Context,
    updateType kubetypes.SyncPodType,
    pod, mirrorPod *v1.Pod,
    podStatus *kubecontainer.PodStatus,
) (isTerminal bool, postSync func(), err error) {
    ...
}
```

這是 high-level 入口，做的事包括：

- 檢查 Pod 是否該被 kill (如果是 Terminated/Failed)
- 計算 Pod 的 cgroup、QoS class
- 建好 Pod 需要的 mount path、emptyDir volume
- 呼叫 container runtime manager 的 SyncPod 處理 sandbox 與 container

container runtime manager 真正跟 CRI 對話：

```go
// File: kubernetes/pkg/kubelet/kuberuntime/kuberuntime_manager.go
// (line 1450)
func (m *kubeGenericRuntimeManager) SyncPod(ctx context.Context,
    pod *v1.Pod,
    podStatus *kubecontainer.PodStatus,
    pullSecrets []v1.Secret,
    backOff *flowcontrol.Backoff,
    restartAllContainers bool,
) (result kubecontainer.PodSyncResult) {
    // 1. Compute sandbox and container changes.
    // 2. Kill the pod sandbox if necessary.
    // 3. Kill any containers that should not be running.
    // 4. Create sandbox if necessary.
    // 5. Create ephemeral containers.
    // 6. Create init containers.
    // 7. Resize running containers (if InPlacePodVerticalScaling).
    // 8. Create normal containers.
    ...
}
```

第 4 步 (建 sandbox) 是關鍵——它建立 Pod 的「殼」並向 CNI 申請 IP，之後 application container 才會掛進這個殼。

## Pod sandbox = pause container 與共用 namespace

Pod 在 OS 層其實是「N 個 container 共用 network/IPC/PID namespace」。問題是：誰持有這些 namespace？如果 application container 是 namespace 擁有者，container 死掉重啟時 namespace 也消失，IP 跟著變——很糟。

解法：每個 Pod 多起一個 `pause` container 當 namespace 持有者。它什麼都不做，只是 sleep + reap zombie。Application container 加入 pause 的 namespace；application 死掉重啟也不影響 namespace。

```go
// File: kubernetes/pkg/kubelet/kuberuntime/kuberuntime_sandbox.go
// (line 38)
func (m *kubeGenericRuntimeManager) createPodSandbox(ctx context.Context,
    pod *v1.Pod, attempt uint32) (string, string, error) {
    podSandboxConfig, err := m.generatePodSandboxConfig(ctx, pod, attempt)
    ...
    podSandBoxID, err := m.runtimeService.RunPodSandbox(ctx,
        podSandboxConfig, runtimeHandler)
    if err != nil {
        ...
    }
    return podSandBoxID, "", nil
}
```

`runtimeService.RunPodSandbox` 是 CRI gRPC call。container runtime (containerd / CRI-O) 收到後：

1. 起一個 `pause` container 並建立 network namespace
2. 透過 CNI plugin 給 namespace 配 IP
3. 回傳 sandbox ID 給 kubelet

之後 kubelet 用同一個 sandbox ID 對 runtime 發 `CreateContainer` 起 application container，container 加入 sandbox 的 namespace。

## CRI gRPC 介面

CRI 是 kubelet 跟 container runtime 之間的契約。它把 runtime 操作分兩組 RPC：

- **RuntimeService**：管 PodSandbox 與 Container (建/列/狀態/移除/exec/log)
- **ImageService**：管 image (pull/list/remove/inspect)

protobuf 定義在 [`https://github.com/kubernetes/cri-api/blob/master/pkg/apis/runtime/v1/api.proto`](https://github.com/kubernetes/cri-api/blob/master/pkg/apis/runtime/v1/api.proto)。

關鍵 RPC：

```protobuf
// File: cri-api/pkg/apis/runtime/v1/api.proto (摘要)
service RuntimeService {
    rpc RunPodSandbox(RunPodSandboxRequest) returns (RunPodSandboxResponse);
    rpc StopPodSandbox(StopPodSandboxRequest) returns (StopPodSandboxResponse);
    rpc CreateContainer(CreateContainerRequest) returns (CreateContainerResponse);
    rpc StartContainer(StartContainerRequest) returns (StartContainerResponse);
    rpc StopContainer(StopContainerRequest) returns (StopContainerResponse);
    rpc ListContainers(ListContainersRequest) returns (ListContainersResponse);
    ...
}
```

containerd / CRI-O 都實作這份介面 (走本機 unix socket，預設 `/run/containerd/containerd.sock`)，所以 kubelet 不在乎背後是誰。`crictl` 命令也是用同一份 protocol，所以你 ssh 到 worker node `crictl ps` 看到的東西就是 kubelet 看到的。

CRI client 程式碼在 v1.36 已經獨立成 staging repo：

```bash
# kubelet 的 CRI client wrapper 在這
kubernetes/staging/src/k8s.io/cri-client/
```

之前在 `kubernetes/pkg/kubelet/cri/`，1.36 之後用 staging 版本。

## CNI 呼叫時機

CNI 不在 CRI 介面內——是 **runtime 自己負責**呼叫 CNI。flow：

1. kubelet `RunPodSandbox` → containerd
2. containerd 建好 network namespace → 在 namespace 內呼叫 CNI plugin (例如 cilium)
3. CNI plugin 給 namespace 配 IP、寫 routing rule、回 result
4. containerd 把 IP 寫進 sandbox status、回給 kubelet
5. kubelet 把 IP 報回 apiserver

這就是為什麼 multus (CNI meta-plugin) 不需要改 kubelet——它只是個 CNI plugin，containerd 呼叫它，它再 delegate 給其他 CNI。

## 整段：worker node 收到新 Pod 之後

```
apiserver watch event ──> kubelet 看到自己 node 多 Pod
              │
              ▼
     SyncPod (kubelet.go:2019)
              │
              ▼
     kuberuntime.SyncPod (kuberuntime_manager.go:1450)
              │
              ├─> createPodSandbox
              │     └─> CRI: RunPodSandbox
              │            └─> containerd 起 pause container
              │                  └─> CNI 配 IP, write routes
              │
              ├─> ImageService.PullImage (CRI)
              │
              ├─> CreateContainer (CRI) — application container
              │     └─> 加入 sandbox 的 network namespace
              │
              └─> StartContainer (CRI)

      Pod Running, kubelet 把 status 寫回 apiserver
```
