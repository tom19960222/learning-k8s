## 場景

這組文章不是單純教你「怎麼裝 Kubernetes」，而是在回答一個更底層的問題：Kubernetes 為什麼能同時接上不同 container runtime、CNI、CSI、Device Plugin，甚至把 VM workload 納入同一套調度模型？

答案是：Kubernetes 把會快速變化、跟硬體或 OS 實作高度相關的部分抽成介面。核心控制平面只管理 API object、期望狀態與調度；真正把 Pod 變成可執行 workload 的細節，交給 kubelet 背後的 CRI runtime、OCI runtime、CNI、CSI 與其他 plugin。

## 這 9 篇的共同主軸

**第一層：Kubernetes core 不直接處理所有底層差異。**  
API server、scheduler、controller-manager 管的是 object 與 reconcile，不應該知道每一種 runtime、CNI 或儲存陣列的細節。

**第二層：kubelet 是 node 端整合點。**  
kubelet 收到 Pod 後，透過 CRI 建 Pod sandbox / container，runtime 再透過 OCI runtime、CNI plugin、filesystem snapshotter 等機制完成真正執行。

**第三層：OCI 是 runtime 共同語言。**  
containerd、CRI-O、Docker stack 最後都會把 container 轉成 OCI runtime bundle，再交給 runc、crun、Kata、gVisor 這類 OCI runtime 執行。

**第四層：runtime 可以因需求替換。**  
一般 workload 用 runc 類 runtime 取得密度與效能；高隔離需求可用 gVisor 或 Kata；VM workload 則由 KubeVirt 這類 CRD/operator 方案接手，而不是只靠舊式 CRI proxy。

## 本專案採用的分析方式

這組文章的角色是提供「問題意識與知識地圖」，不是最終答案。本專案會把它們轉成 source-first 分析：

1. 先從使用者看到的 Pod / RuntimeClass / image pull 症狀切入。
2. 再進 Kubernetes 原始碼找真正的邊界：API type、kubelet manager、CRI client、generated CRI gRPC stub。
3. 最後才回到 runtime 實作，例如 containerd / CRI-O / Kata / gVisor 的差異。

新增頁面的 source anchor 如下：

| 主題 | 先讀哪裡 | 為什麼 |
|---|---|---|
| CRI API | `kubernetes/staging/src/k8s.io/cri-api/pkg/apis/runtime/v1/api.proto` | 這是 kubelet 與 runtime 的正式 gRPC contract。 |
| Pod -> sandbox | `kubernetes/pkg/kubelet/kuberuntime/kuberuntime_manager.go` 的 `SyncPod()` | 這是 kubelet 把 desired Pod 轉成 sandbox/container action 的主入口。 |
| RuntimeClass -> runtime handler | `kubernetes/pkg/kubelet/kuberuntime/kuberuntime_sandbox.go` 的 `createPodSandbox()` | 這裡會 lookup `runtimeClassName`，再把 handler 傳進 `RunPodSandbox()`。 |
| Container lifecycle | `kubernetes/pkg/kubelet/kuberuntime/kuberuntime_container.go` 的 `startContainer()` | 這裡明確分成 pull image、create container、start container。 |
| RuntimeClass API | `kubernetes/staging/src/k8s.io/api/node/v1/types.go` 的 `RuntimeClass` | 這裡定義 `handler`、`overhead`、`scheduling`，不是文章想像出來的概念。 |

## 用現代 Kubernetes 修正舊文章脈絡

這組文章寫於 2019 年，核心概念仍然有效，但有幾個版本差異要更新：

- `dockershim` 已在 Kubernetes 1.24 從 kubelet 移除；現在 kubelet 需要 CRI-compatible runtime，例如 containerd 或 CRI-O。
- Kubernetes 1.26 之後要求 runtime 支援 CRI `v1` API；不再把舊版 CRI negotiation 當作可長期依賴的路徑。
- `RuntimeClass` 已是 stable；選 gVisor、Kata 等 alternate runtime 時，現代做法通常是在 node runtime 設定 handler，再讓 Pod 指定 `runtimeClassName`。
- VM workload 的主流學習路徑應該轉向 KubeVirt：VM 是 CRD 與 virt-launcher Pod，不只是把 kubelet CRI request proxy 到 VM backend。
