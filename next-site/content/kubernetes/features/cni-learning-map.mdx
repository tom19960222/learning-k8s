## 場景

你有網路背景，所以 Kubernetes network 最容易讓人覺得「魔法」的地方不是 IP、route、ARP、VXLAN 本身，而是這些熟悉的東西突然被 kubelet、container runtime、CNI plugin、controller-manager、DaemonSet、Node annotation 串在一起。

這組 CNI 文章剛好可以拆成兩條線：

1. Kubernetes 把「Pod 要有 network namespace 與 IP」抽成 CNI contract。
2. 真正的 datapath 仍然是 Linux network stack：veth、bridge、route、iptables、ARP、FDB、VXLAN、netlink。

本專案會把文章內容轉成 source-first 的分析：先找 Kubernetes source 裡的責任邊界，再把邊界之外的 CNI plugin / flannel / Linux datapath 補上。

## 本專案採用的分析方式

CNI 不能只從一個 repo 讀完，因為責任被刻意拆散：

| 層級 | Source anchor | 讀它是為了回答什麼 |
|---|---|---|
| Kubernetes API | `kubernetes/staging/src/k8s.io/api/core/v1/types.go` 的 `NodeSpec.PodCIDR` / `PodCIDRs` | Node 上的 Pod subnet 是 Kubernetes API object 的哪個欄位？ |
| kubelet / CRI | `kubernetes/staging/src/k8s.io/cri-api/pkg/apis/runtime/v1/api.proto` 的 `RunPodSandbox`、`NamespaceOption`、`NetworkReady` | kubelet 怎麼把「建 Pod network」交給 CRI runtime？ |
| kubelet health | `kubernetes/pkg/kubelet/kubelet.go` 的 `updateRuntimeUp()` | runtime network 沒 ready 時，Node 為什麼會 NotReady？ |
| NodeIPAM | `kubernetes/pkg/controller/nodeipam/ipam/range_allocator.go` | controller-manager 怎麼把 cluster CIDR 切成每 node PodCIDR？ |
| CNI spec | `containernetworking/cni/SPEC.md` | ADD / DEL / CHECK / STATUS / GC / VERSION、stdin、env var、plugin chain 的正式 contract。 |
| CNI plugin | `containernetworking/plugins/plugins/main/bridge/bridge.go`、`plugins/ipam/host-local` | bridge、veth、IPAM 是怎麼真的落到 Linux kernel？ |
| flannel | `flannel-io/flannel` 與 `flannel-io/cni-plugin` | flanneld 怎麼讀 Node、寫 annotation、建立 VXLAN route/ARP/FDB？ |

## 先建立全局心智模型

```
Pod object
  │
  ▼
kubelet SyncPod
  │
  ▼
CRI RunPodSandbox
  │
  ▼
containerd / CRI-O
  │
  ├─ 建 Pod sandbox / network namespace
  ├─ 呼叫 CNI ADD
  │     ├─ bridge / veth / IPAM / route / portmap
  │     └─ Cilium / Calico / Flannel / Multus / cloud CNI
  └─ 回報 PodSandboxStatus 與 runtime NetworkReady
```

所以 Kubernetes network 不是 apiserver 在轉封包，也不是 scheduler 在決定 route。control plane 管的是期望狀態與分配資訊；node 上真正改 kernel datapath 的通常是 runtime、CNI plugin、CNI daemon 與 kube-proxy / Cilium 這類 per-node agent。

## 用現代 Kubernetes 修正舊文章脈絡

這組文章寫於 2019 年，概念仍然有價值，但有幾個地方需要更新：

- `dockershim` 已在 Kubernetes 1.24 移除；現代 kubelet 透過 CRI runtime endpoint 連 containerd / CRI-O。
- 舊文提到的 kubelet `--network-plugin`、`--cni-bin-dir`、`--cni-conf-dir` 屬於 dockershim 時代的 kubelet 直接 CNI integration；現在 CNI conf/bin 位置通常是 containerd / CRI-O 的設定。
- CNI spec 已不是文章中的 `0.4.0` 脈絡；目前 spec 文件標示 `1.1.0`，多了 `STATUS`、`GC` 等現代操作語義。
- NetworkPolicy 不是 CNI spec 的必備欄位，而是 Kubernetes policy API 加上特定 CNI controller/datapath 的實作。
- Service routing 也不是 CNI spec 的基本責任；iptables/IPVS kube-proxy 或 Cilium kube-proxy replacement 是另一條 datapath。

## 接下來怎麼讀

1. 先讀 [pod-network-lifecycle](/kubernetes/features/pod-network-lifecycle)，釐清 kubelet、CRI、CNI 誰呼叫誰。
2. 再讀 [cni-plugin-primitives](/kubernetes/features/cni-plugin-primitives)，把 CNI binary、veth、bridge、IPAM 變成可觀察的 Linux 動作。
3. 接著讀 [node-ipam-and-flannel](/kubernetes/features/node-ipam-and-flannel)，看 Kubernetes 怎麼分 PodCIDR，flannel 怎麼把它交給 host-local。
4. 最後讀 [pod-to-pod-datapath](/kubernetes/features/pod-to-pod-datapath)，把跨 node 封包一路拆到 route、ARP、FDB、VXLAN。
